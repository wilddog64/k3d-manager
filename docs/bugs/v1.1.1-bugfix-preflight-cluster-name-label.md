# Bugfix: v1.1.1 — preflight platform clone never generates (missing `argocd.argoproj.io/cluster-name` label)

**Branch:** `k3d-manager-v1.7.1`
**Files:** `bin/cluster-preflight`

---

## Problem

After the appset-environment isolation fix (`ceffbdb5`) stopped the production `platform-helm`
leak, a default-mode run still times out: the platform **clone** ApplicationSet
(`<name>-preflight-platform-helm`) reports "All applications have been generated successfully"
but generates **zero** apps — `<name>-preflight-platform` never appears, so the platform
app-of-apps (namespaces, ESO/CRDs, the nested ArgoCD) never installs and `data-layer` +
shopping-cart clones stay `Missing`/`OutOfSync`.

**Root cause:** `_cluster_preflight_render_platform_appset` adds a cluster-generator guard
`argocd.argoproj.io/cluster-name In [<name>]` (the `awk` block), but the cluster secret created
by `register_app_cluster` does **not** carry an `argocd.argoproj.io/cluster-name` label —
ArgoCD does not auto-inject it. The guard therefore matches no cluster, ANDed with the
(now-correct) `environment In [preflight]` expression → empty selection → no generated app.

The other two clones are unaffected: `observability-acg` uses a list generator and
`services-git` uses a git-directory generator — neither depends on cluster labels.

---

## Reproduction / live evidence (`pf1`, 2026-06-16)

```bash
# After ceffbdb5, a fresh run leaves the platform clone empty:
kubectl --context k3d-k3d-cluster -n cicd get applicationset pf1-preflight-platform-helm \
  -o jsonpath='{.status.conditions[*].message}'      # → "All applications have been generated successfully"
kubectl --context k3d-k3d-cluster -n cicd get application pf1-preflight-platform   # → NotFound

# Secret labels lack the guard's key:
kubectl --context k3d-k3d-cluster -n cicd get secret cluster-pf1 -o jsonpath='{.metadata.labels}'
#   {argocd-chart-version, argocd-replicas, argocd.argoproj.io/secret-type, environment: preflight}
#   ^ no argocd.argoproj.io/cluster-name

# Adding the missing label makes the clone generate within ~5s:
kubectl --context k3d-k3d-cluster -n cicd label secret cluster-pf1 \
  argocd.argoproj.io/cluster-name=pf1 --overwrite
kubectl --context k3d-k3d-cluster -n cicd get application pf1-preflight-platform
#   → pf1-preflight-platform   owner=pf1-preflight-platform-helm   project=pf1-preflight-platform
#   → platform app-of-apps then begins installing (argocd components Synced, waiting healthy)
```

---

## Fix

### Change 1 — `bin/cluster-preflight`: label the cluster secret after registration

Add the `argocd.argoproj.io/cluster-name=<name>` label to the just-created cluster secret,
immediately after `register_app_cluster` returns and while the kube context is still the hub
(`REGISTER_KUBE_CONTEXT`, switched at lines 576–579). This is the label the platform clone's
generator guard already expects.

**Exact old block (lines 584–586):**

```bash
  ARGOCD_APP_CLUSTER_TOKEN="${VC_TOKEN}" \
    _run_command -- "${REPO_ROOT}/scripts/k3d-manager" register_app_cluster
fi
```

**Exact new block:**

```bash
  ARGOCD_APP_CLUSTER_TOKEN="${VC_TOKEN}" \
    _run_command -- "${REPO_ROOT}/scripts/k3d-manager" register_app_cluster
  _info "[preflight] Labeling ${APP_CLUSTER_SECRET_NAME} with argocd.argoproj.io/cluster-name=${APP_CLUSTER_NAME} (platform clone selector guard)..."
  _kubectl -n "${ARGOCD_NAMESPACE}" label secret "${APP_CLUSTER_SECRET_NAME}" \
    "argocd.argoproj.io/cluster-name=${APP_CLUSTER_NAME}" --overwrite >/dev/null
fi
```
