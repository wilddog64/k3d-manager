# Bugfix: v1.1.1 — preflight wait loop aborts instantly under `set -e` (unguarded `read`)

**Branch:** `k3d-manager-v1.7.1`
**Files:** `bin/cluster-preflight`

---

## Problem

`make preflight NAME=<x> MODE=--auto` reaches the convergence wait, logs
`Waiting up to <N>s for preflight apps to become Synced/Healthy...`, then **fails within
seconds** with `--auto: run FAILED (rc=1)` — *without* ever printing the function's own
`Timed out waiting ...` message. The wait never actually polls; the ArgoCD apps are given no
time to converge.

**Root cause:** in `_cluster_preflight_wait_for_apps`, the per-app status read is unguarded:

```bash
read -r sync_status health_status < <(
  _kubectl -n "${ARGOCD_NAMESPACE}" get application "${app}" \
    -o jsonpath='{.status.sync.status}{" "}{.status.health.status}' 2>/dev/null \
    || printf 'Unknown Unknown'
)
```

`kubectl -o jsonpath` emits **no trailing newline**, and a freshly-created Application has an
**empty `.status`** (jsonpath yields an empty string). Under bash, `read` returns **exit 1** when
it hits EOF without a line delimiter (verified: both empty input and newline-less input return 1;
only newline-terminated input returns 0). The script runs with `set -euo pipefail` (line 2), so
the very first `read` of the first app in the first poll iteration trips `set -e` and aborts the
whole script. The `EXIT` trap (`_cluster_preflight_cleanup`) then fires and reports
`run FAILED (rc=1)` — which is why no timeout message appears.

This masks real progress: with the wait removed, the same run was observed to bring
`green1-preflight-platform` (nested ArgoCD app-of-apps) and the observability stack to
**Synced/Healthy** over ~7 minutes.

---

## Reproduction

```bash
# bash (the script's shebang shell) — read fails on newline-less / empty input under set -e:
bash -c 'set -euo pipefail; read -r a b < <(printf "OutOfSync Missing"); echo ok'; echo "exit=$?"
#   → prints nothing; exit=1   (set -e aborted before the echo)

bash -c 'set -euo pipefail; read -r a b < <(printf ""); echo ok'; echo "exit=$?"
#   → exit=1

# Fix proven to return 0:
bash -c 'set -euo pipefail; read -r a b < <(printf "OutOfSync Missing") || true; echo ok'; echo "exit=$?"
#   → prints ok; exit=0
```

Live: `make preflight NAME=green1 MODE=--auto` failed at `02:24:30Z`, the same instant the
apps were created — total elapsed ~37s against a 900s budget, with no `Timed out` line in the log.

---

## Fix

### Change 1 — `bin/cluster-preflight`: guard the per-app status `read`

Append `|| true` to the `read` so an empty/newline-less status (or a `kubectl` failure) cannot
trip `set -e`. The subsequent `[[ ... != "Synced" ... ]]` test already treats empty/`Unknown`
status as not-ready and keeps polling — that is the correct behavior; the bug is only that the
script aborts before reaching it.

**Exact old block (lines 152–156):**

```bash
      read -r sync_status health_status < <(
        _kubectl -n "${ARGOCD_NAMESPACE}" get application "${app}" \
          -o jsonpath='{.status.sync.status}{" "}{.status.health.status}' 2>/dev/null \
          || printf 'Unknown Unknown'
      )
```

**Exact new block:**

```bash
      read -r sync_status health_status < <(
        _kubectl -n "${ARGOCD_NAMESPACE}" get application "${app}" \
          -o jsonpath='{.status.sync.status}{" "}{.status.health.status}' 2>/dev/null \
          || printf 'Unknown Unknown'
      ) || true
```
