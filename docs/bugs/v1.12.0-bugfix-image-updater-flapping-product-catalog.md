# Bugfix: stop `shopping-cart-product-catalog` Image Updater flapping

The `shopping-cart-product-catalog` ArgoCD Application oscillates
`Synced → OutOfSync` every ~2–4 minutes and never converges. This fixes the
**two code-level root causes**, both in the k3d-manager repo.

**Classification:** observability/correctness bugfix in `docs/bugs/` (exempt from
the max-5-plan limit). Plan-doc count for v1.12.0 unchanged.

---

## Background (read this — it explains both changes)

Live investigation (2026-06-29, hub `k3d-k3d-cluster`, ns `cicd`) found:

- `.spec.source.kustomize.images` on the live Application stays **empty across a
  full Image Updater cycle**, yet Image Updater logs `Successfully updated the
  live application spec` every other cycle (`image_tag=dummy`, resolves
  `latest@sha256:68ec4ed…` vs the git-pinned `sha-6ca5e88d…` base tag). So the
  digest write-back **never durably lands**.

**Root cause 1 — the ApplicationSet reverts the write.** The app is generated by
the `services-git` ApplicationSet with `write-back-method: argocd`, so Image
Updater writes the resolved digest into the live Application's
`.spec.source.kustomize.images`. The ApplicationSet **owns** that Application and
reconciles `.spec.source` back to its template, which has **no `kustomize`
block**. The guard meant to prevent this —

```yaml
ignoreApplicationDifferences:
  - jsonPointers:
      - /spec/source/kustomize/images
```

— **fails**, because a JSON Pointer cannot normalize a path whose **parent
(`/spec/source/kustomize`) does not exist** in the template. The controller sees
Image Updater's freshly-added `kustomize` object as drift and reverts the whole
thing. Next cycle Image Updater re-writes → reverted again → infinite flap.
The documented fix is `jqPathExpressions`, which tolerates the absent parent.

**Root cause 2 — the kustomization has no local image override target.**
`services/shopping-cart-product-catalog/kustomization.yaml` pulls a **remote
base** (`shopping-cart-product-catalog//k8s/base?ref=main`, image pinned to
`sha-6ca5e88d…`) and declares **no local `images:` field**. With `:latest` +
`update-strategy: digest`, the manifest should reference the image by `:latest`
and let Image Updater pin the digest. Adding an explicit `images:` anchor gives
the kustomize override a deterministic, name-matched target and aligns the
manifest with the `image-list` annotation.

The flap is **harmless to the running workload** (the Deployment stays `Healthy`
on the base image); the cost is constant live-spec churn and a perpetual
`OutOfSync`.

## Change 1 — `scripts/etc/argocd/applicationsets/services-git.yaml`

**Find this exact text (lines 10–12):**

```yaml
  ignoreApplicationDifferences:
    - jsonPointers:
        - /spec/source/kustomize/images
```

**Replace with:**

```yaml
  ignoreApplicationDifferences:
    - jqPathExpressions:
        - .spec.source.kustomize.images
```

Do NOT touch the per-app `ignoreDifferences:` block lower in the same file
(lines ~52–78) — that is a different, correct construct. Only the top-level
`spec.ignoreApplicationDifferences` changes.

## Change 2 — `services/shopping-cart-product-catalog/kustomization.yaml`

**Find this exact text (the whole current file):**

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - https://github.com/wilddog64/shopping-cart-product-catalog//k8s/base?ref=main
patches:
  - patch: |-
      apiVersion: v1
      kind: ServiceAccount
      metadata:
        name: product-catalog
      imagePullSecrets:
        - name: ghcr-pull-secret
    target:
      kind: ServiceAccount
      name: product-catalog
```

**Replace with (adds an `images:` block; everything else unchanged):**

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - https://github.com/wilddog64/shopping-cart-product-catalog//k8s/base?ref=main
images:
  - name: ghcr.io/wilddog64/shopping-cart-product-catalog
    newTag: latest
patches:
  - patch: |-
      apiVersion: v1
      kind: ServiceAccount
      metadata:
        name: product-catalog
      imagePullSecrets:
        - name: ghcr-pull-secret
    target:
      kind: ServiceAccount
      name: product-catalog
```

Notes (do NOT add as comments to the file):
- The image name `ghcr.io/wilddog64/shopping-cart-product-catalog` is confirmed
  from the live Image Updater logs (`registry=ghcr.io
  image_name=wilddog64/shopping-cart-product-catalog`).
- `newTag: latest` aligns the manifest with the `image-list` annotation
  (`…:latest`) and the `digest` update-strategy; Image Updater then pins the
  concrete digest via `.spec.source.kustomize.images` (now preserved by Change 1).

## Change 3 — `scripts/tests/plugins/argocd_image_updater_flap_fix.bats` (new)

```bash
#!/usr/bin/env bats

@test "services-git ApplicationSet ignores kustomize image overrides via jqPathExpressions" {
  run grep -nF 'jqPathExpressions' scripts/etc/argocd/applicationsets/services-git.yaml
  [ "$status" -eq 0 ]

  run grep -nF '.spec.source.kustomize.images' scripts/etc/argocd/applicationsets/services-git.yaml
  [ "$status" -eq 0 ]

  run grep -nF '/spec/source/kustomize/images' scripts/etc/argocd/applicationsets/services-git.yaml
  [ "$status" -ne 0 ]
}

@test "product-catalog kustomization declares an image override anchor" {
  run grep -nE '^images:' services/shopping-cart-product-catalog/kustomization.yaml
  [ "$status" -eq 0 ]

  run grep -nF 'ghcr.io/wilddog64/shopping-cart-product-catalog' services/shopping-cart-product-catalog/kustomization.yaml
  [ "$status" -eq 0 ]
}
```
