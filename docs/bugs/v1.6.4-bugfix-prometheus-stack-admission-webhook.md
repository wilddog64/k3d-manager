# Bugfix: v1.6.4 — kube-prometheus-stack-operator stuck in ContainerCreating (admission secret missing)

**Branch:** `k3d-manager-v1.6.4`
**Files:** `scripts/etc/helm/observability/kube-prometheus-stack-acg-values.yaml`

---

## Problem

After every cluster rebuild, `acg-kube-prometheus-stack-operator` is stuck in
`ContainerCreating` indefinitely, which prevents Grafana from loading.

**Root cause:** The chart's `admissionWebhooks` Helm hook jobs (`admission-create`,
`admission-patch`) run once during `helm install` and create the TLS secret
`acg-kube-prometheus-stack-admission`. ArgoCD does not re-run Helm hooks on sync — so on
every cluster rebuild the ValidatingWebhookConfiguration and MutatingWebhookConfiguration
are applied (ArgoCD renders those), but the TLS secret is never recreated. The operator
pod mounts it as `tls-secret` and hangs with `FailedMount`.

---

## Reproduction

1. Run `acg-up` to provision a fresh cluster
2. `kubectl get pods -n monitoring --context ubuntu-k3s` — operator shows `ContainerCreating`
3. `kubectl describe pod -n monitoring -l app=kube-prometheus-stack-operator --context ubuntu-k3s`
   → `MountVolume.SetUp failed for volume "tls-secret" : secret "acg-kube-prometheus-stack-admission" not found`

---

## Fix

### Change 1 — `scripts/etc/helm/observability/kube-prometheus-stack-acg-values.yaml`: disable admission webhooks

**Exact old block (line 1):**

```yaml
defaultRules:
```

**Exact new block:**

```yaml
admissionWebhooks:
  enabled: false

defaultRules:
```

Disabling admission webhooks removes the TLS secret dependency entirely. The webhooks
validate `PrometheusRule` CRDs — acceptable to skip in a dev/ACG cluster.
