# Bugfix: v1.4.11 — ExternalSecret sync fails when ClusterSecretStore not yet validated

**Branch:** `k3d-manager-v1.4.11`
**Files:** `scripts/plugins/shopping_cart.sh`

---

## Problem

On fresh cluster provision, ESO applies ExternalSecrets before the ClusterSecretStore
`vault-backend` is internally validated. ESO records a `SecretSyncedError` on first sync
and with a 24h refresh interval, secrets remain unsynced for the rest of the provision run.
`kubectl wait --for=condition=Ready` on the ExternalSecret then times out after 180s,
aborting `make up`.

**Root cause:** `shopping_cart_sync_vault_backed_secrets()` applies ExternalSecrets without
first confirming the ClusterSecretStore is Ready; ExternalSecrets carry a 24h refresh interval
so ESO never automatically retries the failed sync during the provision window.

---

## Reproduction

1. Fresh `make up` on a new cluster
2. ArgoCD deploys ClusterSecretStore as part of data-layer app
3. `shopping_cart_sync_vault_backed_secrets()` applies ExternalSecrets immediately
4. ESO first-sync attempt hits "ClusterSecretStore not ready" → 24h retry interval
5. `kubectl wait --for=condition=Ready --timeout=180s externalsecret/redis-cart-apps` times out
6. `make: *** [up] Error 1`

---

## Fix

### Change 1 — `scripts/plugins/shopping_cart.sh`: wait for ClusterSecretStore before applying ExternalSecrets; force-annotate each ExternalSecret to trigger fresh sync

**Added before ExternalSecret apply:**
```bash
_info "[shopping_cart] Waiting for ClusterSecretStore vault-backend to be Ready..."
kubectl wait --for=condition=Ready --timeout=120s \
  clustersecretstore/vault-backend --context ubuntu-k3s
```

**Added per-ExternalSecret before `kubectl wait`:**
```bash
kubectl annotate externalsecret "${name}" -n "${namespace}" --context ubuntu-k3s \
  force-sync="${_es_sync_ts}" --overwrite >/dev/null
```
