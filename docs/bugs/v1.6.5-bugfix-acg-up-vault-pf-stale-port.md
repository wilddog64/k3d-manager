# Bug: acg-up Vault port-forward fails — stale kubectl process holds port 18200

**Date:** 2026-06-11
**Branch:** k3d-manager-v1.6.5
**File:** `bin/acg-up`

---

## Symptom

`make up` fails at Step 4 even when the k3d Hub cluster is healthy:

```
INFO: [acg-up] Step 4/12 — Starting Vault port-forward (k3d → localhost:18200)...
INFO: [acg-up] Vault port-forward PID: 30233 (log: ...)
ERROR: [acg-up] Vault not responding on localhost:18200 — port-forward may have failed.
```

The vault-pf.log shows:

```
Unable to listen on port 18200: ... bind: address already in use
error: unable to listen on any of the requested ports: [{18200 8200}]
```

---

## Root Cause

`kubectl port-forward` from a previous failed (or interrupted) run is still holding port
18200. The PID file (`run/vault-pf.pid`) was deleted by `_acg_up_cleanup`, but the process
was not killed — either because the cleanup `kill` failed silently, or because the terminal
session was terminated before the EXIT trap could fire.

The current code at step 4 kills only the PID recorded in the PID file:

```bash
if [[ -f "${_vault_pf_pid_file}" ]]; then
  kill "$(cat "${_vault_pf_pid_file}")" 2>/dev/null || true
  rm -f "${_vault_pf_pid_file}"
fi
```

When the PID file is missing (cleanup already removed it), no kill happens. The new
`kubectl port-forward` then fails immediately with `address already in use` and exits.
After `sleep 3` the health check finds nothing on port 18200.

Confirmed: `lsof -ti:18200` returns PID 31224 (`kubectl port-forward vault-0 18200:8200`)
with no PID file present.

---

## Fix

Two changes to `bin/acg-up`:

### Change 1 — add `lsof`-based stale-holder kill after the PID file kill

After the PID file block (lines 276–278), add an `lsof` sweep so any process holding
the port is killed regardless of whether the PID file exists.

**Exact old block (lines 276–280):**

```bash
if [[ -f "${_vault_pf_pid_file}" ]]; then
  kill "$(cat "${_vault_pf_pid_file}")" 2>/dev/null || true
  rm -f "${_vault_pf_pid_file}"
fi
kubectl port-forward svc/vault -n secrets --context k3d-k3d-cluster "${_vault_local_port}:8200" \
```

**Exact new block:**

```bash
if [[ -f "${_vault_pf_pid_file}" ]]; then
  kill "$(cat "${_vault_pf_pid_file}")" 2>/dev/null || true
  rm -f "${_vault_pf_pid_file}"
fi
_stale_vault_pf=$(lsof -ti:"${_vault_local_port}" 2>/dev/null || true)
if [[ -n "${_stale_vault_pf}" ]]; then
  _info "[acg-up] Port ${_vault_local_port} in use — killing stale vault port-forward holders..."
  echo "${_stale_vault_pf}" | xargs kill 2>/dev/null || true
  sleep 1
fi
kubectl port-forward svc/vault -n secrets --context k3d-k3d-cluster "${_vault_local_port}:8200" \
```

---

### Change 2 — warn user about OrbStack GUI password dialog

When `orbctl start` is called, OrbStack may show a macOS GUI dialog asking for an admin
password to install its helper. The script logs no hint of this, so the user sees the
script appear to hang with no explanation.

**Exact old block (lines 206–208):**

```bash
    if [[ "${_orb_status}" == "Stopped" ]]; then
      _info "[acg-up] OrbStack VM is stopped — starting it now..."
      orbctl start 2>/dev/null || true
```

**Exact new block:**

```bash
    if [[ "${_orb_status}" == "Stopped" ]]; then
      _info "[acg-up] OrbStack VM is stopped — starting it now..."
      _info "[acg-up] OrbStack may show a GUI dialog asking for admin password — approve it to start the VM..."
      orbctl start 2>/dev/null || true
```
