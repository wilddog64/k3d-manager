# Bugfix: v1.6.0 — Shopping-cart product images broken + LDAP login credentials undocumented

**Branch (spec repo):** `k3d-manager-v1.6.0`
**Work repo:** `shopping-cart-infra`
**Work branch:** `fix/product-images-keycloak-login`
**Files (shopping-cart-infra):**
- `data-layer/minio/image-upload-job.yaml`
- `identity/ldap/bootstrap.yaml`

---

## Problem A — Product images not loading

Product catalog images are all broken (browser shows broken-image placeholder).

Image URLs in the database are `/minio/product-images/<slug>.jpg`. The frontend
nginx proxies `^~ /minio/product-images/` to MinIO in-cluster. The MinIO
image-upload PostSync hook Job uses `python:3.12-alpine` as its container image.

**Root cause:** `python:3.12-alpine` is a musl-libc image. Pillow distributes only
`manylinux` (glibc) binary wheels on PyPI. On alpine, `pip install Pillow` must
compile from source, which requires `gcc`, `libjpeg-dev`, `libpng-dev`, and other
build tools that are not present in `python:3.12-alpine`. The install fails silently
(backoffLimit: 3 is exhausted) and no images are uploaded to MinIO.

---

## Problem B — Keycloak login credentials not documented

Users cannot log into the frontend at `https://frontend.3ai-talk.org` via Keycloak
because the LDAP test-user passwords (stored as SSHA hashes in `bootstrap.yaml`)
were generated from unknown plaintext values. There is no documented way to
authenticate as `admin`, `developer`, or `operator` in the `shopping-cart` realm.

**Root cause:** The `06-sample-users.ldif` section in `identity/ldap/bootstrap.yaml`
contains SSHA hashes with no corresponding plaintext record. The bootstrap runs once
at cluster creation via the LDAP deployment initContainer; the original passwords are
not recorded anywhere.

*Note: redirect URI and OIDC client configuration are correct — `https://frontend.3ai-talk.org/callback`
is registered in `realm-shopping-cart.json`, and CI builds the frontend image with
`VITE_KEYCLOAK_URL=https://keycloak.3ai-talk.org`. The failure is purely at the
credential level.*

---

## Reproduction

### Images
1. Open `https://frontend.3ai-talk.org` — all product card images are broken
2. Open DevTools → Network → observe 403 or connection errors for `/minio/product-images/*.jpg`

### Keycloak login
1. Open `https://frontend.3ai-talk.org` → click Sign In
2. Redirected to `https://keycloak.3ai-talk.org/realms/shopping-cart/protocol/openid-connect/auth`
3. Enter `admin` / any password → login fails

### Fix B2 — Reset passwords on existing cluster

The bootstrap LDIF only runs on initial LDAP deployment (initContainer). If the
cluster already has users seeded with old hashes, run this one-time password reset
against the running LDAP pod:

```bash
LDAP_POD=$(kubectl get pods -n identity -l app.kubernetes.io/name=ldap -o jsonpath='{.items[0].metadata.name}')
LDAP_ADMIN_PW=$(kubectl get secret ldap-secrets -n identity -o jsonpath='{.data.LDAP_ADMIN_PASSWORD}' | base64 -d)

kubectl exec -n identity "${LDAP_POD}" -- ldapmodify -x \
  -H ldap://localhost:389 \
  -D "cn=admin,dc=shopping-cart,dc=local" \
  -w "${LDAP_ADMIN_PW}" <<'EOF'
dn: uid=admin,ou=users,dc=shopping-cart,dc=local
changetype: modify
replace: userPassword
userPassword: Shopping1!

dn: uid=developer,ou=users,dc=shopping-cart,dc=local
changetype: modify
replace: userPassword
userPassword: Dev1234!

dn: uid=operator,ou=users,dc=shopping-cart,dc=local
changetype: modify
replace: userPassword
userPassword: Ops5678!
EOF
```

OpenLDAP will hash these automatically using the configured password policy.
