# Bugfix: v1.1.1 — preflight vcluster create fails (invalid `distro.k3s` schema)

**Branch:** `k3d-manager-v1.7.1`
**Files:** `scripts/etc/vcluster/values-preflight.yaml`

---

## Problem

The previous sizing fix (`2b46f0ce`) added `controlPlane.distro.k3s.enabled: true` to
`values-preflight.yaml`. This **breaks vcluster creation outright** — the very first preflight
step now fails:

```
fatal merge values: invalid config: error unmarshaling JSON: while decoding JSON: json: unknown field "k3s"
vcluster command failed (1): vcluster create green1 -n vclusters --chart-version 0.32.1 ...
```

**Root cause:** a CLI/chart **version split**. The installed `vcluster` CLI is **0.35.0**; it
parses and validates the `-f` values file against *its own* config schema before deploying the
chart. In CLI 0.35.0 the **k3s distro was removed** — `controlPlane.distro` only accepts `k8s`.
(The chart 0.32.1 `values.yaml` still lists a `k3s:` block, which is what misled the original
spec, but the CLI validator rejects it.) So `distro.k3s` is an unknown field and the create
aborts.

The k3s distro change was also **unnecessary**: live testing on the host (`ubuntu-hostinger`,
2 vCPU / 7.7Gi) proved the **resource-limit raise alone** stops the crashloop. A vcluster created
with the default **k8s distro** + `limits: 1500m / 1536Mi` ran the control plane **0 restarts at
3 min** (the old 500m-limited pf1 had 8 syncer restarts by the same age). The 500m CPU cap was
the throttle behind the lease-renewal timeout; 1500m removes it.

---

## Reproduction / evidence

```bash
# Broken (current main): create fails immediately
make preflight NAME=green1 MODE=--auto
#   → fatal ... unknown field "k3s"

# Correct config (k8s default + raised limits) — created and stable on the HOST:
kubectl --context ubuntu-hostinger -n vclusters get pod gtest-0
#   gtest-0  1/1  Running  0  2m59s     ← 0 restarts (old pf1: 8 by this age)
```

CLI / chart versions:

```bash
vcluster version          # 0.35.0  ← validates -f against its own schema (no k3s distro)
# chart pinned in bin/cluster-preflight: --chart-version 0.32.1
```

---

## Fix

### Change 1 — `scripts/etc/vcluster/values-preflight.yaml`: drop the invalid distro block

Remove the `distro:` block entirely (the default distro is `k8s`, which the CLI accepts) and
**keep** the raised control-plane limits from `2b46f0ce` — those are the part that actually fixes
the crashloop.

**Exact old file contents:**

```yaml
controlPlane:
  distro:
    k3s:
      enabled: true
  service:
    spec:
      type: NodePort
  statefulSet:
    resources:
      requests:
        cpu: 300m
        memory: 384Mi
      limits:
        cpu: 1500m
        memory: 1536Mi
```

**Exact new file contents:**

```yaml
controlPlane:
  service:
    spec:
      type: NodePort
  statefulSet:
    resources:
      requests:
        cpu: 300m
        memory: 384Mi
      limits:
        cpu: 1500m
        memory: 1536Mi
```
