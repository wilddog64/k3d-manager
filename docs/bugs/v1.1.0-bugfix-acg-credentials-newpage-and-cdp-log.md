## Problem

### Bug 1 — silent CDP failure (line 166)

When Chrome is not running with `--remote-debugging-port=9222`, the inner `try`
block at line 147 throws on `connectOverCDP`. The `catch` at line 166 silently
nulls `_cdpBrowser` and falls through to `launchPersistentContext`. The user
sees no indication of why the fallback happened — if the isolated `playwright-auth/`
profile has no Pluralsight cookies (e.g. after the profile was cleared), the script
silently redirects to `/id` and times out after 300s.

### Bug 2 — tab hijack (line 185)

After `browserContext` is set (from either CDP or `launchPersistentContext`), the
code searches for an existing Pluralsight page. When none is found, it falls back
to `allPages[0]`. In a CDP session, `allPages[0]` is whatever the user currently
has open — navigating it hijacks that tab. In a fresh `launchPersistentContext`,
`allPages[0]` is a blank page — this works but is fragile.

The correct fix for both paths: call `await context.newPage()` when no Pluralsight
page is found. This opens a fresh tab (in the user's Chrome for CDP, or in the
Playwright window for the persistent context) without touching existing tabs.

---

## Fix

### Change 1 — CDP catch block (line 166)

**Exact old block:**

```js
    } catch {
      _cdpBrowser = null;
    }
```

**Exact new block:**

```js
    } catch {
      console.error('INFO: Chrome not running on CDP port 9222 — falling back to isolated Playwright profile.');
      _cdpBrowser = null;
    }
```

### Change 2 — page selection fallback (lines 184–187)

**Exact old block:**

```js
    if (!page) {
      page = allPages[0];
      if (!page) throw new Error('No page found in the browser context');
    }
```

**Exact new block:**

```js
    if (!page) {
      page = await context.newPage();
    }
```
