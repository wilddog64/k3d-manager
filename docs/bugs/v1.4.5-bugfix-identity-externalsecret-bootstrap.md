## Symptoms

```
cicd    shopping-cart-identity    OutOfSync    Degraded
```

Three cascading failures on a fresh `acg-up`:

1. **`ClusterSecretStore/vault-backend` not found** ŌĆö all three keycloak/ldap ExternalSecrets fail with:
   `could not get ClusterSecretStore "vault-backend" ... not found`

2. **`keycloak-secrets` secret missing Vault-fetched keys** ŌĆö even after the store is created, the secret only has 2 of 5 keys (`KC_DB_USERNAME`, `KEYCLOAK_ADMIN`). The three Vault-fetched keys (`KC_DB_PASSWORD`, `KEYCLOAK_ADMIN_PASSWORD`, `LDAP_BIND_CREDENTIAL`) are silently dropped.

3. **`postgres-keycloak` pod `CreateContainerConfigError`** ŌĆö cascades from #2; secret `keycloak-secrets` exists but is missing `KC_DB_PASSWORD`.

Secondary issue (still open):
4. **`ldap-data-pvc` WaitForFirstConsumer deadlock** ŌĆö ArgoCD waits for the PVC to be Healthy (Bound) before applying the `ldap` Deployment, but the PVC (StorageClass: local-path, bindingMode: WaitForFirstConsumer) won't bind until a pod claims it. Circular dependency causes `shopping-cart-identity` to stay `Progressing` indefinitely.

## Root Causes

### Bug 1 ŌĆö Missing ClusterSecretStore (k3d-manager)

The ExternalSecrets in `shopping-cart-infra/identity/{keycloak,ldap}` reference:
```yaml
secretStoreRef:
  kind: ClusterSecretStore
  name: vault-backend
```

`vault-backend` is never provisioned by k3d-manager's ESO setup. Only a namespace-scoped `SecretStore/vault-kv-store` in the `identity` namespace is created (used by the OpenLDAP helm chart). Since `ClusterSecretStore` is cluster-scoped infrastructure, k3d-manager must provision it ŌĆö not shopping-cart-infra.

**Fix (k3d-manager) ŌĆö exact changes:**

**File 1: `scripts/plugins/eso.sh`** ŌĆö add new private function before `_eso_configure_remote_vault`:

```bash
function _eso_apply_vault_cluster_store() {
  local store_name="${ESO_VAULT_CLUSTER_STORE:-vault-backend}"
  local server="${VAULT_ENDPOINT:-http://vault.${VAULT_NS:-secrets}.svc:8200}"
  local path="${LDAP_VAULT_KV_MOUNT:-secret}"
  local role="${LDAP_ESO_ROLE:-eso-ldap-directory}"
  local sa="${LDAP_ESO_SERVICE_ACCOUNT:-eso-ldap-sa}"
  local sa_ns="${LDAP_NAMESPACE:-identity}"

  _info "[eso] Applying ClusterSecretStore/${store_name}"
  cat <<YAML | _kubectl apply -f -
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: ${store_name}
spec:
  provider:
    vault:
      server: "${server}"
      path: "${path}"
      version: v2
      auth:
        kubernetes:
          mountPath: kubernetes
          role: "${role}"
          serviceAccountRef:
            name: ${sa}
            namespace: ${sa_ns}
YAML
}
```

**File 2: `scripts/plugins/ldap.sh`** ŌĆö add eso.sh source guard alongside the existing vault.sh source guard (lines ~14-21):

```bash
# Source ESO plugin if not already loaded
if ! declare -f _eso_apply_vault_cluster_store >/dev/null 2>&1; then
   ESO_PLUGIN="$PLUGINS_DIR/eso.sh"
   if [[ -r "$ESO_PLUGIN" ]]; then
      # shellcheck disable=SC1090
      source "$ESO_PLUGIN"
   fi
fi
```

**File 3: `scripts/plugins/ldap.sh`** ŌĆö add one call in `deploy_ldap()` after `_ldap_apply_eso_resources` (currently line 1129):

```bash
   _ldap_apply_eso_resources "$namespace" || return 1
   _eso_apply_vault_cluster_store || return 1   # ŌåÉ add this line
   _ldap_wait_for_secret "$namespace" "${LDAP_ADMIN_SECRET_NAME}"
```

Must be applied after Vault is initialized and `eso-ldap-directory` role + policy exist, and after `eso-ldap-sa` SA is created in `identity` namespace (all satisfied by `_ldap_apply_eso_resources` running first).

### Bug 2 ŌĆö ExternalSecret template drops Vault-fetched keys (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `main`  
**ArgoCD source path:** `identity/keycloak/` (confirmed from `.status.operationState.syncResult.sources`)

Find the file under `identity/keycloak/` that defines the `keycloak-secrets` ExternalSecret. Its `spec.target.template.data` currently only has two static keys:
```yaml
target:
  template:
    data:
      KC_DB_USERNAME: keycloak
      KEYCLOAK_ADMIN: admin
```

In ESO v1, `spec.target.template.data` **replaces** the secret data entirely ŌĆö `spec.data` Vault-fetched values are available only as template variables and must be explicitly referenced. Result: only 2 of 5 keys exist in the secret; `KC_DB_PASSWORD` missing causes `postgres-keycloak` pod `CreateContainerConfigError`.

**Fix:** Add template references for all three Vault-fetched keys:
```yaml
target:
  template:
    data:
      KC_DB_USERNAME: keycloak
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: "{{ .KEYCLOAK_ADMIN_PASSWORD }}"
      KC_DB_PASSWORD: "{{ .KC_DB_PASSWORD }}"
      LDAP_BIND_CREDENTIAL: "{{ .LDAP_BIND_CREDENTIAL }}"
```

The `spec.data` entries and their `secretKey` names (confirmed from live cluster):
- `secretKey: KEYCLOAK_ADMIN_PASSWORD` ŌåÉ `secret/data/keycloak/admin` ŌåÆ `admin_password`
- `secretKey: KC_DB_PASSWORD` ŌåÉ `secret/data/keycloak/admin` ŌåÆ `db_password`
- `secretKey: LDAP_BIND_CREDENTIAL` ŌåÉ `secret/data/ldap/admin` ŌåÆ `admin_password`

### Bug 3 ŌĆö ldap-data-pvc WaitForFirstConsumer deadlock (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `main`  
**ArgoCD source path:** `identity/ldap/`

`ldap-data-pvc` uses StorageClass `local-path` (`volumeBindingMode: WaitForFirstConsumer`). ArgoCD health assessment: `Pending` PVC = `Progressing`, which blocks sync completion. The `ldap` Deployment (the consumer that would bind the PVC) is in a later sync wave and never gets applied ŌĆö circular deadlock confirmed on live cluster.

**Fix:** In the `ldap-data-pvc` PVC manifest under `identity/ldap/`, add the `Replace` sync option so ArgoCD does not wait for PVC health before continuing:
```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-options: "Replace=true"
```

Alternative (preferred if the `ldap` Deployment is in the same file or manageable): put both PVC and `ldap` Deployment in the same sync wave so ArgoCD applies them together:
```yaml
# ldap-data-pvc metadata.annotations:
argocd.argoproj.io/sync-wave: "0"
# ldap Deployment metadata.annotations:
argocd.argoproj.io/sync-wave: "0"
```

Pick whichever fits the existing wave structure in `identity/ldap/`. If no waves are currently set, the `sync-options: Replace=true` on the PVC is the minimal change.

### Bug 4 ŌĆö Keycloak `--optimized` bypasses PostgreSQL driver selection (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `fix/identity-externalsecret-bootstrap`
**File:** `identity/keycloak/deployment.yaml`

Keycloak 24 (Quarkus) pre-compiles database driver selection at build time. The official
`quay.io/keycloak/keycloak:24.0` image is built for H2. Running `start --optimized` skips
the build phase so `KC_DB=postgres` from the ConfigMap is used for URL construction but not
for driver selection. H2's URL validator then rejects the assembled `jdbc:postgresql://` URL.

**Fix:** Replace `--optimized` with `--db=postgres` as the CLI arg. The explicit `--db` flag
overrides build-time defaults even in optimized mode.

**Exact old block** (`identity/keycloak/deployment.yaml` lines 31ŌĆō33):
```yaml
        args:
        - start
        - --optimized
```

**Exact new block:**
```yaml
        args:
        - start
        - --db=postgres
```

**Commit message (exact):** `fix(keycloak): replace --optimized with --db=postgres to fix PostgreSQL driver selection`

### Bug 5 ŌĆö LDAP ConfigMap LDIF mount causes chown failure on startup (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `fix/identity-externalsecret-bootstrap`
**File:** `identity/ldap/deployment.yaml`

The `osixia/openldap:1.5.0` startup script runs `chown -R openldap:openldap` on the bootstrap
LDIF directory. Kubernetes ConfigMap volumes implement atomic updates via internal `..timestamp`
symlink directories that the kernel marks read-only for ownership changes ŌĆö `chown` fails and
the startup script exits 1.

**Fix:** Add an `initContainer` that copies the LDIF files from the ConfigMap (read-only source)
to an `emptyDir` (writable destination), setting correct ownership. Mount the `emptyDir` in the
main container instead of the ConfigMap directly.

**Change 1 ŌĆö add initContainer** (insert before `containers:` at line 55):

Old:
```yaml
    spec:
      containers:
```

New:
```yaml
    spec:
      initContainers:
      - name: copy-ldif
        image: busybox:1.36
        command: ['sh', '-c', 'cp /ldif-source/* /ldif-dest/ && chown -R 911:911 /ldif-dest/']
        volumeMounts:
        - name: ldap-bootstrap
          mountPath: /ldif-source
          readOnly: true
        - name: ldap-bootstrap-writable
          mountPath: /ldif-dest
      containers:
```

**Change 2 ŌĆö change volumeMount in main container** (lines 80ŌĆō81):

Old:
```yaml
        - name: ldap-bootstrap
          mountPath: /container/service/slapd/assets/config/bootstrap/ldif/custom
```

New:
```yaml
        - name: ldap-bootstrap-writable
          mountPath: /container/service/slapd/assets/config/bootstrap/ldif/custom
```

**Change 3 ŌĆö add emptyDir volume** (after the existing `ldap-bootstrap` configMap volume, lines 98ŌĆō100):

Old:
```yaml
      - name: ldap-bootstrap
        configMap:
          name: ldap-bootstrap
```

New:
```yaml
      - name: ldap-bootstrap
        configMap:
          name: ldap-bootstrap
      - name: ldap-bootstrap-writable
        emptyDir: {}
```

**Commit message (exact):** `fix(ldap): add initContainer to copy LDIF ConfigMap to writable emptyDir`
