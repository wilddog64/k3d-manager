# Bug: shopping-cart-identity ExternalSecrets fail to sync on fresh cluster

**Branch:** k3d-manager-v1.4.5  
**Severity:** High ‚Äî `shopping-cart-identity` ArgoCD app stays `OutOfSync / Degraded` on every fresh cluster bring-up  
**Repos affected:** `k3d-manager` (ClusterSecretStore provisioning), `shopping-cart-infra` (ExternalSecret template bug, PVC sync wave)

---

## Symptoms

```
cicd    shopping-cart-identity    OutOfSync    Degraded
```

Three cascading failures on a fresh `acg-up`:

1. **`ClusterSecretStore/vault-backend` not found** ‚Äî all three keycloak/ldap ExternalSecrets fail with:
   `could not get ClusterSecretStore "vault-backend" ... not found`

2. **`keycloak-secrets` secret missing Vault-fetched keys** ‚Äî even after the store is created, the secret only has 2 of 5 keys (`KC_DB_USERNAME`, `KEYCLOAK_ADMIN`). The three Vault-fetched keys (`KC_DB_PASSWORD`, `KEYCLOAK_ADMIN_PASSWORD`, `LDAP_BIND_CREDENTIAL`) are silently dropped.

3. **`postgres-keycloak` pod `CreateContainerConfigError`** ‚Äî cascades from #2; secret `keycloak-secrets` exists but is missing `KC_DB_PASSWORD`.

Secondary issue (still open):
4. **`ldap-data-pvc` WaitForFirstConsumer deadlock** ‚Äî ArgoCD waits for the PVC to be Healthy (Bound) before applying the `ldap` Deployment, but the PVC (StorageClass: local-path, bindingMode: WaitForFirstConsumer) won't bind until a pod claims it. Circular dependency causes `shopping-cart-identity` to stay `Progressing` indefinitely.

---

## Root Causes

### Bug 1 ‚Äî Missing ClusterSecretStore (k3d-manager)

The ExternalSecrets in `shopping-cart-infra/identity/{keycloak,ldap}` reference:
```yaml
secretStoreRef:
  kind: ClusterSecretStore
  name: vault-backend
```

`vault-backend` is never provisioned by k3d-manager's ESO setup. Only a namespace-scoped `SecretStore/vault-kv-store` in the `identity` namespace is created (used by the OpenLDAP helm chart). Since `ClusterSecretStore` is cluster-scoped infrastructure, k3d-manager must provision it ‚Äî not shopping-cart-infra.

**Fix (k3d-manager) ‚Äî exact changes:**

**File 1: `scripts/plugins/eso.sh`** ‚Äî add new private function before `_eso_configure_remote_vault`:

```bash
function _eso_apply_vault_cluster_store() {
  local store_name="${ESO_VAULT_CLUSTER_STORE:-vault-backend}"
  local server="${VAULT_ENDPOINT:-http://vault.${VAULT_NS:-secrets}.svc:8200}"
  local path="${LDAP_VAULT_KV_MOUNT:-secret}"
  local role="${LDAP_ESO_ROLE:-eso-ldap-directory}"
  local sa="${LDAP_ESO_SERVICE_ACCOUNT:-eso-ldap-sa}"
  local sa_ns="${LDAP_NAMESPACE:-identity}"

  _info "[eso] Applying ClusterSecretStore/${store_name}"
  cat <<YAML | _kubectl apply -f -
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: ${store_name}
spec:
  provider:
    vault:
      server: "${server}"
      path: "${path}"
      version: v2
      auth:
        kubernetes:
          mountPath: kubernetes
          role: "${role}"
          serviceAccountRef:
            name: ${sa}
            namespace: ${sa_ns}
YAML
}
```

**File 2: `scripts/plugins/ldap.sh`** ‚Äî add eso.sh source guard alongside the existing vault.sh source guard (lines ~14-21):

```bash
# Source ESO plugin if not already loaded
if ! declare -f _eso_apply_vault_cluster_store >/dev/null 2>&1; then
   ESO_PLUGIN="$PLUGINS_DIR/eso.sh"
   if [[ -r "$ESO_PLUGIN" ]]; then
      # shellcheck disable=SC1090
      source "$ESO_PLUGIN"
   fi
fi
```

**File 3: `scripts/plugins/ldap.sh`** ‚Äî add one call in `deploy_ldap()` after `_ldap_apply_eso_resources` (currently line 1129):

```bash
   _ldap_apply_eso_resources "$namespace" || return 1
   _eso_apply_vault_cluster_store || return 1   # ‚Üê add this line
   _ldap_wait_for_secret "$namespace" "${LDAP_ADMIN_SECRET_NAME}"
```

Must be applied after Vault is initialized and `eso-ldap-directory` role + policy exist, and after `eso-ldap-sa` SA is created in `identity` namespace (all satisfied by `_ldap_apply_eso_resources` running first).

### Bug 2 ‚Äî ExternalSecret template drops Vault-fetched keys (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `main`  
**ArgoCD source path:** `identity/keycloak/` (confirmed from `.status.operationState.syncResult.sources`)

Find the file under `identity/keycloak/` that defines the `keycloak-secrets` ExternalSecret. Its `spec.target.template.data` currently only has two static keys:
```yaml
target:
  template:
    data:
      KC_DB_USERNAME: keycloak
      KEYCLOAK_ADMIN: admin
```

In ESO v1, `spec.target.template.data` **replaces** the secret data entirely ‚Äî `spec.data` Vault-fetched values are available only as template variables and must be explicitly referenced. Result: only 2 of 5 keys exist in the secret; `KC_DB_PASSWORD` missing causes `postgres-keycloak` pod `CreateContainerConfigError`.

**Fix:** Add template references for all three Vault-fetched keys:
```yaml
target:
  template:
    data:
      KC_DB_USERNAME: keycloak
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: "{{ .KEYCLOAK_ADMIN_PASSWORD }}"
      KC_DB_PASSWORD: "{{ .KC_DB_PASSWORD }}"
      LDAP_BIND_CREDENTIAL: "{{ .LDAP_BIND_CREDENTIAL }}"
```

The `spec.data` entries and their `secretKey` names (confirmed from live cluster):
- `secretKey: KEYCLOAK_ADMIN_PASSWORD` ‚Üê `secret/data/keycloak/admin` ‚Üí `admin_password`
- `secretKey: KC_DB_PASSWORD` ‚Üê `secret/data/keycloak/admin` ‚Üí `db_password`
- `secretKey: LDAP_BIND_CREDENTIAL` ‚Üê `secret/data/ldap/admin` ‚Üí `admin_password`

### Bug 3 ‚Äî ldap-data-pvc WaitForFirstConsumer deadlock (shopping-cart-infra)

**Repo:** `wilddog64/shopping-cart-infra`, branch `main`  
**ArgoCD source path:** `identity/ldap/`

`ldap-data-pvc` uses StorageClass `local-path` (`volumeBindingMode: WaitForFirstConsumer`). ArgoCD health assessment: `Pending` PVC = `Progressing`, which blocks sync completion. The `ldap` Deployment (the consumer that would bind the PVC) is in a later sync wave and never gets applied ‚Äî circular deadlock confirmed on live cluster.

**Fix:** In the `ldap-data-pvc` PVC manifest under `identity/ldap/`, add the `Replace` sync option so ArgoCD does not wait for PVC health before continuing:
```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-options: "Replace=true"
```

Alternative (preferred if the `ldap` Deployment is in the same file or manageable): put both PVC and `ldap` Deployment in the same sync wave so ArgoCD applies them together:
```yaml
# ldap-data-pvc metadata.annotations:
argocd.argoproj.io/sync-wave: "0"
# ldap Deployment metadata.annotations:
argocd.argoproj.io/sync-wave: "0"
```

Pick whichever fits the existing wave structure in `identity/ldap/`. If no waves are currently set, the `sync-options: Replace=true` on the PVC is the minimal change.

## Hotfix Applied (cluster-only, will be overwritten by ArgoCD self-heal)

1. `kubectl apply` ‚Äî created `ClusterSecretStore/vault-backend` manually
2. `kubectl annotate externalsecret ... force-sync=...` ‚Äî forced ESO refresh
3. `kubectl patch externalsecret keycloak-secrets` ‚Äî patched template to include Vault-fetched keys
4. `kubectl delete pod postgres-keycloak-*` ‚Äî recycled pod to pick up fixed secret

**These are not permanent.** ArgoCD self-heal will re-apply the git source and overwrite the ExternalSecret patch. The ClusterSecretStore will persist until deleted.
