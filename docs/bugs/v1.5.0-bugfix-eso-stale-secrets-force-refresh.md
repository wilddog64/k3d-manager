# Bug: ESO secrets stale after `argocd app sync data-layer` — app pods get old Vault values

**Branch:** `k3d-manager-v1.5.0`
**File:** `scripts/plugins/shopping_cart.sh:98–99`
**Severity:** P2 — on a fresh sandbox, app pods may authenticate against PostgreSQL with a stale password
  (race between ESO sync time and Vault write time during provisioning)

---

## Symptom

On a fresh `make up`, ESO secrets for the same Vault path can show different values depending
on when each ExternalSecret was first synced:

```
postgres-orders-admin    password = GLiggl3NG1pVBtCx93orqVICaQ6Ko0x   ← synced later
postgres-orders-readwrite password = DWyhOWXgwuT4Tx6LBYGlRXnyWif04sXN  ← synced earlier (stale)
order-service-secrets    DB_PASSWORD = DWyhOWXgwuT4Tx6LBYGlRXnyWif04sXN ← stale
```

Both ESO objects read from `secret/data/postgres/orders`, property `password`. They disagree
because Vault's value changed (new password written) between the time each secret was first synced.

After `make up` removes the `ALTER USER CHANGE_ME` bug (see sibling spec), PostgreSQL holds
the correct Vault password. But if app secrets were synced from Vault before the final write,
they hold an old password → auth fails.

---

## Root Cause

`argocd app sync data-layer` triggers ESO to create k8s secrets from Vault, but ESO processes
ExternalSecret objects asynchronously. During provisioning, Vault may be written to at multiple
points. ExternalSecrets synced early in the sequence may capture an intermediate (stale) value.
The `refreshInterval: 24h` means ESO will not self-correct until the next day.

---

## Fix

After the `argocd app sync data-layer` call, add a force-refresh loop that annotates all
ExternalSecrets in `shopping-cart-data` and `shopping-cart-apps` to trigger an immediate
re-sync from Vault:

**`scripts/plugins/shopping_cart.sh` — replace lines 98–99 with:**

```bash
  _info "[shopping_cart] Syncing data-layer ArgoCD app before Redis deploy (ensures ESO processes ExternalSecret)..."
  argocd app sync data-layer --server localhost:8080 --insecure --wait --timeout 120 2>/dev/null || true

  _info "[shopping_cart] Force-refreshing all ExternalSecrets to ensure current Vault values..."
  for ns in shopping-cart-data shopping-cart-apps; do
    kubectl get externalsecret -n "${ns}" --context ubuntu-k3s \
      --no-headers -o custom-columns=":metadata.name" 2>/dev/null \
    | while read -r es; do
        kubectl annotate externalsecret "${es}" -n "${ns}" --context ubuntu-k3s \
          force-sync="$(date +%s)" --overwrite >/dev/null 2>&1 || true
      done
  done
  _info "[shopping_cart] Waiting 10s for ESO force-refresh to propagate..."
  sleep 10
```

---

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| ESO secrets synced at different Vault write times | Stale value in app secret → auth fails | All secrets refreshed to current Vault value before PostgreSQL starts |
| No ExternalSecrets in namespace | `kubectl get` returns empty — loop no-ops | Same no-op |
