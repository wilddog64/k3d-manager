# Bugfix: v1.6.2 — Slack thread command `acg-down` (and `acg-up`) not recognized

**Branch:** `k3d-manager-v1.6.2`
**Files:** `bin/k3dm-webhook`

---

## Problem

Typing `/acg-down` (or `acg-down`) as a reply in a Slack job thread does nothing.
The webhook responds with "Unknown command" and lists `kill`, `diagnosis`, `status`, `logs`
as the only valid thread commands.

**Root cause:** `_handle_thread_command` (`bin/k3dm-webhook:372`) does not have a branch
for `acg-down` or `acg-up`. Both verbs fall through to the `else` unknown-command handler.

---

## Reproduction

1. Start an acg-up job via the API — a Slack thread is created.
2. Reply in that thread with `acg-down`.
3. Observe: webhook responds `❓ Unknown command 'acg-down' — available: kill, diagnosis, status, logs`.
4. No cluster down job is started.

---

```python
    elif cmd in ("acg-down", "down"):
        running = _running_cluster_job()
        if running:
            _notify_job(job_id, f"⚠️ Cluster job already running: `{running[0]}` ({running[1]}) — wait for it to finish")
            return
        new_job_id = str(uuid.uuid4())[:8]
        new_job_dir = JOB_DIR / new_job_id
        new_job_dir.mkdir(parents=True, exist_ok=True)
        (new_job_dir / "status").write_text("running")
        (new_job_dir / "action").write_text("down")
        thread_ts_file = JOB_DIR / job_id / "thread_ts"
        if thread_ts_file.exists():
            (new_job_dir / "thread_ts").write_text(thread_ts_file.read_text())
        threading.Thread(
            target=_run_cluster,
            args=(new_job_id, "down", "aws"),
            daemon=True,
        ).start()
        _notify_job(job_id, f"⬇️ *acg-down* queued — job `{new_job_id}`")

    elif cmd in ("acg-up", "up"):
        running = _running_cluster_job()
        if running:
            _notify_job(job_id, f"⚠️ Cluster job already running: `{running[0]}` ({running[1]}) — wait for it to finish")
            return
        new_job_id = str(uuid.uuid4())[:8]
        new_job_dir = JOB_DIR / new_job_id
        new_job_dir.mkdir(parents=True, exist_ok=True)
        (new_job_dir / "status").write_text("running")
        (new_job_dir / "action").write_text("up")
        thread_ts_file = JOB_DIR / job_id / "thread_ts"
        if thread_ts_file.exists():
            (new_job_dir / "thread_ts").write_text(thread_ts_file.read_text())
        threading.Thread(
            target=_run_cluster,
            args=(new_job_id, "up", "aws"),
            daemon=True,
        ).start()
        _notify_job(job_id, f"🚀 *acg-up* queued — job `{new_job_id}`")

    else:
        _notify_job(
            job_id,
            f"❓ Unknown command `{command.strip()}` — available: `kill`, `diagnosis`, `status`, `logs`, `acg-up`, `acg-down`",
        )
```

---

## How it works

Both `acg-down` and `acg-up` follow the same pattern:

1. Check for an already-running cluster job (`_running_cluster_job()`) — reject with a warning if one exists.
2. Create a new `job_id` (8-char UUID) and its `JOB_DIR` entry.
3. Copy the `thread_ts` from the originating job so Slack replies land in the same thread.
4. Spawn `_run_cluster(new_job_id, action, "aws")` in a daemon thread.
5. Post a confirmation to the originating job's thread.

The originating `job_id` (the one passed into `_handle_thread_command`) is used only for
`_notify_job` acknowledgement and `thread_ts` lookup. All cluster work runs under `new_job_id`.
