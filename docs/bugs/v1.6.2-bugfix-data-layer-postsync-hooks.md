# Bugfix: v1.6.2 — Force-sync data-layer to trigger MinIO PostSync hooks

**Branch:** `k3d-manager-v1.6.2`
**Files:** `bin/acg-up`

---

## Problem

On every fresh `acg-up` provision, the MinIO `product-images` bucket may not exist after
`deploy_shopping_cart_data` completes. The root cause is that ArgoCD's `data-layer`
application may already report `Synced` (stale Hub cluster state from the previous
sandbox session). The `acg-up` wait loop (`until ... grep -q "Synced"`) exits immediately
because the status is already `Synced`, and ArgoCD's `selfHeal` sync never re-triggers
PostSync hooks (the `minio-bucket-init` and `minio-image-upload` Jobs).

Without those Jobs running, the MinIO `product-images` bucket does not exist. The
post-provision smoke test then reports `Product images: ❌`.

**Root cause:** `selfHeal` sync does not re-run PostSync hooks that previously ran and
were deleted via `HookSucceeded` policy. Only an explicit `argocd app sync` triggers hook
re-execution.

---

## Reproduction

1. Run `acg-up` on a fresh sandbox (Hub cluster `k3d-k3d-cluster` is already running from
   a previous session — ArgoCD still has `data-layer` in `Synced` state).
2. After provision completes, check:
   ```
   mc ls local/product-images
   ```
3. Expected: 20 product images.
4. Actual: `ERROR: The specified bucket does not exist.`

---

## Fix

### Change — `bin/acg-up`: force-sync data-layer after deploy

After `deploy_shopping_cart_data` completes (line 596), add an explicit force-sync so
ArgoCD re-runs the PostSync hooks regardless of the app's prior Synced state.

**Exact old block (line 596):**

```bash
deploy_shopping_cart_data
```

**Exact new block:**

```bash
deploy_shopping_cart_data
_info "[acg-up] Force-syncing data-layer to trigger PostSync hooks (MinIO bucket init + image upload)..."
argocd app sync data-layer \
  --server localhost:8080 --insecure \
  --force \
  --timeout 180 \
  2>/dev/null \
  || _info "[acg-up] data-layer force-sync returned non-zero — PostSync hooks may still complete"
```
