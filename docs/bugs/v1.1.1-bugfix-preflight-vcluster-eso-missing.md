# Bugfix: v1.1.1 — preflight vCluster has no External Secrets Operator → data-layer cascade

**Branch:** `k3d-manager-v1.7.1`
**Files:** `bin/cluster-preflight` (+ possibly a preflight ESO manifest under `scripts/etc/`) — exact
target to be pinned during the platform-helm trace below.

---

## Problem

A converged-but-failing preflight run (`green1` on `ubuntu-hostinger`) leaves the whole shopping-cart
data tier down. ArgoCD on the hub shows:

```
green1-preflight-data-layer        OutOfSync   Missing
green1-preflight-shopping-cart-basket/frontend   Synced   Degraded
```

`green1-preflight-data-layer` is `SyncError: one or more synchronization tasks are not valid`. The
per-resource sync messages are all variants of:

```
ExternalSecret/redis-cart: The Kubernetes API could not find external-secrets.io/ExternalSecret
for requested resource shopping-cart-data/redis-cart. Make sure the "ExternalSecret" CRD is
installed on the destination cluster.
```

Because the ExternalSecrets are invalid, the data-layer sync aborts → `redis-cart` /
`postgresql-*` Services + StatefulSets never deploy → the app pods cascade:

```
basket-service ... CrashLoopBackOff
  "failed to connect to Redis: lookup redis-cart.shopping-cart-data.svc.cluster.local: no such host"
frontend ...       CrashLoopBackOff
make status → "Data layer: 4 not ready: postgresql-orders, postgresql-payment"
```

**Root cause (confirmed live 2026-06-17):** the **External Secrets Operator is never installed in
the preflight vCluster.** Evidence:
- no `external-secrets` pods syncing for green1 (`kubectl --context ubuntu-hostinger -n vclusters
  get pods | grep external-secret` → empty),
- no `green1-*` ArgoCD Application installs external-secrets,
- the green1 vcluster pod set is argocd / basket / frontend / coredns / prometheus / trivy only.

The data-layer manifests in `shopping-cart-infra` are **correct** — `ExternalSecret` is the intended
way they source Redis/Postgres credentials from Vault via the ClusterSecretStore. The defect is in
the **preflight platform provisioning**: it stands up nested ArgoCD + observability + the app clones
but never installs ESO (operator + CRDs) into the throwaway cluster, so any manifest referencing
`external-secrets.io/ExternalSecret` is rejected as an unknown kind. This is the preflight analogue
of platform bug **#3 (ESO on app cluster)**. It is infra/tooling (k3d-manager), **not** a
shopping-cart-infra bug.

---

## Reproduction

```bash
make preflight NAME=green1 MODE=--keep        # warm ubuntu-hostinger host
# wait for the platform clone to converge, then:
kubectl --context k3d-k3d-cluster -n cicd get application green1-preflight-data-layer \
  -o jsonpath='{.status.conditions[*].message}'
#   -> "one or more synchronization tasks are not valid"
kubectl --context k3d-k3d-cluster -n cicd get application green1-preflight-data-layer \
  -o jsonpath='{range .status.operationState.syncResult.resources[*]}{.message}{"\n"}{end}' | grep ExternalSecret
#   -> "... Make sure the ExternalSecret CRD is installed on the destination cluster."
kubectl --context ubuntu-hostinger -n vclusters get pods | grep external-secret   # -> (nothing)
```

Expected after fix: the preflight installs ESO into the vCluster before data-layer syncs; the
ExternalSecret CRD is present; `green1-preflight-data-layer` reaches Synced/Healthy; `redis-cart` +
`postgresql-*` come up; basket/frontend stop crash-looping.

---

## Fix (direction — pin the exact insertion during implementation)

The persistent clusters get ESO from the platform stack. The preflight must do the equivalent for
the vCluster. **Before writing code, trace where ESO is installed for a normal cluster** and mirror
it for the preflight:

1. Determine how ESO reaches a normal app cluster — is it a member of the `platform-helm`
   ApplicationSet / the platform app-of-apps, or a separate install step? Inspect
   `scripts/etc/argocd/applicationsets/platform-helm*.yaml`, the `platform` AppProject, and any
   `eso`/`external-secrets` plugin (`scripts/plugins/eso.sh`).
2. **If ESO is part of platform-helm:** the preflight platform clone (rendered by `bin/cluster-preflight`)
   must include the external-secrets app so it installs into the vCluster, with a **sync-wave** that
   puts ESO (CRDs + operator ready) **before** `data-layer`. The label/selector isolation fixes
   (`ceffbdb5`, cluster-name-label) already make the clone generate; this adds the missing member.
3. **If ESO is a discrete step:** add an analogous step in `bin/cluster-preflight` that installs ESO
   into the vCluster (targeting the vcluster kubeconfig/context, the same way the ghcr-pull-secret
   step targets it) and a readiness wait for the `externalsecrets.external-secrets.io` CRD before the
   data-layer app is allowed to sync.

Either way: ESO operator + CRDs present in the vCluster, **gated before data-layer**, no hardcoded
IPs, no secrets in argv. Scope stays in `bin/cluster-preflight` (+ a preflight ESO manifest under
`scripts/etc/` if step (3) is chosen). Do NOT edit shopping-cart-infra — the data-layer manifests
are correct.
