# Bugfix: v1.4.5 ‚Äî eso-ldap-directory Vault policy missing keycloak/* paths

**Branch:** `k3d-manager-v1.4.5`
**Files:** `scripts/etc/ldap/vars.sh`

---

## Problem

`keycloak-secrets` and `keycloak-client-secrets` ExternalSecrets in namespace `identity`
remain in `SecretSyncedError` state after the `vault-kv-store` SecretStore fix. Keycloak
and postgres pods cannot start.

**Root cause:** The `eso-ldap-directory` Vault policy (created by `ldap.sh` via
`_vault_configure_secret_reader_role`) only grants read access to `ldap/*` paths.
`LDAP_VAULT_POLICY_PREFIX` defaults to `"ldap"` in `scripts/etc/ldap/vars.sh`.
The Keycloak ExternalSecrets use the same `vault-kv-store` SecretStore (which authenticates
via the `eso-ldap-directory` Kubernetes role), but Vault rejects reads to `keycloak/*`
because the policy has no rule for those paths.

---

## Reproduction

```bash
# Confirm policy is missing keycloak paths:
curl -sf -H "X-Vault-Token: $(kubectl get secret vault-root -n secrets \
  --context k3d-k3d-cluster -o jsonpath='{.data.root_token}' | base64 -d)" \
  "http://localhost:18200/v1/sys/policies/acl/eso-ldap-directory" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['policy'])"
# Actual: only ldap/* paths ‚Äî no keycloak/* paths

kubectl get externalsecret keycloak-secrets -n identity --context k3d-k3d-cluster \
  -o jsonpath='{.status.conditions[0].message}'
# Actual: "could not get secret data from provider"
```

---

## Fix

### Part A ‚Äî Live cluster: extend Vault policy

Obtain the Vault root token:
```bash
_vault_token=$(kubectl get secret vault-root -n secrets \
  --context k3d-k3d-cluster -o jsonpath='{.data.root_token}' | base64 -d)
```

Apply the updated policy via Vault API (Vault is reachable at `localhost:18200`
via the existing port-forward):

```bash
_new_policy='path "secret/data/ldap"           { capabilities = ["read"] }
path "secret/data/ldap/*"          { capabilities = ["read"] }
path "secret/metadata/ldap"        { capabilities = ["read", "list"] }
path "secret/metadata/ldap/*"      { capabilities = ["read", "list"] }
path "secret/data/keycloak"        { capabilities = ["read"] }
path "secret/data/keycloak/*"      { capabilities = ["read"] }
path "secret/metadata/keycloak"    { capabilities = ["read", "list"] }
path "secret/metadata/keycloak/*"  { capabilities = ["read", "list"] }'

curl -sf -X PUT \
  -H "X-Vault-Token: ${_vault_token}" \
  -H "Content-Type: application/json" \
  "http://localhost:18200/v1/sys/policies/acl/eso-ldap-directory" \
  -d "{\"policy\": $(echo "$_new_policy" | python3 -c 'import sys,json; print(json.dumps(sys.stdin.read()))')}"
```

Verify:
```bash
curl -sf -H "X-Vault-Token: ${_vault_token}" \
  "http://localhost:18200/v1/sys/policies/acl/eso-ldap-directory" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['policy'])"
# Expected: policy includes keycloak/* paths
```

Force ESO to re-sync:
```bash
kubectl annotate externalsecret keycloak-secrets -n identity --context k3d-k3d-cluster \
  force-sync="$(date +%s)" --overwrite
kubectl annotate externalsecret keycloak-client-secrets -n identity --context k3d-k3d-cluster \
  force-sync="$(date +%s)" --overwrite
```

Confirm secrets sync:
```bash
kubectl get externalsecret -n identity --context k3d-k3d-cluster
# Expected: keycloak-secrets and keycloak-client-secrets both show SecretSynced / True
kubectl get pods -n identity --context k3d-k3d-cluster
# Expected: postgres-keycloak and keycloak pods reach Running
```

---

### Part B ‚Äî Code fix: `scripts/etc/ldap/vars.sh` line 79

**Exact old line:**
```bash
export LDAP_VAULT_POLICY_PREFIX="${LDAP_VAULT_POLICY_PREFIX:-ldap}"
```

**Exact new line:**
```bash
export LDAP_VAULT_POLICY_PREFIX="${LDAP_VAULT_POLICY_PREFIX:-ldap,keycloak}"
```

This ensures that on the next `make up` (or any call to `ldap_provision`),
`_vault_configure_secret_reader_role` generates a policy covering both `ldap/*`
and `keycloak/*` paths so the cluster state matches the code.
