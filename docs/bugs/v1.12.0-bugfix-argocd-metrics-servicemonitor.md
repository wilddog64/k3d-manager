## Problem

ArgoCD is installed via the Helm chart (`scripts/etc/argocd/values.yaml.tmpl`),
but the chart ships per-component metrics **disabled by default**. As a result:

- No `argocd-*-metrics` Services and **no ServiceMonitor** exist in the `cicd`
  namespace, so Prometheus never scrapes ArgoCD.
- The existing `scripts/etc/argocd/platform-ops/prometheusrule.yaml`
  (`argocd-degraded`) alerts on `argocd_app_info{health_status="Degraded"}`, but
  that series **does not exist** — the rule is dead and can never fire.
- There is no ArgoCD/Image-Updater visibility in Grafana even though a full
  kube-prometheus-stack (Prometheus + Grafana + operator) is running in the
  `monitoring` namespace.

Verified live on `k3d-k3d-cluster` (2026-06-29):

```
$ kubectl -n cicd get servicemonitor
No resources found in cicd namespace.

$ # prometheus query
count(argocd_app_info)  ->  [] (empty vector)
```

**Root cause:** The Helm values template does not enable
`metrics.serviceMonitor` for the ArgoCD components, so the prometheus-operator
has nothing to scrape and the downstream PrometheusRule has no data.

**Why this is the right fix (and the only realistic Grafana path today):**
ArgoCD Image Updater itself exposes **no `/metrics` endpoint** (port 8080 serves
only `/healthz`; `/metrics` returns 404), so it cannot be scraped directly.
Image-updater's effect — it commits a new image digest, which makes ArgoCD sync
the app — is observable through ArgoCD's own metrics
(`argocd_app_info`, `argocd_app_sync_total`). Enabling those metrics is the
zero-new-infrastructure way to see image-updater activity in Grafana.

## Environment facts (already verified — do NOT re-discover or change)

- Prometheus selects ServiceMonitors by label
  **`release: kube-prometheus-stack`**; its `serviceMonitorNamespaceSelector` is
  `{}` (all namespaces). So any ServiceMonitor carrying that label is scraped.
- Grafana dashboard sidecar discovers dashboards by ConfigMap label
  **`grafana_dashboard: "1"`**, `NAMESPACE=ALL` (any namespace).
- The live ArgoCD chart is `argo-cd-9.5.15` (app `v3.4.2`); the script default
  pins `ARGOCD_CHART_VERSION=7.8.1`. The `*.metrics.serviceMonitor.*` value keys
  used below are stable across both. **Do NOT change the chart version.**

## Fix

### Change 1 — `scripts/etc/argocd/values.yaml.tmpl`: enable metrics + ServiceMonitor per component

Make four small, independent edits. Each adds a `metrics:` block with the
prometheus-operator discovery label. The literal strings contain no `$`, so
`envsubst` passes them through unchanged.

**1a — controller**

Exact old block:

```yaml
controller:
  replicas: 1
```

Exact new block:

```yaml
controller:
  replicas: 1

  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: kube-prometheus-stack
```

**1b — applicationSet**

Exact old block:

```yaml
applicationSet:
  replicas: ${ARGOCD_APPLICATIONSET_REPLICAS}
```

Exact new block:

```yaml
applicationSet:
  replicas: ${ARGOCD_APPLICATIONSET_REPLICAS}

  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: kube-prometheus-stack
```

**1c — repoServer**

Exact old block:

```yaml
repoServer:
  replicas: ${ARGOCD_REPO_SERVER_REPLICAS}
```

Exact new block:

```yaml
repoServer:
  replicas: ${ARGOCD_REPO_SERVER_REPLICAS}

  metrics:
    enabled: true
    serviceMonitor:
      enabled: true
      additionalLabels:
        release: kube-prometheus-stack
```

**1d — server**

Exact old block:

```yaml

    - --insecure
```

### Change 4 — new test `scripts/tests/plugins/argocd_metrics_servicemonitor.bats`

Create the file with exactly these contents:

```bash
#!/usr/bin/env bats

VALUES="${BATS_TEST_DIRNAME}/../../etc/argocd/values.yaml.tmpl"
PLUGIN="${BATS_TEST_DIRNAME}/../../plugins/argocd.sh"
DASH="${BATS_TEST_DIRNAME}/../../etc/argocd/platform-ops/grafana-dashboard-argocd.yaml"

@test "metrics: serviceMonitor enabled for all four components" {
  run grep -cF -- 'serviceMonitor:' "${VALUES}"
  [ "${status}" -eq 0 ]
  [ "${output}" -ge 4 ]
}

@test "metrics: prometheus-operator discovery label present per component" {
  run grep -cF -- 'release: kube-prometheus-stack' "${VALUES}"
  [ "${status}" -eq 0 ]
  [ "${output}" -ge 4 ]
}

@test "metrics: dashboard ConfigMap carries grafana sidecar label" {
  run grep -F -- 'grafana_dashboard: "1"' "${DASH}"
  [ "${status}" -eq 0 ]
}

@test "metrics: dashboard targets monitoring namespace" {
  run grep -F -- 'namespace: monitoring' "${DASH}"
  [ "${status}" -eq 0 ]
}

@test "metrics: dashboard applied from argocd.sh platform-ops deploy" {
  run grep -F -- 'grafana-dashboard-argocd.yaml' "${PLUGIN}"
  [ "${status}" -eq 0 ]
}
```
