# Bugfix: ArgoCD/Image Updater Grafana dashboard missing + no data at grafana.3ai-talk.org

**Branch:** `k3d-manager-v1.12.0`
**Type:** Bugfix (regression in dashboard visibility)
**Target file (work):** `scripts/lib/providers/k3s-hostinger.sh`, `scripts/tests/lib/provider_contract.bats`

---

## Problem / Root Cause

The user reports the ArgoCD/Image Updater dashboard is **missing** and shows **no data** at
`grafana.3ai-talk.org`. Investigation (read-only, live cluster) established the exact cause:

There are two Grafana instances:

- **Hub Grafana** — `svc/kube-prometheus-stack-grafana` on context `k3d-k3d-cluster`, namespace
  `monitoring`. The dashboard ConfigMap is applied here (`observability.sh:80`,
  `_observability_apply_argocd_dashboard "${_hub_context}"`), datasource UIDs `prometheus` + `loki`
  both resolve, and every panel returns data. **This Grafana is NOT exposed publicly** — it is only
  reachable by a local port-forward.

- **App-cluster (Hostinger) Grafana** — `svc/acg-kube-prometheus-stack-grafana` on context
  `ubuntu-hostinger`. This is what `grafana.3ai-talk.org` actually shows, via:
  `grafana.3ai-talk.org` → cloudflared (`scripts/etc/cloudflared/config.yml:13-14`) → `localhost:3001`
  → launchd port-forward written in `_hostinger_refresh_access_layer`
  (`scripts/lib/providers/k3s-hostinger.sh:604-610`).

Two prior commits (`0aa8c443` move-to-hub, `72751fc5` remove-from-app-cluster) correctly moved the
dashboard to the hub — because the `argocd_*` / image-updater metrics only exist on the hub
Prometheus — but left an **exposure gap**:

- **"Missing":** the dashboard was deleted from the app-cluster Grafana (the one the user sees) and
  only deployed to the hub Grafana (which is not on the tunnel). Live check of the user's `:3001`
  Grafana: 0 of 24 dashboards are argocd/image-updater.
- **"No data":** the app-cluster Prometheus has no argocd metrics (`argocd_app_info` → 0 results) and
  the app-cluster Grafana has no Loki datasource — so even if the dashboard were present there it
  could never populate.

**Fix:** repoint the existing `:3001` port-forward (which the tunnel already routes to) from the
Hostinger app-cluster Grafana to the **hub Grafana**, where the dashboard exists and has data. This is
a 2-value change to the plist-writer call plus the matching BATS contract assertion. No datasource,
dashboard JSON, or cloudflared change is required.

**Known limitation (do NOT fix in this change):** hub-Grafana exposure remains coupled to the
Hostinger provider's `_hostinger_refresh_access_layer`, and the Hostinger app-cluster's own Grafana is
no longer directly exposed on `:3001` (its node/cluster metrics are still visible on the hub Grafana
via the existing `federate-acg` job, label `cluster: acg`). Decoupling hub-Grafana exposure into the
hub observability deploy path is a future refactor, out of scope here.

## Exact Changes

### Change 1 — `scripts/lib/providers/k3s-hostinger.sh`

Repoint the grafana port-forward plist-writer call (lines 604-610).

**OLD:**
```bash
  _hostinger_write_monitoring_port_forward_plist \
    "${_grafana_pf_plist}" \
    "${_grafana_pf_log}" \
    "svc/acg-kube-prometheus-stack-grafana" \
    "${_HOSTINGER_KUBE_CONTEXT}" \
    "3001" \
    "80"
```

**NEW:**
```bash
  _hostinger_write_monitoring_port_forward_plist \
    "${_grafana_pf_plist}" \
    "${_grafana_pf_log}" \
    "svc/kube-prometheus-stack-grafana" \
    "k3d-k3d-cluster" \
    "3001" \
    "80"
```

Change ONLY the service name (arg 3) and the context (arg 4). Leave all other lines, the
`_grafana_pf_log` / `_grafana_pf_plist` variables, the ports, and the second
`_hostinger_restart_launchd "com.k3d-manager.grafana-port-forward"` block (lines 661-664) untouched.

### Change 2 — `scripts/tests/lib/provider_contract.bats`

Update the two assertions that pin the old grafana port-forward target (lines 556-559).

**OLD:**
```bash
  run grep -F -- 'svc/acg-kube-prometheus-stack-grafana' "${HOME}/Library/LaunchAgents/com.k3d-manager.grafana-port-forward.plist"
  [ "$status" -eq 0 ]
  run grep -F -- '<string>ubuntu-hostinger</string>' "${HOME}/Library/LaunchAgents/com.k3d-manager.grafana-port-forward.plist"
  [ "$status" -eq 0 ]
```

**NEW:**
```bash
  run grep -F -- 'svc/kube-prometheus-stack-grafana' "${HOME}/Library/LaunchAgents/com.k3d-manager.grafana-port-forward.plist"
  [ "$status" -eq 0 ]
  run grep -F -- '<string>k3d-k3d-cluster</string>' "${HOME}/Library/LaunchAgents/com.k3d-manager.grafana-port-forward.plist"
  [ "$status" -eq 0 ]
```

> Note: `svc/kube-prometheus-stack-grafana` is a substring of the old `svc/acg-kube-prometheus-stack-grafana`,
> so the service-name assertion alone is not sufficient — the context assertion
> (`k3d-k3d-cluster` vs `ubuntu-hostinger`) is what proves the repoint. Both lines must change.
