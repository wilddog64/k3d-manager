# Bug: `make up` fails with `Write permissions error` when trusting Vault CA cert

**Branch:** `k3d-manager-v1.5.3`
**File:** `bin/acg-up:353–360`
**Severity:** P2 — blocks `make up` on cluster rebuild

---

## Symptom

```
INFO: [acg-up] Trusting Vault CA cert in macOS System Keychain (may prompt for sudo password)...
SecCertificateAddToKeychain: Write permissions error.
security command failed (1): security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain ...
ERROR: failed to execute security add-trusted-cert ...: 1
make: *** [up] Error 1
```

`make up` exits non-zero at the Vault CA trust step every time the cluster is rebuilt.

---

## Root Cause

`_run_command --prefer-sudo` at `bin/acg-up:355` falls back to running `security` as the current
user when `sudo -n true` has no cached session. The user doesn't have write permission to
`/Library/Keychains/System.keychain`, so `security` exits 1.

`_run_command` without `--soft` calls `_err` on failure, which calls `exit 1` directly — bypassing
the `|| _warn` fallback on line 358–359. The fallback was intended to make this non-fatal but
never fires.

---

## Fix

**`bin/acg-up:353–360` — replace with:**

```bash
    if [[ "$(uname)" == "Darwin" ]]; then
      if ! security verify-cert -c "${_argocd_browser_tls_dir}/ca.crt" >/dev/null 2>&1; then
        _info "[acg-up] Trusting Vault CA cert in macOS System Keychain (may prompt for sudo password)..."
        _run_command --interactive-sudo --soft -- \
          security add-trusted-cert -d -r trustRoot \
            -k /Library/Keychains/System.keychain \
            "${_argocd_browser_tls_dir}/ca.crt" || \
          _warn "[acg-up] Could not add Vault CA to System Keychain — browser may show TLS warning for ArgoCD HTTPS"
      else
        _info "[acg-up] Vault CA cert already trusted in System Keychain — skipping"
      fi
    fi
```

Key changes:
- `security verify-cert` idempotency check — skips `add-trusted-cert` entirely if cert already trusted (no sudo, no PTY needed on repeat runs)
- `--prefer-sudo` → `--interactive-sudo` — prompts for password when needed instead of silently falling back to user
- Added `--soft` — `_run_command` returns non-zero instead of calling `exit 1`, so the `|| _warn` fallback actually fires

---

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| Cert already trusted (repeat run) | `make` fails | Skips silently |
| Cert not trusted, TTY present | `make` fails | Prompts for sudo password, adds cert |
| Cert not trusted, no TTY / sudo fails | `make` fails | WARN logged, `make` continues |
