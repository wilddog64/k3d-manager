# Bugfix: v1.6.3 — acg-resume fails with "ubuntu-k3s context does not exist"

**Branch:** `k3d-manager-v1.6.3`
**Files:** `bin/acg-up`, `bin/k3dm-webhook`

---

## Problem

`/acg-resume` runs `make up CLUSTER_PROVIDER=k3s-aws K3DM_RESUME=1`, which skips step 2
(cluster provisioning) via checkpoint. Step 2 is the only place `add_ubuntu_k3s_cluster`
is called — it SSHs to the remote host, fetches `/home/parallels/.kube/k3s.yaml`, and
merges it into `~/.kube/config` as the `ubuntu-k3s` context.

When resuming after a machine restart (or after `~/.kube/config` was cleared), step 2 is
skipped by checkpoint, so `ubuntu-k3s` context is never written. Step 5
(`shopping_cart_prepare_infra_bootstrap`) then fails immediately:

```
error: context "ubuntu-k3s" does not exist
```

The SSH tunnel is up at this point (step 3 just ran), but the kubeconfig is missing.

**Root cause:** step 2 checkpoint skips both CloudFormation provisioning AND the kubeconfig
merge; the kubeconfig guard to re-fetch from the live cluster is missing.

---

## Reproduction

```bash
# 1. Ensure step-2-cluster checkpoint exists (cluster already provisioned)
ls ~/.local/share/k3d-manager/checkpoints/step-2-cluster.done

# 2. Remove ubuntu-k3s context to simulate the missing-kubeconfig state
kubectl config delete-context ubuntu-k3s

# 3. Run resume
make up CLUSTER_PROVIDER=k3s-aws K3DM_RESUME=1

# Expected: step 5 fails with "error: context "ubuntu-k3s" does not exist"
# Desired: step 3c detects missing context, re-fetches kubeconfig via SSH, continues
```

---

## Fix

### Change 1 — `bin/acg-up`: add kubeconfig guard before step 5

Insert between the GCP early-exit block (line 511) and `shopping_cart_prepare_infra_bootstrap`
(line 513). The guard calls `add_ubuntu_k3s_cluster` (already defined in `shopping_cart.sh`,
already sourced at line 49) when the context is missing. The SSH tunnel is up at this point.

**Exact old block (lines 511–513):**

```bash
fi

shopping_cart_prepare_infra_bootstrap
```

**Exact new block:**

```bash
fi

# ── 3c. ubuntu-k3s kubeconfig guard ──────────────────────────────────────────
if ! kubectl config get-contexts ubuntu-k3s &>/dev/null; then
  _info "[acg-up] ubuntu-k3s context missing — re-fetching kubeconfig via SSH tunnel..."
  add_ubuntu_k3s_cluster
fi

shopping_cart_prepare_infra_bootstrap
```

---

### Change 2 — `bin/k3dm-webhook`: fast-path failure advice for missing context

`_analyze_failure` is called on every resume failure and already posts Gemini output to
Slack. Add a pattern check at the top of the function (before the Gemini call) that
detects the `context "ubuntu-k3s" does not exist` error and returns targeted advice
immediately — no Gemini call needed for this known-bad pattern.

**File:** `bin/k3dm-webhook`
**Function:** `_analyze_failure` (currently line 753)

**Exact old block (first lines of `_analyze_failure`, lines 753–756):**

```python
def _analyze_failure(lines):
    """Feed last 50 non-empty lines + live cluster state to Gemini; return failure diagnosis."""
    last_50 = [l.rstrip() for l in lines[-50:] if l.strip()]
    last_output = "\n".join(last_50)
```

**Exact new block:**

```python
def _analyze_failure(lines):
    """Feed last 50 non-empty lines + live cluster state to Gemini; return failure diagnosis."""
    for _l in lines:
        if 'context "ubuntu-k3s" does not exist' in _l:
            return (
                "CLUSTER_FAILURE\n"
                "PERMANENT\n"
                "ubuntu-k3s kubeconfig context is missing — cluster is provisioned but kubeconfig "
                "was not restored after machine restart or kubeconfig reset.\n"
                "Fix: run /acg-refresh to re-fetch the kubeconfig and restore the ubuntu-k3s "
                "context, then retry /acg-resume"
            )
    last_50 = [l.rstrip() for l in lines[-50:] if l.strip()]
    last_output = "\n".join(last_50)
```
