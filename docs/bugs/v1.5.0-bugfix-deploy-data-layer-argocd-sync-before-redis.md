# Bug: `deploy_shopping_cart_data` copies redis-cart-secret before ArgoCD/ESO creates it

**Branch:** `k3d-manager-v1.5.0`
**File:** `scripts/plugins/shopping_cart.sh:98`
**Severity:** P2 — `make up` fails on fresh sandbox when ExternalSecret hasn't been processed yet

---

## Symptom

```
INFO: [shopping_cart] Waiting for redis-cart-secret to be available in shopping-cart-data (up to 120s)...
error: shopping_cart] redis-cart-secret not found after 120s
make: *** [up] Error 1
```

Occurs on a fresh sandbox or after a full cluster wipe where the `data-layer` ArgoCD app has
never synced (or its last sync is stale). The 120s wait/poll guard added in commit `780930ac`
catches the worst case but doesn't prevent the timeout.

---

## Root Cause

`deploy_shopping_cart_data` at line 98 does `kubectl apply -f redis/cart/` which lands the
`redis-cart` ExternalSecret on the cluster. ESO must then read Vault and create
`redis-cart-secret` — but this takes 15–60s on a cold cluster.

The function immediately starts polling for the secret. On a fast machine the poll wins; on
a slow or loaded sandbox it times out.

The root cause is that we apply the ExternalSecret and immediately poll for its output without
first ensuring ArgoCD (which owns the `data-layer` app) has processed the resource. Triggering
an explicit ArgoCD sync before the apply ensures:
1. ArgoCD declares the app in-sync (meaning ESO is aware of the ExternalSecret).
2. ESO has already started reconciling — the secret appears faster, well within the 120s window.

---

## Fix

Add `argocd app sync data-layer --server localhost:8080 --insecure --wait --timeout 120`
**before** the `kubectl apply -f redis/cart/` line. Use `|| true` so a missing ArgoCD CLI
or an unreachable server degrades gracefully — the existing `kubectl wait` + poll fallback
at line 121 remains as the safety net.

**`scripts/plugins/shopping_cart.sh:94–99` — replace with:**

```bash
  _info "[shopping_cart] Deploying data layer (PostgreSQL, Redis, RabbitMQ)..."
  for pg_dir in orders payment products; do
    _run_command -- kubectl apply --context ubuntu-k3s -f "${infra_root}/postgresql/${pg_dir}/"
  done
  _info "[shopping_cart] Syncing data-layer ArgoCD app before Redis deploy (ensures ESO processes ExternalSecret)..."
  argocd app sync data-layer --server localhost:8080 --insecure --wait --timeout 120 2>/dev/null || true
  _run_command -- kubectl apply --context ubuntu-k3s -f "${infra_root}/redis/cart/"
  _run_command -- kubectl apply --context ubuntu-k3s -f "${infra_root}/rabbitmq/"
```

---

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| Fresh sandbox, cold ESO | Times out after 120s polling | ArgoCD sync warms ESO; secret usually ready before poll starts |
| ArgoCD unreachable (no port-forward) | N/A — same | Degrades to existing 120s poll (no regression) |
| Re-run on existing cluster (secret already present) | Copies immediately | Sync is fast (already in-sync); copies immediately |
