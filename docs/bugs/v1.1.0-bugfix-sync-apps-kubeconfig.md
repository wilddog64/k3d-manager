## Problem

`make sync-apps CLUSTER_PROVIDER=k3s-gcp` calls `register_shopping_cart_apps` without
setting `KUBECONFIG`. The function uses bare `kubectl apply -f` with no `--context` flag,
so it targets whatever cluster the active `KUBECONFIG` points to. In a fresh shell where
KUBECONFIG is unset or points to the k3d local cluster, the ArgoCD Application manifests
are applied to the wrong cluster.

**Root cause:** The Makefile `sync-apps` k3s-gcp case omits `KUBECONFIG="$$HOME/.kube/k3s-gcp.yaml"`, which was required by the preceding spec but not included in the implementation.

---

## Reproduction

```bash
# Open a fresh shell (no prior make up in this session)
make sync-apps CLUSTER_PROVIDER=k3s-gcp
# kubectl apply targets the wrong cluster — no error, wrong cluster
```

---

## Fix

### Change 1 — `Makefile`: add `KUBECONFIG` and echo to the sync-apps k3s-gcp case (line 96)

**Exact old block (lines 95–98):**

```makefile
	@case "$(CLUSTER_PROVIDER)" in \
	  k3s-gcp) CLUSTER_PROVIDER="$(CLUSTER_PROVIDER)" scripts/k3d-manager register_shopping_cart_apps ;; \
	  *)       bin/acg-sync-apps ;; \
	esac
```

**Exact new block:**

```makefile
	@case "$(CLUSTER_PROVIDER)" in \
	  k3s-gcp) echo "[make] CLUSTER_PROVIDER=k3s-gcp — registering shopping-cart apps..."; \
	           CLUSTER_PROVIDER="$(CLUSTER_PROVIDER)" KUBECONFIG="$$HOME/.kube/k3s-gcp.yaml" \
	           scripts/k3d-manager register_shopping_cart_apps ;; \
	  *)       bin/acg-sync-apps ;; \
	esac
```
