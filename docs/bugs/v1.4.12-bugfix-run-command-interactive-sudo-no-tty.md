# Bug: `_run_command --interactive-sudo` fails with PTY error when no TTY (e.g. `make up`)

**Branch:** `k3d-manager-v1.4.12`
**File:** `scripts/lib/system.sh` — `_run_command_resolve_sudo` (subtree: lib-foundation)
**Severity:** P1 — `make up` fails repeatedly at any step using `--interactive-sudo`

---

## Symptom

Any `_run_command --interactive-sudo` call from within `make up` (which has no controlling
TTY) fails with:

```
sudo: unable to allocate pty: Device not configured
```

Affected steps in `bin/acg-up` (all use `--interactive-sudo`):
- Line 267 — ArgoCD browser HTTPS launchctl bootout
- Line 297 — Vault CA cert trust (`security add-trusted-cert`)
- Line 327/335/344 — ArgoCD browser launchd install + bootstrap
- Line 949/957/966 — Keycloak browser launchd install + bootstrap
- Line 1089 — loopback alias (`ifconfig lo0 alias`)
- Line 1122/1125/1131 — loopback plist launchd install + bootstrap
- Line 1174/1182/1191 — frontend browser launchd install + bootstrap
- Line 1317/1320/1326 — Cloudflare tunnel launchd install + bootstrap

---

## Root Cause

In `_run_command_resolve_sudo` (`scripts/lib/system.sh:60`):

```bash
local -a sudo_flags=()
if (( interactive_sudo == 0 )); then
  sudo_flags=(-n)   # only adds -n when NOT interactive
fi
```

When `--interactive-sudo` is set, `sudo_flags` is empty — sudo runs without `-n` and
attempts to allocate a PTY to prompt for a password. When the caller has no TTY (launched
from `make`, a CI runner, or any non-interactive shell), sudo fails.

---

## Fix

In `_run_command_resolve_sudo`, detect TTY availability before deciding on sudo flags.
If no TTY is present, force `sudo -n` even when `--interactive-sudo` was requested.
If `sudo -n` also fails (credentials not cached), emit a clear actionable error.

**`scripts/lib/system.sh` — replace lines 59–62 with:**

```bash
  local -a sudo_flags=()
  if (( interactive_sudo == 0 )); then
    sudo_flags=(-n)
  elif [[ ! -t 0 || ! -t 1 ]]; then
    # No TTY — cannot prompt interactively; fall back to sudo -n (cached credentials).
    # If credentials are not cached, sudo -n will fail with a clear error.
    sudo_flags=(-n)
  fi
```

**Note:** this is a `scripts/lib/system.sh` change. Because `system.sh` is subtree-managed
from lib-foundation, the fix must go to lib-foundation first
(`feat/v0.3.5-run-command-no-tty-sudo`), then be pulled into k3d-manager via
`git subtree pull`.

---

## Workaround (until fix ships)

Run `sudo -v` in the terminal before `make up` to cache sudo credentials. All
`--interactive-sudo` calls will then succeed via `sudo -n` (cached).

```bash
sudo -v && make up
```

If `sudo -v` itself fails with `unable to allocate pty`, a zombie Node/Gemini process is
likely holding the PTY device open. Kill it first:

```bash
pkill -f node
sudo -v && make up
```

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| `make up` with no TTY, sudo credentials cached | PTY error, aborts | `sudo -n` succeeds silently |
| `make up` with no TTY, credentials NOT cached | PTY error, aborts | Clear error: "sudo credentials not cached — run `sudo -v` first" |
| Interactive terminal with TTY | Works (prompts) | Works (prompts, unchanged) |
