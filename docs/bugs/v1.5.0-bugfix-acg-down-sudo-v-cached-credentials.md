# Bug: `bin/acg-down` prompts for password even when sudo credentials are already cached

**Branch:** `k3d-manager-v1.5.0`
**File:** `bin/acg-down:46–50`
**Severity:** P3 — unnecessary password prompt when running `acg-down` after `acg-up`

---

## Symptom

```
INFO: [acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)...
Password:
```

Appears even when the user just ran `acg-up` minutes earlier — sudo credentials are still
cached within the default 5-minute window, but `sudo -v` prompts unconditionally.

---

## Root Cause

The existing TTY guard (`[[ -t 0 && -t 1 ]]`) correctly prevents the prompt in non-TTY
contexts, but does not check whether sudo credentials are **already cached**. When running
`bin/acg-down` directly in a terminal the guard passes and `sudo -v` always asks for a
password, even though the prior `acg-up` run already authorized it.

```bash
# bin/acg-down:44-50 — current
if _is_mac; then
  if [[ -t 0 && -t 1 ]]; then
    _info "[acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)..."
    sudo -v
  fi
fi
```

---

## Fix

Add a `sudo -n true 2>/dev/null` cache probe before prompting. Only pre-authorize when
credentials are **not** cached. `sudo -n` exits 0 if credentials are valid, non-zero if they
need renewal.

**`bin/acg-down:44–50` — replace with:**

```bash
if _is_mac; then
  if [[ -t 0 && -t 1 ]] && ! sudo -n true 2>/dev/null; then
    _info "[acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)..."
    sudo -v
  fi
fi
```

---

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| Credentials cached (e.g. just ran `acg-up`) | `Password:` prompt | Silent — skips pre-auth |
| Credentials expired in terminal (TTY present) | `Password:` prompt | `Password:` prompt (unchanged — still prompts once) |
| No TTY (e.g. `make down`) | Silent (TTY guard) | Silent (unchanged) |
