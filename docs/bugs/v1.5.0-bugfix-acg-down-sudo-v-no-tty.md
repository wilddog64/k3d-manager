# Bug: `make down` prompts for password — bare `sudo -v` in no-TTY subshell

**Branch:** `k3d-manager-v1.5.0`
**File:** `bin/acg-down:46–47`
**Severity:** P2 — interactive password prompt blocks `make down` in non-TTY environments

---

## Symptom

```
INFO: [acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)...
Password:
```

`make down` (or any caller that spawns `bin/acg-down` without a controlling TTY) gets a
dangling `Password:` prompt. The process either hangs waiting for input or fails.

---

## Root Cause

`bin/acg-down:47` calls `sudo -v` directly — outside of `_run_command` — to pre-authorize
sudo credentials before the launchd cleanup steps. `sudo -v` tries to prompt for a password
when credentials are not cached, which requires a TTY. When `make` spawns `bin/acg-down` in
a subshell there is no controlling TTY, so the prompt appears unexpectedly.

The lib-foundation PTY fix (`_run_command_resolve_sudo` falling back to `sudo -n` when no
TTY) does not help here because this `sudo -v` call bypasses `_run_command` entirely.

```bash
# bin/acg-down:45-48 — current
if _is_mac; then
  _info "[acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)..."
  sudo -v
fi
```

---

## Fix

Guard the `sudo -v` call with a TTY check. When no TTY is present, skip the pre-auth
silently — `_run_command --interactive-sudo` already falls back to `sudo -n` in that case.

**`bin/acg-down:45–48` — replace with:**

```bash
if _is_mac; then
  if [[ -t 0 && -t 1 ]]; then
    _info "[acg-down] Pre-authorizing sudo for macOS host operations (launchd cleanup)..."
    sudo -v
  fi
fi
```

---

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| `make down` (no TTY) | `Password:` prompt, hangs | Silent — skips pre-auth |
| `bin/acg-down` run directly in terminal (TTY present) | Pre-auth prompt | Pre-auth prompt (unchanged) |
| Sudo credentials already cached | No prompt | No prompt (unchanged) |
