# Bug: v1.6.0 — webhook Python 3.9 SIGSEGV on macOS 26.5.1 Beta

**Branch:** `k3d-manager-v1.6.0`
**Date:** 2026-06-04
**Files:** `scripts/etc/launchd/com.k3d-manager.webhook.plist.tmpl`, `~/Library/LaunchAgents/com.k3d-manager.webhook.plist`

---

## Symptom

`bin/k3dm-webhook` crashes repeatedly with `EXC_BAD_ACCESS (SIGSEGV)` at address
`0x000000010609c9c5`. macOS generates 5+ crash reports with parent PID matching the
webhook LaunchAgent process. Webhook becomes unresponsive; all Slack commands return
"Something went wrong."

## Root Cause

Python 3.9 (`/usr/bin/python3`, Apple CommandLineTools build) has a fork-safety bug on
macOS 26.5.1 Beta. When `subprocess.run()` is called from a thread inside a heavily
threaded `http.server.HTTPServer`, the forked child process crashes during dynamic
linker initialization. This is a known Python 3.9 regression on recent macOS betas.

## Fix

Changed the LaunchAgent `ProgramArguments` interpreter from `/usr/bin/python3` (Python 3.9)
to `/opt/homebrew/bin/python3` (Python 3.13). Also updated the plist template so fresh
installs via `bin/k3dm-webhook-setup` use the correct interpreter.

**Live plist** (`~/Library/LaunchAgents/com.k3d-manager.webhook.plist`): changed manually,
`make restart-webhook` applied.

**Template** (`scripts/etc/launchd/com.k3d-manager.webhook.plist.tmpl`): updated in commit
on `k3d-manager-v1.6.0`.

## Verification

```bash
ps aux | grep k3dm-webhook   # confirm /opt/homebrew/Cellar/python@3.13/... in process path
tail -20 ~/Library/Logs/k3dm-webhook.log   # no SIGSEGV; only DeprecationWarning for datetime.utcnow()
```

## Notes

- `datetime.utcnow()` DeprecationWarning in Python 3.13 is cosmetic — not blocking.
- The template previously hardcoded `/usr/bin/python3`; any new install on macOS would
  reproduce the crash without this fix.
