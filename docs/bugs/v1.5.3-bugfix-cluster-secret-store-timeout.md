# Bug: `kubectl wait ClusterSecretStore vault-backend` times out at 120s on cluster rebuild

**Branch:** `k3d-manager-v1.5.3`
**File:** `scripts/plugins/shopping_cart.sh:150`
**Severity:** P2 — blocks `make up` on cluster rebuild; self-heals after timeout

---

## Symptom

```
INFO: [shopping_cart] Waiting for ClusterSecretStore vault-backend to be Ready...
error: timed out waiting for the condition on clustersecretstores/vault-backend
ERROR: [shopping_cart] ClusterSecretStore vault-backend never became Ready
make: *** [up] Error 1
```

After `make up` fails, `kubectl get clustersecretstore vault-backend --context ubuntu-k3s`
shows `Ready=True` — the store self-healed a few minutes later.

---

## Root Cause

The chain from ClusterSecretStore to Vault is:

```
ESO (ubuntu-k3s) → vault-bridge K8s Service:8201
  → vault-bridge Endpoints → node IP:8201
  → vault-bridge socat (EC2) → EC2 localhost:8200
  → SSH reverse tunnel (-R 8200:127.0.0.1:18200) → Mac localhost:18200
  → kubectl port-forward → k3d Vault:8200
```

On cluster rebuild, the SSH reverse tunnel (managed by autossh/launchd) may not be
fully re-established by the time ESO validates the ClusterSecretStore. ESO's first
validation attempt fails, and it retries on its default ~30s interval. Four retries = 120s,
which exhausts the `kubectl wait` budget before Vault becomes reachable.

The ClusterSecretStore also already receives a `k3d-manager/reconcile-at` annotation
when it is applied (step 9), but that annotation is set at apply time — ESO may have already
started its retry backoff before the annotation update triggers a resync.

---

## Fix

**`scripts/plugins/shopping_cart.sh:149–152` — replace with:**

```bash
  _info "[shopping_cart] Waiting for ClusterSecretStore vault-backend to be Ready..."
  local _css_ready=0
  for _css_try in 1 2 3; do
    if kubectl wait --for=condition=Ready --timeout=90s \
        clustersecretstore/vault-backend --context ubuntu-k3s 2>/dev/null; then
      _css_ready=1
      break
    fi
    _warn "[shopping_cart] ClusterSecretStore vault-backend not Ready after 90s (attempt ${_css_try}/3) — forcing reconcile..."
    kubectl annotate clustersecretstore vault-backend --context ubuntu-k3s \
      "k3d-manager/reconcile-at=$(date -u +%Y%m%dT%H%M%SZ)" --overwrite >/dev/null 2>&1 || true
  done
  if (( _css_ready == 0 )); then
    _err "[shopping_cart] ClusterSecretStore vault-backend never became Ready after 3 attempts (270s total)"
    return 1
  fi
```

**Why:** Three 90s windows = 270s total. After each failed window, a fresh reconcile
annotation forces ESO to re-validate immediately instead of waiting for its full retry
interval. On happy path (Vault reachable from the start), the first 90s window succeeds
with no extra delay.

## Behaviour After Fix

| Scenario | Before | After |
|----------|--------|-------|
| Vault reachable immediately (happy path) | Ready in <30s, ✅ | Same |
| SSH tunnel slow to reconnect (~90–150s) | Timeout at 120s, ❌ | Force-reconcile fires, Ready by attempt 2, ✅ |
| Vault unreachable entirely | Timeout at 120s, ❌ | Timeout at 270s with 3 clear WARN messages, ❌ |
