# Bugfix: v1.4.11 — Keycloak LDAP bind credential stale when realm already exists

**Branch:** `k3d-manager-v1.4.11`
**Files:** `bin/acg-up`

---

## Problem

After `make up` on a cluster where the Keycloak realm already exists, ArgoCD SSO login
fails with "Invalid username or password." LDAP users are synced and user passwords are
correct, but Keycloak cannot bind to OpenLDAP.

**Root cause:** Step 10d imports the realm with the current `_ldap_admin_pass` substituted
into the realm JSON. When the realm already exists, Keycloak returns HTTP 409 and acg-up
falls back to reconciling only the OIDC client — leaving the LDAP federation component's
`bindCredential` pointing at the previous run's (rotated) password.

---

## Reproduction

1. Run `make up` to completion on a fresh cluster → SSO works
2. Tear down and reprovision → `make up` with the realm already in Keycloak DB
3. Step 10d sees HTTP 409, skips full realm import
4. LDAP bind credential in Keycloak remains from previous run
5. Login fails: "Invalid username or password"

---

## Fix

Added **Step 10d.6** in `bin/acg-up` immediately after the LDAP user password seed step.
Uses `kcadm.sh` inside the Keycloak pod to:
1. Find the LDAP federation component ID in the `shopping-cart` realm
2. Update `config.bindCredential` to the current `_ldap_admin_pass` from Vault
3. Trigger a full LDAP user sync to verify connectivity
