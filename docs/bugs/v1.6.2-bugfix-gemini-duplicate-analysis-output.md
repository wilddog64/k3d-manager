# Bugfix: v1.6.2 — Gemini duplicate analysis + state dir cleanup

**Branch:** `k3d-manager-v1.6.2`
**Files:** `bin/k3dm-webhook`, `bin/k3dm-cleanup` (new), `scripts/etc/launchd/com.k3d-manager.cleanup.plist.tmpl` (new), `Makefile`

---

## Problem

**Problem 1 — Duplicate Gemini analysis in Slack**

The Slack failure analysis message (`🔍 Failure analysis`) contains the Gemini response
repeated 2–3 times with internal tool-use reasoning prose between each block
(e.g. "The read_file tool returned binary content...", "Now I can formulate the response...").

**Root cause:** `_call_gemini` sets `GEMINI_CLI_TRUST_WORKSPACE=true` in the subprocess env,
which enables the Gemini CLI's workspace file-reading tools. `_analyze_failure` appends
`"Please read this image file and describe..."` to the prompt when a screenshot path is found
in the log. The Gemini CLI tries `read_file` on the PNG, receives binary content it cannot
parse, retries with different reasoning, and outputs all its chain-of-thought to stdout on
each iteration — all of which ends up in the single response string posted to Slack.

**Problem 2 — State dir `~/.local/share/k3d-manager/` grows unboundedly**

Failure screenshots (`acg-extend-failure-*.png`) accumulate in the state root and
`logs/` subdirectory — 11 in root and 44 in logs/ observed at ~1-2MB each. Playwright
browser profiles (`profile/` and `playwright-auth/`) accumulate Chromium re-downloadable
cache subdirs (`component_crx_cache`, `optimization_guide_model_store`, `GraphiteDawnCache`,
`WasmTtsEngine`) that add ~240MB over time. No cleanup runs automatically.

## Reproduction

**Problem 1:**
Run `bin/acg-up` against a fresh Pluralsight sandbox. When `make up` fails and a Playwright
screenshot is present in `/tmp/`, the Slack failure analysis message will contain the Gemini
analysis paragraph repeated 2–3 times with tool-use prose between repetitions.

**Problem 2:**
```bash
du -sh ~/.local/share/k3d-manager/
# Expected: <200MB
# Actual:   ~900MB and growing
ls ~/.local/share/k3d-manager/acg-extend-failure-*.png | wc -l  # dozens of PNGs
du -sh ~/.local/share/k3d-manager/profile/component_crx_cache   # 50MB+ Chromium cache
```

## Fix

### Change 1 — `bin/k3dm-webhook`: remove `GEMINI_CLI_TRUST_WORKSPACE` from `_call_gemini` env

Disabling workspace tools prevents the Gemini CLI from attempting file reads. All callers
of `_call_gemini` pass their full context via the prompt string — no caller depends on
Gemini reading files from the workspace.

**Exact old block (lines 518–519):**

```python
    gemini_bin = os.environ.get("K3DM_GEMINI_BIN", "gemini")
    env = {**os.environ, "GEMINI_CLI_TRUST_WORKSPACE": "true"}
```

**Exact new block:**

```python
    gemini_bin = os.environ.get("K3DM_GEMINI_BIN", "gemini")
    env = {k: v for k, v in os.environ.items() if k != "GEMINI_CLI_TRUST_WORKSPACE"}
```

### Change 2 — `bin/k3dm-webhook`: remove "read this image file" instruction from `_analyze_failure` screenshot prompt

Gemini CLI cannot process binary PNG. Replace the read instruction with a plain note that
a screenshot was captured; Gemini can still reason from log context.

**Exact old block (lines 675–682):**

```python
            if os.path.isfile(screenshot_path):
                prompt = (
                    prompt.rstrip()
                    + f"\n\nA browser screenshot was captured at the moment of failure: {screenshot_path}"
                    "\nPlease read this image file and describe in 2-3 sentences what the browser "
                    "was showing — which page or dialog is visible, what UI elements are present, "
                    "and how this visual state relates to the failure."
                )
            break
```

**Exact new block:**

```python
            if os.path.isfile(screenshot_path):
                prompt = (
                    prompt.rstrip()
                    + f"\n\nA browser screenshot was saved at {screenshot_path} at the moment of failure."
                    " Based on the log context above, describe in 1-2 sentences what the browser"
                    " was likely showing and how it relates to the failure."
                )
            break
```

### Change 3 — `bin/k3dm-cleanup` (new file): periodic state dir cleanup script

Create `bin/k3dm-cleanup` as an executable shell script:

```bash
#!/usr/bin/env bash
set -euo pipefail

K3DM_STATE_DIR="${HOME}/.local/share/k3d-manager"

# Failure screenshots — keep most recent 10 in root, 10 in logs/
_prune_pngs() {
  local dir="$1"
  local keep="${2:-10}"
  local count
  count=$(ls -1t "${dir}"/acg-extend-failure-*.png 2>/dev/null | wc -l | tr -d ' ')
  if [[ "${count}" -gt "${keep}" ]]; then
    ls -1t "${dir}"/acg-extend-failure-*.png | tail -n "+$((keep + 1))" | xargs rm -f
    echo "k3dm-cleanup: pruned $((count - keep)) failure PNGs from ${dir}"
  fi
}

# Chromium re-downloadable cache dirs — safe to delete; browser re-creates them
_prune_chromium_cache() {
  local profile_dir="$1"
  local -a cache_dirs=(
    component_crx_cache
    optimization_guide_model_store
    GraphiteDawnCache
    WasmTtsEngine
    Safe\ Browsing
  )
  for cache_dir in "${cache_dirs[@]}"; do
    local target="${profile_dir}/${cache_dir}"
    if [[ -d "${target}" ]]; then
      rm -rf "${target}"
      echo "k3dm-cleanup: removed Chromium cache ${target}"
    fi
  done
}

_prune_pngs "${K3DM_STATE_DIR}"
_prune_pngs "${K3DM_STATE_DIR}/logs"
_prune_chromium_cache "${K3DM_STATE_DIR}/profile"
_prune_chromium_cache "${K3DM_STATE_DIR}/playwright-auth"

echo "k3dm-cleanup: done"
```

File must be executable (`chmod 755 bin/k3dm-cleanup`).

### Change 4 — `scripts/etc/launchd/com.k3d-manager.cleanup.plist.tmpl` (new file): daily LaunchAgent

Create this plist template:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.k3d-manager.cleanup</string>
  <key>ProgramArguments</key>
  <array>
    <string>{{K3DM_REPO_ROOT}}/bin/k3dm-cleanup</string>
  </array>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key>
    <integer>3</integer>
    <key>Minute</key>
    <integer>0</integer>
  </dict>
  <key>RunAtLoad</key>
  <false/>
  <key>StandardOutPath</key>
  <string>{{HOME}}/Library/Logs/k3dm-cleanup.log</string>
  <key>StandardErrorPath</key>
  <string>{{HOME}}/Library/Logs/k3dm-cleanup.log</string>
</dict>
</plist>
```

### Change 5 — `Makefile`: add `install-cleanup` target

Append after the `install-token-rotator` target (find that block and add after it):

```makefile
## Install the daily state dir cleanup LaunchAgent (run once; safe to re-run)
install-cleanup:
	sed \
	  -e "s|{{K3DM_REPO_ROOT}}|$$(pwd)|g" \
	  -e "s|{{HOME}}|$(HOME)|g" \
	  scripts/etc/launchd/com.k3d-manager.cleanup.plist.tmpl \
	  > "$(HOME)/Library/LaunchAgents/com.k3d-manager.cleanup.plist"
	launchctl bootout "gui/$$(id -u)/com.k3d-manager.cleanup" 2>/dev/null || true
	launchctl bootstrap "gui/$$(id -u)" \
	  "$(HOME)/Library/LaunchAgents/com.k3d-manager.cleanup.plist"
	@echo "Cleanup agent installed — fires daily at 03:00"
```

### Change 6 — `bin/acg-down`: prune Docker build cache and unused images after cluster delete

After `acg-down` deletes the k3d Hub cluster, Docker build cache and unused images
accumulate (~1.3GB reclaimable observed). Add a prune step gated on Docker being
available. Uses `docker system prune --force` (no `--volumes` — volumes are cleaned
up by k3d delete and pruning all volumes would affect non-k3d projects).

**Exact old block (lines 221–222):**

```bash
fi

_info "[acg-down] Cleaning up stale /tmp files..."
```

**Exact new block:**

```bash
fi

_info "[acg-down] Pruning Docker build cache and unused images..."
if command -v docker >/dev/null 2>&1; then
  docker system prune --force >/dev/null
  _info "[acg-down] Docker prune complete"
fi

_info "[acg-down] Cleaning up stale /tmp files..."
```
