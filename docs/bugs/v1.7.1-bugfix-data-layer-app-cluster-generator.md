# v1.7.1-bugfix — Parameterize app-cluster destination via ArgoCD cluster generator

**Filed:** 2026-06-19
**Type:** bugfix + small refactor (root-cause fix for data-layer cross-cluster misroute)
**Repos:** `k3d-manager`, `shopping-cart-infra`

## Problem

`data-layer` ArgoCD Application hardcodes its destination:

```yaml
# shopping-cart-infra/argocd/applications/data-layer.yaml:22-24
destination:
  name: ubuntu-k3s
  namespace: shopping-cart-data
```

Every shopping-cart **workload** (basket, frontend, order, payment, product-catalog, namespace)
deploys to `ubuntu-hostinger`, but the **data tier** (PostgreSQL, RabbitMQ, Redis) deploys to a
*different* cluster `ubuntu-k3s` (`https://host.k3d.internal:6443`, currently unreachable). This
cannot work: a Service in cluster A is unresolvable from cluster B, so basket crashes on
`lookup redis-cart.shopping-cart-data.svc.cluster.local: no such host` (1348 restarts / 4d19h),
order is CreateContainerConfigError, postgres-orders/payment never become ready.

Root cause: `data-layer` is a **static** manifest delivered by the `shopping-cart-apps` app-of-apps
(`kubectl apply -f argocd/applications/`, raw git sync — no envsubst), so it never received the
`${APP_CLUSTER_NAME}` parameterization the `services-git` ApplicationSet uses. It is the only app
left on the old cluster.

## Design (chosen): ArgoCD cluster generator keyed on a label

Single source of truth = a label on the registered ArgoCD cluster secret:
`k3d-manager/role: app-cluster`. Exactly one registered cluster carries it at a time = the active
app cluster. All shopping-cart Applications (services **and** data-layer) resolve their destination
from an ApplicationSet `clusters` generator with that selector — ArgoCD resolves the destination,
no envsubst, no per-app hardcoding. Switching clusters = move the label.

`_argocd_set_active_app_cluster` (new) sets the label exclusively and is called from both
`register_app_cluster` and `_argocd_deploy_applicationsets`, with the active cluster derived from
the existing resolver `_acg_provider_context "$(_acg_resolve_provider)"`.

### Edit 4 — `scripts/plugins/argocd.sh` — add `_argocd_set_active_app_cluster`

Insert immediately **before** `function register_app_cluster() {` (i.e. right after
`_argocd_deploy_applicationsets` closes):

```bash
function _argocd_set_active_app_cluster() {
  local _active="${1:-}"
  if [[ -z "${_active}" ]]; then
    _err "[argocd] _argocd_set_active_app_cluster: active cluster name required"
    return 1
  fi
  local _ns="${ARGOCD_NAMESPACE:-cicd}"
  local _s _cname
  while IFS= read -r _s; do
    [[ -z "${_s}" ]] && continue
    _cname="$(_kubectl get "${_s}" -n "${_ns}" -o jsonpath='{.metadata.labels.argocd\.argoproj\.io/cluster-name}' 2>/dev/null)"
    if [[ "${_cname}" == "${_active}" ]]; then
      _kubectl label "${_s}" -n "${_ns}" k3d-manager/role=app-cluster --overwrite >/dev/null 2>&1 || true
    else
      _kubectl label "${_s}" -n "${_ns}" k3d-manager/role- >/dev/null 2>&1 || true
    fi
  done < <(_kubectl get secrets -n "${_ns}" -l argocd.argoproj.io/secret-type=cluster -o name 2>/dev/null)
  _info "[argocd] app-cluster role label set on '${_active}' (cleared from others)"
}
```

## Cutover (Claude runs directly after merge — NOT Codex, NOT Gemini)

**Pre-req:** confirm `ubuntu-hostinger` has PV capacity for postgres-orders/payment/products +
RabbitMQ + Redis.

**Hazards (verified live 2026-06-19):** the existing `data-layer` Application carries
`resources-finalizer.argocd.argoproj.io`, and the `shopping-cart-apps` app-of-apps is
`prune=true selfHeal=true`. So merging the infra PR auto-prunes the old `data-layer`, which then
**hangs on the finalizer** (child-resource deletion targets the unreachable `ubuntu-k3s`). Both the
old (app-of-apps) and new (`data-git`) Applications are named `data-layer` in `cicd` — do not let
them fight. Required order:

1. **Merge shopping-cart-infra PR first** (removes the static `data-layer.yaml`). The app-of-apps
   prunes the old `data-layer` Application → it gets stuck terminating on the finalizer.
2. **Clear the finalizer** so it actually deletes (cannot reach `ubuntu-k3s` to run it):
   `kubectl patch application data-layer -n cicd --type=merge -p '{"metadata":{"finalizers":null}}'`
   then confirm `kubectl get application data-layer -n cicd` is gone.
3. **Merge k3d-manager PR** (services-git matrix + `data-git` ApplicationSet).
4. **Label the active cluster explicitly** (deterministic — do not rely on the resolver), and ensure
   no other cluster carries it:
   `kubectl label secret cluster-ubuntu-hostinger -n cicd k3d-manager/role=app-cluster --overwrite`
   `kubectl label secret cluster-ubuntu-k3s -n cicd k3d-manager/role- 2>/dev/null || true`
5. **Apply the ApplicationSets** (`deploy_argocd` / `_argocd_deploy_applicationsets`). `data-git`
   creates a fresh `data-layer` targeting `ubuntu-hostinger`; `services-git` regenerates services
   on `ubuntu-hostinger`.
6. **Verify:** data tier StatefulSets Ready in `shopping-cart-data` on `ubuntu-hostinger`; basket
   leaves CrashLoop; order leaves CreateContainerConfigError; smoke-test data-layer passes.
