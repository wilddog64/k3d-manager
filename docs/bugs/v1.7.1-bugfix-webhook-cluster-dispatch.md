# Bugfix: v1.7.1 ‚Äî webhook cluster-up/down/resume dispatch (standalone + provider allowlist)

**Branch:** `k3d-manager-v1.7.1`
**Files:** `bin/k3dm-webhook`

Consolidates two filed issues against the Slack webhook dispatcher:
- `docs/issues/2026-06-15-cluster-up-not-in-standalone-slack-commands.md`
- `docs/issues/2026-06-15-cluster-up-hostinger-routes-to-k3s-aws.md`

Both are the same dispatcher, same root-cause class (command-surface drift after the
`acg-*` ‚Üí `cluster-*` rename). ACG logic is unchanged ‚Äî it remains the `aws` provider
under the unified `cluster-*` commands (`cluster-up aws` ‚Üí `make up CLUSTER_PROVIDER=k3s-aws`).

---

## Problem

1. **Top-level `cluster-up` / `cluster-down` are silently dropped.** A fresh
   `/cluster-up <provider>` posted as a top-level Slack message (not a thread reply)
   never reaches the dispatcher, because `cluster-up`, `cluster-down`, and
   `cluster-resume` are absent from `_STANDALONE_CMDS`. They work only as replies
   inside an existing job thread.

2. **`cluster-up|down|resume <hostinger>` silently routes to k3s-aws.** The provider
   allowlist on the up/down/resume branches is `("aws", "gcp", "az")` with an `"aws"`
   fallback. `hostinger` (and any typo) fails the membership check and falls back to
   `aws`, kicking off a real ACG AWS sandbox credential run on a host where the user
   asked for Hostinger. Confirmed by job `4570a5e0` (~12m ACG timeout, `exited -15`).

**Root cause:** `cluster-up/down/resume` were added to `_THREAD_COMMANDS` and the
`_handle_thread_command` dispatch chain during the `acg-*` ‚Üí `cluster-*` rename, but
`_STANDALONE_CMDS` and the up/down/resume provider allowlists were never updated to
match `cluster-status`/`cluster-refresh` (which already include `hostinger` and default
to `hostinger`).

---

## Reproduction

- **Drop:** post top-level `/cluster-up aws` in the channel ‚Üí no webhook response.
  (Same command as a reply in an existing job thread works.)
- **Misroute:** post `/cluster-up hostinger` ‚Üí webhook runs
  `make up CLUSTER_PROVIDER=k3s-aws`, fails in ACG sandbox credential extraction
  (`Terminated: 15`) instead of provisioning k3s-hostinger.

---

## Why these values

- Allowlist `("aws", "gcp", "az", "hostinger")` matches `cluster-status` (617) and
  `cluster-refresh` (649) exactly ‚Äî no more drift.
- Default `"hostinger"` matches status/refresh and is the always-on lab VPS: a bare or
  typo'd `cluster-up` can never accidentally start an ACG AWS sandbox run. ACG stays
  reachable explicitly via `cluster-up aws` / `cluster-down aws` / `cluster-resume aws`.
