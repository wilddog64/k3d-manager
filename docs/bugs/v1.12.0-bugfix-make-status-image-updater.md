## Background (read this — it explains B and why B is docs-only)

During live validation of the metrics-ServiceMonitor change (`60cd500c`), a
manual `helm upgrade argocd ... -n cicd` failed with a server-side-apply
conflict on `argocd-cm` (`.data.oidc.config`, `.data.url`) and `argocd-rbac-cm`
(`.data.policy.csv`, `.data.scopes`). Investigation showed:

- `bin/cluster-up` configures ArgoCD SSO **out-of-band** at Step 10f — it
  `kubectl apply`s `argocd-cm.yaml`/`argocd-rbac-cm.yaml` from shopping-cart-infra
  (owns `policy.csv`/`scopes`) and `kubectl patch`es the Cloudflare public `url`
  + `oidc.config` (owns those). Helm does not own those fields → any later helm
  upgrade conflicts.
- BUT the helm install (`deploy_argocd`) runs at Step **3.6**, *before* Step
  **10f**. The refresh path uses `deploy_argocd_bootstrap`, which does **not**
  re-run helm. And `deploy_argocd` hard-codes `release_exists=0` → plain
  `helm upgrade --install`.

**Therefore the metrics change is safe in every normal flow** (values apply on
fresh install, SSO patches land afterward). The conflict only fires on a
*manual* `helm upgrade`/`deploy_argocd` against an already-SSO-patched live
release. So deliverable B is a **known-issue doc only — do NOT modify
`scripts/plugins/argocd.sh` or the deploy path.**

```bash
kubectl get applicationsets.argoproj.io -A --context "${INFRA_CONTEXT}" 2>/dev/null \
  || echo "Cannot reach Hub cluster or ArgoCD CRDs not installed"

echo ""
echo "=== ArgoCD Image Updater ==="
_iu_ready="$(kubectl get deploy argocd-image-updater -n "${ARGOCD_NAMESPACE}" \
  --context "${INFRA_CONTEXT}" -o jsonpath='{.status.readyReplicas}/{.status.replicas}' 2>/dev/null || true)"
if [[ -z "${_iu_ready}" || "${_iu_ready}" == "/" ]]; then
  echo "Not installed (no argocd-image-updater deployment in ${ARGOCD_NAMESPACE})"
else
  echo "Deployment: ${_iu_ready} ready"

  _iu_logs="$(kubectl logs deploy/argocd-image-updater -n "${ARGOCD_NAMESPACE}" \
    --context "${INFRA_CONTEXT}" --tail=200 2>/dev/null || true)"

  _iu_cycle="$(printf '%s\n' "${_iu_logs}" | grep -F 'Processing results:' | tail -1 \
    | sed 's/.*Processing results: //; s/"$//' || true)"
  if [[ -n "${_iu_cycle}" ]]; then
    echo "Last cycle: ${_iu_cycle}"
  else
    echo "Last cycle: (no completed cycle in recent logs)"
  fi

  _iu_updates="$(printf '%s\n' "${_iu_logs}" | grep -c 'images_updated=[1-9]' || true)"
  if [[ "${_iu_updates}" -ge 3 ]]; then
    echo "WARN Flapping: ${_iu_updates} recent cycles wrote an update — an app may not be converging"
  fi

  echo "Watching:"
  _iu_watch="$(kubectl get applications.argoproj.io -n "${ARGOCD_NAMESPACE}" \
    --context "${INFRA_CONTEXT}" --no-headers \
    -o custom-columns='N:.metadata.name,IL:.metadata.annotations.argocd-image-updater\.argoproj\.io/image-list' 2>/dev/null \
    | awk '$2 != "<none>" && $2 != "" { printf "  %s -> %s\n", $1, $2 }' || true)"
  if [[ -n "${_iu_watch}" ]]; then
    printf '%s\n' "${_iu_watch}"
  else
    echo "  (no applications carry an image-list annotation)"
  fi
fi

if [[ "${CLUSTER_PROVIDER}" == "k3s-aws" ]]; then
```

Notes (do NOT add as comments to the script):
- `|| true` on every command substitution is required — `set -euo pipefail` +
  `pipefail` would otherwise abort on `grep` "no match" (exit 1).
- `grep -c` prints `0` and exits 1 with no match; `|| true` keeps the count as
  `0` so the `-ge 3` test is safe.
- Plain ASCII `WARN` prefix (no emoji) to match the rest of the file.

## Change B — `docs/issues/2026-06-29-argocd-helm-ssa-conflict-sso-cm.md` (new)

Create this file with exactly this content:

```markdown
# ArgoCD helm upgrade SSA field conflict on argocd-cm / argocd-rbac-cm

**Found:** 2026-06-29, during live validation of the ArgoCD metrics
ServiceMonitor change (commit `60cd500c`). Helm v4.2.2.

## Symptom

`helm upgrade argocd argo/argo-cd -n cicd ...` fails:

```
UPGRADE FAILED: conflict occurred while applying object cicd/argocd-cm:
  conflicts with "kubectl-patch": .data.oidc.config, .data.url
conflict occurred while applying object cicd/argocd-rbac-cm:
  conflicts with "kubectl-client-side-apply": .data.policy.csv, .data.scopes
```

The upgrade is non-atomic: chart-templated objects (e.g. the metrics
Services/ServiceMonitors) still apply, but the release is marked `failed`.
`--take-ownership` does NOT clear the field-level conflict.

## Root cause

`bin/cluster-up` configures ArgoCD SSO out-of-band at Step 10f:
- `kubectl apply` of `argocd-cm.yaml` / `argocd-rbac-cm.yaml` from
  shopping-cart-infra → field manager `kubectl-client-side-apply` owns
  `.data.policy.csv` / `.data.scopes`.
- `kubectl patch configmap argocd-cm --type merge` for the Cloudflare public
  `url` + `oidc.config` → field manager `kubectl-patch` owns those fields.

Helm does not own those fields, so any later helm upgrade collides under
server-side apply.

## Why this is NOT a normal-flow bug

- `deploy_argocd` runs the helm install at cluster-up Step **3.6**, before the
  SSO config at Step **10f**, so metrics/chart values apply cleanly on a fresh
  hub.
- The refresh path calls `deploy_argocd_bootstrap`, which does not re-run helm.
- `deploy_argocd` hard-codes `release_exists=0` → plain `helm upgrade --install`.

The conflict only fires on a *manual* `helm upgrade` / `deploy_argocd` against an
already-SSO-patched live release.

## Remediation

- Prefer redeploying from a fresh hub (helm install precedes SSO patches).
- If a release is already `failed`: `helm rollback argocd -n cicd`.
- Do NOT `--force` against a live release — it risks clobbering the live ArgoCD
  OIDC/RBAC config.
- Long-term option (not yet scoped): fold the SSO `argocd-cm`/`argocd-rbac-cm`
  config into the helm values so there is a single field-manager owner.
```
