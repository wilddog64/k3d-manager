## Problem

The ArgoCD Image Updater controller runs on the **hub** (`k3d-k3d-cluster`,
namespace `cicd`). Its registry config
(`scripts/etc/argocd/image-updater/kustomization.yaml:42`) points credentials at:

```
credentials: pullsecret:shopping-cart-apps/ghcr-pull-secret
```

A `pullsecret:<ns>/<name>` reference is resolved **against the cluster the
controller runs on** — i.e. the hub. But on the hub:

- the `shopping-cart-apps` namespace **does not exist** (it is an *app-cluster*
  namespace), and
- `ghcr-pull-secret` exists **only on the app clusters**, created by
  `shopping_cart_create_ghcr_pull_secret` (`shopping_cart.sh:343`) during
  `acg-up` Step 5/12.

Consequences (verified live on `k3d-k3d-cluster`, 2026-06-28):

- `kubectl get ns shopping-cart-apps` → `NotFound`.
- `ghcr-pull-secret` is absent from every namespace on the hub.
- `pull-secret-rbac.yaml` places its Role + RoleBinding in `shopping-cart-apps`,
  so `kubectl apply -k scripts/etc/argocd/image-updater/` **partially fails** on
  the hub (target namespace missing) — currently masked by the non-fatal
  `apply -k` in `_argocd_deploy_image_updater`.
- The shopping-cart app images are **private** on ghcr.io (anonymous manifest
  pulls of `wilddog64/product-catalog|basket|order` all return HTTP 403), so
  credentials are genuinely required — the reference cannot simply be dropped.

**Root cause:** the registry credential reference and its RBAC were written as if
Image Updater runs *beside the apps*. It runs on the *hub*, so it needs a
`ghcr-pull-secret` it can read **in its own cluster**.

## Decision (already made — do not re-litigate)

- **Secret location:** the hub-side `ghcr-pull-secret` lives in `cicd` (the
  controller's own namespace). Repoint the registry config at
  `pullsecret:cicd/ghcr-pull-secret`. The upstream Image Updater install
  (`v0.15.0`) already grants the `argocd-image-updater` ServiceAccount a Role to
  read secrets in its own namespace, so the cross-namespace
  `pull-secret-rbac.yaml` is **deleted**.
- **Provisioning:** a new private `_argocd_ensure_ghcr_pull_secret` in
  `argocd.sh` resolves a GHCR PAT (env → Vault best-effort → `gh auth token`),
  validates it, and creates the docker-registry secret in `cicd`. Non-fatal: if
  no PAT resolves, warn and continue (controller installs; auth works once the
  secret exists).

### Change 3 — `argocd.sh`: add `_argocd_ensure_ghcr_pull_secret`

Insert this new function immediately **before** `_argocd_deploy_image_updater()`
(currently ~line 1074). It mirrors the resolver + create pattern in
`shopping_cart.sh` but is self-contained (argocd bootstrap does not load the
shopping_cart plugin). The docker-registry secret is created with **bare
`kubectl`** (matching `shopping_cart.sh:367`) so the PAT never passes through the
`_kubectl`/`_run_command` trace wrapper — `--docker-password` is not registered
in `_args_have_sensitive_flag` (see SECURITY note below).

**Exact new block:**

```bash
function _argocd_ensure_ghcr_pull_secret() {
   local user="${GITHUB_USERNAME:-wilddog64}"
   local pat="" http netrc

   _ghcr_pat_valid() {
      [[ -z "$1" ]] && return 1
      netrc=$(mktemp) && chmod 0600 "$netrc"
      printf 'machine api.github.com login %s password %s\n' "$user" "$1" > "$netrc"
      http=$(curl -s -o /dev/null -w "%{http_code}" --netrc-file "$netrc" "https://api.github.com/user" 2>/dev/null || true)
      rm -f "$netrc"
      [[ "$http" == "200" ]]
   }

   if _ghcr_pat_valid "${GHCR_PAT:-}"; then
      pat="${GHCR_PAT}"
      _info "[argocd] using validated GHCR_PAT from env for ghcr-pull-secret"
   elif [[ -n "${_vault_local_port:-}" ]]; then
      local vault_token vault_pat
      vault_token=$(_kubectl -n secrets get secret vault-root -o jsonpath='{.data.root_token}' 2>/dev/null | base64 -d 2>/dev/null || true)
      if [[ -n "$vault_token" ]]; then
         vault_pat=$(curl -s -H "X-Vault-Token: ${vault_token}" "http://localhost:${_vault_local_port}/v1/secret/data/github/pat" | jq -r '.data.data.token // empty' 2>/dev/null || true)
         if _ghcr_pat_valid "$vault_pat"; then
            pat="$vault_pat"
            _info "[argocd] using PAT from Vault for ghcr-pull-secret"
         fi
      fi
   fi

   if [[ -z "$pat" ]] && command -v gh >/dev/null 2>&1; then
      local gh_token
      gh_token=$(gh auth token 2>/dev/null || true)
      if [[ -n "$gh_token" ]] && GH_TOKEN="$gh_token" gh api user >/dev/null 2>&1; then
         pat="$gh_token"
         _info "[argocd] using gh CLI token for ghcr-pull-secret"
      fi
   fi

   if [[ -z "$pat" ]]; then
      _warn "[argocd] No valid GHCR PAT (env/Vault/gh) — skipping ghcr-pull-secret; Image Updater auth will fail until it exists. Set GHCR_PAT or run: gh auth login"
      return 0
   fi

   kubectl create secret docker-registry ghcr-pull-secret \
      --docker-server=ghcr.io \
      --docker-username="$user" \
      --docker-password="$pat" \
      -n "$ARGOCD_NAMESPACE" \
      --dry-run=client -o yaml \
      | kubectl apply -n "$ARGOCD_NAMESPACE" -f - >/dev/null 2>&1 \
      && _info "[argocd] ghcr-pull-secret ensured in namespace $ARGOCD_NAMESPACE" \
      || _warn "[argocd] failed to apply ghcr-pull-secret in $ARGOCD_NAMESPACE"
}

```

## SECURITY note (read — do NOT skip)

- The docker-registry secret is created with **bare `kubectl`**, exactly like
  `shopping_cart_create_ghcr_pull_secret`. Do NOT route it through `_kubectl` or
  `_run_command`: `--docker-password` is not registered in
  `_args_have_sensitive_flag` (`scripts/lib/system.sh`, a lib-foundation
  subtree), so a trace-aware wrapper could leak the PAT. Registering
  `--docker-password` belongs upstream in lib-foundation and is **out of scope**
  here — file it as a follow-up, do not edit `scripts/lib/system.sh`.
- Never echo `$pat`, `$GHCR_PAT`, or the gh token. The `mktemp` netrc file is
  mode `0600` and removed immediately after validation — keep it that way.
- Do not add `--insecure`/`-k` to any curl call. `insecure: false` in the
  kustomization stays.
