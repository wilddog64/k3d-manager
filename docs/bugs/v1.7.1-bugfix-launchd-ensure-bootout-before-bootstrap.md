# Bugfix: v1.7.1 ‚Äî `_launchd_ensure` bootstrap fails with error 5 (no bootout first)

**Branch:** `k3d-manager-v1.7.1`
**Files:** `bin/cluster-refresh`

---

## Problem

After `make refresh` regenerates a system daemon's cert + wrapper (e.g. the v1.7.1
argocd-browser-https self-heal), the daemon still will not start. Every system-domain browser
daemon fails to bootstrap:

```
INFO: [acg-refresh] launchd com.k3d-manager.argocd-browser-https: not loaded ‚Äî bootstrapping...
Bootstrap failed: 5: Input/output error
WARN: [acg-refresh] launchd com.k3d-manager.argocd-browser-https: bootstrap failed (plist exists but service could not start ‚Äî check wrapper script and TLS certs)
INFO: [acg-refresh] launchd com.k3d-manager.keycloak-browser-http: not loaded ‚Äî bootstrapping...
Bootstrap failed: 5: Input/output error
INFO: [acg-refresh] launchd com.k3d-manager.frontend-browser-http: not loaded ‚Äî bootstrapping...
Bootstrap failed: 5: Input/output error
```

The `WARN` message blames the wrapper/TLS, but those are valid ‚Äî the cert was just re-issued from
Vault PKI and the wrapper regenerated at the canonical path. The real failure is the bootstrap.

**Root cause:** `_launchd_ensure` (system-domain branch) runs `launchctl bootstrap system "${plist}"`
**without a preceding `launchctl bootout`**. When a daemon was previously bootstrapped and then
crash-looped (e.g. socat exiting on a missing cert until launchd throttled it), launchd retains a
stale service record in the system domain even though `launchctl list <label>` no longer shows it.
Bootstrapping a label already known to the domain returns **error 5 (EIO)**. The sibling functions
already handle this correctly ‚Äî `_launchd_restart` (`bin/cluster-refresh:100-101`) and
`bin/cluster-up:557‚Üí574` both `bootout` then `bootstrap`. `_launchd_ensure` is the only one missing
the bootout.

This is why `make refresh` self-heals the artifacts but the listener never comes back: the plist is
regenerated, the cert is valid, but `_launchd_ensure` cannot replace the stale service record.

---

## Reproduction

```bash
# Drop the listener into a crash-looped/dropped state, then regenerate + try to bring it up:
rm -f ~/.local/share/k3d-manager/argocd-browser-https-tls/*
make refresh
#   -> "Bootstrap failed: 5: Input/output error" for all 3 browser daemons
launchctl list | grep argocd-browser     # -> (nothing)
curl -s -o /dev/null -w "%{http_code}\n" https://argocd.shopping-cart.local/   # -> 000
```

Expected after fix: `make refresh` boots out the stale system-domain record first, then bootstraps
cleanly; the daemon loads and `curl -sk https://argocd.shopping-cart.local/healthz` returns `200`.

---

## Fix

Make `_launchd_ensure`'s system-domain branch `bootout` before `bootstrap`, mirroring the proven
sequence in `_launchd_restart` and `bin/cluster-up`. The bootout is best-effort (`|| true`) so it is
a no-op when no stale record exists.

### Change 1 ‚Äî `bin/cluster-refresh`: bootout stale system-domain record before bootstrap

**Exact old block (lines 81‚Äì84):**

```bash
    if [[ "${domain}" == "system" ]]; then
      _run_command --interactive-sudo --quiet --soft -- launchctl bootstrap system "${plist}" \
        && _info "[acg-refresh] launchd ${label}: bootstrapped" \
        || _warn "[acg-refresh] launchd ${label}: bootstrap failed (plist exists but service could not start ‚Äî check wrapper script and TLS certs)"
```

**Exact new block:**

```bash
    if [[ "${domain}" == "system" ]]; then
      _run_command --interactive-sudo --quiet --soft -- launchctl bootout system "${plist}" || true
      _run_command --interactive-sudo --quiet --soft -- launchctl bootstrap system "${plist}" \
        && _info "[acg-refresh] launchd ${label}: bootstrapped" \
        || _warn "[acg-refresh] launchd ${label}: bootstrap failed (plist exists but service could not start ‚Äî check wrapper script and TLS certs)"
```
