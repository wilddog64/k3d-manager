## Problem

`scripts/plugins/acg.sh` contains functions that are pure AWS cloud operations
(CloudFormation, EC2, STS). These belong in `scripts/plugins/aws.sh` — the
dedicated AWS plugin — not in `acg.sh`, which is the ACG sandbox lifecycle
orchestrator.

Functions currently in `acg.sh` that are AWS-specific:

| Function | Why it belongs in `aws.sh` |
|----------|---------------------------|
| `_acg_check_credentials` | Calls `aws sts get-caller-identity` — pure AWS auth check |
| `_acg_get_instance_id` | Calls `aws ec2 describe-instances` — pure EC2 query |
| `_acg_get_instance_attr` | Calls `aws ec2 describe-instances` — pure EC2 query |
| `_acg_update_ssh_config` | SSH config helper for EC2 IPs — dependency of `_acg_cf_deploy` |
| `_acg_upsert_ssh_host` | SSH config helper for EC2 IPs — dependency of `_acg_cf_deploy` |
| `_acg_check_k3s` | SSH probe for k3s on EC2 — dependency of `acg_status` |
| `_acg_cf_deploy` | Runs CloudFormation deploy + EC2 IP lookup |
| `acg_provision` | Orchestrates CloudFormation provisioning via `_acg_cf_deploy` |
| `acg_status` | Checks EC2 instance state via `aws ec2` |

## Why this works without circular dependencies

`acg.sh` already sources `aws.sh` at line 12:
```bash
source "${SCRIPT_DIR}/plugins/aws.sh"
```

After the move:
- When the dispatcher loads `acg_provision` or `acg_status`, it sources `aws.sh` directly.
  All constants and helpers are defined in `aws.sh` — no `acg.sh` needed.
- When the dispatcher loads any remaining `acg.*` function (e.g. `acg_teardown`), it sources
  `acg.sh`, which sources `aws.sh` → `_acg_check_credentials`, `_ACG_CF_STACK_NAME`, etc.
  are all available to `acg_teardown` via the chain.
- No circular dependency. `aws.sh` never sources `acg.sh`.
