# Retrospective — v0.9.21

**Date:** 2026-03-29
**Milestone:** `_ensure_k3sup` auto-install helper
**PR:** #56 — merged to main (`f98f2a8`)
**Participants:** Claude, Codex, Gemini, Copilot

## What Went Well

- **Codex implementation was clean** — spec was precise enough that Codex produced the right function on the first attempt; only two files touched, commit message exact.
- **Copilot caught real issues** — all three findings were valid: `_run_command` hard-exit swallowing the fallback chain, brittle `curl -sLS` command substitution, and missing BATS coverage for install success paths. All fixed before merge.
- **Ubuntu Parallels smoke test added value** — confirmed the curl installer path actually fires on Linux; also surfaced the sudo password prompt issue which is now documented as a v1.2.0 k3dm-mcp gate.
- **Roadmap versioning resolved** — v0.9.21 / v1.0.0 (AWS) / v1.0.1 (GCP) / v1.0.2 (Azure) decision recorded in roadmap and memory-bank; no ambiguity going into v1.0.0 planning.

## What Went Wrong

- **BATS coverage gap in original spec** — spec only required tests for the "present" and "no-installer" paths. Copilot had to catch the missing install-success path tests. Should have included all three paths in the spec template.
- **`_run_command` hard-exit not caught in spec review** — the spec used `_run_command --` (not `--soft`) for both install calls. This would have silently broken the brew → curl fallback chain in production. Copilot caught it; spec author should have caught it.
- **Gemini cold-path test was incomplete** — macOS cold-path filtered out the entire `/opt/homebrew/bin`, hiding both k3sup and brew. Verified the no-installer error path but not the brew install path. Required a second Gemini run (ubuntu-parallels) to verify the curl path.
- **Uncommitted `progress.md` after Gemini update** — Gemini updated memory-bank files but didn't push; discovered via `git status` after the fact. Process gap: always check `git status` immediately after agent reports done.

## Process Rules Added

| Rule | Context | Detail |
|------|---------|--------|
| BATS spec must cover all install paths | Spec template | For any `_ensure_*` helper with multiple install branches, spec must require tests for: already-present, each install success path, no-installer error |
| Use `--soft` for all auto-install `_run_command` calls | Code pattern | `_run_command --` hard-exits on failure; install helpers must use `--soft` so fallback chains are preserved |
| Check `git status` after every agent completion | Process | Before declaring agent done, verify no unstaged changes remain in the working tree |

## Decisions Made

- **Roadmap versioning:** v0.9.21 = `_ensure_k3sup` (this release); v1.0.0 = AWS/ACG 3-node full automation; v1.0.1 = GCP; v1.0.2 = Azure. `v1.0.0` marks the threshold where k3d-manager provisions and fully configures a remote multi-node cluster end-to-end without manual steps.
- **sudo pre-flight for k3dm-mcp (v1.2.0):** `deploy_cluster`/`deploy_app_cluster` must include `ssh <host> sudo -n true` pre-flight check before v1.2.0 ships — agents run non-interactively and a sudo prompt hangs them with no TTY.
- **Ubuntu passwordless sudo is a target prerequisite:** ACG EC2 ubuntu user is passwordless by default (no change needed); home lab and self-managed VMs must be pre-configured.

## Theme

v0.9.21 was a one-function cleanup sprint that closed a consistency gap (`deploy_app_cluster` was the only tool dependency check that didn't auto-install). Small scope, but Copilot found three real bugs in the implementation — including one (`_run_command` hard-exit) that would have silently broken the brew → curl fallback in production. The milestone also produced a useful side-effect: Ubuntu Parallels smoke testing surfaced the sudo prompt issue, which is now documented as a v1.2.0 gate rather than discovered at k3dm-mcp integration time.
