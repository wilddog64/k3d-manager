# Retrospective — v0.9.13

**Date:** 2026-03-23
**Milestone:** v0.9.12 retro + CHANGE.md backfill + process: mergeable_state check
**PR:** #48 — merged to main (`c54fbe6`)
**Participants:** Claude, Copilot

## What Went Well

- **Copilot findings were precise** — all 3 findings were genuine accuracy issues (stale file ref, wrong filename `CHANGELOG` vs `CHANGE.md`, stale branch header); none were noise
- **Small-scope PR moved fast** — docs-only change: CI ran in ~35s, lint + detect only, no stage2; total cycle from commit to merged ~10 minutes
- **`mergeable_state` check worked first-time** — the new Step 0 in `/create-pr` confirmed `"blocked"` (clean) immediately, no dirty-state surprise this release
- **Three-file fix commit** — all Copilot findings landed in one commit (`d1972ca`), replies + GraphQL resolution in one pass

## What Went Wrong

- **`activeContext.md` branch header not updated on branch cut** — the header still said `k3d-manager-v0.9.12` when the branch was cut; Copilot caught it. Should be updated as part of the post-merge Step 7 (memory-bank update) when creating the next branch.
- **`memory/project_copilot_token.md` ref in retro** — referenced a Claude user-memory path that doesn't exist in the repo. Retro docs must only reference files that exist in the repo.
- **`CHANGELOG` vs `CHANGE.md`** — two occurrences slipped through; the file has always been `CHANGE.md`. Should add a grep check in the pre-PR checklist.

## Process Rules Added

| Rule | File | Description |
|---|---|---|
| Update branch header on branch cut | `memory-bank/activeContext.md` | When starting a new branch, update `## Current Branch:` to the new branch name and today's date as the first commit |
| Retro refs must be in-repo | Retro doc template | Never reference `~/.claude/` paths or user-memory paths; only reference files that exist in the k3d-manager repo |

## Decisions Made

- **v0.9.13 tagged** — docs/process milestones get a version tag when they close a named sprint (retro + CHANGE.md backfill qualifies); untagged releases (v0.9.8, v0.9.12) are mid-sprint intermediate merges
- **`CHANGE.md` (not `CHANGELOG.md`)** — canonical filename confirmed; added to process rules to avoid recurrence

## Theme

v0.9.13 was pure housekeeping: close out v0.9.12 properly with a retro, backfill the changelog, and ship the `mergeable_state` process improvement that came out of the v0.9.12 debugging session. Copilot caught three real accuracy issues (a stale file reference, the wrong changelog filename, and a stale branch header), all of which were fixed in a single commit. The release cycle from PR open to tag was under 15 minutes — the shortest to date.
