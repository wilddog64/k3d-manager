**Date:** 2026-06-28
**Milestone:** Tier 3 hub-Vault canonical seeding (P3) + assisted-failover watchdog (P4)

**Participants:** Claude, Codex, Gemini, Copilot

## What Went Well

- **Tier 3 shipped complete.** P3 (`vault_seed_hub_into_context` — idempotent canonical-source
  seeding of the 13 canonical keys laptop→in-cluster Vault + keyring + native k8s backup) and P4
  (`vault_failover_hub_into_context` + `vault_install_failover_watchdog` + the
  `com.k3d-manager.vault-failover` LaunchAgent + `bin/k3dm-vault-failover` wrapper) both landed,
  closing the Vault-endpoint open seam from v1.10.0.
- **Codex bugfix work verified byte-for-byte.** The watchdog-hardening commit `3b32375a` was
  confirmed on origin, scope held to the two spec files, both diff blocks matched the spec exactly,
  and every static gate (`shellcheck -S warning`, `bash -n`, the two BATS suites, `_agent_audit`)
  was re-run by Claude rather than trusted.
- **Live verification caught real, not theoretical, breakage.** The deferred live checks on the
  `ubuntu-hostinger` cluster proved both watchdog fixes end-to-end: the launchd-PATH fix turned a
  non-functional LaunchAgent into one that logs `profile=hostinger` healthy, and the subshell
  re-seed fix turned an aborting failover into a warn-and-continue that reaches the CSS reconcile —
  all without touching real persisted state.
- **The live cutover to the in-cluster hub Vault succeeded** (attempt #3), retiring the fragile
  laptop reverse-tunnel + socat bridge for the app cluster: CSS on `http://vault.secrets.svc:8200`
  via kubernetes-auth, 17/17 ExternalSecrets synced.
- **spec-before-implement held throughout** — every bugfix went through a `docs/plans/` spec with
  exact old/new blocks before Codex touched code; the deferred Copilot findings were captured in a
  tracked bug spec rather than rushed in.

## What Went Wrong

- **Vault tokens leaked into `curl` argv** (Copilot findings 4/5) — a direct security-rule
  violation that slipped past the initial P3 implementation. Both root tokens were passed via
  `-H "X-Vault-Token: …"`, visible to any `ps` during the seeding loop. Fixed in `b2ed5b80` by
  routing tokens through `curl --config` mode-600 header files (verified the header still
  transmits). The class of bug recurs elsewhere (`_vault_kv_*` in `shopping_cart.sh`).
- **The live cutover took three attempts.** Attempt #1 collided with ArgoCD-managed ESO (no Helm
  release → wrong fresh-install branch); attempt #2 wrote app-cluster auth to the laptop hub Vault
  because the writes ran against the current kube-context. Each surfaced a real bug, each fixed, but
  the path needed live exercise to find them — pure-logic BATS could not.
- **The failover watchdog was non-functional as first installed** — launchd's minimal PATH excluded
  Homebrew, so `kubectl`/`helm`/`jq` were unreachable; and the re-seed `|| _warn` guard was dead
  code because `_err` calls `exit 1`, killing the shell before the `||`. Both only manifested under
  real launchd / real failover, not in tests.
- **The v1.10.0 releases-table rows were missing** — a gap left by the prior release's post-merge;
  backfilled this cycle in README + `docs/releases.md`.
- **The `enforce_admins` DELETE was blocked by the auto-mode classifier**, twice — once for the
  merge unblock, once for the post-merge subagent's token-in-remote-URL fallback. Both were
  surfaced to the user and run directly in-conversation with SSH instead, consistent with the
  never-auto-merge rule.

## Decisions Made

- **Deferred Copilot findings 1–3** (the canonical-source reader `_seed_source_data` is defined in
  `shopping_cart.sh` but never wired into the generate-if-absent branches for `redis/cart`,
  `redis/orders-cache`, `rabbitmq/default`) to `docs/bugs/v1.11.0-bugfix-seed-source-data-unused.md`
  rather than fixing inline during the release. It is a latent gap, **not a regression**: the active
  canonical-copy path for the cutover is the other operator, `vault.sh:vault_seed_hub_into_context`,
  which already reads from the source Vault, and in the default config `SEED_VAULT_SOURCE_*`
  resolves to the same Vault.
- **Left the failover watchdog LaunchAgent loaded** permanently (user decision) now that it is
  functional and the profile is `hostinger`.
- **The in-cluster hostinger hub Vault is now the live hub** for the `ubuntu-hostinger` app cluster;
  the laptop reverse-tunnel bridge is retired for it (fallback profile only).

## Theme

v1.11.0 finished the hub-Vault resilience story that v1.10.0 began. P3 mirrors the canonical
secrets into an in-cluster Vault so the cluster no longer depends on a laptop being reachable, and
P4 adds an assisted-failover watchdog that probes the hub, flips the `HUB_VAULT_PROFILE` seam on
sustained failure, re-seeds, and reconciles the app-cluster ClusterSecretStore. The milestone's real
character was in the bugfixes: six of them, found almost entirely by exercising the code against the
live `ubuntu-hostinger` cluster and a real launchd context — the launchd-PATH gap and the
`_err`-subshell-exit that made the watchdog survive being launched by a LaunchAgent, plus the
context-targeting and ArgoCD-ownership fixes that made the live cutover actually land. The lesson
that keeps repeating: the seam between "passes pure-logic BATS" and "works under real launchd /
real failover / real ArgoCD" is exactly where these bugs live, and only live verification finds them.
