# Retrospective — v0.1.6

**Date:** 2026-06-12
**Milestone:** Azure sandbox SP-empty delete+restart + 3-attempt reopen
**PR:** #43 — merged to main (`696ca83f`)
**Participants:** Claude, Codex, Copilot

## What Went Well

- Copilot caught two real defects in the first review pass: premature timer start and counter placement before the guard
- Fix commit `f0bc7e8` addressed both findings cleanly in a single commit
- CI stayed green on the original commit and the fix commit (46s each)
- Both Copilot threads resolved via GraphQL — no manual GitHub interaction needed
- `vals.some()` gate is the right minimal fix — adds precision without restructuring the loop

## What Went Wrong

- `partialCredsFirstSeen` timer started unconditionally on any incomplete input state, including when 0 fields were populated — timer should only start when `vals.some()` is true
- `deleteCycleCount++` placed before the `if (deleteBtn)` guard — a transient button-not-found failure consumed a delete cycle without doing anything
- CHANGELOG described the trigger as "SP credentials remain empty" — symptom-driven wording rather than code-accurate wording ("credential fields partially populated")

## Process Rules Added

No new rules added. Existing principle reinforced: counter increments that track "attempts made" must be inside the block that actually performs the attempt, not at the top of the decision block before guard checks.

## Decisions Made

- Timer gate: `vals.some(v => v.trim().length > 0)` ensures the 60s clock starts only when the panel has loaded enough to show some credentials. This means a sandbox that never populates any fields won't trigger a delete — it'll just time out normally.
- Counter placement inside `if (deleteBtn)`: a failed button lookup does not count as a delete attempt. The timer resets via `partialCredsFirstSeen = 0` either way, so a fresh 60s window begins on the next cycle.

## Theme

v0.1.6 sharpened the Azure sandbox auto-recovery logic introduced in v0.1.5. The delete+restart feature needed two precision fixes caught by Copilot: the timer now ignores the initial loading state (0 fields populated), and the attempt counter only increments when a delete actually happens. Both were edge-case correctness issues — the happy path would have worked fine — but the Copilot review surfaced them before they could cause confusing failures in the field.
