# Retrospective — v1.5.0

**Date:** 2026-05-31
**Milestone:** OCI provider, observability stack, Alertmanager, template refactor
**PR:** #84 — merged to main (SHA `faa00331214626982fd99d65b3764bca9f4c63ce`)
**Release window:** 2026-05-22 to 2026-05-31

---

## Executive Summary

v1.5.0 shipped the k3s-oci Always Free provider as the stable foundation for cloud-native k3d-manager deployments. The release includes OCI single-node and two-node cluster support with Cilium CNI, comprehensive observability stack (Prometheus + Grafana + Trivy + Alertmanager), and architectural pattern refinements (template files, multi-env ArgoCD, state-dir reorganization, passwordless sudo). Nine completed specs, 12 bug fixes, and BATS coverage for the new OCI provider validate the release.

---

## What Went Well

1. **OCI provider architecture is solid** — single-node and two-node deployments both provision and validate cleanly; Cilium networking is stable; smoke tests pass on first run after provider landing. The two-node agent wait loop fix (`grep -v master` → 2-node Ready count) was caught during BATS coverage work before production use.

2. **Observability stack unblocked full-stack monitoring** — v1.5.4 (Prometheus + Grafana + Trivy) and v1.5.5 (Alertmanager + PrometheusRules) landed back-to-back with working Hub + ACG ApplicationSets and Vault-backed SMTP configuration. Template refactor (v1.5.6) proved moving config to `scripts/etc/prometheus/` eliminated duplication.

3. **Spec sequencing discipline prevented rework** — the v1.5.0 implementation order (OCI first, platform-helm multi-env, bug fixes, observability) ensured each layer had dependencies resolved before landing. No conflicts or cross-blocking issues.

4. **Passwordless sudo on macOS reduced friction** — `bin/install-sudoers.sh` + scoped `/etc/hosts` helper + three prewarm sites consolidated a messy sudo story into a clean NOPASSWD policy model. BATS coverage in `test_install_sudoers.bats` validated the implementation.

5. **Copilot review findings were actionable** — PR #84 Copilot review (`docs/issues/2026-05-31-copilot-pr84-review-findings.md`) flagged shell injection patterns, incomplete app lists in `make status`, and doc accuracy issues; all were addressed before merge.

---

## What Went Wrong

1. **PyYAML validation gap** — three specs (`platform-helm`, `services-git`, `shopping-cart-platform`) required PyYAML YAML validation that was unavailable in the environment. Fallback to Ruby YAML parsing and manual inspection masked potential issues. Issue docs created: `docs/issues/2026-05-30-platform-helm-services-git-pyyaml-missing.md`, `docs/issues/2026-05-30-shopping-cart-platform-project-pyyaml-missing.md`.

2. **BATS test path mismatch** — `test_install_sudoers.bats` repo-root path was wrong on first run (`../../..` instead of `../..`); caught immediately and fixed, but showed path assumptions weren't validated early.

3. **ArgoCD browser login URL test drift** — `scripts/tests/bin/acg_up.bats` has a stale assertion (`Step 10e/14 — Installing Istio ingress HTTP listener` vs actual output); noted in `docs/issues/2026-05-30-acg-up-bats-argocd-browser-login-url-test-drift.md`. Low priority but blocking full BATS validation.

4. **Frontend e2e integration stalled** — shopping-cart-frontend and shopping-cart-e2e-tests PRs (e.g., frontend PR #18) fixed E2E mock response format and OIDC parameter mismatches but left unresolved test failures in related suites (`eso.bats`, `gcp.bats`, `keycloak.bats`). Issue doc: `docs/issues/2026-05-22-frontend-e2e-copilot-pr18-oidc-param-fixes.md`.

5. **Shopping-cart frontend no-products issue unresolved** — live cluster showed frontend returning empty product list post-deploy. Issue doc created (`docs/issues/2026-05-30-shopping-cart-frontend-no-products.md`) but root cause (backend init data, API contract, or ESO sync timing) was not fully validated before release.

## Process Rules Added or Reinforced

1. **Multi-agent workflow discipline** — Claude (specs), Codex (implementation), Gemini (verification), Copilot (review). Clear handoff points and tool scoping (read-only for verify tasks) prevented tool misuse and kept output focused.

2. **Template file pattern for configs** — v1.5.6 established that Kubernetes/config templates belong in `scripts/etc/<subsystem>/` with envsubst rendering in plugin code. Replaces the old heredoc-in-plugin pattern.

3. **OCI provider as gate for v1.6.0+** — architectural decision: no webhook, CVE scan, or self-healing work until OCI provider is stable and live. v1.5.0 merge unblocks the pipeline.

4. **Spec exemptions for issue docs** — `docs/bugs/` and `docs/issues/` specs do NOT count against the max-5-plan-docs release limit. v1.5.0 shipped 12 bug/issue docs to cleanup debt without violating scope constraints.

---

## Decisions Made

1. **Alertmanager on Hub cluster, not OCI** — observability stack (Prometheus + Grafana + Alertmanager + Trivy) runs on Hub and ACG via ApplicationSets, not on OCI. Monitoring survivability is a v1.7.0 decision (after OCI provider proven). This simplifies initial deployment.

2. **No Istio ambient mesh in v1.5.0** — deferred to v1.8.0 or later. Current sidecar model is stable; ambient mesh adds complexity and requires OCI cluster to validate ARM64 ztunnel behavior.

3. **State-dir layout cleanup as refactor, not feature** — `bin/`, `run/`, `logs/` subdirectories are a housekeeping move, not a new capability. Bundled with OCI provider as a cleanup task while the feature branch was active.

4. **Copilot auto-review on every PR** — enforced in PR #84 workflow. Copilot bot must post review before merge. This became a gate in downstream specs (v1.6.0+).

## Theme

**Foundation Complete — Ready for Extension**

v1.5.0 establishes k3d-manager as a multi-cloud, multi-provider Kubernetes bootstrap tool with production-grade observability. The OCI Always Free provider removes cost as a barrier; passwordless sudo removes manual friction on macOS; observability stack removes monitoring as a blocker. Next: v1.6.0 (webhook-driven self-healing) and v1.7.0 (monitoring on OCI) extend the platform without revisiting core patterns.

## Follow-Up Blockers

1. **PyYAML validation environment** — next release must either install PyYAML in Claude session or validate YAML via alternate means (e.g., `kubectl apply --dry-run=server`).

2. **BATS test suite cleanup** — merge `scripts/tests/bin/acg_up.bats` assertion drift fix and root-cause the `eso.bats`, `gcp.bats`, `keycloak.bats` failures before v1.6.0.

3. **Shopping-cart e2e-tests integration** — unify frontend + e2e test suite versioning; eliminate duplicate test infrastructure across repos. Prerequisite for v1.9.0 (blue-green + stress testing).
