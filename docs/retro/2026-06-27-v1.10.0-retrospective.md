**Date:** 2026-06-27

## What Went Well

- **Provider-agnostic app-cluster auth:** refactored Vault Kubernetes auth into `configure_vault_app_auth_for_context` — portable across EKS/AKS/ACG/Azure/OCI/Hostinger instead of hardcoded to Hostinger. Removed infrastructure-as-liability coupling.
- **In-cluster unseal watchdog:** CronJob replica of the shamir-shard autounseal pattern, namespace-scoped and self-contained; never auto-invoked (P3-gated via `HUB_VAULT_INCLUSTER=1`). Proved the hub Vault can move without blocking the app cluster.
- **Hub Vault relocation envelope:** three design knobs (`HUB_VAULT_PROFILE`, `HUB_VAULT_INCLUSTER`, `HUB_VAULT_CSS_AUTH`) allow blue-green cutover (laptop → hostinger) without breaking existing deployments. Default `laptop` preserves byte-for-byte behavior.
- **Copilot findings triaged and fixed pre-merge:** three real defects caught in PR review (unchecked `kubectl apply` rc, kubeconfig context-vs-cluster-name trap, macOS `base64 -d` portability). Fixed in single commit `1fd01e24`, validated with BATS + `_agent_audit` before merge. Zero defects merged.
- **Data-layer ArgoCD drift root-cause + fix:** benign OutOfSync (7 StatefulSets) traced to controller-injecting `.status`/`.apiVersion`/`.kind` into `volumeClaimTemplates[]` — the ApplicationSet `ignoreDifferences` was incomplete. Fixed by adding 3 new `jqPathExpressions` + BATS assertion test.
- **Process rules documented:** kubeconfig context-vs-cluster-name trap and `base64 -d`/`--decode`/`-D` portability recorded in review findings for future multi-provider work.

## What Went Wrong

- **Version numbering skip (1.8.0 → 1.10.0):** intentional blue-green hold on v1.9.0 to defer hub-Vault relocation P3/P4 (seeding + failover) to the next milestone. Documented in spec but a footnote on the retro for clarity.
- **Data-layer ApplicationSet `ignoreDifferences` incomplete:** pre-existing multi-cloud tech debt. The immutable field gap meant permanent OutOfSync + self-healing powerless. Addressed in v1.10.0 bugfix; no customer impact (all pods healthy, AppSet status correct).
- **YAML validation via PyYAML unavailable:** spec outlined `python3 -c "import yaml; yaml.safe_load(...)"` validation step, but PyYAML is not a standard library on the CI runner. Escalated to `docs/issues/` for future CI env clarification; spec's real work (argocd ApplicationSet fix) is complete and verified.

## Process Rules Added

| Rule | Source | Applicability |
|------|--------|----------------|
| Kubeconfig context ≠ cluster name | Copilot Finding 2 | All multi-provider kubeconfig reads; resolve `.contexts[].context.cluster` first, never key `.clusters[]` on context name |
| macOS `base64 -d` fallback | Copilot Finding 3 | All new base64 decodes in shell; use `base64 --decode \|\| base64 -D` dual-flag idiom per `scripts/lib/identity_tools.sh:36` |
| Vault policy least-privilege pinning | v1.10.0 app-cluster-reader | New Vault policies: exact-match `secret/{data,metadata}/*` prefixes, no wildcard (7 prefixes for read-only app-cluster access) |

## Decisions Made

- **`HUB_VAULT_PROFILE` default stays `laptop`:** production cutover (hostinger) is P3-gated. No auto-wiring of `vault_deploy_hub_into_context`; users opt-in via env override. Preserves stability for existing deployments.
- **App-cluster Vault auth provider-agnostic, not provider-selected:** the auth is portable; Vault endpoint is the seam (resolves to `localhost:18200` on laptop, Hostinger IP on hostinger). Future provider additions inherit the portable pattern automatically.
- **Version gap (v1.9.0 deferred):** P3 CSS seeding + P4 failover moved to v1.11.0 to hold v1.10.0 scope at Tier 3 P1/P2a/P2b. Respects the max-5-plan-docs release boundary and sequence risk (hub Vault move is a critical operation).
- **BATS coverage baseline:** all three Copilot findings were caught by code review, not test gaps; BATS suites for watchdog, app-auth, and hub-profile are complete (11/11, 5/5, 5/5 pass).

## Theme

**Portable and Safe by Design.** v1.10.0 decouples app-cluster Kubernetes auth from infrastructure provider and operator experience constraints. The hub Vault relocation machinery (`configure_vault_app_auth_for_context`, unseal watchdog, `HUB_VAULT_PROFILE` seams) is battle-hardened against multi-cloud pitfalls (kubeconfig topology, CA handling, base64 portability). The P3/P4 cutover is feature-complete but operator-gated, so existing deployments stay stable. Copilot's pre-merge findings (unchecked rc, context-name trap, macOS base64) prove the review-and-fix loop works — three real defects stopped at CI, zero merged. Data-layer drift was a benign tech-debt surface (permanent OutOfSync) that's now cleared. Next milestone (v1.11.0) unlocks the cutover with P3 CSS seeding and P4 failover validation.
