# Retrospective — v0.9.5

**Date:** 2026-03-21
**Milestone:** deploy_app_cluster — k3sup install + kubeconfig merge
**PR:** #38 — merged to main (`573c0ac`)
**Participants:** Claude, Codex, Gemini, Copilot

## What Went Well

- **Codex delivered the core implementation correctly** — `deploy_app_cluster`, BATS tests, and kubeconfig merge logic were all correct on the first pass. Code quality was high enough that only minor Copilot findings needed fixing.
- **11 Copilot threads resolved cleanly** — most were real issues (hardcoded basename exception, mkdir placement, kubeconfig merge order, missing `uname` stub in BATS). Fast feedback loop with no re-review cycles needed.
- **`_run_command` kubeconfig dir fix was non-trivial and correct** — switching from `dirname` (external command, fails under `PATH=/dev/null` in BATS) to `${local_kubeconfig%/*}` (bash parameter expansion) was the right call and verified immediately.
- **deploy_app_cluster milestone gate achieved** — automation replaces manual Gemini rebuild. This was the v0.9.5 primary deliverable.
- **CI debugging produced durable fixes** — kubectl context pinning (`KUBE_CONTEXT=k3d-k3d-cluster`), API server readiness retry loop, `kubectl wait --for=condition=Available` replacing fragile `rollout status` — all real improvements to check_cluster_health.sh.

## What Went Wrong

- **Codex committed locally but did not push** — SHA `13c79b3` existed on local branch only; CI could not see it. Required manual push. Root cause: Codex's push step is not reliable without explicit enforcement.
- **CI flakiness traced to M2 Air resource exhaustion** — Parallels Ubuntu VM was consuming ~9GB RAM. k3d cluster + Istio + Vault + ESO + ArgoCD + CI runner simultaneously on 24GB M2 Air hit the ceiling. Vault readiness probe (3s timeout, failureThreshold=2) timed out under load avg 21+. PR ultimately merged with `--admin` to bypass infrastructure-only CI failures.
- **Multiple CI fix rounds** — four CI runs were needed to identify: wrong kubectl context → rollout status hang → readyReplicas empty when 0 → M2 resource exhaustion (not fixable in code). The progression was correct but costly.
- **`continue-on-error` push blocked by gh token scope** — attempted to use `continue-on-error: true` in the CI workflow but token lacked `workflow` scope. Reverted the change; merged with `--admin` instead. Scope limitation discovered late.
- **enforce_admins left disabled after merge** — was not re-enabled immediately after `--admin` merge. Required explicit cleanup in post-merge. Should be part of the merge action.

## Decisions Made

- **M2 Air is at resource ceiling for CI** — running k3d infra cluster + Parallels Ubuntu + GitHub Actions runner simultaneously is not sustainable. Decision: shift development focus to ACG AWS sandbox rather than continuing to test against M2 local cluster.
- **`--admin` merge is acceptable for infrastructure-only CI failures** — when code is verified correct and CI failures are traceable to runner resource exhaustion, not code defects, `--admin` merge with explicit documentation is appropriate.
- **v1.0.0 is the ACG multi-node milestone** — 3-node k3sup cluster replaces single t3.medium; resolves frontend/payment/order resource exhaustion structurally.
- **lib-foundation TTY fix upstream** — `_run_command_has_tty()` + `_run_command_resolve_sudo()` need to go to lib-foundation `feat/v0.3.4` via Codex spec; too diverged for a simple cherry-pick.

## Theme

v0.9.5 was a focused, single-deliverable sprint: automate the EC2 cluster rebuild that Gemini was doing manually. Codex implemented it correctly on the first pass. The real work was in CI hardening — the cluster health check accumulated four rounds of fixes that uncovered a deeper truth: the M2 Air is no longer viable as both the dev environment and the CI runner. The milestone closes with `deploy_app_cluster` merged, CI infrastructure improved, and a clear directive: build on ACG AWS sandbox, not the local machine.
