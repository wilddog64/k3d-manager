# Retrospective — v1.1.0

**Date:** 2026-04-25
**Milestone:** Unified ACG automation AWS + GCP
**PR:** #65 — merged to main (`e013d23b`)
**Participants:** Claude, Codex, Gemini, Copilot

## What Went Well

- **E2E cycle closed on AWS.** `make down → make up → make sync-apps` ran end-to-end on a live ACG sandbox. Every failure produced a spec, every spec got a Codex fix.
- **Codex discipline held.** Commit messages matched specs exactly on every fix. Push-before-done was enforced. No fabricated SHAs.
- **Copilot found real bugs.** 9 findings on PR #65; 8 were real bugs (the 9th was the CHANGE.md path, a pure docs error). The `--confirm` to `deploy_vault/deploy_ldap` and the PID leak were both latent issues that would have hit in production.
- **Debug loop was tight.** The pattern — user pastes error → Claude diagnoses and specs → Codex fixes → Claude verifies SHA → resume testing — produced ~20 fixes without any regressions slipping through.
- **Port-forward reuse in `bin/acg-sync-apps`.** The full state-file lifecycle (create, reuse, replace-foreign, cleanup on exit) is a solid pattern and worth reusing elsewhere.

## What Went Wrong

- **`--no-sandbox` shipped without a gate.** Merged unconditionally on Linux. Copilot caught it on review — should have been gated from the start.
- **`--confirm` copied to wrong call sites.** `argocd.sh` called `deploy_vault --confirm` and `deploy_ldap --confirm` — neither option parser accepts `--confirm`. Bug only dormant because the namespaces existed in all live tests. Spec review loop should have caught this before Codex committed.
- **CHANGE.md paths stale at PR creation.** Listed `scripts/etc/antigravity/` when the actual files were under `scripts/playwright/`. Pre-PR checklist did not include path validation against the actual file tree.
- **GCP provisioning incomplete.** GCP cluster provisioning is still PARTIAL. Full E2E (k3s-gcp `make up` end-to-end) was not achieved in v1.1.0. CDP startup on Linux blocked the smoke test.
- **`acg-sync-apps` accumulated 12 spec/fix cycles.** The bin script started with barely any hardening and grew into a production-quality port-forward manager across a single session. A pre-built, port-forward-aware `bin/` pattern would have caught most of these gaps before coding started.

## Process Rules Added

| Rule | Location | Why |
|---|---|---|
| Spec must include exact old/new code blocks | handoff skill | Codex without exact blocks drifts toward interpretation |
| Push before done is non-negotiable | handoff / specs | SHA on origin, not just locally |
| Check `deploy_*` option parsers before passing new flags | code review mental model | `--confirm` propagated to functions that don't accept it |
| Log messages must not interpolate `process.env` values | JS spec template | CodeQL finding: clear-text env logging |
| `--no-sandbox` must be gated on `$EUID -eq 0` or explicit env var | antigravity.sh pattern | Chrome sandbox is security-critical; must not disable globally |

## Decisions Made

- **ACG extraction deferred to v1.2.0.** The extraction plan exists (`docs/plans/v1.1.0-acg-extraction-repo-split.md`) but bundling a repo split into the same tag as a live E2E milestone is too risky. v1.1.0 ships the AWS E2E story; v1.2.0 will split browser automation into its own repo.
- **k3dm-mcp gates on two working cloud backends.** v1.2.0 gate: AWS E2E + GCP E2E both fully passing. Until GCP E2E completes, k3dm-mcp stays queued.
- **No `--confirm` in plugin-to-plugin calls.** The `--confirm` flag is a dispatcher-level concern consumed by `scripts/k3d-manager`. Plugin functions (`deploy_vault`, `deploy_ldap`, etc.) should not be called with `--confirm` from within other plugins. Internal calls go directly without the flag.

## Theme

v1.1.0 was the reliability sprint that `bin/acg-sync-apps` needed to graduate from a stub into a production tool. The AWS ACG E2E cycle now runs end-to-end: credentials extracted, Hub provisioned, ArgoCD bootstrapped, app cluster registered, sync confirmed. GCP automation got its identity bridge and OAuth flow; cluster provisioning is partially proven but not yet smoke-tested end-to-end. The net result: one cloud backend fully verified, one partially built — a solid foundation for v1.2.0 to complete GCP E2E and begin the ACG extraction.
