# Retrospective — v1.0.6

**Date:** 2026-04-11
**Milestone:** AWS SSM support for k3s-aws provider
**PR:** #64 — merged to main (`279db18c`)
**Participants:** Claude, Copilot

## What Went Well

- SSM plugin is cleanly isolated — `scripts/plugins/ssm.sh` required no lib changes; the dispatcher pattern absorbed it without friction
- `make up` end-to-end verification succeeded on the first run after the `CAPABILITY_NAMED_IAM` fix — cluster up, nodes Ready, pods Running
- Copilot caught 6 real issues that would have caused silent failures in the field (None/null sentinel, missing `mkdir -p`, bare sudo, indentation bug, inaccurate docs)
- `K3S_AWS_SSM_ENABLED` opt-in design preserved backwards compatibility — existing SSH path unchanged

## What Went Wrong

- **`--capabilities CAPABILITY_NAMED_IAM` missing at first** — CloudFormation deploy failed with `InsufficientCapabilitiesException` on first `make up` run. The fix was a one-line addition but it blocked end-to-end verification until discovered.
- **`_ssm_get_instance_id` returned AWS "None" as a valid ID** — AWS CLI `--output text` returns the string `"None"` (not empty) for null JMESPath results. Not caught in spec review; Copilot found it.
- **Lost indentation in `deploy_app_cluster`** — post-join block was pasted at column 0 instead of inside the function. Visual regression, caught by Copilot.
- **CHANGE.md entry described `make ssm` as "open SSM shell"** — spec intent was broader; final implementation was scoped to plugin install only; description not updated.

## Process Rules Added

| Rule | Source |
|------|--------|
| Any function writing to `~/.kube/` must `mkdir -p "${HOME}/.kube"` first | Copilot finding #3 |
| All AWS CLI `--output text` calls returning an ID must guard against `"None"` and `"null"` sentinels | Copilot finding #4 |
| Makefile targets that install software must not call `sudo` directly — macOS via `brew` only; all other platforms print the manual install URL | Copilot finding #1 |
| CHANGE.md entries must be written (or re-verified) after final implementation, not carried over from spec intent | Copilot finding #6 |

## Decisions Made

- **Option B for Makefile multi-cloud** — generic `make up`/`make down`/`make refresh` targets will dispatch via `CLUSTER_PROVIDER` env var rather than creating cloud-prefixed targets (`make acg-up`, `make gcp-up`). GCP provider (v1.0.7) is the first to exercise this pattern. Provider-specific targets (`make provision`, `make ssm`) stay as-is since they are genuinely AWS-only.
- **SSM as opt-in path only** — `K3S_AWS_SSM_ENABLED=true` activates the SSM bootstrap; default is SSH. Vault reverse bridge remains SSH-only for v1.0.6.

## Theme

v1.0.6 laid the SSM foundation for zero-SSH EC2 access. The CAPABILITY_NAMED_IAM failure was the kind of one-line fix with outsized impact — it blocked the entire `make up` path until found. The milestone surfaced the multi-cloud Makefile target question that will drive v1.0.7 design: rather than proliferating cloud-prefixed targets, the generic targets will become `CLUSTER_PROVIDER`-aware dispatchers, mirroring how the plugin system already works.
