# Retrospective — v1.4.11

**Date:** 2026-05-29
**Milestone:** v1.4.11 Data Layer GitOps + RBAC Stabilization
**PR:** #82 — merged to main (`57cd3bc3`)
**Participants:** Claude, Codex, Copilot

## What Went Well
- Annotate-all-before-wait ESO pattern eliminates controller saturation — clean design that composes with future ESO additions
- `--prefer-sudo` over `--interactive-sudo` is the right default for all non-interactive paths; macOS Tahoe PTY allocation failure surfaced the latent bug
- ArgoCD duplicate-app root cause was cleanly identified (two competing app-of-apps layers) and fixed at the source (delete legacy yamls from shopping-cart-infra)
- Codex followed specs exactly on all three tasks (ESO fix, remove legacy apps, acg-down fix) with correct commit messages and push confirmation
- Copilot caught missing `|| return 1` on all three kubectl calls — legitimate correctness issues, not false positives

## What Went Wrong
- data-layer StatefulSet race fix was already committed (51355904) but memory-bank still listed it as "pending commit" — stale OPEN row not cleaned up after Codex verify
- Two stale realm-json bug docs (`2026-05-18`, `2026-05-25`) had internally inconsistent old/new code blocks — should have been caught before committing those docs
- `ExternalSecret/product-catalog-secrets` SharedResourceWarning persists after legacy app removal — the ESO is claimed by both `cicd/product-catalog` and `data-layer` ArgoCD apps; this is a separate conflict from the Deployment/Service ones we fixed

## Process Rules Added
No new rules added to CLAUDE.md this milestone. Existing rules held.

## Decisions Made
- `services-git` ApplicationSet (k3d-manager) is the canonical ArgoCD app layer for shopping-cart services; legacy direct-app definitions in shopping-cart-infra are permanently removed
- `--prefer-sudo` is the correct flag for all macOS LaunchDaemon operations in `bin/acg-down` and `bin/acg-up` — `--interactive-sudo` is only for truly interactive flows (not teardown)
- `kubectl wait` and `kubectl annotate` must have explicit `|| return 1` / `|| _warn` guards in provision functions; silent continuation on timeout is a reliability bug

## Theme
v1.4.11 was a reliability sprint. Three fresh-cluster failures (ESO sync saturation, StatefulSet not-found race, macOS PTY allocation) and two ArgoCD correctness issues (duplicate app layers, RBAC name mismatch) were all root-caused and fixed at the source rather than patched with retries or ignores. The recurring theme: provisioning functions that silently continued on failure — kubectl wait timeouts, annotate failures — were upgraded to explicit error handling. One known issue remains (product-catalog-secrets SharedResourceWarning) for follow-on work in v1.4.12.
