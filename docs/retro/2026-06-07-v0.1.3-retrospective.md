# Retrospective — v0.1.3

**Date:** 2026-06-07
**Milestone:** v0.1.3 — restore force:true on Start/Resume Sandbox clicks
**PR:** #38 — merged to main (`ea76204a`)
**Participants:** Claude, Copilot

## What Went Well
- Root cause identified quickly from Slack failure screenshot — viewport shift pattern recognized from v0.1.2 history
- Fix was minimal (3 one-line changes) with no risk of behavior regression
- Copilot caught all 7 doc consistency issues: Files header, Rules contradiction, What NOT to Do handoff guard, CHANGELOG path prefix, memory-bank stale status block
- All threads resolved cleanly before merge

## What Went Wrong
- v0.1.2 spec added `scrollIntoViewIfNeeded()` but omitted `{ force: true }` — the spec was half-correct and the code reviewer (Claude) missed the omission
- Bug spec initially had "Do NOT create a PR" in the committed doc — Copilot correctly flagged this as a handoff-only guard that should not appear in checked-in specs
- Bug spec Files header only listed the code file, not CHANGELOG/memory-bank — Copilot correctly flagged this as internally inconsistent

## Process Rules Added
| Rule | Applies To |
|------|------------|
| Bug spec Files header must list ALL expected touched files (code + CHANGELOG + memory-bank) | All future bug specs |
| Rules section must not say "No other files touched" when memory-bank/CHANGELOG updates are expected | All future bug specs |
| "Do NOT create a PR" must not appear in committed bug docs — it is a Codex handoff guard only | All future bug specs |

## Decisions Made
- `{ force: true }` is the correct pattern for all Start/Resume Sandbox button clicks when `scrollIntoViewIfNeeded()` is also present — the scroll positions the element, force bypasses the post-scroll viewport re-check that fails after layout shift
- `openButton` already used this pattern (added PR #35); Start/Resume paths were consistently missed across two fix iterations

## Theme
A one-line regression that took two releases to fully fix. v0.1.2 added the scroll but missed the force flag; v0.1.3 added the force flag. The lesson: when fixing a viewport click failure, both `scrollIntoViewIfNeeded()` AND `{ force: true }` are required together — one without the other is incomplete.
