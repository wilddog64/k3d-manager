# Retrospective — v0.9.9

**Date:** 2026-03-22
**Milestone:** if-count allowlist elimination (ldap + vault)
**PR:** #43 — merged to main (`c1043175`)
**Participants:** Claude, Codex, Copilot

## What Went Well

- **Codex delivered on ldap spec cleanly** — `ba6f3a9` landed exactly as written; 11 helpers extracted, 7 allowlist entries removed, net -291 lines on first attempt
- **Vault refactor also clean** — `365846c`; 6 helpers extracted, 5 allowlist entries removed, net -81 lines
- **Pre-commit hook caught working-tree inconsistency** — Codex's partial jenkins work (reverted but not fully cleaned) left an inconsistent working tree; the hook blocked the commit, which was the right behavior; restored with `git restore` and proceeded
- **Copilot caught 9 real bugs** — `set +x` vs `set -x` trace inversion, `jq .sealed // empty` boolean misdetection, `printf` format splits, missing `local` declaration, double-value `grep -c || echo "0"` — all legitimate; none were false positives
- **Spec quality held** — exact code blocks in specs meant Codex had nothing to interpret; finding rate (9 Copilot findings) was lower than previous refactor PRs

## What Went Wrong

- **Jenkins refactor deferred mid-task** — Codex started jenkins refactor, hit scope issues (functions not found in current jenkins.sh), left working tree in partial state. Root cause: spec was written against a version of jenkins.sh that had those functions; the current file doesn't. Wasted one Codex cycle.
- **Vault commit (`365846c`) silently removed jenkins allowlist entries** — the vault commit cleared the jenkins entries from the allowlist alongside vault entries, leaving jenkins.sh above threshold with no exemptions. Only caught when pre-commit hook blocked a later commit. Should have been caught during vault verification.
- **`printf '%s\n'` split pattern** — 3 instances of the same formatting bug (multi-line printf) in the generated code. Codex consistently splits `printf '%s\n'` across lines when wrapping long lines. Should be in spec template as an explicit rule.

## Process Rules Added

| Rule | Where |
|------|--------|
| `printf '%s\n'` must be single-line — never split format string across lines | Spec template |
| Never use `jq -r '.field // empty'` on boolean fields — use `jq -r '.field'` | Spec template + CLAUDE.md security rules |
| `(( restore_trace )) && set -x` re-enables; `set +x` disables — add inline comment | Spec template |
| Verify allowlist diff after every commit — check no unintended entries removed | Verification checklist |

## Decisions Made

- **Jenkins refactor → v0.9.10** — the 4 jenkins functions (`_jenkins_run_saved_trap_literal`, `_jenkins_run_smoke_test`, `deploy_jenkins`, `_deploy_jenkins`) are not present in the current jenkins.sh. The spec is obsolete. v0.9.10 will either re-verify jenkins.sh state or close the spec as moot.
- **`system.sh` entries remain in allowlist** — `_run_command` and `_ensure_node` are blocked on lib-foundation `feat/v0.3.4` PR; not actionable in k3d-manager directly.
- **v0.9.9 scope: ldap + vault only** — net 372 LOC reduction; goal of "eliminate all plugin allowlist entries" partially achieved (12 of 16 non-system entries removed).

## Theme

v0.9.9 was a structural cleanup sprint: two plugin files refactored into narrow helpers, allowlist shrunk from 18 to 4 entries (ldap and vault fully cleared). The session demonstrated that spec-driven Codex handoffs work cleanly for pure refactors — when the spec has exact code, Codex has nothing to misinterpret. The main friction was peripheral: a stale spec for jenkins, a working-tree inconsistency from a partial Codex run, and Copilot catching the same `printf` formatting bug three times. The pre-commit hook and Copilot review caught everything before merge — the enforcement layer did its job.
