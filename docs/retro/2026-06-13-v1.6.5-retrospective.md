# v1.6.5 Retrospective

**Date:** 2026-06-13  
**Milestone:** v1.6.5 — k3s-az Azure provider + provisioning hardening  
**PR:** #94 merged to main (`56eeb58c`)  
**Participants:** Claude, Codex, Gemini, Copilot

## What Went Well

- **k3s-az Azure provider landed** — VM provision, k3sup install, and shopping-cart deploy fully operational; wired into `bin/acg-up` and `bin/acg-down`
- **Provider-aware refresh/status via recorded active-provider state** — recorded state file eliminates AWS fall-through defaults; provider stored at provision time with kube-context probe fallback and explicit env override support
- **Large batch of ACG/OrbStack provisioning hardening** — recovered stopped OrbStack VM via `orbctl start`, dismissed update dialogs, fixed stale Vault port-forward binding, non-interactive host keys in shopping-cart deploy, known_hosts prune during `acg-up` preflight
- **shopping-cart cross-cluster deploy hardening** — StatefulSet presence polling instead of one-shot check, ArgoCD re-login before cluster add, stale default cluster/user purge before kubeconfig flatten, SSH alias resolution to IP, k3s kubeconfig copy to user home for sudo-free re-export, product-catalog-seed job deletion wait, defaulted SSH user to `ubuntu`
- **CI fully green** — all integration and unit tests passed
- **Copilot caught a real case-sensitivity bug** — slack-relay provider normalization issue found and fixed promptly (`eb0dfca2`); demonstrates effective PR review workflow
- **Subtree discipline held** — Copilot findings in `scripts/lib/acg/` deferred upstream to lib-acg feat/v0.1.8 rather than patched in place, enforcing proper subtree workflow

## What Went Wrong

- **ACG sandbox provisioning tripped abuse limiter** — A Cloud Guru's rate-limiting blocked the sandbox ~72h (resolution ~2026-06-16) from an Azure SP-credential recreate storm; root cause was device-code wait having no timeout and chasing intermittently-rendered SP panel via sandbox recreate
- **Live smoke test could not run** — no functional cluster available (ACG blocked + Azure sandbox expired), so v1.6.5 shipped with smoke test deferred, relying on static validation (shellcheck, bats, node --check) instead of live verification
- **Release docs drifted** — README releases table and docs/releases.md were missing v1.6.4 / v1.6.3 rows and had to be backfilled during PR prep

## Process Rules Added

| Rule | Context |
|------|---------|
| Copilot findings in subtree paths are filed upstream, never patched in place | `scripts/lib/acg/` and `scripts/lib/foundation/` findings go to lib-acg / lib-foundation feat branches; consuming repo defers |
| Provider-token normalization must case-fold at the boundary | Before alias/membership tests, normalize provider tokens (e.g., `Azure` → `az`) |

## Decisions Made

- **ACG demoted from automation backend to occasional learning surface** — hard abuse limits and temporary blocks make ACG unsuitable for production automation; retained for on-demand learning labs only
- **Chosen durable remote app-cluster:** Hostinger KVM 2 single node (2 vCPU / 8GB, $215/2yr) replacing the ACG EC2 app-cluster leg; infra (k3d hub, Slack webhook, Prometheus) stays on laptop
- **lib-acg Azure SP re-extract spec written, awaiting Codex** — page-reload re-extract before capped recreate + bounded device-code wait to prevent credential storms
- **Recreate cap kept at 1 with cooldown per user** — limit ACG sandbox recreates to 1 per session with cooldown to prevent abuse-limiter triggers
- **Free-tier squeeze acknowledged** — OCI A1 capacity-blocked, home lab Mac Mini M5 target ~October 2026

## Theme

v1.6.5 brought Azure as a first-class k3s provider and hardened the ACG/OrbStack provisioning path, but the milestone's real lesson was that **ACG is a learning sandbox, not infrastructure**: chasing flaky Azure SP credentials by recreating sandboxes tripped an abuse block, which crystallized the decision to move the permanent app cluster onto a cheap owned host (Hostinger) and keep ACG for occasional labs.
