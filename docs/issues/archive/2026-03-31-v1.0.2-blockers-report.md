# Issue: v1.0.2 3-node Cluster Deployment Blockers

**Date:** 2026-03-31
**Branch:** `k3d-manager-v1.0.2`

## Problem
The goal of getting all 5 pods `Running` on the 3-node AWS cluster is currently blocked by several infrastructure and configuration issues.

## Analysis

### 1. ESO CRD Version Mismatch
- **Symptoms:** ArgoCD reported `The Kubernetes API could not find version "v1beta1" of external-secrets.io/ExternalSecret`.
- **Root Cause:** App cluster ESO installation only serves `v1`, while manifests use `v1beta1`.
- **Status:** Sync for `data-layer` and secrets is completely blocked.

### 2. Missing Image Pull Secrets
- **Symptoms:** Application pods showed `ImagePullBackOff`.
- **Root Cause:** `ghcr-pull-secret` was not provisioned on the fresh 3-node cluster.
- **Status:** Patched manually during the session but needs a permanent automated fix in the provider.

### 3. Vault Reachability
- **Symptoms:** `ClusterSecretStore` reported `connection refused` to Vault.
- **Root Cause:** Lack of a stable network bridge between the remote AWS cluster and the local infra Vault.
- **Status:** Requires a reverse SSH tunnel and potentially a `socat` bridge or a formal ingress solution.

### 4. RBAC for ArgoCD Manager
- **Symptoms:** `register_app_cluster` failed due to missing service account on the remote cluster.
- **Root Cause:** The `argocd-manager` ServiceAccount and its `cluster-admin` binding were not present.
- **Status:** Manually created to enable cluster registration.

## Impact
Milestone gate for v1.0.2 cannot pass until the infrastructure "bridge" (Secrets + Registry + RBAC) is fully automated and stable.

## Recommended Follow-up
1. Align ESO API versions between manifests and cluster installation.
2. Automate `ghcr-pull-secret` injection in `k3s-aws` provider.
3. Establish a standard pattern for remote-to-local Vault connectivity (e.g., automated reverse tunnel).
