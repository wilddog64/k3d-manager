# v1.1.1 — vCluster Preflight (webhook-as-runner)

**Roadmap slot:** v1.1.1 (k3d-console CI/CD pipeline — PR gate tier)
**Branch:** `k3d-manager-v1.7.1` (or a dedicated `feat/vcluster-preflight` branch)
**Status:** PLAN — phased; Phase 1 is the first handoff unit

---

## Goal

Validate the full shopping-cart stack in a throwaway vCluster **before** a PR merges to
main. No self-hosted GitHub Actions runner — the existing **k3dm-webhook server** (already
running on the warm host that has Istio + Vault + ESO) is the execution engine. One
preflight script, three entry points (make, Slack, GitHub stub), one durable log link.

This is the validation environment for the order/payment **Go rewrite**: the e2e suite is
the acceptance contract; each Go port is proven against it here before merge.

---

## Deploy mechanism — DECIDED: ArgoCD-registered vcluster (Option 2)

The cart stack is deployed **the same way prod is** — GitOps via the hub ArgoCD, not a
side-channel helm apply. The preflight registers the throwaway vcluster as an ArgoCD
destination and points the existing ApplicationSets at it via `APP_CLUSTER_NAME`.

**Why this is clean for vcluster specifically:** a vcluster runs *inside* the hub cluster
(shares host nodes), so the hub's ArgoCD reaches the vcluster API at its **in-cluster
service URL** — no SSH tunnel, unlike `ubuntu-k3s`/Hostinger which need
`register_app_cluster` over a tunnel. Same registration primitive (`register_app_cluster`,
`argocd.sh:1119` — applies an ArgoCD cluster Secret), different (simpler) server URL.

**Registration building blocks (reuse, do not reinvent):**
- `register_app_cluster` applies a cluster Secret: `name` / `server` / `bearerToken` /
  `tlsClientConfig.insecure`. Driven by env: `ARGOCD_APP_CLUSTER_NAME`,
  `ARGOCD_APP_CLUSTER_SERVER`, `ARGOCD_APP_CLUSTER_TOKEN`, `ARGOCD_APP_CLUSTER_SECRET_NAME`.
- ApplicationSets (`scripts/etc/argocd/applicationsets/services-git.yaml`,
  `observability*.yaml`) already template `destination.name: ${APP_CLUSTER_NAME}`
  (`argocd.sh:1079`, default `ubuntu-hostinger`).

### Host = Hostinger k3s (DECIDED 2026-06-15) — NOT the local Mac k3d hub

The preflight vcluster runs **inside the Hostinger k3s cluster** (`ubuntu-hostinger`
context), not the local k3d hub. Reason: vcluster's embedded control plane crash-loops on
k3d-on-Docker-on-Mac — its kube-controller-manager loses leader election when lease renewal
to `https://127.0.0.1:6443` exceeds the 5s deadline (disk-IO latency on the Docker-on-Mac
filesystem). On a real Linux host (Hostinger) that latency is gone. Confirmed live
2026-06-15: `phase1new-0` restarted 65× with `leaderelection lost … context deadline
exceeded`. (NOT OCI — OCI is not a target; only local-Mac and Hostinger are our hosts.)

**Consequence for ArgoCD registration — the in-cluster URL trick no longer applies.** When
the vcluster ran inside the same cluster as the hub ArgoCD, `https://<name>.vclusters.svc:443`
worked. With the vcluster on Hostinger and ArgoCD on the Mac hub, that `.svc` DNS is NOT
resolvable from the hub. The vcluster API must be **exposed externally from Hostinger**
(ingress hostname / NodePort / LB) and registered by that external URL — exactly how
`ubuntu-hostinger` itself is registered with the hub.

**Three details the implementer MUST pin from the live cluster (do NOT guess in code):**
1. **vcluster host context** = `ubuntu-hostinger`. `vcluster_create`/`vcluster_destroy`
   must run against it (`HOST_CONTEXT=ubuntu-hostinger`), and `_vcluster_check_prerequisites`
   needs a clean host-context parameter (see note on the reverted `vcluster.sh` hack below).
2. **vcluster external API URL** the hub ArgoCD uses → confirm the exposure path on
   Hostinger (`kubectl --context ubuntu-hostinger -n vclusters get svc/ingress`). This
   becomes `ARGOCD_APP_CLUSTER_SERVER` — an external `https://…` reachable from the Mac hub,
   NOT a `.svc` address.
3. **vcluster bearer token.** Mint an `argocd-manager` SA + ClusterRoleBinding *inside the
   vcluster*, then `kubectl --kubeconfig <vcluster-kcfg> create token argocd-manager
   -n kube-system --duration=8760h`. This becomes `ARGOCD_APP_CLUSTER_TOKEN`.

> **Resource note:** Hostinger is 2 vCPU / 7.8 Gi and already runs the app cluster. A
> vcluster + full stack on top is tight — serialize preflight (one vcluster at a time) and
> consider a reduced stack for the gate.

### Isolation pattern — KEEP Codex's prefixed-clone approach (do NOT re-use deploy_argocd_bootstrap)

The first attempt ran `deploy_argocd_bootstrap` with `APP_CLUSTER_NAME=<probe>`, which
**re-applies the shared live `services-git`/`observability-acg` ApplicationSets in place and
retargeted the live `ubuntu-hostinger` apps onto the throwaway vcluster** (incident
`docs/issues/2026-06-14-vcluster-preflight-phase1-live-retarget-failure.md`). NEVER do that.

The working `bin/cluster-preflight` (uncommitted, 2026-06-14) instead **renders prefixed
clones** of the AppProject + ApplicationSets + data-layer Application
(`<name>-preflight-*`, `<name>-preflight-platform` project) via envsubst/sed/awk and applies
those — leaving the live shared resources untouched. That approach is CORRECT and is the
base to build on; only the host model (→ Hostinger) and the external-URL registration need
rework. Brittleness to address: the sed/awk template-munging keys off exact line patterns
(e.g. `name: ubuntu-k3s`) — harden or template properly.

## Architecture — one script, three triggers, on-demand or gated

```
make preflight NAME=test --keep   ─┐   (on-demand: create + deploy, LEAVE UP for poking)
make preflight PR=<n>             ─┼─→  bin/cluster-preflight <ref> [--keep] [--destroy]
Slack /cluster-preflight <ref>    ─┤        vcluster_create <name>
GitHub PR stub  curl /preflight   ─┘        register_app_cluster (vcluster, in-cluster URL)
                                            apply ApplicationSets APP_CLUSTER_NAME=<name>
                                            wait ArgoCD sync+health
                                            [gate mode] run shopping-cart-e2e-tests
                                            [--destroy] vcluster_destroy + deregister
                                            emit: result + log link
```

- **On-demand testing (explicit user requirement):** `bin/cluster-preflight <name> --keep`
  creates the vcluster, deploys the full stack via ArgoCD, and **leaves it running** so it
  can be inspected / poked at any time. `--destroy` (the CI/gate default) tears it down on
  exit. This makes "spin up a vcluster with the whole stack to test, anytime" a first-class
  mode, not just a PR side effect.
- **make** runs `bin/cluster-preflight` directly on the warm host (dev loop).
- **Slack** → worker → webhook `/api/v1/preflight` → background job runs the same script,
  streams progress to the Slack thread (same machinery as `/api/v1/cluster`).
- **GitHub stub** → a 5-line GitHub-hosted workflow that just `curl`s the webhook with the
  PR number + head SHA. No runner. The webhook reports a commit status back so the PR's
  `preflight/vcluster` required check goes green/red.

**Why webhook-as-runner:** the warm host already has the platform stack + hub ArgoCD, and
the webhook already spawns background jobs with Slack-thread continuity (`_run_cluster`
pattern, `bin/k3dm-webhook:281`). vCluster gives cluster-scoped isolation (own
CRDs/webhooks/RBAC) so per-PR runs don't collide on Istio/ESO/Vault CRDs — a bare namespace
can't.

## Phases

### Phase 1a — DE-RISK SPIKE (HANDOFF TARGET, 2026-06-15) — vcluster on Hostinger + NodePort expose + register, then STOP

Goal: prove the two unknowns introduced by the Hostinger-host decision, in the smallest
verifiable step, BEFORE building the heavy in-vcluster stack:
1. a vcluster's control plane is **stable on Hostinger** (the Mac-k3d crash-loop is gone);
2. the vcluster API, exposed as **NodePort on the Hostinger public IP**, is **reachable from
   the Mac hub** (the VPS firewall does not block it).

Work from the **current uncommitted `bin/cluster-preflight`** (do not rewrite from scratch —
its prefixed-clone rendering is correct and reused in Phase 1b). Spike changes:

**1. Host context → Hostinger.** Change the default:
```bash
# old (line ~43):
HOST_CONTEXT="${HOST_CONTEXT:-${INFRA_CONTEXT:-k3d-k3d-cluster}}"
# new:
HOST_CONTEXT="${HOST_CONTEXT:-ubuntu-hostinger}"
```
`vcluster_create`/`vcluster_destroy` and the RBAC/token steps run against this context.

**2. Expose the vcluster API as NodePort.** New values file
`scripts/etc/vcluster/values-preflight.yaml` (vcluster 0.34 / v0.20+ schema):
```yaml
controlPlane:
  service:
    spec:
      type: NodePort
  statefulSet:
    resources:
      requests: { cpu: 200m, memory: 256Mi }
      limits:   { cpu: 500m, memory: 512Mi }
```
Pass it to `vcluster create` (extend `vcluster_create` to accept an override values file, or
set `VCLUSTER_VALUES_FILE` — confirm the cleanest hook in `vcluster.sh`).

**3. Derive the external API URL from config — do NOT hardcode the IP.** After create, read
the assigned NodePort and the Hostinger host from existing config (`HOSTINGER_HOST` in
`scripts/etc/hostinger/vars.sh`, or parse the `ubuntu-hostinger` context server host):
```bash
NODE_PORT="$(_kubectl --context ubuntu-hostinger -n "${VCLUSTER_NAMESPACE}" \
  get svc "${PREFLIGHT_NAME}" -o jsonpath='{.spec.ports[?(@.name=="https")].nodePort}')"
APP_CLUSTER_SERVER="https://${HOSTINGER_HOST_OR_IP}:${NODE_PORT}"
```

**4. FIREWALL GUARDRAIL — verify reachability, STOP if blocked.** Before registering:
```bash
if ! curl -ksS --max-time 10 "${APP_CLUSTER_SERVER}/healthz" >/dev/null 2>&1; then
  _err "[preflight] vcluster NodePort ${APP_CLUSTER_SERVER} not reachable from this host — \
open the NodePort in the Hostinger VPS firewall before proceeding. STOPPING (do not guess)."
fi
```

**5. Mint token + register, then STOP.** Keep the existing RBAC-apply + token-mint +
`register_app_cluster` steps. For the spike, **do NOT call** `_cluster_preflight_install_eso`
or `_cluster_preflight_apply_argocd_resources` — stop after registration and verify the hub
ArgoCD lists the vcluster as a healthy registered cluster.

**6. Revert the bad `vcluster.sh` edit.** Restore the original
`_vcluster_check_prerequisites` (the host-unreachable case must `_err`, not `_warn`), and add
a CLEAN host-context parameter (`VCLUSTER_HOST_CONTEXT`, default current context) instead of
the hardcoded `k3d-k3d-cluster`. Also revert the `eso.sh` timeout band-aids (not needed on a
stable Hostinger host; Phase 1b revisits ESO).

**Phase 1a DoD:**
- `make preflight NAME=spike` creates a vcluster **on `ubuntu-hostinger`** that stays Ready
  with **0 restarts** for ≥10 min (proves stability — contrast `phase1new-0`'s 65 restarts).
- The vcluster API NodePort is reachable from the Mac (`curl …/healthz` succeeds), OR the
  script STOPS with the firewall message.
- Hub ArgoCD shows `cluster-spike` registered and reachable (`argocd cluster list` / the
  cluster Secret resolves).
- `make preflight NAME=spike MODE=--destroy` removes the vcluster + cluster Secret cleanly.
- `vcluster.sh` + `eso.sh` band-aid edits reverted; `shellcheck -S warning bin/cluster-preflight`
  and `_agent_audit` clean.
- Report back the live evidence: `kubectl --context ubuntu-hostinger -n vclusters get pod`
  (restart count), the `curl …/healthz` result, and `argocd cluster list`.

**Phase 1b (AFTER spike proves out):** restore the ESO install + prefixed-clone
ArgoCD apply (the existing functions) so the full stack deploys into the Hostinger-hosted
vcluster; `--keep` leaves it up. This is where the in-vcluster platform weight
(Istio/Vault/ESO on 2 cores) gets validated against the resource note above.

### Phase 1 (original framing — superseded by 1a/1b above) — on-demand deploy via ArgoCD

Prove the deploy path first: create a vcluster, register it with the hub ArgoCD, deploy the
full stack (PINNED release images — no PR override yet), wait for sync, and **leave it up**.
This is the standalone "spin up a vcluster + stack to test anytime" capability. No e2e, no
Slack, no teardown-by-default, no image override yet.

**New file: `bin/cluster-preflight`** (sources lib-foundation; `_kubectl`/`_run_command`
only). Skeleton — implementer fills the two pinned-from-live values (server URL, token):

```bash
#!/usr/bin/env bash
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
# shellcheck source=/dev/null
source "${REPO_ROOT}/scripts/lib/foundation/system.sh"

PREFLIGHT_NAME="${1:?Usage: cluster-preflight <name> [--keep|--destroy]}"
PREFLIGHT_MODE="${2:---keep}"   # Phase 1 default: leave the vcluster up

# 1. throwaway vcluster (name validated to DNS-label by vcluster_create)
"${REPO_ROOT}/scripts/k3d-manager" vcluster_create "$PREFLIGHT_NAME"
VC_KCFG="${VCLUSTER_KUBECONFIG_DIR:-${HOME}/.kube/vclusters}/${PREFLIGHT_NAME}.yaml"

# 2. mint argocd-manager SA + token INSIDE the vcluster  (pin RBAC manifest in impl)
KUBECONFIG="$VC_KCFG" _kubectl apply -f "${REPO_ROOT}/scripts/etc/vcluster/argocd-manager-rbac.yaml"
VC_TOKEN="$(KUBECONFIG="$VC_KCFG" _kubectl create token argocd-manager -n kube-system --duration=8760h)"

# 3. register the vcluster with the hub ArgoCD (in-cluster URL — confirm svc name live)
ARGOCD_APP_CLUSTER_NAME="$PREFLIGHT_NAME" \
ARGOCD_APP_CLUSTER_SECRET_NAME="cluster-${PREFLIGHT_NAME}" \
ARGOCD_APP_CLUSTER_SERVER="https://${PREFLIGHT_NAME}.${VCLUSTER_NAMESPACE:-vclusters}.svc:443" \
ARGOCD_APP_CLUSTER_TOKEN="$VC_TOKEN" \
  "${REPO_ROOT}/scripts/k3d-manager" register_app_cluster

# 4. point the ApplicationSets at the vcluster + wait for sync/health
APP_CLUSTER_NAME="$PREFLIGHT_NAME" \
  "${REPO_ROOT}/scripts/k3d-manager" deploy_argocd_bootstrap --skip-appproject
# (impl: argocd app wait <apps> --health --timeout … against APP_CLUSTER_NAME)

[[ "$PREFLIGHT_MODE" == "--destroy" ]] && {
  "${REPO_ROOT}/scripts/k3d-manager" vcluster_destroy "$PREFLIGHT_NAME"
  _kubectl delete secret "cluster-${PREFLIGHT_NAME}" -n "${ARGOCD_NAMESPACE:-cicd}" --ignore-not-found
}
```

> **Implementer pins (do NOT guess):** (a) the in-cluster vcluster API service name/URL via
> `kubectl -n vclusters get svc` after create; (b) the `argocd-manager-rbac.yaml` SA +
> ClusterRoleBinding to create; (c) how `deploy_argocd_bootstrap` consumes `APP_CLUSTER_NAME`
> for an *additional* destination without disturbing the existing `ubuntu-hostinger`
> ApplicationSets (may need a vcluster-scoped ApplicationSet or an app-of-apps override —
> confirm in `argocd.sh` + `scripts/etc/argocd/applicationsets/`).

**New file: `scripts/etc/vcluster/argocd-manager-rbac.yaml`** — SA `argocd-manager` in
`kube-system` + ClusterRoleBinding. Scope to what ArgoCD needs (avoid blanket
cluster-admin if a narrower ClusterRole suffices — but match what the hub expects).

**Makefile target:**

```makefile
.PHONY: preflight
preflight: ## Spin a vcluster + full stack via ArgoCD (NAME=<name>; --keep by default)
	@bin/cluster-preflight "$(NAME)" "$(MODE)"
```

**Phase 1 DoD:** `make preflight NAME=test` creates a vcluster, registers it with the hub
ArgoCD, the cart ApplicationSets sync onto it (`argocd app list` shows them Healthy against
destination `test`), and the vcluster stays up. `make preflight NAME=test MODE=--destroy`
also tears it down + removes the cluster Secret. `shellcheck -S warning bin/cluster-preflight`
clean; `_agent_audit` clean.

### Phase 3 — commit-status merge gate + GitHub PR stub (auto-per-PR)

**Merge gate** — the webhook job posts a commit status to the PR head SHA so branch
protection can require it:

```python
def _post_commit_status(sha, state, target_url):
    # state in {"success","failure"}; context "preflight/vcluster"; target_url = log link
    subprocess  # NOT allowed in webhook — use _posix_spawn_capture
    # cmd: gh api repos/<owner>/<repo>/statuses/<sha> -f state=.. -f context=preflight/vcluster -f target_url=..
```

Mark `preflight/vcluster` a **required status check** on `main` branch protection.

**GitHub stub workflow** (`.github/workflows/preflight-dispatch.yml`) — GitHub-hosted, no
runner, only makes an HTTP call:

```yaml
name: vcluster-preflight-dispatch
on:
  pull_request:
    types: [opened, synchronize, reopened]
permissions:
  contents: read
jobs:
  dispatch:
    runs-on: ubuntu-latest
    steps:
      - name: trigger webhook preflight
        run: |
          curl -fsS -X POST "${{ secrets.K3DM_WEBHOOK_URL }}/api/v1/preflight" \
            -H "Authorization: Bearer ${{ secrets.K3DM_WEBHOOK_TOKEN }}" \
            -H "Content-Type: application/json" \
            -d "{\"ref\":\"${{ github.head_ref }}\",\"head_sha\":\"${{ github.event.pull_request.head.sha }}\"}"
```

Plus a `make preflight-dispatch REF=<branch>` convenience target that POSTs the same
payload from a dev machine.

**Phase 3 DoD:** opening a PR fires the stub → webhook runs preflight → PR shows a
`preflight/vcluster` check that flips to ✅/❌ with the log URL as its target link; a red
check blocks merge.

## Rules

- `shellcheck -S warning` clean on `bin/cluster-preflight` and any edited bash.
- `bin/cluster-preflight` sources lib-foundation and uses `_kubectl`/`_run_command` — no
  bare `kubectl`/`sudo` (per bin/ convention).
- Webhook: use `_posix_spawn_capture` / `_posix_spawn_job` only — never `subprocess.run`.
- Run `make restart-webhook` after any `bin/k3dm-webhook` edit.
- GitHub Actions steps pinned to version tags; `permissions: contents: read`.
- Webhook GH token needs `statuses:write` (Phase 3) — keychain/env, never in argv.
- `_agent_audit` gate (`./scripts/k3d-manager _agent_audit`) before reporting done.
- Subtree discipline: all target files are native/editable
  (`bin/cluster-preflight` new, `Makefile`, `bin/k3dm-webhook`,
  `workers/slack-relay/index.js`, `scripts/plugins/vcluster.sh` if touched). Do NOT edit
  anything under `scripts/lib/foundation/` or `scripts/lib/acg/`.

## Security notes

- The webhook runs untrusted PR code on a host with cluster creds. vCluster isolation
  (own RBAC/CRDs) is the boundary; still scope the host ServiceAccount minimally and
  destroy on `trap EXIT` always.
- Validate `ref` before use — `vcluster_create` already enforces a DNS-label regex on the
  derived name; keep that as the sanitization point (`_vcluster_kubeconfig_path`).
- Slack signature verification already exists (`_verify_slack_signature`,
  `bin/k3dm-webhook:516`; worker `verifySlack`) — reuse, do not bypass.
