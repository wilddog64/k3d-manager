# Bugfix: v1.6.3 — DNS short-circuit advice incomplete

**Branch:** `k3d-manager-v1.6.3`
**Files:** `bin/acg-refresh`, `bin/k3dm-webhook`

---

## Problem

When ArgoCD on the k3d hub cluster cannot resolve `host.k3d.internal`, the
`_analyze_failure` short-circuit in `bin/k3dm-webhook` fires and advises the user to
run `/acg-refresh`. But `/acg-refresh` only re-registers the ArgoCD cluster secret — it
does not restart CoreDNS, which is the actual step needed to restore `host.k3d.internal`
resolution.

`host.k3d.internal` is injected into k3d node `/etc/hosts` via `hostAliases` in
`cluster.yaml.tmpl`. CoreDNS reads that entry via its `NodeHosts` plugin. After a
CoreDNS pod restart or update, the NodeHosts cache is stale and `host.k3d.internal`
stops resolving — even though the entry is still in `/etc/hosts`. Restarting CoreDNS
forces a re-read and restores resolution.

**Root cause:** `/acg-refresh` step 2b registers the ArgoCD cluster secret but never
restarts CoreDNS, so the DNS failure persists and the cluster secret registration silently
succeeds against an unreachable server.

---

## Reproduction

1. Restart the CoreDNS deployment on the hub cluster:
   `kubectl rollout restart deployment/coredns -n kube-system --context k3d-k3d-cluster`
2. Before rollout completes, run an ArgoCD sync — observe `no such host` for
   `host.k3d.internal`.
3. Run `/acg-refresh` without this fix — DNS failure persists; ArgoCD still can't connect.

---

## Fix

### Change 1 — `bin/acg-refresh`: add CoreDNS restart before step 2b

Insert between line 187 (blank line after tunnel-ready check) and line 188 (`# ── 2b.`).

**Exact old block (lines 187–188):**

```bash

# ── 2b. ArgoCD cluster secret ────────────────────────────────────────────────
```

**Exact new block:**

```bash

# ── 2b-pre. CoreDNS restart on hub cluster ───────────────────────────────────
_info "[acg-refresh] Restarting CoreDNS on hub cluster to restore host.k3d.internal resolution..."
if kubectl rollout restart deployment/coredns -n kube-system --context k3d-k3d-cluster >/dev/null 2>&1; then
  kubectl rollout status deployment/coredns -n kube-system --context k3d-k3d-cluster --timeout=30s >/dev/null 2>&1 \
    && _info "[acg-refresh] CoreDNS restarted — host.k3d.internal resolution restored" \
    || _warn "[acg-refresh] CoreDNS rollout status timed out — DNS may still be recovering"
else
  _warn "[acg-refresh] CoreDNS restart failed — hub cluster may not be running"
fi

# ── 2b. ArgoCD cluster secret ────────────────────────────────────────────────
```

### Change 2 — `bin/k3dm-webhook`: update short-circuit advice to reflect the real fix

**Exact old block (lines 834–841):**

```python
        if 'host.k3d.internal' in _l and 'no such host' in _l:
            return (
                "CLUSTER_FAILURE\n"
                "TRANSIENT\n"
                "ArgoCD cannot resolve host.k3d.internal — the k3d CoreDNS host injection "
                "was lost (cluster restart or CoreDNS update).\n"
                "Fix: run /acg-refresh to re-register the ArgoCD cluster secret, then retry /acg-resume"
            )
```

**Exact new block:**

```python
        if 'host.k3d.internal' in _l and 'no such host' in _l:
            return (
                "CLUSTER_FAILURE\n"
                "TRANSIENT\n"
                "ArgoCD cannot resolve host.k3d.internal — CoreDNS NodeHosts cache is stale "
                "(pod restart or update flushed the host.k3d.internal entry from /etc/hosts).\n"
                "Fix: run /acg-refresh — it now restarts CoreDNS on the hub cluster to restore "
                "host.k3d.internal resolution, then re-registers the ArgoCD cluster secret; "
                "then retry /acg-resume"
            )
```
