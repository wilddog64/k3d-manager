# Bugfix: v1.11.0 — failover aborts at re-seed when laptop source is unreachable

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/plugins/vault.sh`

---

## Problem

During a `laptop -> hostinger` assisted failover, `vault_failover_hub_into_context` re-seeds the
in-cluster Vault from the laptop hub before repointing the CSS:

```bash
vault_seed_hub_into_context "${_app_ctx}" \
  || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
```

`vault_seed_hub_into_context` reads its canonical source from the **laptop** hub Vault. But a
`laptop -> hostinger` failover only fires **because the laptop hub is unreachable** — so the
re-seed's source read fails and the function calls `_err "[vault] could not read source
vault-root token ..."`. `_err` (`scripts/lib/system.sh:1717`) does **`exit 1`**, which terminates
the entire process. The `|| _warn "...continuing"` guard is therefore **dead code** — the
`exit 1` kills the shell before `||` can catch a nonzero status.

**Consequence:** the new profile is persisted to the state file (that write happens *before* the
re-seed), but the failover then **aborts before the CSS reconcile** (`_hostinger_reconcile_vault_cluster_store`),
so the app-cluster ClusterSecretStore is never repointed at the in-cluster Vault. The failover
defeats its own purpose in exactly the scenario it exists for.

**Visible symptom (live, `ubuntu-hostinger`):** forcing a `laptop -> hostinger` failover with the
laptop probe failing produced:

```
WARN: [vault] hub Vault probe 1/2 failed (profile=laptop)
WARN: [vault] hub Vault probe 2/2 failed (profile=laptop)
WARN: [vault] hub Vault unreachable after 2 probes — failing over laptop -> hostinger
INFO: [vault] re-seeding in-cluster Vault on ubuntu-hostinger before repointing CSS
ERROR: [vault] could not read source vault-root token from k3d-k3d-cluster/secrets
```

— then **nothing**. No "re-seed reported issues" warn, no reconcile, no "failover complete". The
process exited at the `_err`. (No damage occurred only because the CSS was already pointing at
the in-cluster Vault in this environment.)

**Root cause:** `_err` exits the process; the re-seed call in `vault_failover_hub_into_context` is
not isolated, so its `_err`-driven `exit 1` aborts the failover instead of degrading to the
intended best-effort warn-and-continue. (`vault_seed_hub_into_context` itself pairs every `_err`
with an unreachable `return 1`, confirming the intended contract was "return non-zero", but `_err`
exits — so the call site must isolate it.)

## Reproduction

```bash
# Temp kubeconfig WITHOUT the laptop context so the laptop probe fails deterministically
# (no Vault pod is touched); temp state file marks laptop as the active profile:
TMPKUBE=/tmp/kc-no-laptop.yaml; cp ~/.kube/config "$TMPKUBE"; chmod 600 "$TMPKUBE"
kubectl --kubeconfig="$TMPKUBE" config delete-context k3d-k3d-cluster
TMPSTATE=/tmp/hub-vault-profile-test; printf 'laptop\n' > "$TMPSTATE"

env -u HUB_VAULT_PROFILE KUBECONFIG="$TMPKUBE" HUB_VAULT_PROFILE_STATE_FILE="$TMPSTATE" \
  HUB_VAULT_FAILOVER_THRESHOLD=2 HUB_VAULT_FAILOVER_INTERVAL=2 CLUSTER_PROVIDER=k3s-hostinger \
  ./scripts/k3d-manager vault_failover_hub_into_context ubuntu-hostinger

# Actual:   aborts after "could not read source vault-root token" (process exits at _err)
# Expected: warns "re-seed reported issues ... continuing", then runs the reconcile and prints
#           "failover complete: HUB_VAULT_PROFILE=hostinger"
```

---

## Fix

### Change 1 — `scripts/plugins/vault.sh`: isolate the re-seed in a subshell

Run `vault_seed_hub_into_context` in a subshell so its `_err`-driven `exit 1` only terminates the
subshell, yielding a nonzero status the existing `|| _warn` guard catches — letting the failover
degrade to best-effort re-seed and continue to the CSS reconcile. Do NOT modify
`vault_seed_hub_into_context` (its exit behavior is relied on by the P3 seeding/operator path).

**Exact old block (lines 1209–1213):**

```bash
  if [[ "${_fallback}" == "hostinger" ]]; then
    _info "[vault] re-seeding in-cluster Vault on ${_app_ctx} before repointing CSS"
    vault_seed_hub_into_context "${_app_ctx}" \
      || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
  fi
```

**Exact new block:**

```bash
  if [[ "${_fallback}" == "hostinger" ]]; then
    _info "[vault] re-seeding in-cluster Vault on ${_app_ctx} before repointing CSS"
    ( vault_seed_hub_into_context "${_app_ctx}" ) \
      || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
  fi
```

The only change is wrapping the call in `( … )`. A subshell contains the `exit 1` from `_err`,
so the parent receives exit status 1 and the `|| _warn` runs; execution then continues past the
`fi` to the profile flip + `_hostinger_reconcile_vault_cluster_store`. External side effects of a
successful re-seed (writes to the in-cluster Vault / keyring / k8s backup) are unaffected — the
subshell does not need to export shell state back to the caller.
