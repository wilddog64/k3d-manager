# Bugfix: v1.6.0 — Hub cluster health check is binary-only and non-recovering

**Branch:** `k3d-manager-v1.6.0`
**Files:** `bin/acg-up`

---

## Problem

`bin/acg-up` Step 3.5 checks whether the Hub k3d cluster is registered via
`k3d cluster list`, but does not check whether the cluster is actually reachable.
If the cluster is listed but its containers are stopped (common after Mac sleep/wake),
`acg-up` logs an unreachable warning and continues — then silently fails later when
`kubectl` calls hit a dead API server. A second bug: the `k3d cluster list` call
itself requires the k3d binary to be in PATH; if PATH is wrong (e.g. under the
webhook LaunchAgent), it silently exits non-zero and grep finds nothing, causing
`acg-up` to treat an existing cluster as missing and attempt a full
`deploy_cluster` — which fails with "Failed to install k3d".

**Root cause:** Step 3.5 uses a single `k3d cluster list | grep` gate for cluster
existence without a recovery path for the "listed but stopped" state, and the
downstream `kubectl get nodes` check only warns instead of acting.

---

## Reproduction

```
# Trigger via PATH bug (webhook daemon missing ~/.local/bin):
/acg-up from Slack
# Result: "Failed to install k3d" after ~2 min

# Trigger via sleep/wake (cluster listed but stopped):
# 1. Put Mac to sleep with k3d running
# 2. Wake Mac
# 3. /acg-up from Slack
# Result: acg-up continues past Step 3.5 warning, then later kubectl calls fail
```

---

## Fix

### Change 1 — `bin/acg-up`: replace Step 3.5 single-gate check with three-tier logic

**Exact old block (lines 159–170):**

```bash
_HUB_CLUSTER_CTX="k3d-${HUB_CLUSTER_NAME:-k3d-cluster}"
_hub_newly_created=0
_info "[acg-up] Step 3.5/12 — Verifying local Hub cluster (create if missing)..."
if ! k3d cluster list 2>/dev/null | grep -q "^${HUB_CLUSTER_NAME:-k3d-cluster}[[:space:]]"; then
  _info "[acg-up] Local Hub cluster '${HUB_CLUSTER_NAME:-k3d-cluster}' not found — creating it now..."
  deploy_cluster --provider k3d "${HUB_CLUSTER_NAME:-k3d-cluster}"
  _hub_newly_created=1
fi
if ! kubectl get nodes --context "${_HUB_CLUSTER_CTX}" --request-timeout=10s >/dev/null 2>&1; then
  _err "[acg-up] Local Hub cluster '${_HUB_CLUSTER_CTX}' is unreachable — OrbStack/k3d may be in a broken state after sleep. Check 'orbctl status'."
fi
_info "[acg-up] Local Hub cluster verified."
```

**Exact new block:**

```bash
_HUB_CLUSTER_CTX="k3d-${HUB_CLUSTER_NAME:-k3d-cluster}"
_hub_newly_created=0
_info "[acg-up] Step 3.5/12 — Verifying local Hub cluster (create if missing)..."
if ! _command_exist k3d; then
  _err "[acg-up] k3d binary not found in PATH — cannot manage Hub cluster. Check PATH includes ~/.local/bin."
  exit 1
fi
if ! k3d cluster list 2>/dev/null | grep -q "^${HUB_CLUSTER_NAME:-k3d-cluster}[[:space:]]"; then
  _info "[acg-up] Local Hub cluster '${HUB_CLUSTER_NAME:-k3d-cluster}' not found — creating it now..."
  deploy_cluster --provider k3d "${HUB_CLUSTER_NAME:-k3d-cluster}"
  _hub_newly_created=1
elif ! kubectl get nodes --context "${_HUB_CLUSTER_CTX}" --request-timeout=10s >/dev/null 2>&1; then
  _info "[acg-up] Local Hub cluster '${HUB_CLUSTER_NAME:-k3d-cluster}' is registered but not reachable — starting stopped containers..."
  k3d cluster start "${HUB_CLUSTER_NAME:-k3d-cluster}"
  local _retry=0
  until kubectl get nodes --context "${_HUB_CLUSTER_CTX}" --request-timeout=10s >/dev/null 2>&1 || [[ $_retry -ge 12 ]]; do
    _info "[acg-up] Waiting for Hub cluster API server... (${_retry}/12)"
    sleep 5
    (( _retry++ )) || true
  done
  if ! kubectl get nodes --context "${_HUB_CLUSTER_CTX}" --request-timeout=10s >/dev/null 2>&1; then
    _err "[acg-up] Hub cluster API server did not come up after cluster start. Check 'k3d cluster list' and 'docker ps'."
    exit 1
  fi
fi
_info "[acg-up] Local Hub cluster verified."
```
