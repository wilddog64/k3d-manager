## Background

ArgoCD→Ubuntu k3s gRPC connectivity fails with persistent i/o timeout.

Confirmed facts:
- TCP reachable: `curl -k https://10.211.55.14:6443/healthz` → 401 from M2 Air ✅
- TLS SAN: `IP Address:10.211.55.14` already in k3s cert ✅
- Still failing: ArgoCD gRPC handshake times out on every attempt

Three suspects:

| # | Suspect | Theory |
|---|---|---|
| A | MTU fragmentation | gRPC TLS handshake packets exceed MTU between k3d overlay and Parallels NIC — fragments dropped |
| B | Source IP / routing | ArgoCD pod reaches Ubuntu with a k3d pod IP (10.42.x.x) — Ubuntu has no route back |
| C | iptables on Ubuntu | k3s or ufw rules drop inbound packets from k3d pod CIDR |

Run all three diagnostics, report results, then attempt fix only for confirmed causes.

## Diagnostic A — MTU Fragmentation

### A1 — Check MTU on both sides

On Ubuntu:
```bash
ssh ubuntu "ip link show eth0 | grep mtu"
# Typical Parallels: 1500
```

Inside the ArgoCD pod (on M2 Air):
```bash
ARGOCD_POD=$(kubectl get pod -n cicd -l app.kubernetes.io/name=argocd-server \
  -o jsonpath='{.items[0].metadata.name}')

kubectl exec -n cicd "${ARGOCD_POD}" -- ip link show eth0 2>/dev/null \
  || kubectl exec -n cicd "${ARGOCD_POD}" -- cat /sys/class/net/eth0/mtu
# k3d overlay typically 1450 or lower
```

### A2 — Test large packet delivery (PMTU)

From inside the ArgoCD pod, ping Ubuntu with DF bit set at decreasing sizes:
```bash
kubectl exec -n cicd "${ARGOCD_POD}" -- \
  ping -M do -s 1400 -c 3 10.211.55.14 2>/dev/null \
  || kubectl exec -n cicd "${ARGOCD_POD}" -- \
  ping -c 3 -s 1400 10.211.55.14
# If this fails but -s 1200 succeeds → MTU fragmentation confirmed
```

### A3 — Test HTTPS from inside the pod

```bash
kubectl exec -n cicd "${ARGOCD_POD}" -- \
  wget -qO- --no-check-certificate https://10.211.55.14:6443/healthz 2>&1 \
  || kubectl exec -n cicd "${ARGOCD_POD}" -- \
  curl -sk https://10.211.55.14:6443/healthz
# Expected: ok  (confirms basic HTTPS works from pod)
# If this FAILS → routing/firewall issue (jump to Diagnostic B/C)
# If this PASSES but ArgoCD still fails → gRPC-specific issue
```

**STOP — report MTU values and wget/curl result before continuing.**

## Diagnostic B — Source IP and Return Routing

### B1 — Find ArgoCD pod's source IP as seen by Ubuntu

On Ubuntu, start a tcpdump in the background, then trigger ArgoCD cluster list:
```bash
ssh ubuntu "sudo tcpdump -i eth0 -n 'port 6443' -c 20 2>&1 &"
sleep 2
argocd cluster list   # triggers a connection attempt
sleep 5
ssh ubuntu "sudo pkill tcpdump; sudo journalctl -u tcpdump --since '1 min ago' 2>/dev/null || true"
```

Look for source IPs hitting 10.211.55.14:6443. Expected source will be either:
- `10.211.55.2` (Parallels host gateway — OrbStack routing correctly)
- `10.42.x.x` or `172.x.x.x` (k3d pod CIDR leaking through — routing problem)

### B2 — Check Ubuntu's routing table

```bash
ssh ubuntu "ip route show"
# Check: is there a route for the k3d pod CIDR (10.42.0.0/16 or similar)?
# If missing → Ubuntu cannot reply to ArgoCD pod directly
```

**STOP — report source IP from tcpdump and Ubuntu routing table before continuing.**

## Diagnostic C — iptables / ufw on Ubuntu

### C1 — Check ufw status


### C2 — Check iptables INPUT chain


### C3 — Check iptables for port 6443 specifically


### C4 — Check k3s firewall config


**STOP — report ufw status and any DROP/REJECT rules near port 6443 before continuing.**

## Fix — Apply Only for Confirmed Causes

### Fix A (if MTU confirmed) — Clamp MSS on Ubuntu

```bash
ssh ubuntu "sudo iptables -t mangle -C FORWARD -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu 2>/dev/null \
  || sudo iptables -t mangle -A FORWARD -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu"
# Persists until reboot — report if this resolves the issue
```

### Fix B (if source IP leaking) — Add return route on Ubuntu

First identify the OrbStack/Parallels host gateway IP from the tcpdump output.
Report to Claude before adding any route — do not add blindly.

### Fix C (if iptables DROP confirmed) — Allow k3d pod CIDR

Report the exact DROP rule to Claude before modifying iptables.
