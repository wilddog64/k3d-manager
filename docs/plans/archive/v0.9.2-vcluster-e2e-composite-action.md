# Task: Convert vCluster E2E Workflow to Composite Actions (v0.9.2)

**Assignee:** Codex
**Repo:** k3d-manager
**Branch:** k3d-manager-v0.9.2

---

## Why This Change

The reusable workflow `.github/workflows/vcluster-e2e-setup.yml` has a fundamental
architectural flaw: each `workflow_call` job runs on a separate GitHub-hosted runner.
The `setup` job creates a k3d cluster and exports a kubeconfig pointing to `127.0.0.1`
on runner A. The caller job runs on runner B — that address is unreachable. Teardown
runs on runner C — the cluster doesn't exist there.

**Fix:** Replace the reusable workflow with two composite actions. Composite actions
run inline in the caller's job — same runner, same Docker network, kubeconfig works.

---

## What To Do

### 1. Delete the reusable workflow

Remove `.github/workflows/vcluster-e2e-setup.yml` entirely.

### 2. Create two composite actions

**`.github/actions/vcluster-e2e-setup/action.yml`** — provisions k3d + vCluster:

```yaml
name: vCluster E2E Setup
description: Provision a k3d cluster with a vCluster for E2E testing

inputs:
  run-id:
    description: 'Unique ID for this run (use github.run_id)'
    required: true
  vcluster-version:
    description: 'vCluster chart version'
    required: false
    default: '0.32.1'
  k3d-version:
    description: 'k3d version to install'
    required: false
    default: 'v5.7.4'

outputs:
  vcluster-name:
    description: 'Name of the provisioned vCluster'
    value: ${{ steps.setup.outputs.vcluster-name }}

runs:
  using: composite
  steps:
    - name: Install k3d
      shell: bash
      run: |
        curl -s https://raw.githubusercontent.com/k3d-io/k3d/${{ inputs.k3d-version }}/install.sh \
          | sudo bash

    - name: Install kubectl
      uses: azure/setup-kubectl@v4

    - name: Setup k3d cluster and vCluster
      id: setup
      shell: bash
      env:
        RUN_ID: ${{ inputs.run-id }}
        VCLUSTER_VERSION: ${{ inputs.vcluster-version }}
      run: |
        set -euo pipefail
        cluster="e2e-${RUN_ID}"
        k3d cluster create "$cluster" --no-lb --wait
        ./scripts/k3d-manager vcluster_install_cli
        ./scripts/k3d-manager vcluster_create "$cluster"
        ./scripts/k3d-manager vcluster_use "$cluster"
        echo "vcluster-name=$cluster" >> "$GITHUB_OUTPUT"
```

**`.github/actions/vcluster-e2e-teardown/action.yml`** — destroys vCluster + k3d:

```yaml
name: vCluster E2E Teardown
description: Destroy the vCluster and k3d cluster created by vcluster-e2e-setup

inputs:
  run-id:
    description: 'Same run-id passed to vcluster-e2e-setup'
    required: true

runs:
  using: composite
  steps:
    - name: Teardown vCluster and k3d cluster
      shell: bash
      env:
        RUN_ID: ${{ inputs.run-id }}
      run: |
        set -euo pipefail
        cluster="e2e-${RUN_ID}"
        kubectl config use-context "k3d-${cluster}" || true
        ./scripts/k3d-manager vcluster_destroy "$cluster" || true
        k3d cluster delete "$cluster" || true
```

### 3. How the caller uses these actions

The caller (shopping-cart-e2e-tests) runs everything in **one job**:

```yaml
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          repository: wilddog64/k3d-manager
          path: k3d-manager

      - uses: wilddog64/k3d-manager/.github/actions/vcluster-e2e-setup@k3d-manager-v0.9.2
        with:
          run-id: ${{ github.run_id }}

      # caller deploys apps and runs tests here

      - uses: wilddog64/k3d-manager/.github/actions/vcluster-e2e-teardown@k3d-manager-v0.9.2
        if: always()
        with:
          run-id: ${{ github.run_id }}
```

Same runner throughout — k3d cluster is reachable for the entire job.

## Security Requirements

- All `uses:` steps pin to a version tag — never `@main` or `@latest` (OWASP A08)
- k3d install script: use tag-specific URL
  `https://raw.githubusercontent.com/k3d-io/k3d/${{ inputs.k3d-version }}/install.sh`
  NOT the `main` branch URL — supply chain risk (Copilot Finding 2)
- `permissions: contents: read` is set at the caller level — composite actions inherit
  the calling job's permissions, no declaration needed in the action itself
