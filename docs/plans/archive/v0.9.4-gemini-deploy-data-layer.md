## Context

The shopping-cart app cluster (Ubuntu k3s on EC2) has 3 services in CrashLoopBackOff
because the data layer was never deployed:

| Service | Missing dependency |
|---|---|
| basket-service | Redis (`redis-cart`) + RabbitMQ |
| order-service | PostgreSQL (`orders-db`) + RabbitMQ |
| product-catalog | PostgreSQL (`products-db`) |

All manifests are in `shopping-cart-infra/data-layer/`:
- `postgresql/orders/` ‚Äî PostgreSQL for order-service
- `postgresql/payment/` ‚Äî PostgreSQL for payment-service
- `postgresql/products/` ‚Äî PostgreSQL for product-catalog
- `redis/cart/` ‚Äî Redis for basket-service cart data
- `redis/orders-cache/` ‚Äî Redis for order-service cache
- `rabbitmq/` ‚Äî RabbitMQ for basket + order async messaging

All components target namespace `shopping-cart-data`.

---

## Task

### Step 1 ‚Äî Ensure namespace exists on ubuntu-k3s

```bash
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/namespaces/namespaces.yaml
kubectl --context ubuntu-k3s get ns shopping-cart-data
```

### Step 2 ‚Äî Copy ghcr-pull-secret to shopping-cart-data

The secret already exists in `shopping-cart-apps`. Copy it:

```bash
kubectl --context ubuntu-k3s get secret ghcr-pull-secret -n shopping-cart-apps -o yaml \
  | sed 's/namespace: shopping-cart-apps/namespace: shopping-cart-data/' \
  | kubectl --context ubuntu-k3s apply -f -
```

### Step 3 ‚Äî Apply PostgreSQL manifests

```bash
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/postgresql/orders/
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/postgresql/payment/
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/postgresql/products/
```

### Step 4 ‚Äî Apply Redis manifests

```bash
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/redis/cart/
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/redis/orders-cache/
```

### Step 5 ‚Äî Apply RabbitMQ with reduced replicas

The manifest has `replicas: 3` (1Gi each) ‚Äî too heavy for `t3.medium`. Apply then scale to 1:

```bash
kubectl --context ubuntu-k3s apply -f shopping-cart-infra/data-layer/rabbitmq/
kubectl --context ubuntu-k3s scale statefulset rabbitmq -n shopping-cart-data --replicas=1
```

### Step 6 ‚Äî Wait for data layer pods Running

```bash
kubectl --context ubuntu-k3s get pods -n shopping-cart-data -w
```

Wait until all pods show `Running`. Expected: 5 PostgreSQL pods, 2 Redis pods, 1 RabbitMQ pod.
Timeout: 5 minutes. If any pod stays in `Pending` or `CrashLoopBackOff`, paste the describe output.

### Step 7 ‚Äî Force ArgoCD sync for app services

On infra cluster context (`k3d-k3d-cluster`):

```bash
kubectl config use-context k3d-k3d-cluster
argocd app sync basket-service --force
argocd app sync order-service --force
argocd app sync product-catalog --force
```

If `argocd` CLI is unavailable, use port-forward:
```bash
kubectl port-forward svc/argocd-server -n cicd 8080:443 &
argocd login localhost:8080 --insecure --username admin --password $(kubectl get secret argocd-initial-admin-secret -n cicd -o jsonpath='{.data.password}' | base64 -d)
```

### Step 8 ‚Äî Verify all 5 app pods Running

```bash
kubectl --context ubuntu-k3s get pods -n shopping-cart-apps
kubectl --context ubuntu-k3s get pods -n shopping-cart-payment
```

Expected: basket-service, order-service, product-catalog, frontend all `Running` in `shopping-cart-apps`; payment-service `Running` in `shopping-cart-payment`.
