## Task 3 — Investigate ArgoCD→Ubuntu Connectivity

Previous attempt failed with gRPC i/o timeout. The Ubuntu k3s API at `10.211.55.14:6443`
is reachable at TCP level (confirmed: `curl -k https://10.211.55.14:6443/healthz` returns 401
from M2 Air). The issue is ArgoCD's gRPC connection being rejected or timing out.

### 3a — Check k3s API server TLS SANs

ArgoCD validates the server certificate. The k3s API cert may not include `10.211.55.14`
as a SAN — only `127.0.0.1` by default.

```bash
ssh ubuntu "sudo openssl s_client -connect 10.211.55.14:6443 </dev/null 2>/dev/null \
  | openssl x509 -noout -text | grep -A5 'Subject Alternative Name'"
```

Expected: `IP Address:10.211.55.14` in the SAN list.
If missing — this is the root cause. Report to Claude.

### 3b — Check ArgoCD cluster status

On M2 Air with port-forward active:

```bash
nohup kubectl port-forward svc/argocd-server -n cicd 8080:443 > /tmp/argocd-pf.log 2>&1 &
sleep 3
curl -k https://localhost:8080/healthz
argocd login localhost:8080 --insecure
argocd cluster list
```

Expected: `ubuntu-k3s` showing `Ready` or a specific error message.

### 3c — Re-export kubeconfig and re-register

```bash
ssh ubuntu "sudo cat /etc/rancher/k3s/k3s.yaml \
  | sed 's|127.0.0.1|10.211.55.14|g'" > ~/.kube/k3s-ubuntu.yaml

cd ~/src/gitrepo/personal/k3d-manager
./scripts/k3d-manager add_ubuntu_k3s_cluster 2>&1 \
  | tee scratch/logs/gemini-task7-$(date +%s).log
```

```bash
argocd cluster list
# Expected: https://10.211.55.14:6443  ubuntu-k3s  Ready
```
