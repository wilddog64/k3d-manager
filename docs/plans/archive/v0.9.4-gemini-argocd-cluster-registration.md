# Gemini Task: ArgoCD Cluster Registration + App Sync Verification (v0.9.4)

## Before You Start

1. Read `memory-bank/activeContext.md` and `memory-bank/progress.md`
2. **First command:** `hostname && uname -n` — confirm you are on M2 Air
3. Run `git pull origin k3d-manager-v0.9.4` on M2 Air
4. Verify infra cluster is healthy:
   ```bash
   kubectl --context k3d-k3d-cluster get pods -n cicd
   ```
   ArgoCD pods must be Running before proceeding.
5. Verify tunnel is running:
   ```bash
   ./scripts/k3d-manager tunnel_status   # must show: process: running
   ```
   If not running: `./scripts/k3d-manager tunnel_start`

---

## Context

ArgoCD apps (`basket-service`, `order-service`, `payment-service`, `product-catalog`,
`frontend`) are in `Unknown` sync status because the cluster secret `cluster-ubuntu-k3s`
has no bearer token. A new `register_app_cluster` command is now available in
`scripts/k3d-manager` that applies a properly formatted cluster secret including a
bearer token.

The `frontend` app also has a stale `destination.server` pointing to
`https://192.168.50.40:6443` (direct VM IP — unreachable from ArgoCD pods). This must
be patched to `https://host.k3d.internal:6443`.

---

## Step 1 — Create argocd-manager service account on ubuntu k3s

Run on M2 Air (via SSH to ubuntu):

```bash
ssh ubuntu kubectl create serviceaccount argocd-manager -n kube-system
ssh ubuntu kubectl create clusterrolebinding argocd-manager \
  --clusterrole=cluster-admin \
  --serviceaccount=kube-system:argocd-manager
```

Verify:
```bash
ssh ubuntu kubectl get serviceaccount argocd-manager -n kube-system
```

---

## Step 2 — Run register_app_cluster

Get a token and register the cluster:

```bash
TOKEN=$(ssh ubuntu kubectl create token argocd-manager -n kube-system --duration=8760h)
ARGOCD_APP_CLUSTER_TOKEN="${TOKEN}" ./scripts/k3d-manager register_app_cluster
```

Verify the secret was applied:
```bash
kubectl --context k3d-k3d-cluster get secret cluster-ubuntu-k3s -n cicd -o jsonpath='{.data.config}' | base64 -d
```

The output must contain `bearerToken` — if it only shows `tlsClientConfig`, the old
secret is still in place. Delete and reapply:
```bash
kubectl --context k3d-k3d-cluster delete secret cluster-ubuntu-k3s -n cicd
ARGOCD_APP_CLUSTER_TOKEN="${TOKEN}" ./scripts/k3d-manager register_app_cluster
```

---

## Step 3 — Patch frontend destination

```bash
kubectl --context k3d-k3d-cluster patch application frontend -n cicd \
  --type merge -p '{"spec":{"destination":{"server":"https://host.k3d.internal:6443"}}}'
```

Verify:
```bash
kubectl --context k3d-k3d-cluster get application frontend -n cicd \
  -o jsonpath='{.spec.destination.server}'
```

Must return `https://host.k3d.internal:6443`.

---

## Step 4 — Force refresh and verify app sync

```bash
kubectl --context k3d-k3d-cluster annotate application -n cicd \
  basket-service order-service payment-service product-catalog frontend \
  argocd.argoproj.io/refresh=normal --overwrite
```

Wait 30 seconds, then check:
```bash
kubectl --context k3d-k3d-cluster get applications -n cicd
```

Expected: all apps show `Synced` or `OutOfSync` (not `Unknown`). `Unknown` means the
cluster connection still failed — recheck the bearer token in the secret.

## Step 5 — Run smoke test

Expected output:
