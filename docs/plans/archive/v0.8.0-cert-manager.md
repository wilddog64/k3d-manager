# v0.8.0: deploy_cert_manager Plugin

## Problem

CA/Browser Forum Ballot SC-081 compresses public TLS certificate lifetimes to
47 days by 2029. Manual renewal at that cadence is not viable. k3d-manager
already handles internal service mesh certs via Vault PKI. This plan adds
ACME-based auto-renewal for external-facing ingress certs via cert-manager +
Let's Encrypt.

---

## Two-Issuer Architecture

| Issuer | Handles | CA ownership |
|---|---|---|
| **Vault PKI** (existing) | Internal service mesh certs (Jenkins, ArgoCD, Keycloak, LDAP) | Owned by us — unchanged |
| **cert-manager + ACME** (new) | External-facing ingress certs | Let's Encrypt |

These two issuers are independent. `deploy_cert_manager` does not touch Vault PKI.

## HTTP-01 Challenge Routing (Istio)

cert-manager's ACME HTTP-01 solver creates a temporary Pod + Service for each
certificate request. Let's Encrypt sends an HTTP request to:

```
http://<domain>/.well-known/acme-challenge/<token>
```

The cluster uses Istio as the ingress controller — there is no separate nginx or
traefik. The `istio-ingressgateway` service exposes HTTP on port 80 (NodePort
31827 locally). For cert-manager to route the challenge through Istio:

- The `ClusterIssuer` solver must specify `ingressClassName: istio`
- This causes cert-manager to create temporary `Ingress` objects
- Istio's ingress controller converts those to `VirtualService` entries
- Let's Encrypt's verification request is routed to the acmesolver Pod

**Istio ingress class prerequisite:** Istio must be configured to watch and
process Kubernetes `Ingress` resources. The plugin must verify this at runtime
by checking whether the `istiod` deployment has `PILOT_ENABLE_INGRESS_GATEWAY_SDS`
or equivalent enabled, or by checking if `IngressClass` resource `istio` exists.
If Istio ingress support is not active, the plugin must print an actionable error
rather than silently deploying a ClusterIssuer that will never issue certificates.

**Prerequisite:** A shared HTTP Gateway must exist in `istio-system` with
`port 80 / HTTP` open. The plugin creates `cert-manager-http-gw` — a
lightweight Gateway dedicated to ACME challenge traffic. It is only used
during certificate issuance and has no impact on other services.

## Local Lab Constraint

| Environment | HTTP-01 | Notes |
|---|---|---|
| k3d / OrbStack (local) | Staging only | Domain not internet-reachable — Let's Encrypt cannot verify. Use staging issuer to validate plumbing. |
| Ubuntu k3s (externally reachable) | Staging + Production | Domain must resolve to the node's public IP on port 80. |
| Cloud (EKS/GKE/AKS — v1.0.0) | Staging + Production | Use cloud-native cert managers instead (ACM, GCP, Key Vault). |

The plugin deploys staging by default. Pass `--production` only when the cluster
is internet-accessible with a valid domain pointing at the ingress gateway.

## New Templates

All templates rendered via `envsubst` with an explicit variable whitelist —
same pattern as `deploy_argocd`. This prevents accidental substitution of
unrelated `$` patterns in YAML.

### `scripts/etc/cert-manager/clusterissuer-staging.yaml.tmpl`

envsubst whitelist: `$CERT_MANAGER_STAGING_ISSUER $ACME_STAGING_SERVER $ACME_EMAIL $ACME_INGRESS_CLASS`

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: ${CERT_MANAGER_STAGING_ISSUER}
spec:
  acme:
    server: ${ACME_STAGING_SERVER}
    email: ${ACME_EMAIL}
    privateKeySecretRef:
      name: letsencrypt-staging-account-key
    solvers:
      - http01:
          ingress:
            ingressClassName: ${ACME_INGRESS_CLASS}
```

### `scripts/etc/cert-manager/clusterissuer-production.yaml.tmpl`

envsubst whitelist: `$CERT_MANAGER_PRODUCTION_ISSUER $ACME_PRODUCTION_SERVER $ACME_EMAIL $ACME_INGRESS_CLASS`

```yaml

  acme:
    server: ${ACME_PRODUCTION_SERVER}
    email: ${ACME_EMAIL}
```

### `scripts/etc/cert-manager/http-gateway.yaml.tmpl`

envsubst whitelist: `$CERT_MANAGER_HTTP_GATEWAY $CERT_MANAGER_HTTP_GATEWAY_NS`

Dedicated Istio Gateway for ACME HTTP-01 challenge traffic. Accepts all hosts
on port 80 so it can route challenges for any domain cert-manager is issuing.

```yaml
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: ${CERT_MANAGER_HTTP_GATEWAY}
  namespace: ${CERT_MANAGER_HTTP_GATEWAY_NS}
spec:
  selector:
    istio: ingressgateway
  servers:
    - port:
        number: 80
        name: http
        protocol: HTTP
      hosts:
        - "*"
```

## Function: `deploy_cert_manager`

Location: `scripts/plugins/cert-manager.sh` (replace stub)

### Steps performed (in order)

1. **Source vars** — load `$SCRIPT_DIR/etc/cert-manager/vars.sh` first so defaults are available
2. **Validate** — `ACME_EMAIL` must be non-empty after vars are loaded; exit 1 with a clear message if still empty
3. **Helm install/upgrade** — `jetstack/cert-manager` with `--set crds.enabled=true` (installs CRDs as part of the chart, no separate kubectl apply needed in v1.15+)
4. **Wait for webhook** — `kubectl wait --for=condition=available deployment/cert-manager-webhook -n cert-manager --timeout=120s` (webhook must be ready before any CRD resources are applied)
5. **Skip steps 5-7 if `--skip-issuer`** — exit cleanly after step 4
6. **Check Istio ingress class** — verify `IngressClass` resource `istio` exists; exit 1 with actionable message if missing
7. **Create HTTP Gateway** — render and apply `http-gateway.yaml.tmpl` (idempotent)
8. **Apply ClusterIssuer** — staging by default; production if `--production` flag is set
9. **Verify ClusterIssuer** — `kubectl wait --for=condition=Ready clusterissuer/<name> --timeout=60s`
10. **Print next steps** — how to annotate an Ingress/Certificate to use the issuer

### STOP gates

- Exit 1 if `ACME_EMAIL` is empty
- Exit 1 if cert-manager webhook is not Ready within 120s
- Warn (do not exit) if `--production` is used with a local provider (`CLUSTER_PROVIDER=k3d` or `orbstack`) — print: "Production issuers require an internet-accessible domain. Use staging to validate plumbing locally."

## Annotating Ingress Resources

After `deploy_cert_manager`, annotate any Ingress (or create a `Certificate` CR
directly) to request a cert:

```yaml
# Option A — annotate an Ingress (cert-manager reads the annotation)
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-staging   # or letsencrypt-production

# Option B — explicit Certificate CR
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: my-service-tls
  namespace: istio-system
spec:
  secretName: my-service-tls
  issuerRef:
    name: letsencrypt-staging
    kind: ClusterIssuer
  dnsNames:
    - my-service.example.com
```

The resulting TLS secret can be referenced in an Istio Gateway's `credentialName`
exactly as the existing `jenkins-tls` secret is used today.

## Security Notes (OWASP)

- **A02 Cryptographic Failures:** Chart version pinned to `v1.20.0` — no floating `latest`. ACME account private key stored as a K8s secret, never exposed in logs or args.
- **A03 Injection:** `ACME_EMAIL` validated before being passed to `envsubst` — must match `[^@]+@[^@]+` pattern; fail if it contains shell metacharacters.
- **A05 Misconfiguration:** cert-manager deployed to its own `cert-manager` namespace. HTTP Gateway accepts all hosts (`"*"`) only on port 80 — no TLS termination at the gateway level (ACME challenges are plain HTTP by design).
- **A08 Supply Chain:** Helm chart pinned to `v1.20.0` tag. CRDs installed via `crds.enabled=true` (chart-managed, not from a raw URL).
