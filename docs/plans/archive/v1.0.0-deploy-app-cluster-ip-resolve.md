## Problem

`deploy_app_cluster` resolves the k3sup `--ip` argument as:

```bash
local external_ip="${UBUNTU_K3S_EXTERNAL_IP:-${ssh_host}}"
```

When `UBUNTU_K3S_EXTERNAL_IP` is not set, it falls back to the SSH host alias (`ubuntu`).
k3sup passes this value as both:
- The SSH connection target (`ssh ubuntu@<ip>`)
- The `server:` address written into the kubeconfig (`https://<ip>:6443`)

The SSH alias works for SSH (resolved via `~/.ssh/config` `HostName`), but it is written
literally into the kubeconfig. `kubectl --context ubuntu-k3s` then sends requests to
`https://ubuntu:6443` — not a valid address outside the SSH config, causing:

```
the server has asked for the client to provide credentials
```

`acg_provision` already writes the real public IP into `~/.ssh/config` via
`_acg_update_ssh_config`. The fix is for `deploy_app_cluster` to read the `HostName`
back from `~/.ssh/config` when `UBUNTU_K3S_EXTERNAL_IP` is not set.

Discovered during Gemini e2e smoke test (2026-03-29). Gemini worked around it by passing
`UBUNTU_K3S_EXTERNAL_IP` manually.

---

## Fix

After resolving `ssh_host`, add an IP lookup from `~/.ssh/config` before falling back to
the alias:

```bash
local external_ip="${UBUNTU_K3S_EXTERNAL_IP:-}"
if [[ -z "$external_ip" ]]; then
  external_ip=$(awk "/^Host ${ssh_host}\$/{found=1} found && /HostName/{print \$2; exit}" \
    "${HOME}/.ssh/config" 2>/dev/null)
fi
: "${external_ip:=${ssh_host}}"
```
