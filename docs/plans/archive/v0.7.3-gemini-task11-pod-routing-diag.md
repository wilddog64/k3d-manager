## Background

Tasks 8–10 ruled out TLS SAN, iptables/ufw, and MTU fragmentation. Two diagnostics from
Task 9 were never executed:

1. `curl` from inside the ArgoCD pod → Ubuntu API (Diagnostic A3)
2. `tcpdump` on Ubuntu to capture what source IP ArgoCD uses (Diagnostic B1)

**Hypothesis**: ArgoCD pod has a k3d overlay IP (e.g. `10.42.x.x`). When it connects to
Ubuntu at `10.211.55.14:6443`, Ubuntu receives the SYN with source IP `10.42.x.x`. Ubuntu
has no route back to the k3d CIDR — its default gateway is the Parallels router which
doesn't know about k3d overlay addresses. The SYN-ACK goes nowhere. Connection hangs.

This would explain why `curl` from the M2 Air host works (host IP is on the Parallels
network) but ArgoCD pod connections always time out.

## Task 3 — tcpdump: Capture Source IP Ubuntu Sees

Run tcpdump on Ubuntu in background, trigger a connection from ArgoCD, capture source IP.

```bash

ssh ubuntu "sudo tcpdump -i eth0 -n 'dst port 6443' -c 10 2>&1" &

# Trigger connection attempt from M2 Air
```

## Task 4 — Fix: Add Static Route on Ubuntu for k3d Pod CIDR

Only proceed if Task 3 shows k3d pod IP (`10.42.x.x`) as the source.

```bash
# Values from Task 1:
K3D_CIDR="<pod CIDR from Task 1>"       # e.g. 10.42.0.0/16
M2AIR_PARALLELS_IP="<M2 Air IP from Task 1>"  # e.g. 10.211.55.2

# Add route on Ubuntu
```

## Task 5 — Make Route Persistent (only if Task 4 succeeds)

A route added with `ip route add` is lost on reboot. Persist it:

```bash
# Check Ubuntu's netplan config location
ssh ubuntu "ls /etc/netplan/"

# Add persistent route via netplan
ssh ubuntu "sudo tee /etc/netplan/99-k3d-route.yaml > /dev/null <<'EOF'
network:
  version: 2
  ethernets:
    eth0:
      routes:
        - to: ${K3D_CIDR}
          via: ${M2AIR_PARALLELS_IP}
EOF"

ssh ubuntu "sudo netplan apply"
ssh ubuntu "ip route show | grep 10.42"
# Verify route still present after apply
```
