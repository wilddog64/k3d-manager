# Bugfix: v1.11.0 — `launchctl enable` before `bootstrap` in `_hostinger_restart_launchd`

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/lib/providers/k3s-hostinger.sh`

---

## Problem

When a k3dm LaunchAgent is left in launchd's **`disabled`** state (e.g. after an
interrupted bootout, a crash, or a manual `launchctl disable`), `_hostinger_restart_launchd`
cannot recover it. The function does `bootout` → `bootstrap`, but `bootstrap` on a
**disabled** label fails with `Bootstrap failed: 5: Input/output error`. The function
swallows this into a `restart failed` warning and continues, so `make refresh` silently
leaves the agent down.

This is the durable root cause behind the recurring Hostinger ESO outage: a laptop reboot
that leaves `com.k3d-manager.ssh-tunnel` `disabled` breaks the hub-Vault bridge, and the
provider's own refresh path cannot heal it — an operator must manually run
`launchctl enable` first (as was required during the 2026-06-28 incident recovery).

**Root cause:** `_hostinger_restart_launchd` never calls `launchctl enable`, so it cannot
clear a `disabled` state before `bootstrap`.

---

## Reproduction

```bash
launchctl disable "gui/$(id -u)/com.k3d-manager.ssh-tunnel"
# now drive a refresh that calls _hostinger_restart_launchd for that label
# expected: agent comes back up
# actual:   "launchd com.k3d-manager.ssh-tunnel: restart failed" (Bootstrap failed: 5)
launchctl print-disabled "gui/$(id -u)" | grep ssh-tunnel   # => "...ssh-tunnel" => disabled
```

---

## Fix

### Change 1 — `scripts/lib/providers/k3s-hostinger.sh`: enable before bootstrap (both domains)

**Exact old block (lines 222-238):**

```bash
  if [[ "${domain}" == "system" ]]; then
    _run_command --interactive-sudo --quiet --soft -- launchctl bootout system "${plist}" >/dev/null 2>&1 || true
    if [[ "${label}" == "com.k3d-manager.argocd-port-forward" ]]; then
      _hostinger_wait_for_port_free 8080 30
    fi
    _run_command --interactive-sudo --quiet --soft -- launchctl bootstrap system "${plist}" >/dev/null 2>&1 \
      && _info "[k3s-hostinger] launchd ${label}: restarted" \
      || _warn "[k3s-hostinger] launchd ${label}: restart failed"
  else
    launchctl bootout "gui/$(id -u)" "${plist}" >/dev/null 2>&1 || true
    if [[ "${label}" == "com.k3d-manager.argocd-port-forward" ]]; then
      _hostinger_wait_for_port_free 8080 30
    fi
    launchctl bootstrap "gui/$(id -u)" "${plist}" >/dev/null 2>&1 \
      && _info "[k3s-hostinger] launchd ${label}: restarted" \
      || _warn "[k3s-hostinger] launchd ${label}: restart failed"
  fi
```

**Exact new block:**

```bash
  if [[ "${domain}" == "system" ]]; then
    _run_command --interactive-sudo --quiet --soft -- launchctl bootout system "${plist}" >/dev/null 2>&1 || true
    if [[ "${label}" == "com.k3d-manager.argocd-port-forward" ]]; then
      _hostinger_wait_for_port_free 8080 30
    fi
    _run_command --interactive-sudo --quiet --soft -- launchctl enable "system/${label}" >/dev/null 2>&1 || true
    _run_command --interactive-sudo --quiet --soft -- launchctl bootstrap system "${plist}" >/dev/null 2>&1 \
      && _info "[k3s-hostinger] launchd ${label}: restarted" \
      || _warn "[k3s-hostinger] launchd ${label}: restart failed"
  else
    launchctl bootout "gui/$(id -u)" "${plist}" >/dev/null 2>&1 || true
    if [[ "${label}" == "com.k3d-manager.argocd-port-forward" ]]; then
      _hostinger_wait_for_port_free 8080 30
    fi
    launchctl enable "gui/$(id -u)/${label}" >/dev/null 2>&1 || true
    launchctl bootstrap "gui/$(id -u)" "${plist}" >/dev/null 2>&1 \
      && _info "[k3s-hostinger] launchd ${label}: restarted" \
      || _warn "[k3s-hostinger] launchd ${label}: restart failed"
  fi
```

`launchctl enable` takes a **service target** (`system/<label>` or `gui/<uid>/<label>`),
not a plist path — distinct from `bootstrap`, which takes a domain + plist path. The
`|| true` keeps it non-fatal (enable on an already-enabled label is a no-op success;
on a not-yet-known label it harmlessly errors).
