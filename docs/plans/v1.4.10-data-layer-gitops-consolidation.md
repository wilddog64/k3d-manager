# Plan: Data Layer GitOps Consolidation

## Objective
Transition the deployment and management of stateful data services (PostgreSQL, Redis, RabbitMQ, MinIO) from imperative Bash scripts (`scripts/plugins/shopping_cart.sh`) to declarative GitOps management using ArgoCD.

## Background & Motivation
Currently, the data layer resources are deployed by running explicit `kubectl apply` commands inside the `acg-up` bootstrap sequence. This approach has several drawbacks:
1.  **Lack of Visibility:** ArgoCD has no knowledge of these resources, meaning they do not appear in the UI, and their health cannot be monitored natively.
2.  **Drift:** If a data layer resource is modified or deleted manually, ArgoCD cannot self-heal it.
3.  **Imperative Technical Debt:** The Bash plugins are bloated with deployment logic that should be handled by standard Kubernetes controllers.

By registering the `data-layer/` directory as an ArgoCD Application, we consolidate infrastructure management, gain UI visibility, and decouple infrastructure creation from imperative state configuration (like DB password reconciliation).

## Key Files & Context
*   `scripts/plugins/shopping_cart.sh`: Contains the imperative deployment logic that needs to be removed.
*   `shopping-cart-infra` repository: Contains the `data-layer/` directory and ArgoCD application definitions.
*   `argocd/applications/data-layer.yaml` (in `shopping-cart-infra`): Will be the master registration file for this tier.

## Implementation Steps

### Phase 1: Establish the Data Layer Application (shopping-cart-infra)
Ensure the `data-layer` Application is fully defined in the infrastructure repository.
1.  Verify/Create `argocd/applications/data-layer.yaml` to point to the `data-layer/` path.
2.  **Safety Gate:** Ensure `syncPolicy.automated.prune` is explicitly set to `false`. This prevents ArgoCD from accidentally deleting stateful resources (like PVCs) if a folder is temporarily missing from Git.
3.  Ensure MinIO, RabbitMQ, Redis, and PostgreSQL manifests are all correctly organized under the `data-layer/` root.

### Phase 2: Refactor Bootstrap Logic (k3d-manager)
Remove the imperative deployment commands from the Bash plugin, delegating the responsibility to ArgoCD.
1.  Modify `scripts/plugins/shopping_cart.sh` to **remove** the `kubectl apply -f` commands for orders, payment, products, redis, rabbitmq, and minio.
2.  Modify the `deploy_argocd_bootstrap` function (or equivalent location) to ensure the `data-layer.yaml` application is applied alongside the App of Apps.
3.  **Keep the Readiness Waits:** The Bash script must *still* execute `kubectl rollout status` for these databases to ensure they are fully running before it attempts imperative tasks like password reconciliation or seeding.

## Verification & Testing
1.  **Cold Run:** Execute `make down` followed by a full `make up`.
2.  **ArgoCD Visibility:** Open the ArgoCD UI and verify the appearance of a new `data-layer` application card.
3.  **Health Check:** Ensure MinIO, all PostgreSQL instances, Redis, and RabbitMQ appear within the `data-layer` card and report as `Synced` and `Healthy`.
4.  **Integration:** Verify that downstream services (`order-service`, `payment-service`) still connect successfully, proving the data layer was provisioned correctly before they started.

## Migration & Rollback Strategy
*   **Migration:** Because the resources already exist in the cluster with matching configurations, ArgoCD will seamlessly "adopt" the live resources without restarting them or deleting their data volumes.
*   **Rollback:** Revert the commits in both repositories to restore the `kubectl apply` lines in the Bash script and remove the `data-layer.yaml` registration.
