# Task: Fix ImagePullBackOff — Shopping Cart Services

**Assignee:** Codex
**Repo scope:** shopping-cart-basket, shopping-cart-product-catalog, shopping-cart-order,
shopping-cart-payment, shopping-cart-frontend, shopping-cart-infra

---

## Root Cause Diagnosis

Two distinct problems found:

### Problem 1 — Missing `newName` in kustomization `images:` blocks

The reusable `build-push-deploy.yml` workflow pushes images to ghcr.io but only updates
`newTag` via sed. It never sets `newName`. Without `newName`, kustomize applies the tag to
the bare image name in `deployment.yaml` (e.g. `shopping-cart-basket:sha-abc123`) instead
of the fully-qualified registry path (`ghcr.io/wilddog64/shopping-cart-basket:sha-abc123`).
The pod then tries to pull from Docker Hub and fails.

**Affected repos:**
| Repo | Current kustomization images block | Missing |
|---|---|---|
| basket | `name: shopping-cart-basket` + correct SHA newTag | `newName: ghcr.io/wilddog64/shopping-cart-basket` |
| frontend | `name: shopping-cart/frontend` + `newTag: latest` | `newName: ghcr.io/wilddog64/shopping-cart-frontend` |
| product-catalog | no images block at all | entire images block |
| order | no images block at all | entire images block |
| payment | no images block at all | entire images block |

### Problem 2 — `build-push-deploy.yml` sed only updates `newTag`, never `newName`

The reusable workflow in `shopping-cart-infra` does:
```bash
sed -i "s|newTag:.*|newTag: sha-${{ github.sha }}|" k8s/base/kustomization.yaml
```
This can never set `newName`. Fix: add a second sed (or use yq) to set `newName` if it
does not already exist, or rely on the static `newName` being pre-set in each repo and
the workflow only updating `newTag`.

**Recommended approach:** Static `newName` in each repo's kustomization (set once, never
overwritten by CI). CI only updates `newTag`. Simpler and more reliable.
