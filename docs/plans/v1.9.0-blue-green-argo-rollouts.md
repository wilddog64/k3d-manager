# v1.9.0 — Blue/green deployment on OCI via Argo Rollouts + k6 stress testing

## Context

OCI is the permanent production cluster. When shopping-cart apps run there, deployments
need a safe promotion gate: spin up green, run a k6 stress test against it, promote only
if error rate and latency are within bounds — otherwise auto-abort and stay on blue.

**Decision recorded:** 2026-05-29

### Why Argo Rollouts over plain Istio split

Argo Rollouts integrates natively with ArgoCD (visible in the UI, controllable via
`kubectl argo rollouts` or the dashboard), provides `AnalysisTemplate` CRDs that run
k6 as a Kubernetes Job during pre-promotion, and works with Istio for traffic mirroring.
Flagger is heavier and Istio-only; plain Istio split requires manual weight management.

### Blue/green strategy (not canary)

```
Push new image tag → ArgoCD syncs Rollout CRD
  → Rollouts controller creates green ReplicaSet (inactive)
  → preview-service selector points to green pods
  → AnalysisTemplate fires:
      1. k6 Job hits preview-service (in-cluster)
         checks: throughput > 50 RPS, error rate < 1%, p95 latency < 500ms
      2. Prometheus query: http_server_errors_total rate < 0.01
  → PASS → active-service selector flips to green; blue RS scaled to 0
  → FAIL → abort; green RS deleted; blue stays active; alert sent
```

No Istio traffic splitting needed for blue/green — Rollouts handles promotion via
service selector swap. Istio traffic mirroring (copy of production traffic to green)
is an optional enhancement noted below but not required for v1.9.0.

**Prerequisites:**
- OCI cluster live (`CLUSTER_PROVIDER=k3s-oci`; v1.5.0)
- kube-prometheus-stack on OCI (v1.7.0) — provides Prometheus metrics for AnalysisTemplate
- ArgoCD on OCI (v1.5.0 platform-helm)

## File 1: `scripts/etc/argocd/applicationsets/rollouts.yaml`

Deploy Argo Rollouts controller only to `environment: prod` (OCI):

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: rollouts
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: platform
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            environment: prod
  goTemplate: true
  template:
    metadata:
      name: rollouts
      labels:
        cluster: '{{.name}}'
        environment: prod
    spec:
      project: platform
      destination:
        server: '{{.server}}'
        namespace: argo-rollouts
      source:
        repoURL: https://argoproj.github.io/argo-helm
        chart: argo-rollouts
        targetRevision: '${ARGO_ROLLOUTS_CHART_VERSION}'
        helm:
          releaseName: argo-rollouts
          parameters:
            - name: controller.replicas
              value: "1"
            - name: dashboard.enabled
              value: "true"
            - name: dashboard.service.type
              value: ClusterIP
      syncPolicy:
        automated:
          prune: false
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true
```

Pin `ARGO_ROLLOUTS_CHART_VERSION` to latest ARM64-compatible chart at deploy time

## File 2: `scripts/etc/rollouts/analysis-template-k6.yaml`

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: k6-stress-test
  namespace: "{{ROLLOUT_NAMESPACE}}"
  labels:
    managed-by: k3d-manager
spec:
  args:
    - name: preview-service
    - name: namespace
    - name: duration
      value: "60s"
    - name: vus
      value: "20"

  metrics:
    # Metric 1: k6 Job must exit 0 (all thresholds pass)
    - name: k6-load-test
      provider:
        job:
          spec:
            template:
              spec:
                restartPolicy: Never
                containers:
                  - name: k6
                    image: grafana/k6:0.56.0
                    args:
                      - run
                      - --vus
                      - "{{args.vus}}"
                      - --duration
                      - "{{args.duration}}"
                      - --env
                      - TARGET_URL=http://{{args.preview-service}}.{{args.namespace}}.svc.cluster.local
                      - /scripts/stress-test.js
                    volumeMounts:
                      - name: k6-script
                        mountPath: /scripts
                volumes:
                  - name: k6-script
                    configMap:
                      name: k6-stress-script
              metadata:
                labels:
                  app: k6-analysis
            backoffLimit: 0
      successCondition: result == "Succeeded"
      failureLimit: 0

    # Metric 2: Prometheus — HTTP error rate on green must be < 1%
    - name: prometheus-error-rate
      provider:
        prometheus:
          address: http://kube-prometheus-stack-prometheus.monitoring.svc:9090
          query: |
            sum(rate(http_server_requests_seconds_count{
              service="{{args.preview-service}}",
              namespace="{{args.namespace}}",
              status=~"5.."
            }[2m])) /
            sum(rate(http_server_requests_seconds_count{
              service="{{args.preview-service}}",
              namespace="{{args.namespace}}"
            }[2m]))
      successCondition: result[0] < 0.01
      failureLimit: 1
      interval: 30s
      count: 3
```

## File 3: `scripts/etc/rollouts/k6-stress-script.js`

Stored as a ConfigMap (`k6-stress-script`) in the rollout namespace:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

const errorRate = new Rate('errors');
const latencyP95 = new Trend('latency_p95', true);

export const options = {
  thresholds: {
    errors:       ['rate<0.01'],           // < 1% error rate
    http_req_duration: ['p(95)<500'],      // p95 latency < 500ms
    http_reqs:    ['rate>50'],             // throughput > 50 RPS
  },
};

const TARGET = __ENV.TARGET_URL || 'http://localhost:8080';

export default function () {
  // Health check
  const health = http.get(`${TARGET}/actuator/health`);
  check(health, { 'health 200': (r) => r.status === 200 });
  errorRate.add(health.status !== 200);

  // Main endpoint
  const res = http.get(`${TARGET}/`);
  check(res, {
    'status 2xx': (r) => r.status >= 200 && r.status < 300,
    'latency ok': (r) => r.timings.duration < 500,
  });
  errorRate.add(res.status >= 400);
  latencyP95.add(res.timings.duration);

  sleep(0.1);
}
```

Deploy the ConfigMap as part of `deploy_rollouts`:

```bash
kubectl create configmap k6-stress-script \
  --from-file=stress-test.js="${_dir}/k6-stress-script.js" \
  --namespace "${_namespace}" \
  --dry-run=client -o yaml | _kubectl apply -f -
```

## File 4: `scripts/etc/rollouts/rollout-bluegreen.yaml.tmpl`

Template for converting a shopping-cart app Deployment to a Rollout.
`envsubst` fills in `APP_NAME`, `NAMESPACE`, `IMAGE`, `PORT`:

```yaml
# Two Services required: active (stable/blue) + preview (green)
---
apiVersion: v1
kind: Service
metadata:
  name: ${APP_NAME}
  namespace: ${NAMESPACE}
  labels:
    app: ${APP_NAME}
    managed-by: k3d-manager
spec:
  selector:
    app: ${APP_NAME}
  ports:
    - port: ${PORT}
      targetPort: ${PORT}
---
apiVersion: v1
kind: Service
metadata:
  name: ${APP_NAME}-preview
  namespace: ${NAMESPACE}
  labels:
    app: ${APP_NAME}
    managed-by: k3d-manager
spec:
  selector:
    app: ${APP_NAME}
  ports:
    - port: ${PORT}
      targetPort: ${PORT}
---
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: ${APP_NAME}
  namespace: ${NAMESPACE}
  labels:
    app: ${APP_NAME}
    managed-by: k3d-manager
spec:
  replicas: 2
  selector:
    matchLabels:
      app: ${APP_NAME}
  template:
    metadata:
      labels:
        app: ${APP_NAME}
    spec:
      containers:
        - name: ${APP_NAME}
          image: ${IMAGE}
          ports:
            - containerPort: ${PORT}
          readinessProbe:
            httpGet:
              path: /actuator/health
              port: ${PORT}
            initialDelaySeconds: 10
            periodSeconds: 5
  strategy:
    blueGreen:
      activeService: ${APP_NAME}
      previewService: ${APP_NAME}-preview
      autoPromotionEnabled: false
      prePromotionAnalysis:
        templates:
          - templateName: k6-stress-test
        args:
          - name: preview-service
            value: ${APP_NAME}-preview
          - name: namespace
            value: ${NAMESPACE}
          - name: duration
            value: "120s"
          - name: vus
            value: "20"
      postPromotionAnalysis:
        templates:
          - templateName: k6-stress-test
        args:
          - name: preview-service
            value: ${APP_NAME}
          - name: namespace
            value: ${NAMESPACE}
          - name: duration
            value: "30s"
          - name: vus
            value: "10"
      scaleDownDelaySeconds: 30
```

**`autoPromotionEnabled: false`** — promotion requires an explicit command after pre-analysis
passes. To enable fully-automated promotion (analysis passes → promote without human step),
change to `true`. For stress testing, keep `false` so you can inspect green before
committing.

## File 5: `scripts/plugins/rollouts.sh`

```bash
#!/usr/bin/env bash
# rollouts.sh — Argo Rollouts management on OCI

function deploy_rollouts() {
  local _dir="${SCRIPT_DIR}/etc/rollouts"
  local _ns="${ROLLOUTS_NAMESPACE:-shopping-cart}"

  _info "[rollouts] Deploying k6-stress-script ConfigMap..."
  _kubectl create configmap k6-stress-script \
    --from-file=stress-test.js="${_dir}/k6-stress-script.js" \
    --namespace "${_ns}" \
    --dry-run=client -o yaml | _kubectl apply -f -

  _info "[rollouts] Deploying AnalysisTemplate..."
  _kubectl apply -f "${_dir}/analysis-template-k6.yaml" \
    --namespace "${_ns}"

  _info "[rollouts] Rollouts controller deployed via ApplicationSet (ArgoCD manages chart)"
  _info "[rollouts] Dashboard: https://rollouts.3ai-talk.org"
}

function rollout_status() {
  local _app="${1:?app name required}"
  local _ns="${2:-shopping-cart}"
  _kubectl argo rollouts status "${_app}" --namespace "${_ns}" --watch=false \
    --context "${OCI_CONTEXT:-}"
}

function rollout_promote() {
  local _app="${1:?app name required}"
  local _ns="${2:-shopping-cart}"
  _info "[rollouts] Promoting ${_app} in ${_ns}..."
  _kubectl argo rollouts promote "${_app}" --namespace "${_ns}" \
    --context "${OCI_CONTEXT:-}"
}

function rollout_abort() {
  local _app="${1:?app name required}"
  local _ns="${2:-shopping-cart}"
  _info "[rollouts] Aborting ${_app} in ${_ns} — rolling back to blue..."
  _kubectl argo rollouts abort "${_app}" --namespace "${_ns}" \
    --context "${OCI_CONTEXT:-}"
}
```

## Migration: Deployment → Rollout

For each shopping-cart app moving to OCI:

1. Delete the existing `Deployment` (ArgoCD will recreate as `Rollout`):
   ```bash
   kubectl delete deployment <app> -n <namespace>
   ```

2. In the shopping-cart-infra kustomize config, replace:
   ```yaml
   # k8s/base/deployment.yaml → k8s/base/rollout.yaml
   kind: Deployment  →  kind: Rollout
   apiVersion: apps/v1  →  apiVersion: argoproj.io/v1alpha1
   ```
   Add the `strategy.blueGreen` block from the template above.

3. Add the preview Service to the app's kustomize base.

4. ArgoCD syncs → Rollouts controller manages subsequent deployments.

This migration is a shopping-cart-infra change — write a spec in that repo when ready.
The k3d-manager pieces (controller, AnalysisTemplate, k6 script) are self-contained here.

## Optional Enhancement: Istio Traffic Mirroring (post-v1.9.0)

To send a shadow copy of live production traffic to green during analysis (without
affecting users):

```yaml
# VirtualService addition — add mirror block
http:
  - route:
      - destination:
          host: ${APP_NAME}
    mirror:
      host: ${APP_NAME}-preview
    mirrorPercentage:
      value: 100
```

This lets k6 AND real traffic hit green simultaneously. Useful for catching bugs that
synthetic load doesn't reproduce. Defer to v1.10.0 — requires Istio sidecar injection
on app pods.
