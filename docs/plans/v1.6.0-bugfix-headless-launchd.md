# Bugfix: v1.6.0 — headless LaunchDaemon exits + cloudflared privilege escalation

**Branch:** `k3d-manager-v1.6.0`
**Files:** `bin/acg-up`, `bin/acg-down`

---

## Problem

`bin/acg-up` exits 1 when invoked from the webhook LaunchAgent context (no TTY, no cached
sudo credential). Three browser listener install blocks call `_err` → `exit 1` when
`install -m 644 … /Library/LaunchDaemons/` fails with "Permission denied".

`bin/acg-down` surfaces "Boot-out failed: 1: Operation not permitted" to Slack stdout/stderr
because it tries to `launchctl bootout system` as an unprivileged user and the output is
tailed to stderr even though the failure is soft.

`cloudflared` has no reason to be a system LaunchDaemon — it only dials outbound (no
privileged port binding). It should be a user LaunchAgent to avoid needing root at all.

**Root cause:** `_run_command --prefer-sudo` probes `sudo -n true`; in the webhook
LaunchAgent context (no TTY, no NOPASSWD sudoers), that probe always fails, so privileged
commands fall back to running as the user → `/Library/LaunchDaemons/` permission denied.

---

## Reproduction

```
# Trigger via Slack /acg-up command (runs headlessly as LaunchAgent):
# → Boot-out failed: 1: Operation not permitted
# → WARN: [acg-up] existing ArgoCD browser HTTPS listener was not loaded; continuing
# → make: *** [up] Error 1
```
