## Problem

ESO (External Secrets Operator) is installed **imperatively** by `deploy_eso` (Helm, run
from the operator's machine). It runs on the hub but was **never bootstrapped on the
`ubuntu-hostinger` app cluster**, so:

```
ExternalSecret/product-catalog-secrets: SyncFailed
  The Kubernetes API could not find external-secrets.io/ExternalSecret ...
  Make sure the "ExternalSecret" CRD is installed on the destination cluster.
```

Consequence on the app cluster: ESO CRDs absent → ExternalSecrets never reconcile →
`payment-db-credentials` etc. never created → order/payment `CreateContainerConfigError`,
product-catalog `Missing`, frontend/basket `CrashLoopBackOff`.

The imperative path is also **OS-coupled**: it runs Helm from the local CLI, so it
depends on the operator's host toolchain and must be re-run per cluster.

## Goal

Install the **ESO operator + CRDs declaratively via an ArgoCD ApplicationSet cluster
generator**, so every cluster labeled `k3d-manager/role: app-cluster` gets ESO installed
in-cluster by ArgoCD. This is portable across clusters and host OS (install happens
in-cluster, not from the local CLI) and across CPU arch (external-secrets ships
multi-arch images). Mirrors the existing `platform-helm` generator that installs
`argo-cd` per cluster.

## Scope

**IN scope (this spec):** the ESO **operator + CRDs** ApplicationSet only.
**OUT of scope (Phase 2, separate spec):** the `ClusterSecretStore` + Vault connectivity
+ per-cluster Vault auth. ESO has no usable backend until Phase 2; that is expected and
fine — this spec removes the "CRD not installed" failure and makes ESO present on app
clusters. See "Phase 2 — next" at the end. Do NOT implement Phase 2 here.

## Change — NEW FILE `scripts/etc/argocd/applicationsets/eso.yaml`

Create the file with exactly this content:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: eso
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: platform
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            k3d-manager/role: app-cluster
  goTemplate: true
  goTemplateOptions:
    - missingkey=error
  template:
    metadata:
      name: '{{.name}}-eso'
      labels:
        cluster: '{{.name}}'
        app-type: platform
    spec:
      project: platform
      destination:
        server: '{{.server}}'
        namespace: secrets
      source:
        repoURL: https://charts.external-secrets.io
        chart: external-secrets
        targetRevision: 1.0.0
        helm:
          releaseName: external-secrets
          parameters:
            - name: installCRDs
              value: 'true'
      syncPolicy:
        automated:
          prune: false
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true
```

**Why `ServerSideApply=true`:** the external-secrets CRDs are large; client-side apply
can fail with `metadata.annotations: Too long`. Server-side apply avoids this. (Same
reason kube-prometheus-stack uses it.)

## Phase 2 — next (separate spec; NOT this task)

To make ESO actually deliver secrets on app clusters:
1. **ClusterSecretStore via GitOps** — a second ApplicationSet (or git generator) applying
   a `ClusterSecretStore` per app cluster, templated from cluster-secret label values.
   The shapes already exist in `eso.sh` (`_eso_apply_vault_cluster_store` /
   `_eso_configure_remote_vault` — `REMOTE_VAULT_ADDR`, `kubernetes-app` mount,
   `eso-app-cluster` role). Convert these to value-templated GitOps manifests.
2. **External Vault endpoint** — the hub Vault is in-cluster (`vault.secrets.svc:8200`),
   unreachable from app clusters. Expose Vault via the existing tunnel/ingress, TLS only
   (no `--insecure`); feed the URL through generator `values`/labels as `REMOTE_VAULT_ADDR`.
3. **Per-cluster Vault auth registration** — register each app cluster's Kubernetes auth
   (`kubernetes-app` mount, `eso-app-cluster` role bound to the cluster's JWT issuer/CA)
   in the hub Vault. This is the one irreducible non-GitOps step; wire it into
   `k3s-hostinger` provider registration (cf. the JWT-issuer prior art in
   `docs/bugs/2026-05-17-keycloak-jwt-issuer-mismatch-app-cluster.md`). Prefer k8s-auth
   (no secret-zero) over AppRole. Least-privilege Vault policy per app cluster.
