## Context

v1.0.1 shipped a 3-node CloudFormation cluster (1 server + 2 agents, t3.medium each).
This resolves the resource exhaustion that caused `frontend`, `payment-service`, and
`order-service` to fail on the single-node cluster.

Four services are currently not Running. Root causes are known:

| Service | Namespace | Root Cause | Known Fix |
|---------|-----------|------------|-----------|
| `frontend` | `apps` | Resource exhaustion (t3.medium single node) | 3-node cluster now available — reschedule |
| `order-service` | `apps` | RabbitMQ `Connection refused` | Verify RabbitMQ pod Running; check service DNS; check `RABBITMQ_HOST` env var |
| `payment-service` | `payment` | Memory constraints | 3-node cluster — reschedule; check resource requests |
| `product-catalog` | `apps` | ESO key mismatch: `RABBITMQ_USERNAME` vs `RABBITMQ_USER` | Check ESO SecretStore sync; verify secret key name in ESO mapping |

## Step-by-Step

### Step 1 — Refresh credentials and verify cluster

```bash
# Get fresh AWS credentials
./scripts/k3d-manager acg_get_credentials <sandbox-url>

# Verify 3 nodes Ready
kubectl get nodes --context ubuntu-k3s

# Verify tunnel up
./scripts/k3d-manager tunnel_status
# If down: ./scripts/k3d-manager tunnel_start
```

### Step 2 — Check current pod state

```bash
kubectl get pods -A --context ubuntu-k3s
kubectl get pods -n apps --context ubuntu-k3s
kubectl get pods -n payment --context ubuntu-k3s
kubectl get pods -n data --context ubuntu-k3s
```

Note which pods are `Running`, `CrashLoopBackOff`, `Pending`, or `Error`.

### Step 3 — Fix frontend (resource exhaustion)

With 3 nodes available, frontend should reschedule automatically. If still Pending:

```bash
kubectl describe pod -n apps -l app=frontend --context ubuntu-k3s | grep -A 10 Events
# If FailedScheduling: check node capacity
kubectl describe nodes --context ubuntu-k3s | grep -A 5 "Allocated resources"
```

If still failing — check read-only filesystem issue:
```bash
kubectl logs -n apps deploy/frontend --context ubuntu-k3s
# If "Read-only file system": nginx needs writable /tmp and /var/cache/nginx
# Fix: patch deployment to add emptyDir volumes for /tmp and /var/cache/nginx
```

### Step 4 — Fix order-service (RabbitMQ)

```bash
# Check RabbitMQ pod
kubectl get pods -n data --context ubuntu-k3s | grep rabbit

# Check order-service logs
kubectl logs -n apps deploy/order-service --context ubuntu-k3s | tail -30

# Verify RabbitMQ DNS resolves from order-service namespace
kubectl exec -n apps deploy/order-service --context ubuntu-k3s -- nslookup rabbitmq.data.svc.cluster.local

# Check RABBITMQ_HOST env var
kubectl exec -n apps deploy/order-service --context ubuntu-k3s -- env | grep RABBIT
```

If `Connection refused` — verify RabbitMQ service port and that the service selector matches:
```bash
kubectl get svc -n data --context ubuntu-k3s
kubectl get endpoints -n data --context ubuntu-k3s
```

### Step 5 — Fix payment-service (memory)

```bash
kubectl describe pod -n payment -l app=payment-service --context ubuntu-k3s | grep -A 5 "Limits\|Requests\|OOMKilled"

# If OOMKilled: patch memory limit up temporarily
kubectl patch deployment payment-service -n payment --context ubuntu-k3s \
  --patch '{"spec":{"template":{"spec":{"containers":[{"name":"payment-service","resources":{"limits":{"memory":"512Mi"},"requests":{"memory":"256Mi"}}}]}}}}'
```

If a manifest change is needed — commit it to `shopping-cart-payment` on the `docs/next-improvements` branch and push; ArgoCD will sync.

### Step 6 — Fix product-catalog (ESO key mismatch)

```bash
# Check what key ESO is syncing
kubectl get externalsecret -n apps --context ubuntu-k3s -o yaml | grep -A 5 rabbit

# Check what the pod sees
kubectl exec -n apps deploy/product-catalog --context ubuntu-k3s -- env | grep RABBIT

# If RABBITMQ_USER vs RABBITMQ_USERNAME mismatch:
# Fix in shopping-cart-infra ESO SecretStore mapping or in the app manifest
```

### Step 7 — Force ArgoCD sync

```bash
argocd app sync order-service
argocd app sync product-catalog
argocd app sync payment
argocd app sync frontend
argocd app list
```

### Step 8 — Verify Istio sidecar injection

```bash
kubectl get pods -n apps --context ubuntu-k3s -o wide
# Each pod should show 2/2 READY (app container + istio-proxy)

# If pods show 1/1 — sidecar injection may be disabled for the namespace
kubectl get namespace apps --context ubuntu-k3s --show-labels | grep istio-injection
# If missing: kubectl label namespace apps istio-injection=enabled --context ubuntu-k3s
# Then restart pods: kubectl rollout restart deployment -n apps --context ubuntu-k3s
```

### Step 9 — Re-enable shopping-cart-e2e-tests

```bash
# Check current state
kubectl get cronjob -n apps --context ubuntu-k3s | grep e2e

# Re-enable if suspended
kubectl patch cronjob shopping-cart-e2e-tests -n apps --context ubuntu-k3s \
  --patch '{"spec":{"suspend":false}}'

# Trigger a manual run to verify
kubectl create job --from=cronjob/shopping-cart-e2e-tests e2e-manual-$(date +%s) -n apps --context ubuntu-k3s
kubectl logs -n apps job/e2e-manual-<id> --context ubuntu-k3s -f
```

If E2E is not ready (upstream app bugs still open) — document reason and defer explicitly.
