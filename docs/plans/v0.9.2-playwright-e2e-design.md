# Design: vCluster + Playwright E2E Workflow

## Status: DESIGN — awaiting approval before implementation

---

## Problem

Shopping-cart services are deployed and ArgoCD-synced but untested end-to-end.
No automated test verifies that the full stack (frontend → basket → order →
product-catalog → payment) actually works together. `ImagePullBackOff` has blocked
live testing; once images land in ghcr.io, there is no test harness to run.

---

## Goals

1. Ephemeral, isolated E2E environment per CI run — no shared cluster state between runs.
2. Playwright tests cover the user-facing shopping flow (browse → add to cart → checkout).
3. AI-assisted test authoring — Playwright MCP for interactive exploration, Codex for
   implementation.
4. CI runs headlessly in GitHub Actions — no human in the loop.

---

## Architecture Decision: Where Does vCluster Run?

### Option A — k3d inside GitHub Actions → vCluster on top
- Self-contained: no external cluster dependency
- GH Actions runner provisions k3d, creates vCluster, deploys apps, runs tests, tears down
- No M2 Air / Ubuntu k3s dependency in CI
- Adds ~3-5 min to CI per run (k3d + vCluster startup)

### Option B — Live Ubuntu k3s + ephemeral vCluster per run
- Faster startup (host cluster already running)
- Ties CI to physical machine availability — fragile
- Requires self-hosted runner on M2 Air

**Decision: Option A — k3d in GH Actions runner.**
Matches existing BATS CI pattern. Self-contained. No machine dependency.

---

## Architecture Decision: Secrets in vCluster

The live shopping-cart stack uses ExternalSecrets + Vault for all credentials.
Replicating Vault inside a vCluster for every CI run is too heavy.

**Decision: Static test secrets injected as Kubernetes Secrets directly.**
- CI creates `kubectl create secret` commands from env vars (GitHub Actions secrets)
- No ESO, no Vault needed in the vCluster
- Test credentials are pre-defined, scoped to the vCluster lifetime

---

## Architecture Decision: Service Connectivity for Playwright

vCluster services (ClusterIP) are not reachable from outside without extra work.

Options:
- `kubectl port-forward` — simple, works for CI, no ingress needed
- NodePort — requires vCluster NodePort exposure config
- Ingress inside vCluster — heavy, needs ingress controller

**Decision: `kubectl port-forward` per service.**
Frontend on `localhost:3000`. No ingress controller needed in vCluster.

## Full CI Flow

```
GitHub Actions Runner
│
├── 1. Provision k3d cluster (single-node, lightweight)
│
├── 2. Create vCluster via k3d-manager
│      ./scripts/k3d-manager vcluster_create e2e-${{ github.run_id }}
│
├── 3. Switch context to vCluster
│      ./scripts/k3d-manager vcluster_use e2e-${{ github.run_id }}
│
├── 4. Create namespaces + static test secrets
│      kubectl create namespace shopping-cart-apps
│      kubectl create namespace shopping-cart-data
│      kubectl create namespace shopping-cart-payment
│      kubectl create namespace identity
│      kubectl create secret generic ... (from GH Actions secrets)
│
├── 5. Deploy Keycloak (dev mode, realm import — no PostgreSQL needed)
│      kubectl apply -k identity/keycloak/ci/     # lightweight dev-mode overlay
│      kubectl wait --for=condition=Ready pod -l app=keycloak -n identity --timeout=120s
│
├── 6. Deploy shopping-cart services
│      kubectl apply -k shopping-cart-frontend/k8s/base   (VITE_KEYCLOAK_URL=http://localhost:8080)
│      kubectl apply -k shopping-cart-basket/k8s/base
│      kubectl apply -k shopping-cart-product-catalog/k8s/base
│      kubectl apply -k shopping-cart-order/k8s/base
│      kubectl apply -k shopping-cart-payment/k8s/base
│      kubectl wait --for=condition=Ready pod --all -n shopping-cart-apps --timeout=120s
│
├── 7. Port-forward (two only — nginx proxies backends internally)
│      kubectl port-forward -n identity svc/keycloak 8080:80 &
│      kubectl port-forward -n shopping-cart-apps svc/frontend 3000:80 &
│      # Note: local port 3000/8080 — no root required
│      # Backend services (order :8081, product-catalog :8082, basket :8083)
│      # are reached by nginx inside the frontend pod via cluster DNS — no separate port-forward
│
├── 8. Run Playwright tests (base URL: http://localhost:3000)
│      npx playwright test --reporter=github
│
└── 9. Destroy vCluster (always, even on failure — use job: always())
       ./scripts/k3d-manager vcluster_destroy e2e-${{ github.run_id }}
```

## Port Map

| Service | In-cluster port | Local port | Who uses it |
|---|---|---|---|
| frontend (nginx) | 80 | 3000 | Playwright base URL |
| keycloak | 80 | 8080 | OIDC redirect target (browser) |
| order-service | 80 | — | nginx proxy (cluster DNS, no port-forward) |
| product-catalog | 80 | — | nginx proxy (cluster DNS, no port-forward) |
| basket-service | 8083 | — | nginx proxy (cluster DNS, no port-forward) |
| payment-service | 8084 | — | not tested in initial scope |

**Free port selection:** Local ports 3000 and 8080 are used by convention. On a fresh
GitHub Actions runner these are reliably free. If a conflict is detected at runtime:
```bash
FREE_PORT=$(python3 -c "import socket; s=socket.socket(); s.bind(('',0)); print(s.getsockname()[1]); s.close()")
```
Pass the result as an env var to both `kubectl port-forward` and Playwright's `baseURL`.

## Playwright Test Scope (initial)

Keycloak auth is **included** — tests log in with a pre-seeded test user from the realm
import before exercising the shopping flow.

| Test | Flow |
|---|---|
| Login | Navigate to `/` → redirected to Keycloak → log in as `testuser` → redirected back |
| Product listing | Authenticated landing page — product catalog renders, ≥1 product visible |
| Add to cart | Click product → Add to cart → cart badge increments |
| View cart | Navigate to cart → added item appears with correct price |
| Checkout flow | Proceed to checkout → order confirmation page renders |

**Test user:** `testuser@shopping-cart.local / Test1234!` — seeded in Keycloak realm export.
Credentials stored as GitHub Actions secret `E2E_TEST_USER` / `E2E_TEST_PASSWORD`.

**Out of scope for v0.9.2:**
- Payment processing (requires live payment gateway)
- Cross-browser (Chromium only for CI)
- Performance / load tests

## Playwright MCP: AI-Assisted Authoring

**What it is:** A local MCP server (`@playwright/mcp`) that lets Claude drive a real
Chromium browser interactively — navigate pages, inspect DOM, generate selectors.

**When to use it:**
- During test *authoring* sessions to explore the UI and discover stable selectors
- When a test fails and the failure reason isn't clear from logs alone
- NOT during CI runs — CI runs headless without MCP

**Workflow:**
```
1. Port-forward frontend locally (against live cluster or local vCluster)
2. Start Claude Code session with Playwright MCP configured
3. Claude navigates the app, inspects elements, proposes test scenarios
4. Codex implements the Playwright spec from Claude's exploration output
5. Tests committed to shopping-cart-infra/e2e/
```

### Playwright MCP Setup

**Step 1 — Install:**
```bash
npm install -g @playwright/mcp
npx playwright install chromium
```

**Step 2 — Configure in `~/.claude/settings.json`:**
```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp", "--headless"]
    }
  }
}
```

Use `--headless` for normal use; omit for visual debugging sessions.
