## Problem

Phase 1 installs the ESO **operator + CRDs** on every `k3d-manager/role: app-cluster`
cluster, but ESO has **no backend** there yet:

1. **No ClusterSecretStore on app clusters.** The hub has a `vault-backend`
   ClusterSecretStore (`_eso_apply_vault_cluster_store`, `eso.sh`) pointing at the
   in-cluster hub Vault `http://vault.${VAULT_NS:-secrets}.svc:8200`. That DNS name does
   not resolve from the remote Hostinger app cluster, so the hub CSS is useless there.
2. **Hub Vault is not reachable from app clusters.** It listens only in-cluster on the
   mac/OrbStack hub. The Hostinger box (remote VPS) cannot reach `vault.secrets.svc`.
3. **No per-cluster Vault auth.** The hub Vault has no Kubernetes auth backend that trusts
   the Hostinger cluster's ServiceAccount JWTs, so even a reachable CSS could not log in.

Net effect: app-cluster `ExternalSecret`s (e.g. `payment-db-credentials`,
`product-catalog-secrets`) never reconcile → `CreateContainerConfigError` / `Missing`.

## Goal

Give app clusters a working Vault backend, declaratively where possible:

1. **External Vault endpoint** — expose the hub Vault at a dedicated public host
   `vault.3ai-talk.org` (TLS terminated at Cloudflare edge), via the existing cloudflared
   tunnel. (User decision 2026-06-20: **dedicated ingress host**, not a shared route.)
2. **GitOps ClusterSecretStore** — a new ApplicationSet (cluster generator on
   `k3d-manager/role: app-cluster`) that deploys a `ClusterSecretStore` to each app cluster,
   pointing at the external Vault with Kubernetes auth.
3. **Per-cluster Vault Kubernetes auth registration** — register each app cluster's
   Kubernetes auth in the hub Vault. (User decision 2026-06-20: **k8s-auth per cluster**,
   not AppRole — no secret-zero.) This is the one irreducible non-GitOps step.

2. **Public Vault guardrail — Cloudflare WAF / IP allowlist** (user decision 2026-06-20).
   A Cloudflare WAF custom rule allows only the Hostinger egress IP(s) to reach
   `vault.3ai-talk.org`; everything else blocked. Vault's own k8s-auth + least-privilege
   policy remain the primary control. No Cloudflare Access service-token plumbing. (No
   `--insecure`/`-k`; CSS uses `https://`.) This is an operator step (Cloudflare dashboard) —
   no code change. See operator step 2.
3. **KV paths — GAP CONFIRMED; policy widened (Change 5).** The app-cluster `ExternalSecret`s
   under `shopping-cart-infra/data-layer/secrets/*` read these KV v2 paths, NONE of which the
   current `eso-reader` policy (`secret/data/eso/*` only) covers:
   - `secret/data/payment/{encryption,stripe,paypal}`
   - `secret/data/postgres/{orders,payment,products}`
   - `secret/data/rabbitmq/default`
   - `secret/data/redis/{cart,orders-cache}`

   `eso-app-cluster` role (Change 5) — named prefixes only, no `secret/*` blanket. Hub
   `eso-reader` (`eso/*`) is left untouched.
4. **KV mount + version — CONFIRMED `secret`, KV v2.** The existing data-layer CSS uses
   `path: "secret"` / `version: "v2"`; every app `remoteRef.key` is `secret/data/...` (v2
   form); `vault secrets enable -path=secret kv-v2`. Change 1's CSS (`path: "secret"`,
   `version: v2`) already matches — no change.

### CSS ownership decision (collision resolution, user 2026-06-20)

`shopping-cart-infra/data-layer/secrets/cluster-secret-store.yaml` ALREADY ships a
`ClusterSecretStore` **named `vault-backend`** (old ubuntu-k3s shape:
`server: http://vault-bridge.secrets.svc.cluster.local:8201`, **token auth**), synced to the
app cluster by the `data-git` ApplicationSet. Phase 2's Change 1 deploys a CSS with the SAME
name (k8s-auth, `https://vault.3ai-talk.org`) → name collision on one cluster.

**Decision: k3d-manager owns the CSS.** Keep Changes 1 + 2 (new CSS + new ApplicationSet in
k3d-manager) and DELETE the stale data-layer CSS from `shopping-cart-infra` (Change 6).
**Ordering (critical):** merge Change 6 (infra CSS deletion) BEFORE the k3d-manager

manually (`kubectl --context <app> delete clustersecretstore vault-backend`). Sequence: merge Change 6

## Change 4 — Per-cluster Vault k8s-auth registration (wire-in, non-GitOps)

**EDIT `scripts/lib/providers/k3s-hostinger.sh`** — in `_hostinger_register_cluster`, after
the ArgoCD cluster secret is applied, register the cluster's Kubernetes auth in the hub Vault
by calling the existing `configure_vault_app_auth`. Derive its required inputs from the
Hostinger kubeconfig already in scope:

- `APP_CLUSTER_API_URL` — the Hostinger API server URL. The function already has `server`
  (read via `kubectl config view ... .cluster.server`) in scope — reuse it.
- `APP_CLUSTER_CA_CERT_PATH` — the function already has `ca_data`
  (`.cluster.certificate-authority-data`, base64) in scope — base64-decode it to a temp file
  and pass that path.

**Critical — guard with a SUBSHELL, not `||`.** `configure_vault_app_auth` calls `_err` on any
failure, and `_err` runs `exit 1` (`scripts/lib/system.sh`). A plain
`configure_vault_app_auth || _info ...` is therefore **defeated** — the `exit` aborts the whole
provider deploy before `||` can fire. Run it in `( … )` so a Vault outage is contained.

Insert the block immediately after the existing `Registered — verify` `_info` line, i.e. just
before the closing `}` of `_hostinger_register_cluster`.

**OLD:**
```bash
  _info "[k3s-hostinger] Registered — verify: kubectl get secret ${secret_name} -n ${argocd_ns}"
}
```

**NEW:**
```bash
  _info "[k3s-hostinger] Registered — verify: kubectl get secret ${secret_name} -n ${argocd_ns}"

  # Phase 2 — register this app cluster's Kubernetes auth in the hub Vault for ESO.
  # Subshell-guarded: configure_vault_app_auth calls _err (exit 1) on failure, so a Vault
  # outage must be contained here or it would abort the whole registration flow.
  local app_ca
  app_ca="$(mktemp -t hostinger-app-ca.XXXXXX.crt)"
  if printf '%s' "${ca_data}" | base64 -d >"${app_ca}" 2>/dev/null; then
    if ( APP_CLUSTER_API_URL="${server}" \
         APP_CLUSTER_CA_CERT_PATH="${app_ca}" \
         configure_vault_app_auth ); then
      _info "[k3s-hostinger] Vault app-auth registered for ${_HOSTINGER_KUBE_CONTEXT}"
    else
      _info "[k3s-hostinger] WARN: Vault app-auth registration failed (Vault down?) — run configure_vault_app_auth later"
    fi
  else
    _info "[k3s-hostinger] WARN: could not decode CA data for Vault app-auth; skipping"
  fi
  rm -f "${app_ca}"
}
```

> `server` and `ca_data` are the locals already populated near the top of
> `_hostinger_register_cluster` — do NOT re-derive them. `base64 -d` matches the repo idiom
> (`scripts/etc/jenkins/cert-rotator.sh`). Do NOT change `configure_vault_app_auth` for this
> change (Change 5 is the only edit to it).

## Change 7 — extract provider-agnostic Vault app-auth helper (refactor Change 4)

**Why:** Change 4 wired `configure_vault_app_auth` INLINE into `_hostinger_register_cluster`, pinning
the per-cluster Vault registration to Hostinger. Factor it into ONE provider-agnostic helper so any
provider that registers an app cluster against the hub Vault (Hostinger now; AWS, Azure, OCI, EKS, AKS,
home-lab later) calls a single line. k8s-auth is already portable; the `(API URL, base64 CA)` pair —
which every `_<provider>_register_cluster` already derives from its kubeconfig — is the universal
contract. (Investigated 2026-06-20: only Hostinger + OCI have register functions today; `k3s-aws.sh` /
`k3s-az.sh` have none; OCI runs its own ArgoCD/Vault so it is intentionally NOT wired here.)

**7a — NEW helper in `scripts/plugins/vault.sh`** — place immediately AFTER the
`configure_vault_app_auth` function's closing `}`:

```bash
function register_app_cluster_vault_auth() {
  local api_url="${1:-}"
  local ca_b64="${2:-}"
  local label="${3:-app-cluster}"
  if [[ -z "${api_url}" || -z "${ca_b64}" ]]; then
    _info "[vault] WARN: register_app_cluster_vault_auth needs <api_url> <ca_base64>; skipping ${label}"
    return 0
  fi
  local app_ca
  app_ca="$(mktemp -t app-cluster-ca.XXXXXX.crt)"
  if printf '%s' "${ca_b64}" | base64 -d >"${app_ca}" 2>/dev/null; then
    if ( APP_CLUSTER_API_URL="${api_url}" \
         APP_CLUSTER_CA_CERT_PATH="${app_ca}" \
         configure_vault_app_auth ); then
      _info "[vault] app-auth registered for ${label}"
    else
      _info "[vault] WARN: app-auth registration failed (Vault down?) for ${label} — run configure_vault_app_auth later"
    fi
  else
    _info "[vault] WARN: could not decode CA data for ${label}; skipping app-auth"
  fi
  rm -f "${app_ca}"
}
```
