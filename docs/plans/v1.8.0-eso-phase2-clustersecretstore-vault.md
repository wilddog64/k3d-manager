## Status of user decisions (resolved 2026-06-22 — baked in below)

1. **Vault endpoint hop = Istio ingress + VirtualService** (NOT the `:18200` port-forward).
   Vault is exposed via a VirtualService on the wildcard `istio-system/default-gateway`
   (HTTP:80, `hosts: "*"`), and cloudflared routes `vault.3ai-talk.org` → the local
   istio-ingressgateway HTTP endpoint. TLS terminates at the Cloudflare edge.
2. **Public Vault guardrails = all three** — Cloudflare Access policy **and** Cloudflare WAF/IP
   rule in front of `vault.3ai-talk.org`, **and** Vault's own k8s-auth + least-privilege policy.
3. **KV paths = enumerated** (see table) — `eso-reader` is widened to the exact app prefixes.
4. **KV mount (Phase 2) = `secret`, KV v2** — every *live* app-cluster ExternalSecret reads from
   `secret/data/...`. Migrating Postgres creds to Vault's dynamic `database/` engine is a
   **separate Phase 3 spec** (see end); the CSS, Istio ingress, and per-cluster k8s-auth built
   here are reused unchanged by that migration.

### Enumerated app-cluster KV paths (from shopping-cart-infra `data-layer/`)

All use `secretStoreRef.name: vault-backend`, KV v2 mount `secret`:

| Vault path | Domain |
|---|---|
| `secret/data/postgres/orders` `payment` `products` | postgres |
| `secret/data/payment/encryption` `stripe` `paypal` | payment |
| `secret/data/rabbitmq/default` | rabbitmq |
| `secret/data/redis/cart` `orders-cache` | redis |
| `secret/data/minio/credentials` | minio |

(Keycloak/LDAP ExternalSecrets use a different store `vault-kv-store` in the identity namespace
— hub-side, out of scope for the app-cluster `eso-reader` policy.)

## Problem

Phase 1 installs the ESO **operator + CRDs** on every `k3d-manager/role: app-cluster` cluster, but
there is no backend there: (1) no ClusterSecretStore on app clusters; (2) the hub Vault is not
reachable from the remote Hostinger box; (3) no per-cluster Vault k8s auth. Net effect:
app-cluster `ExternalSecret`s never reconcile → `CreateContainerConfigError` / `Missing`.

## Goal

1. **External Vault endpoint** at `https://vault.3ai-talk.org` via Istio ingress (VirtualService on
   `default-gateway`) + cloudflared, TLS at the Cloudflare edge.
2. **GitOps ClusterSecretStore** — an ApplicationSet (cluster generator on `app-cluster`) that
   deploys a `ClusterSecretStore` named `vault-backend` to each app cluster.
3. **Per-cluster Vault k8s auth** — register each app cluster's Kubernetes auth in the hub Vault
   via the existing `configure_vault_app_auth` (no AppRole, no secret-zero). The one irreducible
   non-GitOps step.

## Existing machinery to reuse (do NOT reinvent or rewrite)

- **`configure_vault_app_auth`** (`scripts/plugins/vault.sh:1315`) — enables the `kubernetes-app`
  auth mount with local JWT validation, creates the `eso-reader` policy, and the `eso-app-cluster`
  role bound to SA `external-secrets`/`secrets`. Inputs: `APP_CLUSTER_API_URL`,
  `APP_CLUSTER_CA_CERT_PATH` (defaults already correct for mount/role/SA/policy). **The only edit
  here is widening the `eso-reader` policy HCL** (Change 4b) — do not restructure the function.
- **`_eso_configure_remote_vault`** (`scripts/plugins/eso.sh:199`) — the CSS shape for a remote
  Vault. Phase 2 freezes this shape into a committed GitOps manifest.
- **Phase 1 generator** `scripts/etc/argocd/applicationsets/eso.yaml` — mirror its cluster-generator
  structure for the new CSS ApplicationSet.
- **`services-git.yaml`** — precedent for sourcing manifests from this repo
  (`repoURL: https://github.com/wilddog64/k3d-manager`, `targetRevision: ${K3D_MANAGER_BRANCH}`).
- **`scripts/etc/argocd/virtualservice.yaml.tmpl` / `keycloak/virtualservice.yaml.tmpl`** — the VS
  pattern to mirror for Change 3a.

## Architecture

```
app-cluster (Hostinger)                         hub (mac / OrbStack)
┌────────────────────────────┐                 ┌──────────────────────────────┐
│ ESO operator (Phase 1)     │                 │ Cloudflare edge (TLS, Access, │
│  + ClusterSecretStore ─────┼── HTTPS ───────►│  WAF/IP)  vault.3ai-talk.org  │
│    (Phase 2, GitOps)       │                 │   → cloudflared               │
│    auth: kubernetes-app    │                 │   → localhost:<istio fwd>     │
│    role: eso-app-cluster   │                 │   → istio-ingressgateway :80  │
│    SA: external-secrets    │                 │   → VirtualService (vault)    │
│                            │                 │   → vault.secrets.svc:8200    │
│ ExternalSecret ────────────┼── login (JWT) ─┤  k8s auth mount kubernetes-app│
└────────────────────────────┘                 │  validates SA JWT via app CA  │
        ▲ registered hub-side by configure_vault_app_auth (per cluster)
```

## Change 1 — GitOps ClusterSecretStore manifest (NEW)

**NEW FILE `platform/eso-clustersecretstore/clustersecretstore.yaml`** (served from this repo's git,
NOT under `services/*` so `services-git` does not pick it up):

```yaml
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "https://vault.3ai-talk.org"
      path: "secret"
      version: v2
      auth:
        kubernetes:
          mountPath: "kubernetes-app"
          role: "eso-app-cluster"
          serviceAccountRef:
            name: external-secrets
            namespace: secrets
```

> CSS name `vault-backend` matches the app `ExternalSecret`s' `secretStoreRef.name` (verified in
> shopping-cart-infra) so they work unchanged. `path: secret` / `version: v2` per decision #4.
> Single hub Vault + single public host → identical for every app cluster → static manifest.

## Change 4 — Per-cluster Vault k8s-auth registration

### 4a — wire-in `_hostinger_register_cluster`

**EDIT `scripts/lib/providers/k3s-hostinger.sh`**. After the cluster Secret is applied successfully
(the `_info "[k3s-hostinger] Registered — verify..."` line, currently line 151), and before the
closing `}` (line 152), register the cluster's Kubernetes auth in the hub Vault. Reuse
`configure_vault_app_auth`; derive its inputs from variables already in scope (`server`, `ca_data`):

**Exact new block to insert after line 151:**

```bash
  # Register this app cluster's Kubernetes auth in the hub Vault (Phase 2).
  # Guarded: a Vault-down/unreachable hub must not hard-fail cluster registration.
  if declare -f configure_vault_app_auth >/dev/null 2>&1; then
    local app_ca_file
    app_ca_file="$(mktemp -t app-cluster-ca.XXXXXX.crt)"
    printf '%s' "${ca_data}" | base64 --decode > "${app_ca_file}" 2>/dev/null || true
    if [[ -s "${app_ca_file}" ]]; then
      APP_CLUSTER_API_URL="${server}" \
      APP_CLUSTER_CA_CERT_PATH="${app_ca_file}" \
        configure_vault_app_auth \
        || _warn "[k3s-hostinger] Vault app-auth registration skipped (hub Vault unreachable?)"
    else
      _warn "[k3s-hostinger] could not decode CA for Vault app-auth; skipping"
    fi
    rm -f "${app_ca_file}"
  fi
```

> `configure_vault_app_auth` lives in `scripts/plugins/vault.sh`; confirm it is sourced in the
> provider's runtime path. If not, source the vault plugin (lazy) before the call — match how other
> providers pull in plugin functions; do NOT duplicate the function.

### 4b — widen the `eso-reader` policy (least-privilege, explicit)

       path "secret/data/eso/*"        { capabilities = ["read"] }
       path "secret/metadata/postgres/*" { capabilities = ["read","list"] }
       path "secret/metadata/payment/*"  { capabilities = ["read","list"] }
       path "secret/metadata/rabbitmq/*" { capabilities = ["read","list"] }
