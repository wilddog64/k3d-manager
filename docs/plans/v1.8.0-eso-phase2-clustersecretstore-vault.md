## Problem

Phase 1 installs the ESO **operator + CRDs** on every `k3d-manager/role: app-cluster`
cluster, but ESO has **no backend** there yet:

1. **No ClusterSecretStore on app clusters.** The hub has a `vault-backend`
   ClusterSecretStore (`_eso_apply_vault_cluster_store`, `eso.sh`) pointing at the
   in-cluster hub Vault `http://vault.${VAULT_NS:-secrets}.svc:8200`. That DNS name does
   not resolve from the remote Hostinger app cluster, so the hub CSS is useless there.
2. **Hub Vault is not reachable from app clusters.** It listens only in-cluster on the
   mac/OrbStack hub. The Hostinger box (remote VPS) cannot reach `vault.secrets.svc`.
3. **No per-cluster Vault auth.** The hub Vault has no Kubernetes auth backend that trusts
   the Hostinger cluster's ServiceAccount JWTs, so even a reachable CSS could not log in.

Net effect: app-cluster `ExternalSecret`s (e.g. `payment-db-credentials`,
`product-catalog-secrets`) never reconcile → `CreateContainerConfigError` / `Missing`.

## Goal

Give app clusters a working Vault backend, declaratively where possible:

1. **External Vault endpoint** — expose the hub Vault at a dedicated public host
   `vault.3ai-talk.org` (TLS terminated at Cloudflare edge), via the existing cloudflared
   tunnel. (User decision 2026-06-20: **dedicated ingress host**, not a shared route.)
2. **GitOps ClusterSecretStore** — a new ApplicationSet (cluster generator on
   `k3d-manager/role: app-cluster`) that deploys a `ClusterSecretStore` to each app cluster,
   pointing at the external Vault with Kubernetes auth.
3. **Per-cluster Vault Kubernetes auth registration** — register each app cluster's
   Kubernetes auth in the hub Vault. (User decision 2026-06-20: **k8s-auth per cluster**,
   not AppRole — no secret-zero.) This is the one irreducible non-GitOps step.

## Existing machinery to reuse (do NOT reinvent)

- **`configure_vault_app_auth`** (`scripts/plugins/vault.sh:1314`) — already implements the
  per-cluster registration: copies the app CA into `vault-0`, enables a `kubernetes-app`
  auth mount with **local JWT validation** (`kubernetes_host` + `kubernetes_ca_cert`, no
  TokenReview callback to the app cluster), creates the `eso-reader` policy, and creates the
  `eso-app-cluster` role bound to SA `external-secrets`/`secrets`. Requires:
  - `APP_CLUSTER_API_URL` — the app cluster's API server URL
  - `APP_CLUSTER_CA_CERT_PATH` — path to the app cluster CA cert file
  - (defaults already correct: mount `kubernetes-app`, role `eso-app-cluster`,
    SA `external-secrets`/`secrets`, policy `eso-reader`)
- **`_eso_configure_remote_vault`** (`scripts/plugins/eso.sh:199`) — the exact CSS shape for
  a remote Vault (`provider.vault` + `auth.kubernetes` with `mountPath: kubernetes-app`,
  `role: eso-app-cluster`). Phase 2 converts this shape into a committed GitOps manifest.
- **Phase 1 generator** `scripts/etc/argocd/applicationsets/eso.yaml` — mirror its structure
  for the new CSS ApplicationSet.
- **services-git** `scripts/etc/argocd/applicationsets/services-git.yaml` — the precedent for
  sourcing manifests from the k3d-manager git repo
  (`repoURL: https://github.com/wilddog64/k3d-manager`, `targetRevision: ${K3D_MANAGER_BRANCH}`).

## Architecture

```
app-cluster (Hostinger)                         hub (mac / OrbStack)
┌────────────────────────────┐                 ┌──────────────────────────────┐
│ ESO operator (Phase 1)     │                 │ cloudflared tunnel            │
│  + ClusterSecretStore ─────┼── HTTPS ───────►│  vault.3ai-talk.org           │
│    (Phase 2, GitOps)       │  Cloudflare TLS │   → http://localhost:18200    │
│    auth: kubernetes-app    │                 │      (vault port-forward LA)  │
│    role: eso-app-cluster   │                 │   → vault-0 :8200 (secrets ns)│
│    SA: external-secrets    │                 │ Vault k8s auth mount          │
│                            │                 │  kubernetes-app / eso-app-... │
│ ExternalSecret ────────────┼── login (JWT) ─┤  validates SA JWT via app CA  │
└────────────────────────────┘                 └──────────────────────────────┘
        ▲ registered hub-side by configure_vault_app_auth (per cluster, non-GitOps)
```

4. **KV mount + version.** Confirm the app secrets live under KV v2 mount `secret`
   (the `_eso_configure_remote_vault` default `REMOTE_VAULT_KV_PATH=secret`, `version: v2`).
   If a different mount is used for app secrets, set it in the CSS + policy.

## Change 1 — GitOps ClusterSecretStore manifest (NEW)

**NEW FILE `platform/eso-clustersecretstore/clustersecretstore.yaml`** (served from this
repo's git, NOT under `services/*` so `services-git` does not pick it up):

```yaml
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "https://vault.3ai-talk.org"
      path: "secret"
      version: v2
      auth:
        kubernetes:
          mountPath: "kubernetes-app"
          role: "eso-app-cluster"
          serviceAccountRef:
            name: external-secrets
            namespace: secrets
```

> Single hub Vault → single public host → the CSS is identical for every app cluster, so the
> manifest is static (no per-cluster templating). `path`/`version` from
> "Confirm Before Handoff" #4. CSS name `vault-backend` matches the hub CSS name so app
> `ExternalSecret`s referencing `secretStoreRef.name: vault-backend` work unchanged.

## Change 2 — ClusterSecretStore ApplicationSet (NEW)

**NEW FILE `scripts/etc/argocd/applicationsets/eso-clustersecretstore.yaml`** — mirrors
Phase 1's `eso.yaml` generator + `services-git`'s git source:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: eso-clustersecretstore
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: platform
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            k3d-manager/role: app-cluster
  goTemplate: true
  goTemplateOptions:
    - missingkey=error
  template:
    metadata:
      name: '{{.name}}-eso-css'
      labels:
        cluster: '{{.name}}'
        app-type: platform
    spec:
      project: platform
      destination:
        server: '{{.server}}'
        namespace: secrets
      source:
        repoURL: https://github.com/wilddog64/k3d-manager
        targetRevision: '${K3D_MANAGER_BRANCH}'
        path: platform/eso-clustersecretstore
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true
```

> Deployed automatically by `_argocd_deploy_applicationsets` (globs `applicationsets/*.yaml`).
> `ServerSideApply=true` for consistency with the ESO CRD-heavy install.
