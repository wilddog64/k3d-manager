# Plan: v1.1.0 — Extract ACG Browser Automation Into Its Own Repo

## Problem Statement

`k3d-manager` currently carries two subsystems with very different failure modes:

1. **Cluster orchestration** — provider dispatch, cluster provisioning, Vault/ESO/ArgoCD setup.
2. **Browser automation** — Chrome CDP bootstrap, Playwright session reuse, login recovery, sandbox navigation, and provider-specific credential scraping.

The browser automation layer is no longer a small helper. During GCP work it repeatedly destabilized the AWS flow because shared ACG/Playwright code was changed in place.

## Goal

Protect `k3d-manager` stability by moving ACG browser automation into a dedicated repo with its own test loop, release cadence, and provider-isolated code structure.

## Why Now

- **AWS stability is regressing during GCP work** — shared Playwright/CDP changes are leaking across providers.
- **One shared script is overloaded** — CDP attach, page selection, navigation, login recovery, sandbox start/resume, and AWS/GCP extraction all live in the same path.
- **`make up` is the wrong test surface** — browser automation should be verified independently before it is allowed to affect provisioning.
- **Roadmap sequencing changed in practice** — waiting for all providers to finish before extraction has proven riskier than extracting the subsystem early.

## Proposed Repo Boundary

### Move into the new ACG repo

- Chrome/CDP bootstrap and profile management
- Playwright scripts and browser helpers
- Provider-specific credential extraction flows (AWS, GCP, future Azure)
- Session/login recovery logic
- Sandbox-page navigation plus open/start/resume automation

### Keep in `k3d-manager`

- Provider dispatch and cluster lifecycle orchestration
- `make up` / `bin/acg-up` workflow coordination
- Consumption of extracted credentials and provider outputs
- Cluster provisioning, Vault, ESO, ArgoCD, and app registration logic

## Contract Between Repos

`k3d-manager` should stop depending on Chrome/CDP internals directly.

Example interface:

- `acg credentials aws --sandbox-url <url>`
- `acg credentials gcp --sandbox-url <url>`
- `acg ensure-session --provider aws|gcp`

Output should be machine-readable and provider-specific, with no cluster-orchestration concerns in the ACG repo.

## Refactor Strategy

### Phase 1 — Freeze and isolate

- Stop expanding shared Playwright logic inside `k3d-manager`
- Identify the last stable AWS browser-automation baseline
- Inventory all ACG-related files and their dependencies

### Phase 2 — Extract repo skeleton

- Create a new repo for ACG automation
- Move shared browser/CDP primitives first
- Split provider-specific flows into separate modules/files
- Add standalone test fixtures for sandbox-page behavior

### Phase 3 — Reconnect `k3d-manager`

- Replace direct Playwright/CDP internals with a thin CLI or script boundary
- Keep `k3d-manager` focused on orchestration only
- Add smoke coverage that verifies integration, not browser internals

## Success Criteria

- AWS browser-automation changes do not require edits to GCP flow files, and vice versa
- `k3d-manager` no longer carries shared Playwright/CDP implementation details
- Browser automation can be tested without running full `make up`
- AWS stability is restored before additional GCP/Azure automation work proceeds

## Out of Scope

- New cloud-provider features beyond stabilizing the ACG automation boundary
- Broad refactors in cluster provisioning unrelated to browser automation extraction
- Immediate TypeScript migration; codebase split and provider isolation come first

## Immediate Next Step

Use `origin/main` (`279db18c`) as the clean orchestration baseline, then rebuild only the ACG integration boundary after the browser automation layer has been extracted and stabilized in its own repo.
