## Architectural Boundary (from roadmap)

k3d-manager does **not** compete with Terraform, Pulumi, or cloud provisioners.
Those tools own VPCs, node groups, IAM, and networking.

**The handoff point is a kubeconfig:**
```
eksctl / az aks / gcloud  →  provisions cluster  →  outputs kubeconfig
k3d-manager               →  receives kubeconfig →  deploys service stack
```

`CLUSTER_PROVIDER` controls only three things per provider:
1. `_provider_<name>_create_cluster` — provision a cluster, write kubeconfig
2. `_provider_<name>_destroy_cluster` — tear down the cluster
3. `_provider_<name>_get_kubeconfig` — refresh kubeconfig if it expires

The plugin layer (Vault, ESO, Istio, ArgoCD) speaks only Kubernetes primitives and
works identically against k3d, k3s, EKS, GKE, or AKS.

## Primary Use Case: ACG Sandbox

A Cloud Guru (ACG) provides temporary AWS / Azure / GCP sandbox environments
that expire after a few hours. They are free, but:
- Everything is provisioned via the ACG web UI — no public API
- Credentials are temporary (STS tokens for AWS, short-lived for Azure/GCP)
- Quotas are limited — cluster must be minimal

**Goal:** One command to spin up the full standard stack on a fresh ACG sandbox,
tear it down cleanly before expiry, and reuse it for hands-on learning.

## Provider 1 — AWS EKS

### Provisioning tool: `eksctl`

`eksctl` is the official CLI for EKS. Single command, no Terraform needed for
a dev/learning cluster.

```bash
eksctl create cluster \
  --name "${CLUSTER_NAME}" \
  --region "${AWS_DEFAULT_REGION:-us-east-1}" \
  --nodegroup-name standard-workers \
  --node-type t3.medium \
  --nodes 1 \
  --nodes-min 1 \
  --nodes-max 1 \
  --managed
```

**Teardown:**
```bash
eksctl delete cluster --name "${CLUSTER_NAME}" --region "${AWS_DEFAULT_REGION:-us-east-1}"
```

**Kubeconfig:**
```bash
aws eks update-kubeconfig --name "${CLUSTER_NAME}" --region "${AWS_DEFAULT_REGION:-us-east-1}"
```

### Authentication

| Source | Env vars required |
|---|---|
| ACG sandbox | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN` (STS) |
| Personal AWS | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_DEFAULT_REGION` |
| CI / production | IAM role via OIDC (no static keys) |

Credentials sourced from Vault at runtime — never in config files or CLI args.
`AWS_SESSION_TOKEN` must be set for ACG sandbox STS tokens.

### Node sizing for ACG sandbox

| Component | Requirement |
|---|---|
| Vault + ESO | ~512Mi |
| Istio | ~512Mi |
| ArgoCD | ~512Mi |
| OpenLDAP | ~128Mi |
| Buffer | ~512Mi |
| **Total** | ~2Gi |

`t3.medium` (2 vCPU / 4Gi) is the minimum that runs the full stack.
ACG sandboxes typically allow t3.medium in us-east-1.

## Provider 2 — Google GKE

### Provisioning tool: `gcloud` CLI

```bash
gcloud container clusters create "${CLUSTER_NAME}" \
  --zone "${GCP_ZONE:-us-central1-a}" \
  --num-nodes 1 \
  --machine-type e2-standard-2 \
  --no-enable-autoupgrade \
  --no-enable-autorepair
```

**Teardown:**
```bash
gcloud container clusters delete "${CLUSTER_NAME}" \
  --zone "${GCP_ZONE:-us-central1-a}" --quiet
```

**Kubeconfig:**
```bash
gcloud container clusters get-credentials "${CLUSTER_NAME}" \
  --zone "${GCP_ZONE:-us-central1-a}"
```

### Authentication

| Source | Method |
|---|---|
| ACG sandbox | `gcloud auth activate-service-account --key-file <(echo $GOOGLE_CREDENTIALS_JSON)` |
| Personal GCP | `gcloud auth login` (interactive) or service account JSON |
| CI | Workload Identity Federation (no key file) |

`GOOGLE_CREDENTIALS_JSON` stored in Vault — piped via process substitution,
never written to disk.

### Node sizing

`e2-standard-2` (2 vCPU / 8Gi) — comfortable for full stack on GKE.
ACG GCP sandboxes typically allow this in us-central1.

## Provider 3 — Azure AKS

### Provisioning tool: `az` CLI

```bash
# Resource group first
az group create \
  --name "${AZ_RESOURCE_GROUP:-k3dm-rg}" \
  --location "${AZ_LOCATION:-eastus}"

# Cluster
az aks create \
  --resource-group "${AZ_RESOURCE_GROUP:-k3dm-rg}" \
  --name "${CLUSTER_NAME}" \
  --node-count 1 \
  --node-vm-size Standard_B2s \
  --generate-ssh-keys \
  --no-wait
```

**Teardown:**
```bash
az aks delete \
  --resource-group "${AZ_RESOURCE_GROUP:-k3dm-rg}" \
  --name "${CLUSTER_NAME}" --yes --no-wait
az group delete --name "${AZ_RESOURCE_GROUP:-k3dm-rg}" --yes --no-wait
```

**Kubeconfig:**
```bash
az aks get-credentials \
  --resource-group "${AZ_RESOURCE_GROUP:-k3dm-rg}" \
  --name "${CLUSTER_NAME}" --overwrite-existing
```

### Authentication

| Source | Method |
|---|---|
| ACG sandbox | `az login --service-principal -u $AZ_CLIENT_ID -p $AZ_CLIENT_SECRET --tenant $AZ_TENANT_ID` |
| Personal Azure | `az login` (interactive browser) |
| CI | Managed Identity or OIDC federated credential |

`AZ_CLIENT_ID`, `AZ_CLIENT_SECRET`, `AZ_TENANT_ID` sourced from Vault.

### Node sizing

`Standard_B2s` (2 vCPU / 4Gi) — minimum for full stack on AKS.
AKS has more overhead than EKS/GKE; `Standard_B2ms` (2 vCPU / 8Gi) preferred
if ACG quota allows.

## ACG Sandbox Credential Lifecycle

ACG provides no public API — credentials are extracted from the web UI.

**Google Antigravity** handles this layer:
```
Slack: "Start an AWS sandbox on ACG"
→ Antigravity: logs into ACG web UI (ACG credentials from Vault)
→ Antigravity: starts sandbox, extracts cloud credentials + expiry timestamp
→ k3dm-mcp: receives structured credentials, injects into Vault
→ k3d-manager: CLUSTER_PROVIDER=eks deploy_cluster
→ SQLite: records expiry timestamp
→ Slack: "EKS stack ready. Sandbox expires in 4h. Auto-teardown at HH:MM."
```

**Expiry handling (SQLite state cache):**

| Field | Description |
|---|---|
| `sandbox_provider` | `aws` / `azure` / `gcp` |
| `sandbox_expiry` | ISO 8601 timestamp from Antigravity |
| `cluster_name` | name of provisioned cluster |
| `auto_teardown` | boolean — trigger destroy before expiry |

`sync_state` tool warns at 30min remaining.
`destroy_cluster` cleans up before sandbox expires — avoids orphaned cloud resources.

## Tool Selection Rationale

Each provider uses the purpose-built, cloud-vendor CLI — not generic infrastructure
tools (Terraform, Pulumi, Crossplane). This matches k3d-manager's philosophy:
minimal surface, fast, no state files, no provisioner overhead.

| Provider | Tool | Why |
|---|---|---|
| EKS | `eksctl` | Purpose-built for EKS by Weaveworks. Single command handles VPC, IAM, node groups. Direct equivalent of what `k3d` does locally. Better than raw `aws eks`. |
| GKE | `gcloud` (container component) | No dedicated `gkectl` exists for standard GKE (only Anthos bare-metal). `gcloud container clusters` is the vendor-maintained, purpose-built tool. |
| AKS | `az aks` | Azure CLI's dedicated AKS subcommand. No widely-used `aksctl` equivalent. Maintained by Microsoft. |

**All three are lazy-loaded** — installed on demand when the provider is first used,
following the `_vcluster_install_cli` pattern. A user running
`CLUSTER_PROVIDER=k3d` never needs any cloud CLI installed.

## k3d-manager Provider Script Layout

```
scripts/lib/
└── providers/
    ├── k3d.sh          # existing
    ├── orbstack.sh     # existing
    ├── k3s.sh          # existing
    ├── eks.sh          # new — auto-installs eksctl, then create/delete/get-kubeconfig
    ├── gke.sh          # new — auto-installs gcloud SDK, then create/delete/get-credentials
    └── aks.sh          # new — auto-installs azure-cli, then create/delete/get-credentials
```

Each provider script follows the same pattern:

```bash
# 1. Install CLI on first use (lazy — only installs when this provider is selected)
function _eks_install_cli() {
  if _command_exist eksctl; then return 0; fi
  if _is_mac; then
    _run_command -- brew install eksctl
  else
    # detect arch, download binary — same pattern as _vcluster_install_cli
  fi
}

# 2. Three-function public contract
function _provider_eks_create_cluster() {
  _eks_install_cli
  _run_command -- eksctl create cluster ...
}
function _provider_eks_destroy_cluster() { ... }
function _provider_eks_get_kubeconfig()  { ... }
```

`CLUSTER_PROVIDER=eks` triggers lazy-load of `eks.sh` — no changes to dispatcher.

## New Environment Variables

| Variable | Default | Description |
|---|---|---|
| `AWS_DEFAULT_REGION` | `us-east-1` | EKS region |
| `AWS_SESSION_TOKEN` | — | Required for ACG STS credentials |
| `GCP_ZONE` | `us-central1-a` | GKE zone |
| `GOOGLE_CREDENTIALS_JSON` | — | Service account JSON (from Vault) |
| `AZ_RESOURCE_GROUP` | `k3dm-rg` | AKS resource group |
| `AZ_LOCATION` | `eastus` | AKS location |
| `AZ_CLIENT_ID` | — | Service principal client ID |
| `AZ_CLIENT_SECRET` | — | Service principal secret (from Vault) |
| `AZ_TENANT_ID` | — | Azure tenant ID |

All secret variables (`AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`,
`GOOGLE_CREDENTIALS_JSON`, `AZ_CLIENT_SECRET`) must be registered in
`_args_have_sensitive_flag` to suppress trace logging.

## Toolchain Requirements (per provider)

| Provider | Required CLIs | Install |
|---|---|---|
| EKS | `aws`, `eksctl` | `brew install awscli eksctl` |
| GKE | `gcloud` | `brew install --cask google-cloud-sdk` |
| AKS | `az` | `brew install azure-cli` |

Auto-install helpers follow the same pattern as `_vcluster_install_cli`:
check for binary, install if missing, error on unsupported arch.

## Implementation Phases

- `scripts/lib/providers/gke.sh`
- `gcloud` auth with JSON key via process substitution
- Live smoke test on ACG GCP sandbox

- `scripts/lib/providers/aks.sh`
- Service principal auth
- Resource group lifecycle (create on deploy, delete on destroy)
- Live smoke test on ACG Azure sandbox

## Out of Scope

- Production-grade cluster hardening (private nodes, VPC CNI, PodSecurityPolicy)
- Multi-node clusters — single node only for ACG sandbox use
- Managed node group autoscaling
- Cloud-native ingress (ALB, Application Gateway) — Istio ingress used as-is
- Terraform/Pulumi integration — may be added in v1.2.0+
