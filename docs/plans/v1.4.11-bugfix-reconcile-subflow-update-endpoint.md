## Problem

The Keycloak reconcile job creates the `browser-with-conditional-otp` flow and the
`otp-conditional-subflow` sub-flow, but then calls
`PUT /authentication/executions/{id}` to set the sub-flow's requirement to CONDITIONAL.
Keycloak 24.0 returns HTTP 404 for that endpoint when the target entry is a sub-flow
(`authenticationFlow: true`) â€” the correct endpoint for sub-flow requirement updates is
`PUT /authentication/flows/{parentFlowAlias}/executions` with a JSON body containing the
execution ID and the updated requirement.

Because `kcadm.sh` exits non-zero on 404 and `set -euo pipefail` is active, the job aborts
immediately, leaving `otp-conditional-subflow` DISABLED and empty. On the next ArgoCD sync,
the idempotency guard sees `browser-with-conditional-otp` already exists and skips creation
entirely, so the broken sub-flow is never repaired. MFA is never activated.

**Root cause:** Line 126 uses `authentication/executions/${_otp_subflow_exec_id}` â€” an
endpoint that only works for leaf executions, not sub-flow entries.

4. Check: `kcadm.sh get authentication/flows/otp-conditional-subflow/executions -r shopping-cart`

   `otp-conditional-subflow` as DISABLED.

## Fix

### Change 1 â€” `identity/keycloak/keycloak-reconcile-hook-job.yaml` (lines 125â€“127)

Replace the `authentication/executions/{id}` call with the flow-level executions endpoint,
which is required for sub-flow (`authenticationFlow: true`) entries.

**Exact old block (lines 125â€“127):**
```bash
            /opt/keycloak/bin/kcadm.sh update \
              "authentication/executions/${_otp_subflow_exec_id}" \
              -r "${KC_REALM}" -s requirement=CONDITIONAL
```

**Exact new block:**
```bash
            /opt/keycloak/bin/kcadm.sh update \
              "authentication/flows/browser-with-conditional-otp/executions" \
              -r "${KC_REALM}" \
              -b "{\"id\":\"${_otp_subflow_exec_id}\",\"requirement\":\"CONDITIONAL\",\"authenticationFlow\":true,\"displayName\":\"otp-conditional-subflow\"}"
```
