# v0.8.0: Vault-Managed ArgoCD Deploy Keys

## Problem

Shopping cart ArgoCD Application CRs currently reference HTTPS repo URLs with no
credential management. The v0.7.3 investigation found empty-passphrase SSH deploy
keys stored on disk — insecure and hard to rotate. This plan eliminates disk-based
credentials entirely: deploy keys live in Vault, ESO syncs them to ArgoCD repo
secrets in the `cicd` namespace.

---

## Solution Overview

```
GitHub deploy key (read-only, per repo)
  → stored in Vault KV at secret/argocd/deploy-keys/<repo-name>
  → ESO ExternalSecret syncs → argocd-repo-<repo-name> secret in cicd ns
  → ArgoCD reads the secret automatically
  → Application CRs updated to SSH URLs
```

**Rotation:** `vault kv put secret/argocd/deploy-keys/<repo> private_key=@<new-key>`
→ ESO syncs → ArgoCD picks up automatically → update GitHub deploy key. One operation.

## Vault KV Schema

Mount: `secret` (existing KV v2 mount)

```
secret/argocd/deploy-keys/basket           → { private_key: "-----BEGIN OPENSSH..." }
secret/argocd/deploy-keys/frontend         → { private_key: "-----BEGIN OPENSSH..." }
secret/argocd/deploy-keys/order            → { private_key: "-----BEGIN OPENSSH..." }
secret/argocd/deploy-keys/payment          → { private_key: "-----BEGIN OPENSSH..." }
secret/argocd/deploy-keys/product-catalog  → { private_key: "-----BEGIN OPENSSH..." }
```

Key name is always `private_key`. Value is the PEM-encoded SSH private key (no passphrase).

## New Vault Policy: `argocd-deploy-key-reader`

Create as a **separate** policy from the existing `eso-argocd-admin` policy
(least-privilege — deploy key reader has no access to admin credentials).

```hcl
# argocd-deploy-key-reader
path "secret/data/argocd/deploy-keys/*" {
  capabilities = ["read"]
}
path "secret/metadata/argocd/deploy-keys/*" {
  capabilities = ["read", "list"]
}
```

Bound to a dedicated Kubernetes service account `eso-argocd-deploy-keys-sa` in the `cicd` namespace.

## ArgoCD Repo Secret Format

ArgoCD discovers repo credentials via secrets labeled
`argocd.argoproj.io/secret-type: repository` in its namespace (`cicd`).

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: argocd-repo-<repo-name>
  namespace: cicd
  labels:
    argocd.argoproj.io/secret-type: repository
stringData:
  type: git
  url: git@github.com:wilddog64/shopping-cart-<repo-name>.git
  sshPrivateKey: |
    -----BEGIN OPENSSH PRIVATE KEY-----
    ...
```

## New Template: `scripts/etc/argocd/externalsecret-deploy-key.yaml.tmpl`

Parameterized by `ARGOCD_REPO_NAME` and `ARGOCD_REPO_SSH_URL`.

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: argocd-deploy-key-${ARGOCD_REPO_NAME}
  namespace: ${ARGOCD_NAMESPACE}
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: ${ARGOCD_DEPLOY_KEY_SECRETSTORE}
    kind: SecretStore
  target:
    name: argocd-repo-${ARGOCD_REPO_NAME}
    creationPolicy: Owner
    template:
      metadata:
        labels:
          argocd.argoproj.io/secret-type: repository
      data:
        type: git
        url: "${ARGOCD_REPO_SSH_URL}"
        sshPrivateKey: "{{ .private_key }}"
  data:
    - secretKey: private_key
      remoteRef:
        key: argocd/deploy-keys/${ARGOCD_REPO_NAME}
        property: private_key
```

## New SecretStore: `argocd-deploy-key-store`

A dedicated SecretStore bound to the `eso-argocd-deploy-keys-sa` service account.
Lives alongside the existing `vault-kv-store` in the `cicd` namespace.

```yaml
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: argocd-deploy-key-store
  namespace: ${ARGOCD_NAMESPACE}
spec:
  provider:
    vault:
      server: "${VAULT_ENDPOINT}"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "argocd-deploy-key-reader"
          serviceAccountRef:
            name: eso-argocd-deploy-keys-sa
```

## New Function: `configure_vault_argocd_repos`

Location: `scripts/plugins/argocd.sh`

```
configure_vault_argocd_repos [--seed-vault] [--dry-run] [-h|--help]
```

### Flags

| Flag | Behaviour |
|---|---|
| *(none)* | Apply ESO resources only — assumes Vault KV entries already exist |
| `--seed-vault` | Also write placeholder entries into Vault KV (for fresh installs) |
| `--dry-run` | Print what would be applied; make no changes |

### Steps performed (in order)

1. **Verify prerequisites** — ArgoCD running in `cicd` ns, ESO CRDs present, Vault reachable
2. **Create service account** — `eso-argocd-deploy-keys-sa` in `cicd` ns (idempotent)
3. **Create Vault policy** — `argocd-deploy-key-reader` via `_vault_exec_stream`
4. **Bind Kubernetes role** — map SA to policy in Vault kubernetes auth
5. **Create SecretStore** — `argocd-deploy-key-store` in `cicd` ns via template render
6. **If `--seed-vault`** — write a placeholder `private_key=REPLACE_ME` into each Vault KV path (so ExternalSecrets can be applied without error; operator replaces with real keys)
7. **For each repo** — render `externalsecret-deploy-key.yaml.tmpl` and apply
8. **Wait for sync** — `kubectl wait --for=condition=Ready externalsecret/argocd-deploy-key-<repo> --timeout=60s`
9. **Print rotation instructions**

### STOP gate

If any ExternalSecret fails to reach `Ready` within 60s, print a warning and
exit 1. Do not proceed silently with broken repo credentials.

## Application CR URL Update

The Application CRs in `shopping-cart-infra/argocd/applications/` must be
updated from HTTPS to SSH URLs. This is a **separate git commit** in
`shopping-cart-infra` — outside the scope of this function.

After `configure_vault_argocd_repos` completes, the operator must:

```bash
# In shopping-cart-infra repo — update each Application CR:
# repoURL: https://github.com/wilddog64/shopping-cart-basket.git
#        → git@github.com:wilddog64/shopping-cart-basket.git
```

The function prints a reminder with the exact URLs after it runs.

## Vault KV Seeding Procedure (Manual — Before First Run)

```bash
# Generate a new deploy key (no passphrase)
ssh-keygen -t ed25519 -C "argocd-basket" -f /tmp/basket-deploy -N ""

# Store in Vault (stdin — key never appears in shell history)
kubectl exec -n vault vault-0 -- vault kv put \
  secret/argocd/deploy-keys/basket \
  private_key=- < /tmp/basket-deploy

# Add public key to GitHub repo as a Deploy Key (read-only)
# Settings → Deploy keys → Add deploy key → paste /tmp/basket-deploy.pub

# Clean up local key files
rm /tmp/basket-deploy /tmp/basket-deploy.pub
```

Repeat for each of the 5 repos.

## Security Notes (OWASP)

- **A01 Least Privilege:** `argocd-deploy-key-reader` reads only `secret/data/argocd/deploy-keys/*`. No access to admin credentials or other paths.
- **A02 Secret Hygiene:** Private keys flow Vault → ESO → K8s secret. Never written to disk, never appear in shell history, never logged by `_run_command` (register `private_key` in `_args_have_sensitive_flag` if passed as a flag).
- **A03 Injection:** All `envsubst` template variables validated against the whitelist pattern used in `deploy_argocd`. No user-supplied input passed to `eval`.
- **A05 Misconfiguration:** All resources deployed to `cicd` namespace. SecretStore uses its own SA — not the cluster-admin SA.
