## Problem

The hub Grafana (`kube-prometheus-stack` on `k3d-k3d-cluster`) shows **no metrics for the
k3s-hostinger cluster**. The hub Grafana has exactly one local datasource plus a federation of
the **ACG** sandbox (`ubuntu-k3s`) — there is no path at all for `ubuntu-hostinger` series.

**Root cause (architecture, confirmed live 2026-06-17):** two distinct app clusters are registered
on the hub — `cluster-ubuntu-k3s` (ACG AWS sandbox, reached via SSH tunnel at `host.k3d.internal`)
and `cluster-ubuntu-hostinger` (persistent VPS at its public API). The hub Prometheus federates
**only ACG** via the established pattern:

- `scripts/etc/helm/observability/kube-prometheus-stack-values.yaml` has a `federate-acg` scrape
  job (`host.internal:19090/federate`, label `cluster: acg`) and an `acg-prometheus` Grafana
  datasource.
- `bin/cluster-up` / `bin/cluster-refresh` run `kubectl port-forward svc/prometheus-operated
  19090:9090 --context ubuntu-k3s` so the hub Prometheus (in k3d on the Mac) can reach the remote
  Prometheus over `host.internal:19090`.
- `scripts/plugins/observability.sh::deploy_observability_acg` deploys a kube-prometheus-stack onto
  the ACG cluster so there is something to federate.

`ubuntu-hostinger` has **no equivalent of any of these three pieces**: no Prometheus deployed on
the host, no port-forward bridging it to the hub, and no `federate-hostinger` job / datasource.
So the hub Grafana cannot show Hostinger metrics.

---

## Design

Mirror the proven ACG federation pattern for Hostinger, end to end. Three pieces:

1. **Stand up a Prometheus on `ubuntu-hostinger`** — a new `observability-hostinger` ApplicationSet
   + a Prometheus-only values file (no Grafana/Alertmanager — the hub already has those), deployed
   by a new `deploy_observability_hostinger` public function (auto-dispatched via
   `./scripts/k3d-manager deploy_observability_hostinger`).
2. **Bridge it to the hub** — a `_pf_ensure` port-forward `host.internal:29090 → ubuntu-hostinger
   svc/prometheus-operated:9090` in `bin/cluster-refresh` (port `29090` chosen to avoid the ACG
   `19090`).
3. **Federate + expose in Grafana** — a `federate-hostinger` scrape job (label
   `cluster: ubuntu-hostinger`) and a `hostinger-prometheus` Grafana datasource in the hub values,
   both pointing at `host.internal:29090`.

No hardcoded IPs (uses the `host.internal:<port>` tunnel alias, identical to ACG). No new secrets.

## Change 1 — new file `scripts/etc/argocd/applicationsets/observability-hostinger.yaml`

Mirror `observability-acg.yaml` but a single Prometheus-only element with the destination pinned to
`ubuntu-hostinger`:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: observability-hostinger
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: observability
spec:
  generators:
    - list:
        elements:
          - name: hostinger-kube-prometheus-stack
            namespace: monitoring
            chart: kube-prometheus-stack
            repoURL: https://prometheus-community.github.io/helm-charts
            targetRevision: 67.9.0
            valuesFile: scripts/etc/helm/observability/kube-prometheus-stack-hostinger-values.yaml
  goTemplate: true
  template:
    metadata:
      name: '{{.name}}'
    spec:
      project: platform
      destination:
        name: ubuntu-hostinger
        namespace: '{{.namespace}}'
      sources:
        - repoURL: https://github.com/wilddog64/k3d-manager
          targetRevision: '${K3D_MANAGER_BRANCH}'
          ref: values
        - repoURL: '{{.repoURL}}'
          chart: '{{.chart}}'
          targetRevision: '{{.targetRevision}}'
          helm:
            valueFiles:
              - '$values/{{.valuesFile}}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true
```

  : "${ARGOCD_NAMESPACE:=cicd}"
