# Bugfix: v1.0.2 — Vault Bridge Not Automated; ClusterSecretStore Wrong Server Address

**Branch (k3d-manager):** `k3d-manager-v1.0.2`
**Branch (shopping-cart-infra):** `docs/next-improvements`
**Files (k3d-manager):** `scripts/plugins/shopping_cart.sh`, `bin/acg-up`
**Files (shopping-cart-infra):** `data-layer/secrets/cluster-secret-store.yaml`, NEW `data-layer/secrets/vault-bridge-svc.yaml`

---

## Problem

ESO `ClusterSecretStore` remains `InvalidProviderConfig` after every sandbox rebuild because:

1. **Wrong server address** — `cluster-secret-store.yaml` uses `server: "http://localhost:8200"`.
   ESO pods resolve `localhost` as the pod's own loopback, not the EC2 node's localhost. Vault is
   unreachable from any pod with this address.

2. **socat bridge not automated** — the reverse SSH tunnel (`-R 8200:localhost:8200`) only binds
   on `localhost` of the EC2 server node. Pods need a routable address. The socat bridge
   (`0.0.0.0:8201 → localhost:8200`) provides this, but it was run manually via `nohup` with no
   restart policy — it dies on SSH disconnect and is never set up by any script.

3. **No Kubernetes Service for the bridge** — pods have no stable DNS name to reach the bridge.
   A headless `vault-bridge` Service + imperatively managed Endpoints provides
   `vault-bridge.secrets.svc.cluster.local:8201` as the stable address.

**Root causes:**
- `deploy_app_cluster` never SSHes to the server node to install socat or write a systemd unit.
- `bin/acg-up` never creates the `vault-bridge` Kubernetes Service/Endpoints.
- `cluster-secret-store.yaml` was written with a conceptually incorrect server address.

## Architecture After Fix

```
Mac Vault (k3d/orbstack)
  → kubectl port-forward → Mac localhost:8200
  → autossh -R 8200 → EC2 server localhost:8200     (reverse SSH tunnel — already works)
  → socat systemd unit → EC2 server 0.0.0.0:8201    (NEW: persistent, auto-restart)
  → K8s Endpoints vault-bridge → node-ip:8201        (NEW: created by bin/acg-up)
  → K8s Service vault-bridge.secrets.svc:8201        (NEW: in Git, deployed by ArgoCD)
  → ESO ClusterSecretStore server address             (FIXED: uses k8s service DNS)
```

### Change 1 — `scripts/plugins/shopping_cart.sh`: add `_setup_vault_bridge`

Insert this new private function **immediately before** `function deploy_app_cluster()`:

**Exact old block:**

```bash
function deploy_app_cluster() {
```

**Exact new block:**

```bash
function _setup_vault_bridge() {
  local ssh_host="${1}"
  local ssh_key="${2}"
  _info "[shopping_cart] Installing socat and vault-bridge systemd unit on ${ssh_host}..."
  # SC2087: single-quoted heredoc intentionally prevents local expansion
  # shellcheck disable=SC2087
  ssh -i "${ssh_key}" "${ssh_host}" bash <<'REMOTE'
set -euo pipefail
if ! command -v socat >/dev/null 2>&1; then
  sudo apt-get update -qq
  sudo apt-get install -y socat
fi
sudo tee /etc/systemd/system/vault-bridge.service >/dev/null <<'UNIT'
[Unit]
Description=Vault reverse tunnel bridge (socat)
After=network.target

[Service]
ExecStart=/usr/bin/socat TCP-LISTEN:8201,fork,bind=0.0.0.0 TCP:localhost:8200
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
UNIT
sudo systemctl daemon-reload
sudo systemctl enable vault-bridge
sudo systemctl restart vault-bridge
REMOTE
  _info "[shopping_cart] vault-bridge active on ${ssh_host}:8201"
}

function deploy_app_cluster() {
```

### Change 3 — `bin/acg-up`: add vault-bridge Endpoints step

**Exact old block (lines 90–91):**

```bash
_info "[acg-up] Step 5/5 — Installing sandbox TTL watcher (launchd)..."
acg_watch_start "${sandbox_url:-${_ACG_SANDBOX_LIST_URL}}"
```

**Exact new block:**

```bash
_info "[acg-up] Step 5/6 — Creating vault-bridge Endpoints in secrets namespace..."
_vault_node_ip=$(kubectl get nodes --context ubuntu-k3s \
  -l node-role.kubernetes.io/control-plane \
  --no-headers \
  -o jsonpath='{.items[0].status.addresses[?(@.type=="InternalIP")].address}')
if [[ -z "${_vault_node_ip}" ]]; then
  _err "[acg-up] Could not determine server node internal IP — vault-bridge Endpoints skipped"
else
  kubectl create namespace secrets --context ubuntu-k3s \
    --dry-run=client -o yaml | kubectl apply --context ubuntu-k3s -f - >/dev/null
  kubectl apply --context ubuntu-k3s -f - <<EOF
apiVersion: v1
kind: Endpoints
metadata:
  name: vault-bridge
  namespace: secrets
subsets:
- addresses:
  - ip: ${_vault_node_ip}
  ports:
  - port: 8201
EOF
  _info "[acg-up] vault-bridge Endpoints → ${_vault_node_ip}:8201"
fi

_info "[acg-up] Step 6/6 — Installing sandbox TTL watcher (launchd)..."
acg_watch_start "${sandbox_url:-${_ACG_SANDBOX_LIST_URL}}"
```

### Change 5 — `shopping-cart-infra/data-layer/secrets/vault-bridge-svc.yaml`: NEW file

Create this file at `data-layer/secrets/vault-bridge-svc.yaml`:

```yaml
---
# Service: vault-bridge
#
# Provides a stable cluster-internal DNS name for the socat bridge on the k3s server node.
# The socat bridge forwards 0.0.0.0:8201 → localhost:8200 on the EC2 server, making
# the reverse SSH tunnel reachable from pods without exposing localhost directly.
#
# Endpoints are created imperatively by bin/acg-up (dynamic node IP changes per sandbox).
# This Service provides the stable DNS name: vault-bridge.secrets.svc.cluster.local:8201
apiVersion: v1
kind: Service
metadata:
  name: vault-bridge
  namespace: secrets
spec:
  ports:
  - port: 8201
    protocol: TCP
    targetPort: 8201
  # No selector — Endpoints managed imperatively by bin/acg-up with the current node IP
```
