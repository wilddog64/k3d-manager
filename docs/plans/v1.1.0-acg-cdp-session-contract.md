# Spec: ACG credential extraction CDP/session contract and unified browser orchestration

**Branch:** `k3d-manager-v1.1.0`

---

## Goal

Fix the **current** ACG credential-extraction blocker by making browser attach/launch behavior explicit, unified, and testable.

This spec is about the live failure mode:

- Chrome running without CDP
- CDP attachability not matching the user's visible signed-in browser state
- successful attach landing on unusable session pages (`/id`, stale tabs, landing pages)

It is **not** a resurrection of the old ProcessSingleton profile-lock bug.

---

## Scope

### Files to change

- `scripts/plugins/acg.sh`
- `scripts/plugins/gcp.sh`
- `scripts/playwright/acg_credentials.js`
- `scripts/tests/plugins/acg.bats`

### Files not to change

- `scripts/playwright/gcp_login.js`
- provider deploy/destroy logic unrelated to browser/session attach
- shopping-cart repos

---

## Problem Statement

Today the system mixes three concerns in a way that obscures failures:

1. Chrome process presence
2. CDP endpoint availability
3. ACG/Pluralsight session usability after attach

These must be separated so failures are actionable instead of ambiguous.

---

## Required Changes

### Change 1 — Introduce one browser/CDP contract helper

Create a single helper in `scripts/plugins/acg.sh` that is responsible for all of the following:

- detect whether Chrome is running;
- detect whether CDP port `9222` is reachable;
- launch Chrome with CDP if Chrome is not running;
- if Chrome is already running **without** CDP, print exact manual recovery instructions and return non-zero;
- never kill Chrome automatically.

`scripts/plugins/gcp.sh` must call this helper instead of carrying its own divergent Chrome/CDP orchestration.

### Change 2 — Standardize user-facing error modes

The shell layer must report three distinct error classes:

1. **No CDP endpoint available**
   - Chrome not running and auto-launch failed, or launch timed out.

2. **Chrome running without CDP**
   - print exact relaunch instructions;
   - explain that visible browser login state cannot be reused until Chrome is relaunched with CDP.

3. **CDP reachable but session unusable**
   - Playwright attached, but page state is on `/id`, generic landing page, or another non-sandbox state.

These messages must be explicit enough that the user can tell **which layer failed**.

### Change 3 — Add post-attach session-state classification in `acg_credentials.js`

After CDP attaches and before button search begins, `acg_credentials.js` must classify the active state into one of:

- ready for sandbox interaction;
- manual sign-in required (`/id` or equivalent);
- landing page / intermediate page still loading;
- unexpected page set with no eligible ACG tab.

This classification must be logged clearly and must throw actionable errors rather than generic timeout failures.

### Change 4 — Make AWS and GCP share the same extraction path

Any provider-specific code may choose the target URL, but the browser attach/launch contract must be shared.

No separate ad hoc CDP launch logic remains in `gcp.sh` once the refactor is complete.

### Change 5 — Add focused tests

Add/update BATS coverage to validate shell-level behavior for:

- Chrome not running → helper attempts launch;
- Chrome running without CDP → helper returns non-zero and prints recovery instructions;
- GCP path delegates to the shared helper rather than duplicating logic.

Do not add fake end-to-end browser tests in BATS.

---

## Validation

The implementation is not complete until all of the following are run:

1. `shellcheck -S warning scripts/plugins/acg.sh scripts/plugins/gcp.sh`
2. `bats scripts/tests/plugins/acg.bats`
3. `rg -n "remote-debugging-port=9222|CDP" scripts/plugins/acg.sh scripts/plugins/gcp.sh scripts/playwright/acg_credentials.js`

Plus one manual smoke note must be recorded in memory-bank:

- whether the system now clearly distinguishes:
  - Chrome-no-CDP,
  - CDP-attach-success-but-session-invalid,
  - CDP-attach-success-and-page-ready.

---

## Definition of Done

- one shared browser/CDP helper exists;
- `gcp.sh` no longer owns divergent CDP launch behavior;
- user-facing failures identify the exact failing layer;
- `acg_credentials.js` logs session classification before sandbox-button waits;
- validations above pass;
- memory-bank records the result with the real commit SHA.
