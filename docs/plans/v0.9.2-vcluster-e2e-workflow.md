# Task: Reusable vCluster E2E Setup Workflow (v0.9.2)

**Assignee:** Codex
**Repo:** k3d-manager
**File:** `.github/workflows/vcluster-e2e-setup.yml`

---

## Purpose

A reusable GitHub Actions workflow that provisions an isolated vCluster environment
for E2E testing. Called by `shopping-cart-e2e-tests` (and any future repo needing
ephemeral cluster environments). The caller owns app deployment and test execution;
this workflow owns only cluster lifecycle.

```
k3d-manager/.github/workflows/vcluster-e2e-setup.yml   ← this file (cluster lifecycle)
shopping-cart-e2e-tests/.github/workflows/e2e-tests.yml ← caller (deploys apps + runs tests)
```

---

## Design

### Separation of concerns

| Responsibility | Owner |
|---|---|
| Install k3d, create host cluster | this workflow |
| Install vcluster CLI, create vCluster | this workflow (via `./scripts/k3d-manager`) |
| Switch kubeconfig to vCluster | this workflow |
| Export kubeconfig for caller | this workflow (base64-encoded output) |
| Deploy namespaces, secrets, apps | caller |
| Port-forward services | caller |
| Run Playwright tests | caller |
| Destroy vCluster + k3d cluster | this workflow (teardown job, `if: always()`) |

### Two-job structure

```
setup  →  (caller deploys + tests)  →  teardown (always)
```

The workflow exposes two jobs: `setup` and `teardown`. The caller sequences them:

```yaml
jobs:
  setup:
    uses: wilddog64/k3d-manager/.github/workflows/vcluster-e2e-setup.yml@main
    with:
      action: setup
      run-id: ${{ github.run_id }}

  e2e:
    needs: setup
    runs-on: ubuntu-latest
    steps:
      # deploy + test using setup.outputs.kubeconfig

  teardown:
    needs: [setup, e2e]
    if: always()
    uses: wilddog64/k3d-manager/.github/workflows/vcluster-e2e-setup.yml@main
    with:
      action: teardown
      run-id: ${{ github.run_id }}
```

## Workflow Spec

### Trigger

```yaml
on:
  workflow_call:
    inputs:
      action:
        description: '"setup" or "teardown"'
        required: true
        type: string
      run-id:
        description: 'Unique ID for this run (use github.run_id)'
        required: true
        type: string
      vcluster-version:
        description: 'vCluster chart version'
        required: false
        type: string
        default: '0.32.1'
      k3d-version:
        description: 'k3d version to install'
        required: false
        type: string
        default: 'v5.7.4'
    outputs:
      kubeconfig:
        description: 'base64-encoded kubeconfig scoped to the vCluster'
        value: ${{ jobs.run.outputs.kubeconfig }}
      vcluster-name:
        description: 'Name of the provisioned vCluster'
        value: ${{ jobs.run.outputs.vcluster-name }}
```

### Permissions

```yaml
permissions:
  contents: read
```

### Job: `run`

Single job that branches on `inputs.action`.

#### Setup steps

```
1. Checkout k3d-manager (actions/checkout@v4)
2. Install k3d
   curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | TAG=${{ inputs.k3d-version }} bash
3. Create k3d cluster
   k3d cluster create e2e-${{ inputs.run-id }} --no-lb --wait
4. Install kubectl (if not present — use azure/setup-kubectl@v4)
5. Install vcluster CLI
   ./scripts/k3d-manager _vcluster_install_cli   (calls the existing helper)
   NOTE: _vcluster_install_cli is a private function — expose a thin public wrapper
   OR: replicate the download logic inline (preferred — avoids sourcing the full plugin)
6. Create vCluster
   VCLUSTER_VERSION=${{ inputs.vcluster-version }} \
   ./scripts/k3d-manager vcluster_create e2e-${{ inputs.run-id }}
7. Switch context to vCluster
   ./scripts/k3d-manager vcluster_use e2e-${{ inputs.run-id }}
8. Export kubeconfig
   kubeconfig=$(kubectl config view --raw | base64 -w0)
   echo "kubeconfig=$kubeconfig" >> "$GITHUB_OUTPUT"
   echo "vcluster-name=e2e-${{ inputs.run-id }}" >> "$GITHUB_OUTPUT"
```

#### Teardown steps

```
1. Checkout k3d-manager
2. Install k3d (same as setup — runner is fresh)
3. Install kubectl
4. Restore kubeconfig to host cluster context
   kubectl config use-context k3d-e2e-${{ inputs.run-id }}
5. Destroy vCluster
   ./scripts/k3d-manager vcluster_destroy e2e-${{ inputs.run-id }}
6. Delete k3d cluster
   k3d cluster delete e2e-${{ inputs.run-id }}
```

## Security Requirements (CLAUDE.md)

- `permissions: contents: read` — no write access needed
- All `uses:` steps must pin to a version tag (`@v4`) — never `@main` or `@latest`
- No secrets passed as CLI arguments — kubeconfig exported via `$GITHUB_OUTPUT` only
- k3d install script: pin via `TAG=` — do not use unversioned `install.sh | bash`

---

## Public Wrapper for vcluster CLI install

`_vcluster_install_cli` is a private function (underscore prefix). The workflow
should NOT call private functions directly. Two options:

**Option A (preferred):** Add a thin public wrapper in `vcluster.sh`:
```bash
function vcluster_install_cli() {
  _vcluster_install_cli
}
```

**Option B:** Replicate the download inline in the workflow YAML.

Use Option A — keeps the logic in one place.

---

## Naming Convention

- k3d cluster name: `e2e-${{ inputs.run-id }}`
- vCluster name: `e2e-${{ inputs.run-id }}`
- vCluster namespace: `vclusters` (default `VCLUSTER_NAMESPACE`)
- k3d context: `k3d-e2e-${{ inputs.run-id }}`

Unique per `run-id` — parallel runs never collide.

## BATS Coverage

No new BATS tests required for the workflow YAML itself (GitHub Actions workflows
are not testable via BATS). However, if Option A is chosen (add `vcluster_install_cli`
public wrapper), add one BATS test in `scripts/tests/plugins/vcluster.bats`:

```bash
@test "vcluster_install_cli is a public function" {
  run grep -n "^function vcluster_install_cli" scripts/plugins/vcluster.sh
  [ "$status" -eq 0 ]
}
```
