# Plan: lib-acg Provider-Plugin Architecture

## Objective
Evolve lib-acg into a modular Provider-Plugin architecture to support AWS, GCP, and Azure independently. This ensures that existing AWS stability is preserved while enabling the development of GCP and Azure features.

## Background
Currently, lib-acg is AWS-centric, with AWS-specific logic mixed into the core acg.sh plugin. As we expand to GCP and Azure, this monolithic approach increases the risk of regressions in the stable AWS code path. A dispatcher-based plugin system will provide isolation and a consistent interface.

## Architecture Design

### 1. The Dispatcher (scripts/plugins/acg.sh)
Transform acg.sh into a thin orchestrator that dispatches calls to provider-specific plugins based on the ACG_PROVIDER environment variable (defaulting to aws).

### 2. The Plugins (scripts/plugins/{aws,gcp,azure}.sh)
- aws.sh: Contains all existing AWS logic.
- gcp.sh: Contains existing and future GCP logic.
- azure.sh: Skeleton for future Azure support.

### 3. Unified Test Interface (bin/acg-*)
Refactor the test binaries and Makefile to provide a consistent testing experience across all clouds.

## Implementation Steps

### Phase 1: AWS Isolation (Stability First)
1. Extract AWS-specific functions from acg.sh to scripts/plugins/aws.sh.
2. Rename functions to use the aws_ prefix.
3. Implement dispatcher logic in acg.sh.

### Phase 2: Refactor Test Binaries
1. bin/acg-credential-test: Replace hardcoded aws sts with dynamic validation.
2. bin/acg-extend-test: Implement dynamic validation.
3. Makefile: Update targets to pass PROVIDER and URL variables.

### Phase 3: GCP and Azure Development
1. Update gcp.sh to match naming convention.
2. Implement gcp_status and gcp_teardown.
3. Implement Azure Playwright extraction and azure.sh skeleton.

## Verification & Testing
1. AWS Regression: Confirm aws provider tests pass.
2. GCP Integration: Confirm gcp extraction and validation works.
3. Cross-Cloud Interface: Verify Makefile handles all provider strings.
