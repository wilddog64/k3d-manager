**Branch:** `k3d-manager-v0.7.3`

**Machines:** macOS M2 Air (infra cluster — OrbStack/k3d) + Ubuntu VM (`ssh ubuntu` from M2 Air)

---

## Background

Task 9 narrowed root cause to Parallels network bridge interfering with large TLS/gRPC
payloads. Basic TCP (`dev/tcp`) works but gRPC streams — which use large TLS records
during the handshake — time out consistently.

Fix: clamp TCP MSS (Maximum Segment Size) on Ubuntu so that TLS records are fragmented
to fit the effective MTU across the Parallels bridge.

## Task 2 — Apply MSS Clamp on Ubuntu

```bash
# Apply MSS clamp on FORWARD chain (handles transit)
ssh ubuntu "sudo iptables -t mangle -A FORWARD -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu"

# Also clamp on INPUT/OUTPUT to cover k3s as endpoint
ssh ubuntu "sudo iptables -t mangle -A INPUT -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu"
ssh ubuntu "sudo iptables -t mangle -A OUTPUT -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu"

# Verify rules were added
ssh ubuntu "sudo iptables -t mangle -L -n -v | grep -A2 TCPMSS"
```

**STOP — confirm rules are in place before continuing.**

## Task 5 — Alternative Fix: Reduce Ubuntu eth0 MTU (only if Task 3 still fails)

A lower MTU on Ubuntu forces smaller IP packets end-to-end, avoiding fragmentation:

```bash
ssh ubuntu "sudo ip link set eth0 mtu 1450"
ssh ubuntu "ip link show eth0 | grep mtu"
# Then retry Task 3 registration
```

If this also fails, report back to Claude — the issue may require a Parallels-level
network configuration change outside the scope of this task.
