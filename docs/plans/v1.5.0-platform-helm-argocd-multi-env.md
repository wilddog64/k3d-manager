# v1.5.0 — platform-helm ApplicationSet: multi-environment ArgoCD with per-cluster versioning

## Context

`scripts/etc/argocd/applicationsets/platform-helm.yaml` is the GitOps mechanism for
deploying ArgoCD to registered clusters. It currently has three problems that must be fixed
before OCI cluster work begins:

1. **`targetRevision: HEAD`** — `HEAD` is invalid for a Helm chart source; ArgoCD will fail
   to sync. Must be a real Helm chart semver (e.g. `7.8.1`).
2. **Only selects `environment: dev`** — OCI cluster will be labeled `environment: prod`;
   it will never be picked up by the current single-generator config.
3. **Replica count is hardcoded to 2** — OCI Always Free is ARM64 / resource-constrained;
   it needs `server.replicas: 1` and `repoServer.replicas: 1`.

### Design: cluster secret labels drive version and replicas (Option A)

Each ArgoCD cluster secret carries two labels that the ApplicationSet reads:

| Label | ACG (dev) | Laptop infra (infra) | OCI (prod) |
|---|---|---|---|
| `argocd-chart-version` | one patch behind latest | one patch behind latest | latest |
| `argocd-replicas` | `"2"` | `"2"` | `"1"` |

The upgrade order enforced by the CVE scan CronJob: **ACG → laptop infra → OCI**.

The ApplicationSet generator reads these labels via `{{.metadata.labels.argocd-chart-version}}`
and `{{.metadata.labels.argocd-replicas}}`. Upgrading a cluster = patch its cluster secret
label. No ApplicationSet YAML change required for routine upgrades.

Cluster secrets are registered in `cicd` namespace by:
- `register_app_cluster` (ACG) — sets `environment: dev` + both new labels
- Hub cluster in-cluster secret — sets `environment: infra` + both new labels (registered
  during `deploy_argocd` bootstrap when hub registers itself)
- `deploy_cluster` for `CLUSTER_PROVIDER=k3s-oci` (OCI, future) — sets `environment: prod`

## File 1: `scripts/etc/argocd/applicationsets/platform-helm.yaml`

Replace the entire file with:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: platform-helm
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: platform
spec:
  generators:
    - clusters:
        selector:
          matchExpressions:
            - key: environment
              operator: In
              values: [dev, infra, prod]
        values:
          chartVersion: '{{index .metadata.labels "argocd-chart-version"}}'
          replicas: '{{index .metadata.labels "argocd-replicas"}}'
  goTemplate: true
  goTemplateOptions:
    - missingkey=error
  template:
    metadata:
      name: '{{.name}}-platform'
      labels:
        cluster: '{{.name}}'
        environment: '{{.metadata.labels.environment}}'
    spec:
      project: platform
      destination:
        server: '{{.server}}'
        namespace: cicd
      ignoreDifferences:
        - group: apps
          kind: Deployment
          jsonPointers:
            - /spec/replicas
      source:
        repoURL: https://argoproj.github.io/argo-helm
        chart: argo-cd
        targetRevision: '{{.values.chartVersion}}'
        helm:
          releaseName: argocd-{{.name}}
          parameters:
            - name: server.replicas
              value: '{{.values.replicas}}'
            - name: repoServer.replicas
              value: '{{.values.replicas}}'
      syncPolicy:
        automated:
          prune: false
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

**Note:** This file is a template — `${ARGOCD_NAMESPACE}` is substituted by `envsubst`
in `_argocd_deploy_applicationsets()` before `kubectl apply`. Do not apply it directly.

## File 2: `scripts/plugins/argocd.sh` — `register_app_cluster`

Find the `kubectl patch secret` or cluster secret creation block inside `register_app_cluster`.
Add `argocd-chart-version` and `argocd-replicas` labels alongside the existing
`environment: dev` label.

The exact patch depends on whether the secret uses `kubectl create` or `kubectl patch`.
Search for the `argocd.argoproj.io/secret-type: cluster` label application — the two new
labels go in the same `metadata.labels` block.

Target labels for ACG cluster secret after the change:
```yaml
metadata:
  labels:
    argocd.argoproj.io/secret-type: cluster
    environment: dev
    argocd-chart-version: "${ARGOCD_CHART_VERSION:-7.8.1}"
    argocd-replicas: "2"
```

Also find the hub cluster in-cluster secret registration (search for
`https://kubernetes.default.svc` in `argocd.sh`) and ensure it sets:
```yaml
metadata:
  labels:
    argocd.argoproj.io/secret-type: cluster
    environment: infra
    argocd-chart-version: "${ARGOCD_CHART_VERSION:-7.8.1}"
    argocd-replicas: "2"
```

Use `ARGOCD_CHART_VERSION` env var with a sensible default (pin to the chart version that
ships ArgoCD v3.4.2, currently `7.8.1` — verify at https://artifacthub.io/packages/helm/argo/argo-cd
before committing).

## What NOT to Do

- Do NOT hardcode the ArgoCD chart version as a literal in the ApplicationSet YAML —
  it must come from the cluster secret label so upgrades are a label patch, not a YAML edit
- Do NOT add a `goTemplateOptions: missingkey=error` that would break if the label is
  absent — ensure `register_app_cluster` always sets both labels before this ApplicationSet
  is deployed
- Do NOT modify `_argocd_deploy_applicationsets()` — it already handles `envsubst` correctly
- Do NOT create a PR
