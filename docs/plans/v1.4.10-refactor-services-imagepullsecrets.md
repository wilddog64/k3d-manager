## Problem

Each `services/shopping-cart-*/kustomization.yaml` patches `imagePullSecrets` onto the app's
Deployment to inject `ghcr-pull-secret`. This creates per-app coupling: k3d-manager must know
about every Deployment in every shopping-cart repo and maintain a patch for each one.

Adding a new shopping-cart service currently requires adding a new `services/<service>/kustomization.yaml`
with the same boilerplate patch. Removing the patch breaks image pulls.

**Root cause:** `imagePullSecrets` is a cluster/namespace infrastructure concern (the secret is
provisioned by `bin/acg-up`, not the app repo), but it is currently expressed at the app layer.

---

## Fix

Kubernetes propagates `imagePullSecrets` from a ServiceAccount to all pods that use it.
Patching the `default` ServiceAccount in `shopping-cart-apps` once eliminates the need for
per-Deployment patches entirely.

### Change 1 â€” `bin/acg-up`: patch default ServiceAccount after ghcr-pull-secret creation

Find the step that creates `ghcr-pull-secret` in `shopping-cart-apps` and add a
`kubectl patch serviceaccount` call immediately after:

**Exact old block** (the line that creates the secret â€” locate by grepping for `ghcr-pull-secret`):

```bash
kubectl create secret docker-registry ghcr-pull-secret \
```

**Exact new block** (add after the secret creation block, before the next step):

```bash
kubectl patch serviceaccount default -n shopping-cart-apps \
  -p '{"imagePullSecrets": [{"name": "ghcr-pull-secret"}]}'
```

### Change 2 â€” Remove `patches` block from all 5 kustomization.yaml files

**Exact old block (same pattern in all 5 files):**

```yaml
patches:
  - patch: |-
      - op: add
        path: /spec/template/spec/imagePullSecrets
        value:
          - name: ghcr-pull-secret
    target:
      kind: Deployment
```

**Exact new block (each file becomes):**

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

  - https://github.com/wilddog64/<repo-name>//k8s/base?ref=main
```
