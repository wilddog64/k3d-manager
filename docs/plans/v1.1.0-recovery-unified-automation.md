# Plan: v1.1.0 — Unified ACG Automation Recovery

## Problem Statement

Prior `k3d-manager-v1.1.0` accumulated "system mess":

1. **URL Mismatch** — AWS dashboard at `cloud-playground/...`, GCP at `hands-on/playground/...` — no shared constants.
2. **Browser Hijacking** — `browser.close()` on a CDP-attached browser killed the user's Chrome and destroyed open tabs.
3. **Identity Seesaw** — `gcloud auth activate-service-account` overwrote the user's CLI identity; teardown used `revoke --all` and nuked the real login.
4. **Path Conflicts** — competing Homebrew vs. snap gcloud installs on Linux; competing auth dirs (`playwright-auth` vs. Gemini-seeded `profile`).
5. **Script hang** — `setTimeout` in a `Promise.race` kept Node alive for minutes after a successful extraction, making `make up` look frozen.

## Goal

Restore 100% AWS stability first, then add GCP via surgical primitives that
never touch the user's Chrome or CLI identity.

## Strategy

- **Fresh branch** — `recovery-v1.1.0-aws-first` cut from `main`. AWS path verified functionally (3-node k3s, `aws sts get-caller-identity` OK, shopping-cart pods Healthy) on 2026-04-19.
- **Split into 3 surgical phases** — each a standalone Codex-ready spec under `docs/bugs/`. Commit per phase; push after each; verify before starting the next.
- **No code direct-edits** — all implementation goes through Codex per the `Spec Before Implement` rule.

## Sequencing — one phase at a time, E2E-gated

Phases must land in order (B depends on A; C depends on B). Each spec ends with
an **E2E Verification** section whose tests must all pass before the phase is
considered done. Do not hand the next phase to any agent until the previous
phase's commit is on `origin/recovery-v1.1.0-aws-first` AND its E2E outputs
have been pasted into the completion report.

No squashing — keep the per-phase history so any regression bisects to one
phase. Memory-bank updates are a separate commit per phase (keeps the
spec-to-code commit diff clean).

## Out of Scope (deferred to later v1.1.x or v1.2.0)

- `gcp_provision_stack`, `_gcp_seed_vault_kv`, ArgoCD wiring on GCP — these require `gcp.sh` primitives to be proven first.
- Homebrew gcloud path ordering on Linux — handled when GCP E2E shows a path conflict, not pre-emptively.
- Persistent automation profile migration (if any) — `vars.sh` standardizes on the existing `playwright-auth` dir; no migration needed.

## Verification (End-to-End, after all 3 phases land)

1. **AWS regression test** — `CLUSTER_PROVIDER=k3s-aws make up` on a fresh sandbox. Pass criteria: 3-node k3s Ready, `aws sts get-caller-identity` OK.
2. **GCP happy path** — `CLUSTER_PROVIDER=k3s-gcp make up`. Pass criteria: `gcp_get_credentials` writes the SA key, `gcp_login` latches to `cloud_user`, no Chrome restart prompt.
3. **Safety audit** — During both runs, confirm: (a) user's Chrome remains open with existing tabs; (b) Pluralsight session cookies persist; (c) active gcloud CLI account is the user's real account, not the SA.
