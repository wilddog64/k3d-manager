# Plan: v1.1.0 — Unified ACG Automation Recovery

## Problem Statement
Previous attempts to integrate GCP sandbox support led to "System Mess":
1. **URL Mismatch:** AWS and GCP dashboards use different URL paths.
2. **Browser Hijacking:** Use of .close() killed user sessions.
3. **Identity Seesaw:** CLI was left in a restricted SA state for GCP.
4. **Path Conflicts:** Conflicting gcloud SDK installations.

## Goal
Restore 100% stability to AWS while enabling GCP via a "Surgical Latch-on" strategy.

## Phase A: Infrastructure (Foundation)
- [x] Create `scripts/etc/playwright/vars.sh` (Single Source of Truth for URLs and Profiles).
- [ ] Refactor `scripts/plugins/acg.sh` to use shared variables.
- [ ] Refactor `scripts/plugins/gcp.sh` to prioritize Homebrew gcloud path.

## Phase B: The "Robot Engine" (acg_credentials.js)
- [ ] **Safe Cleanup:** Replace .close() with .disconnect() for all CDP sessions.
- [ ] **Provider Awareness:** Use --provider flag to toggle AWS/GCP extraction logic.
- [ ] **Patient Bridge:** Implement 300s polling loop for manual sign-in on redirects.
- [ ] **IPv4 Force:** Use 127.0.0.1 to bypass macOS IPv6/localhost issues.

## Phase C: Identity & Access (GCP)
- [ ] **Surgical Latch-on:** Automate `gcloud auth login` for the extracted cloud_user.
- [ ] **ADC for SA:** Use environment variables for the SA key during k3sup, keeping the CLI identity as the user.
- [ ] **Surgical Revoke:** Protect the user session by revoking ONLY the specific SA email.

## Verification (End-to-End)
1. **AWS Test:** `CLUSTER_PROVIDER=k3s-aws make up` (Must work 100%).
2. **GCP Test:** `CLUSTER_PROVIDER=k3s-gcp make up` (Must work 100%).
3. **Safety Audit:** Confirm browser stays open and cookies persist in the profile.
