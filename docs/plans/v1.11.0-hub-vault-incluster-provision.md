## Why (grounding — 2026-06-27 live pass)

P2b completes the `hostinger` profile so the hub Vault can run **in** the Hostinger app
cluster with **no laptop dependency**. Two blockers were ground out of real code; both
are resolved by locked decisions below.

**Blocker 1 — `deploy_vault` runs on the ambient kube-context.**
`deploy_vault` (vault.sh:902) and its whole call tree (`_deploy_vault_ha`→`_helm`,
`_vault_bootstrap_ha`→`_vault_operator_init`→`_vault_exec`, `_enable_kv2_k8s_auth`,
`_vault_setup_pki`) use `_kubectl`/`_helm`, which are **pure passthroughs to the ambient
current-context** (system.sh:926/959 — no `--context` injected). `_kubectl`/`_helm` live in
`scripts/lib/system.sh`, a **lib-foundation subtree** (edits would be upstream-first), so
threading a context arg through them is out. Also, `deploy_vault` **early-returns when
`CLUSTER_ROLE=app`** (vault.sh:919) — and Hostinger *is* the app cluster.

**Locked decision (1):** new **vault.sh wrapper** `vault_deploy_hub_into_context` that
saves/restores the current-context (k3s-hostinger.sh:113-115 idiom) and runs `deploy_vault`
with a new **`HUB_VAULT_INCLUSTER=1`** guard that bypasses the `CLUSTER_ROLE=app`
early-return **only for this intentional path**. No subtree edits, minimal surface,
operator-dispatchable (no auto-caller).

**Blocker 2 — the k8s-auth CSS would 403 on every business path.**
The runtime CSS (`shopping_cart.sh:475`) uses `path: "secret"` with **token auth** via the
laptop root token. The app's real `remoteRef` keys (shopping-cart-infra) are
`secret/data/{postgres,payment,redis,rabbitmq,keycloak,ldap,minio}/*` — **none under
`eso/*`**. The `eso-reader` policy grants **only** `secret/data/eso/*`, so a k8s-auth CSS
bound to the `eso-app-cluster` role would be denied on every path.

**Locked decision (2):** new least-privilege **`app-cluster-reader`** policy granting read on
exactly the 7 prefixes the manifests use, bound to the `eso-app-cluster` role. `eso-reader`
is left untouched.

**Behavior-preserving:** every change is gated on `HUB_VAULT_PROFILE` /
`HUB_VAULT_CSS_AUTH` / `HUB_VAULT_INCLUSTER`. With the default `laptop` profile, all P2b
machinery is **inert** — the CSS still emits token auth and reads the laptop root token
byte-for-byte as today. The default flip to `hostinger` is a separate **cutover** step,
gated on P3 seeding (see runbook note).

## Change 3 — `scripts/plugins/vault.sh`: new `vault_deploy_hub_into_context` wrapper

Insert this function **immediately before** `function _vault_wait_ready() {` (currently
vault.sh:972). This anchor is untouched by P1/P2a, so it is conflict-safe regardless of
P2a landing order.

```bash
function vault_deploy_hub_into_context() {
   local app_context="${1:-}"
   [[ -n "${app_context}" ]] || _err "[vault] vault_deploy_hub_into_context requires an app cluster kube-context"
   shift || true
   kubectl config get-contexts "${app_context}" >/dev/null 2>&1 \
      || _err "[vault] kube-context '${app_context}' not found in kubeconfig"

   local prev_ctx
   prev_ctx="$(kubectl config current-context 2>/dev/null || true)"
   _VAULT_DEPLOY_PREV_CTX="${prev_ctx}"
   trap '[[ -n "${_VAULT_DEPLOY_PREV_CTX:-}" ]] && kubectl config use-context "${_VAULT_DEPLOY_PREV_CTX}" >/dev/null 2>&1 || true' EXIT TERM

   kubectl config use-context "${app_context}" >/dev/null 2>&1 \
      || _err "[vault] failed to switch to kube-context '${app_context}'"
   _info "[vault] deploying hub Vault into '${app_context}' (in-cluster relocation)"

   local rc=0
   HUB_VAULT_INCLUSTER=1 CLUSTER_ROLE=infra deploy_vault "$@" || rc=$?

   if [[ -n "${prev_ctx}" ]] && kubectl config get-contexts "${prev_ctx}" >/dev/null 2>&1; then
      kubectl config use-context "${prev_ctx}" >/dev/null 2>&1 || true
   fi
   trap - EXIT TERM
   unset _VAULT_DEPLOY_PREV_CTX
   return "${rc}"
}
```

Notes for the implementer (do NOT add as inline comments in the code):
- Raw `kubectl config …` mirrors the existing cross-context idiom in k3s-hostinger.sh
  (`_kubectl` routes through `_run_command` sudo-probing — not wanted for config ops).
- The trap uses single quotes (SC2064-clean) and reads `_VAULT_DEPLOY_PREV_CTX` at
  signal time, so context is restored even if `deploy_vault` aborts via `_err`.

function _shopping_cart_css_auth_block() {
  local mode="${1:-token}"
  if [[ "${mode}" == "kubernetes" ]]; then
    cat <<K8SAUTH
        kubernetes:
          mountPath: "${APP_K8S_AUTH_MOUNT:-kubernetes-app}"
          role: "${APP_ESO_VAULT_ROLE:-eso-app-cluster}"
          serviceAccountRef:
            name: "${APP_ESO_SA_NAME:-external-secrets}"
            namespace: "${APP_ESO_SA_NS:-secrets}"

  else
    cat <<TOKENAUTH
        tokenSecretRef:
          name: vault-token
          namespace: secrets

TOKENAUTH

}

## Cutover runbook (live, P3-gated — NOT part of this spec's code)

Once P3 makes seeding idempotent + canonical-source-driven, the operator cutover is:

1. `./scripts/k3d-manager vault_deploy_hub_into_context ubuntu-hostinger` — provision the
   in-cluster hub Vault on Hostinger (HA, auto-unseal watchdog from P2a applies).
2. Seed the in-cluster Vault from the canonical source (P3).
3. Ensure app-cluster k8s-auth: the Hostinger reconcile already calls
   `configure_vault_app_auth_for_context ubuntu-hostinger` (k3s-hostinger.sh:170), which now
   creates `app-cluster-reader` and binds the `eso-app-cluster` role to it.
4. Flip the default: `export HUB_VAULT_PROFILE=hostinger` (or set it in config) and re-run
   the reconcile — the CSS re-applies with `kubernetes` auth and ESO re-syncs from the
   in-cluster Vault. The laptop path remains the selectable fallback.
