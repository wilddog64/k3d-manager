# Bugfix: v1.11.0 ÔÇö `deploy_eso` must skip Helm install when ESO is externally managed (ArgoCD)

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/plugins/eso.sh`

---

## Problem

`deploy_eso` decides "ESO already installed" **only** by checking for a Helm release
(`helm -n "$ns" status "$release"`). When ESO is present but managed by something other
than this tool's Helm release ÔÇö e.g. an ArgoCD `Application` ÔÇö `helm status` returns
non-zero, so `release_exists=0`. With CRDs already present, the function falls through to
`_eso_install_chart`, which runs `helm upgrade --install external-secrets`. Helm then
refuses to adopt the externally-owned objects:

```
ServiceAccount "external-secrets-cert-controller" in namespace "secrets" exists and cannot
be imported into the current release: invalid ownership metadata; annotation validation
error: missing key "meta.helm.sh/release-name": must be set to "external-secrets"
```

This blocks the live Hostinger hub-Vault cutover: `vault_deploy_hub_into_context
ubuntu-hostinger` Ôćĺ `deploy_vault` Ôćĺ `deploy_eso` aborts **before** any Vault object is
provisioned, because Hostinger's ESO is ArgoCD-managed
(`argocd.argoproj.io/tracking-id: ubuntu-hostinger-eso:apps/Deployment:secrets/external-secrets`).

**Root cause:** the idempotency guard keys only on Helm-release existence, not on a
present-and-healthy ESO Deployment, so an externally-managed ESO is misread as absent and
a colliding Helm install is attempted.

---

## Reproduction

```bash
# On a cluster whose ESO was installed by ArgoCD (not this tool's Helm release):
kubectl --context ubuntu-hostinger -n secrets get deploy external-secrets \
  -o jsonpath='{.metadata.annotations.argocd\.argoproj\.io/tracking-id}'   # => ubuntu-hostinger-eso:...
helm --kube-context ubuntu-hostinger -n secrets status external-secrets    # => Error: release: not found

./scripts/k3d-manager vault_deploy_hub_into_context ubuntu-hostinger
# expected: detects existing ESO, skips ESO install, provisions Vault
# actual:   "cannot be imported into the current release: invalid ownership metadata"
#           deploy aborts before provisioning Vault
```

---

## Fix

### Change 1 ÔÇö `scripts/plugins/eso.sh`: treat a present Deployment as installed

**Exact old block (lines 135ÔÇô147):**

```bash
  local -a missing_crds=()
  local release_exists=0
  mapfile -t missing_crds < <(_eso_missing_crds)

  if _run_command --no-exit -- helm -n "$ns" status "$release" > /dev/null 2>&1 ; then
      release_exists=1
  fi

  if (( release_exists )) && (( ${#missing_crds[@]} == 0 )); then
      _eso_wait_for_core_readiness "$ns"
      echo "ESO already installed in namespace $ns"
      return 0
  fi
```

**Exact new block:**

```bash
  local -a missing_crds=()
  local release_exists=0
  local deploy_exists=0
  mapfile -t missing_crds < <(_eso_missing_crds)

  if _run_command --no-exit -- helm -n "$ns" status "$release" > /dev/null 2>&1 ; then
      release_exists=1
  fi

  if _kubectl --no-exit -n "$ns" get deploy/"$release" > /dev/null 2>&1 ; then
      deploy_exists=1
  fi

  if (( release_exists )) && (( ${#missing_crds[@]} == 0 )); then
      _eso_wait_for_core_readiness "$ns"
      echo "ESO already installed in namespace $ns"
      return 0
  fi

  if (( ! release_exists )) && (( deploy_exists )) && (( ${#missing_crds[@]} == 0 )); then
      _eso_wait_for_core_readiness "$ns"
      echo "ESO already present in namespace $ns (externally managed; skipping Helm install)"
      return 0
  fi
```

The new branch fires only when there is **no** Helm release but the ESO Deployment **does**
exist and **all** CRDs are present ÔÇö i.e. ESO is externally managed (ArgoCD/manual) and
healthy. In that case the bundled Helm install is skipped so it cannot collide with the
externally-owned objects. Every other path (no release + no deploy, or missing CRDs) is
unchanged, so first-time installs and CRD-repair reinstalls behave exactly as before.
