# Bugfix: v1.11.0 — `configure_vault_app_auth_for_context` must target the in-cluster Vault context in the `hostinger` profile

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/plugins/vault.sh`

---

## Problem

During the live Hostinger hub-Vault cutover, flipping `HUB_VAULT_PROFILE=hostinger` and
running the reconcile drove all 17 ExternalSecrets to `SecretSyncedError` with the
ClusterSecretStore reporting `Ready=False / InvalidProviderConfig: unable to create client`.

The CSS was repointed correctly to the in-cluster Vault
(`http://vault.secrets.svc:8200`, kubernetes auth, mount `kubernetes-app`, role
`eso-app-cluster`). But the in-cluster Hostinger Vault had **only** `kubernetes/` + `token/`
auth methods and **no** `app-cluster-reader` policy — the `kubernetes-app` mount,
`eso-app-cluster` role, and `app-cluster-reader`/`eso-reader` policies were never created
there.

**Root cause:** `configure_vault_app_auth` (vault.sh:1622) performs all its Vault writes via
bare `_vault_exec` / `_kubectl`, which act on the **current kube-context**. During the
Hostinger reconcile the current context is `k3d-k3d-cluster` (the laptop) — the provider's
kubeconfig merge explicitly *preserves* the current context. So the auth objects were
(re)written on the **laptop** hub Vault (which already had `kubernetes-app` → "path already
in use", then a misleading "configured successfully"), while the in-cluster Vault that the
CSS actually points at got nothing.

In the `laptop` profile this is correct — the hub Vault *is* on the laptop (current context),
and the app cluster is the remote one. Only the `hostinger` (in-cluster) profile mis-targets:
there the hub Vault lives in the app cluster, so the auth writes must run against the
app-cluster context, not the laptop.

---

## Reproduction

```bash
# in-cluster Vault already provisioned + seeded on ubuntu-hostinger
printf 'hostinger\n' > "${HOME}/.local/share/k3d-manager/hub-vault-profile"
CLUSTER_PROVIDER=k3s-hostinger make refresh     # current context stays k3d-k3d-cluster

ROOT=$(kubectl --context ubuntu-hostinger -n secrets get secret vault-root \
  -o go-template='{{ index .data "root_token" }}' | base64 -d)
kubectl --context ubuntu-hostinger -n secrets exec vault-0 -- \
  sh -c "VAULT_TOKEN='$ROOT' vault auth list"      # expected: kubernetes-app/ present
                                                   # actual:   only kubernetes/ + token/
kubectl --context ubuntu-hostinger get clustersecretstore vault-backend \
  -o jsonpath='{.status.conditions[0].reason}'     # actual: InvalidProviderConfig
```

---

## Fix

### Change 1 — `scripts/plugins/vault.sh`: run the inner auth config against the in-cluster context

In `configure_vault_app_auth_for_context`, when the in-cluster profile is active
(`HUB_VAULT_USE_BRIDGE=0`), switch the active kube-context to the app cluster
(`app_context`) for the duration of the `configure_vault_app_auth` call, then restore it.
This mirrors the save/restore pattern already used by `vault_deploy_hub_into_context`.

**Exact old block (lines 1761–1768):**

```bash
  local rc=0
  (
    APP_CLUSTER_API_URL="${server}" \
    APP_CLUSTER_CA_CERT_PATH="${ca_path}" \
    configure_vault_app_auth
  ) || rc=$?

  [[ -n "${tmp_ca}" ]] && rm -f "${tmp_ca}"
```

**Exact new block:**

```bash
  local rc=0
  local _prev_ctx="" _switched_ctx=0
  if [[ "${HUB_VAULT_USE_BRIDGE:-1}" == "0" ]]; then
    _prev_ctx="$("${kctl[@]}" config current-context 2>/dev/null || true)"
    if [[ -n "${_prev_ctx}" && "${_prev_ctx}" != "${app_context}" ]]; then
      if "${kctl[@]}" config use-context "${app_context}" >/dev/null 2>&1; then
        _switched_ctx=1
        _info "[vault] in-cluster profile: configuring app-cluster auth on '${app_context}' Vault"
      else
        _warn "[vault] could not switch to context '${app_context}' for in-cluster Vault auth"
      fi
    fi
  fi
  (
    APP_CLUSTER_API_URL="${server}" \
    APP_CLUSTER_CA_CERT_PATH="${ca_path}" \
    configure_vault_app_auth
  ) || rc=$?
  if (( _switched_ctx )) && [[ -n "${_prev_ctx}" ]]; then
    "${kctl[@]}" config use-context "${_prev_ctx}" >/dev/null 2>&1 || true
  fi

  [[ -n "${tmp_ca}" ]] && rm -f "${tmp_ca}"
```

The guard keys on `HUB_VAULT_USE_BRIDGE=0` (the in-cluster/`hostinger` profile). In the
default `laptop` profile (`HUB_VAULT_USE_BRIDGE=1`) the block is byte-for-byte equivalent to
today — no context switch — so existing behavior is preserved. The restore runs
unconditionally when a switch happened, including on the failure path.

## Note for the live retry (Claude, not Codex)

This fix corrects the *definite* bug — auth objects landing on the wrong Vault. After it
lands, the cutover retries from Step 3 (k8s-auth + flip). The kubernetes-auth JWT path for an
in-cluster CSS has never been load-bearing before (the `laptop` profile CSS uses token auth),
so the retry must verify ESO can actually authenticate (audience / SA-JWT signing). If k8s
auth still fails after the context fix, the fallback is token auth for the in-cluster CSS —
a design decision to surface to the user, not to make in this bugfix.
