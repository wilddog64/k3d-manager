# v1.0.2 — Fix `bin/` Script SCRIPT_DIR Contamination

**Date**: 2026-04-03
**Branch**: k3d-manager-v1.0.2
**Status**: ASSIGNED to Codex

## Problem

All `bin/` scripts set `SCRIPT_DIR` to their own directory (`bin/`):

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

Any plugin that uses `${SCRIPT_DIR}/plugins/` as a path anchor breaks when sourced from a `bin/`
script — `SCRIPT_DIR` is `bin/`, so the path resolves to `bin/plugins/` (does not exist).

**Failing example:** `bin/acg-up` sources `scripts/plugins/acg.sh`. Inside `acg.sh` line 7-12:
```bash
if [[ -z "${SCRIPT_DIR:-}" ]]; then
  SCRIPT_DIR="$(cd -P "$(dirname "${BASH_SOURCE[0]}")/.." >/dev/null 2>&1 && pwd)"
fi
source "${SCRIPT_DIR}/plugins/aws.sh"   # → bin/plugins/aws.sh — does not exist
```

The guard fires false (SCRIPT_DIR is already set to `bin/`) so it tries `bin/plugins/aws.sh`.

## Root Cause

`bin/` scripts should never own `SCRIPT_DIR`. That variable is the root of the plugin system
and must always point to `scripts/` — the same convention used by `scripts/k3d-manager`.

## Fix

In all three affected `bin/` scripts, swap the order: compute `REPO_ROOT` from `BASH_SOURCE`,
then derive `SCRIPT_DIR` from `REPO_ROOT` — not from the script's own location.

**Old pattern (all 3 files):**
```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

**New pattern (all 3 files):**
```bash
REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
SCRIPT_DIR="${REPO_ROOT}/scripts"
```
