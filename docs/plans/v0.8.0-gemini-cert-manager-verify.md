## Steps (run in order, one at a time)

### Step 1 ‚Äî Confirm cluster context
```bash
kubectl config current-context
kubectl get nodes
```
Expected: context is `k3d-k3d-cluster`, node is Ready.

### Step 2 ‚Äî Confirm Istio IngressClass exists
```bash
kubectl get ingressclass istio
```
Expected: `istio` IngressClass present. If missing, STOP and report.

### Step 3 ‚Äî Run deploy_cert_manager (staging)
```bash
ACME_EMAIL=test@example.com ./scripts/k3d-manager deploy_cert_manager
```
Use staging issuer (default ‚Äî do NOT pass `--production`).

### Step 4 ‚Äî Verify cert-manager pods
```bash
kubectl get pods -n cert-manager
```
Expected: `cert-manager`, `cert-manager-cainjector`, `cert-manager-webhook` all
`Running` with `1/1` ready.

### Step 5 ‚Äî Verify webhook deployment Available
```bash
kubectl get deployment cert-manager-webhook -n cert-manager
```
Expected: `AVAILABLE` = 1.

### Step 6 ‚Äî Verify ClusterIssuer Ready
```bash
kubectl get clusterissuer letsencrypt-staging
kubectl describe clusterissuer letsencrypt-staging | grep -A5 "Conditions:"
```
Expected: condition `Ready=True`. The ACME account registration may show a
`pending` or network error since the cluster is not internet-accessible ‚Äî that is
expected for a local lab. The issuer object existing and not erroring on creation
is the pass condition.

### Step 7 ‚Äî Verify HTTP Gateway created
```bash
kubectl get gateway cert-manager-http-gw -n istio-system
```
Expected: gateway exists.

### Step 8 ‚Äî Verify idempotency
```bash
ACME_EMAIL=test@example.com ./scripts/k3d-manager deploy_cert_manager
```
Re-run the same command. Expected: completes without error (Helm upgrade is
idempotent, `kubectl apply` is idempotent).
