# Bugfix: v1.11.0 — failover watchdog hardening (launchd PATH + re-seed isolation)

**Branch:** `k3d-manager-v1.11.0`
**Files:** `bin/k3dm-vault-failover`, `scripts/plugins/vault.sh`

---

## Overview

Two independent defects, both surfaced by live-exercising the P4 assisted-failover path, both
preventing the failover from doing its job in exactly the scenario it exists for. They are bundled
into one fix because they are two facets of the same "make the watchdog actually work end-to-end"
hardening pass and touch one line each in two different files.

1. **Change 1 — `bin/k3dm-vault-failover`:** under launchd's minimal PATH the watchdog can't find
   `kubectl`, so it aborts at the kube-context check before it ever probes.
2. **Change 2 — `scripts/plugins/vault.sh`:** during a real `laptop -> hostinger` failover the
   re-seed source read fails with `_err` (which `exit 1`s the process), aborting the failover before
   the CSS reconcile instead of degrading to warn-and-continue.

---

## Problem 1 — watchdog fails under launchd (Homebrew tools not on PATH)

The `com.k3d-manager.vault-failover` LaunchAgent runs `bin/k3dm-vault-failover` under launchd's
minimal PATH (`/usr/bin:/bin:/usr/sbin:/sbin`). `kubectl` (and `helm`/`jq`) live in
`/opt/homebrew/bin` (Apple Silicon) or `/usr/local/bin` (Intel), which are **not** on that PATH.
Inside `vault_failover_hub_into_context`, `_kubectl config get-contexts` finds no `kubectl`,
returns empty, and the kube-context check fails.

**Visible symptom (live):** after `vault_install_failover_watchdog ubuntu-hostinger`, forcing a run
with `launchctl kickstart gui/<uid>/com.k3d-manager.vault-failover` logged:

```
ERROR: [vault] kube-context 'ubuntu-hostinger' not found in kubeconfig
```

even though the context is present in `~/.kube/config` and an interactive
`vault_failover_hub_into_context ubuntu-hostinger --probe-only` reports
`hub Vault healthy (profile=hostinger)`. The watchdog errors out before it ever probes, so the
LaunchAgent is **non-functional**.

**Root cause:** `bin/k3dm-vault-failover` does not augment PATH, so under launchd's minimal PATH the
Homebrew-installed `kubectl` is not found.

---

## Problem 2 — failover aborts at re-seed when laptop source is unreachable

During a `laptop -> hostinger` assisted failover, `vault_failover_hub_into_context` re-seeds the
in-cluster Vault from the laptop hub before repointing the CSS:

```bash
vault_seed_hub_into_context "${_app_ctx}" \
  || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
```

`vault_seed_hub_into_context` reads its canonical source from the **laptop** hub Vault. But a
`laptop -> hostinger` failover only fires **because the laptop hub is unreachable** — so the
re-seed's source read fails and the function calls `_err "[vault] could not read source
vault-root token ..."`. `_err` (`scripts/lib/system.sh:1717`) does **`exit 1`**, which terminates
the entire process. The `|| _warn "...continuing"` guard is therefore **dead code** — the `exit 1`
kills the shell before `||` can catch a nonzero status.

**Consequence:** the new profile is persisted to the state file (that write happens *before* the
re-seed), but the failover then **aborts before the CSS reconcile**
(`_hostinger_reconcile_vault_cluster_store`), so the app-cluster ClusterSecretStore is never
repointed at the in-cluster Vault. The failover defeats its own purpose in exactly the scenario it
exists for.

**Visible symptom (live, `ubuntu-hostinger`):** forcing a `laptop -> hostinger` failover with the
laptop probe failing produced:

```
WARN: [vault] hub Vault probe 1/2 failed (profile=laptop)
WARN: [vault] hub Vault probe 2/2 failed (profile=laptop)
WARN: [vault] hub Vault unreachable after 2 probes — failing over laptop -> hostinger
INFO: [vault] re-seeding in-cluster Vault on ubuntu-hostinger before repointing CSS
ERROR: [vault] could not read source vault-root token from k3d-k3d-cluster/secrets
```

— then **nothing**. No "re-seed reported issues" warn, no reconcile, no "failover complete". The
process exited at the `_err`. (No damage occurred only because the CSS was already pointing at the
in-cluster Vault in this environment.)

**Root cause:** `_err` exits the process; the re-seed call in `vault_failover_hub_into_context` is
not isolated, so its `_err`-driven `exit 1` aborts the failover instead of degrading to the intended
best-effort warn-and-continue.

## Fix

### Change 1 — `bin/k3dm-vault-failover`: prepend Homebrew prefixes to PATH

Prepend the common Homebrew bin directories to PATH right after `set -euo pipefail`, before any tool
is invoked. This covers both Apple Silicon (`/opt/homebrew/bin`) and Intel (`/usr/local/bin`), is
harmless for interactive runs (those dirs are already on PATH), and fixes every Homebrew tool the
failover path needs (`kubectl`, `helm`, `jq`).

**Exact old block:**

```bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

**Exact new block:**

```bash
set -euo pipefail

# launchd runs with a minimal PATH (/usr/bin:/bin:/usr/sbin:/sbin); Homebrew tools
# (kubectl, helm, jq) live outside it. Prepend the common Homebrew prefixes so the
# watchdog finds them when launched by the LaunchAgent.
export PATH="/opt/homebrew/bin:/usr/local/bin:${PATH}"

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

The 3-line comment is intentional — it documents a non-obvious launchd quirk so the PATH
manipulation is not mistaken for an error.

### Change 2 — `scripts/plugins/vault.sh`: isolate the re-seed in a subshell

Run `vault_seed_hub_into_context` in a subshell so its `_err`-driven `exit 1` only terminates the
subshell, yielding a nonzero status the existing `|| _warn` guard catches — letting the failover
degrade to best-effort re-seed and continue to the CSS reconcile. Do NOT modify
`vault_seed_hub_into_context` (its exit behavior is relied on by the P3 seeding/operator path).

**Exact old block (lines 1209–1213):**

```bash
  if [[ "${_fallback}" == "hostinger" ]]; then
    _info "[vault] re-seeding in-cluster Vault on ${_app_ctx} before repointing CSS"
    vault_seed_hub_into_context "${_app_ctx}" \
      || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
  fi
```

**Exact new block:**

```bash
  if [[ "${_fallback}" == "hostinger" ]]; then
    _info "[vault] re-seeding in-cluster Vault on ${_app_ctx} before repointing CSS"
    ( vault_seed_hub_into_context "${_app_ctx}" ) \
      || _warn "[vault] re-seed reported issues — relying on existing in-cluster data / DR backup; continuing"
  fi
```

The only change is wrapping the call in `( … )`. A subshell contains the `exit 1` from `_err`, so
the parent receives exit status 1 and the `|| _warn` runs; execution then continues past the `fi`
to the profile flip + `_hostinger_reconcile_vault_cluster_store`. External side effects of a
successful re-seed (writes to the in-cluster Vault / keyring / k8s backup) are unaffected — the
subshell does not need to export shell state back to the caller.
