# v1.6.0 — Webhook failure analysis — AI triage on any `make up` exit

**Branch:** `k3d-manager-v1.6.0`
**File:** `bin/k3dm-webhook`

---

## Problem

When `make up` exits non-zero (any failure — permission error, timeout waiting for ArgoCD
StatefulSet, CloudFormation stack error, etc.), the webhook posts:

```
❌ acg-up (aws) — failed in Nm
```

No Haiku analysis is performed. The user gets a bare failure with no root cause.

Example failure that should have been analyzed:
```
ERROR: [shopping_cart] Timed out waiting for StatefulSet postgresql-orders
       to be created by ArgoCD
WARN: [acg-up] failed (exit 1) — cleaning up local processes...
make: *** [up] Error 1
```

The output IS captured (the `lines` list in `_run_cluster` accumulates all stdout/stderr).
The gap: the `except` block calls `_finish("failed")` immediately without feeding `lines`
to Haiku.

**Goal:** for every `make up` failure, call Haiku with the last 30 lines, post a diagnosis
to Slack before the ❌ summary, so the user knows whether to retry or investigate.

---

## Changes

Two targeted additions to `bin/k3dm-webhook`. No other file is touched.

---

### Change 1 — Add `_analyze_failure` function after `_analyze_stall`

Insert this function immediately after the closing `return _call_claude(prompt)` line of
`_analyze_stall` (and before the `_POST_PROVISION_DELAY_SECS = 90` constant):

**Exact insertion point — after this line:**
```python
    return _call_claude(prompt)


_POST_PROVISION_DELAY_SECS = 90  # ESO sync + pod startup grace period
```

**Insert this block between them:**
```python
    return _call_claude(prompt)


def _analyze_failure(lines):
    """Feed last 30 non-empty lines to Claude Haiku; return failure diagnosis."""
    last_30 = [l.rstrip() for l in lines[-30:] if l.strip()]
    last_output = "\n".join(last_30)
    prompt = (
        "acg-up failed (make up exited non-zero). "
        f"Last 30 lines of output:\n\n{last_output}\n\n"
        "What failed and why? Is this TRANSIENT (a retry may succeed — e.g. timeout "
        "waiting for a resource that wasn't ready) or PERMANENT (a configuration or "
        "infrastructure problem that requires human action — e.g. permission denied, "
        "missing secret, quota exceeded)?\n"
        "Reply with TRANSIENT or PERMANENT, the root cause in one sentence, and the "
        "recommended fix in one sentence. Be brief."
    )
    return _call_claude(prompt)


_POST_PROVISION_DELAY_SECS = 90  # ESO sync + pod startup grace period
```

---

### Change 2 — Call `_analyze_failure` in the `except` block before `_finish("failed")`

**Exact old block (lines 224–226):**
```python
    except Exception as exc:
        _log(f"ERROR: {exc}")
        _finish("failed")
```

**Exact new block:**
```python
    except Exception as exc:
        _log(f"ERROR: {exc}")
        analysis = _analyze_failure(lines)
        _notify(f"🔍 *Failure analysis* (job `{job_id}`)\n{analysis}")
        _finish("failed")
```

Note: `_notify` is defined inside `_run_cluster` at the start of the function — it is
in scope at the `except` block. Do NOT define a new `_notify` — reuse the existing one.

## Behaviour

```
make up exits non-zero
  → _log("ERROR: …")
  → _analyze_failure(lines) — last 30 lines → Haiku → returns diagnosis string
  → Slack: "🔍 Failure analysis (job abc123)
             PERMANENT: Timed out waiting for StatefulSet postgresql-orders to be created by
             ArgoCD. Check if the shopping-cart-orders ArgoCD Application is Degraded and
             re-sync it manually."
  → _finish("failed") fires
  → Slack: "❌ acg-up (aws) — failed in 12m"
```

The analysis fires before the ❌ summary so the user reads: root cause → conclusion.
If the Anthropic API is unavailable, `_call_claude` returns `"ANTHROPIC_API_KEY not
configured — skipping AI analysis"` — the webhook continues without crashing.
