# Bugfix: v1.6.0 — Add node state to failure analysis; distinguish cluster vs app failure

**Branch:** `k3d-manager-v1.6.0`
**Files:** `bin/k3dm-webhook`

---

## Problem

When `acg-up` fails, Gemini classifies the failure as TRANSIENT or PERMANENT but cannot
distinguish between two fundamentally different failure modes:

1. **Cluster not provisioned** — CloudFormation failed, EC2 nodes never came up; retry of
   `acg-up` will rebuild from scratch.
2. **Cluster up, apps failing** — nodes are Ready but ArgoCD sync timed out or pods are
   CrashLooping; cluster already exists, only app deployment needs to be retried.

Without node state in the prompt, Gemini cannot tell which mode occurred and gives
generic advice that applies to neither.

**Root cause:** `_collect_cluster_state` does not run `kubectl get nodes` on the remote
cluster, so there is no signal in the Gemini prompt about whether nodes exist.

---

## Fix

### Change 1 — `bin/k3dm-webhook`: add node check to `_collect_cluster_state`

**Exact old block (first `try` block inside `_collect_cluster_state`):**

```python
    try:
        result = subprocess.run(
            ["kubectl", "get", "pods", "-A", "--context", "ubuntu-k3s", "--no-headers"],
            capture_output=True, text=True, timeout=15,
        )
        if result.returncode == 0 and result.stdout.strip():
            bad = [
                ln for ln in result.stdout.splitlines()
                if any(s in ln for s in (
                    "CrashLoopBackOff", "Error", "OOMKilled",
                    "ImagePullBackOff", "ErrImagePull", "Pending",
                ))
                and "Completed" not in ln
            ]
            if bad:
                sections.append("Unhealthy pods (ubuntu-k3s):\n" + "\n".join(bad))
    except Exception:
        pass
```

**Exact new block:**

```python
    try:
        result = subprocess.run(
            ["kubectl", "get", "nodes", "--context", "ubuntu-k3s", "--no-headers"],
            capture_output=True, text=True, timeout=15,
        )
        if result.returncode == 0 and result.stdout.strip():
            sections.append("Remote cluster nodes (ubuntu-k3s):\n" + result.stdout.strip())
        else:
            sections.append("Remote cluster nodes (ubuntu-k3s): UNREACHABLE — cluster may not be provisioned")
    except Exception:
        sections.append("Remote cluster nodes (ubuntu-k3s): UNREACHABLE — cluster may not be provisioned")

    try:
        result = subprocess.run(
            ["kubectl", "get", "pods", "-A", "--context", "ubuntu-k3s", "--no-headers"],
            capture_output=True, text=True, timeout=15,
        )
        if result.returncode == 0 and result.stdout.strip():
            bad = [
                ln for ln in result.stdout.splitlines()
                if any(s in ln for s in (
                    "CrashLoopBackOff", "Error", "OOMKilled",
                    "ImagePullBackOff", "ErrImagePull", "Pending",
                ))
                and "Completed" not in ln
            ]
            if bad:
                sections.append("Unhealthy pods (ubuntu-k3s):\n" + "\n".join(bad))
    except Exception:
        pass
```

### Change 2 — `bin/k3dm-webhook`: update `_analyze_failure` prompt to distinguish failure modes

**Exact old string in prompt (the "Diagnose the failure" block):**

```python
        "Diagnose the failure:\n"
        "1. TRANSIENT or PERMANENT?\n"
        "   TRANSIENT = a retry of acg-up is likely to fix it (e.g. ArgoCD sync lag, "
        "pod not ready yet, API server briefly unreachable).\n"
        "   PERMANENT = human action is required before a retry will work (e.g. "
        "misconfigured manifest, missing secret, quota exceeded, SSH key rejected, "
        "CloudFormation stack in ROLLBACK state).\n"
        "2. Root cause: exactly what component failed and why (name the StatefulSet, "
        "pod, ArgoCD app, or AWS resource if visible in the logs).\n"
        "3. Fix: the single most actionable next step — e.g. 'Run acg-up again', "
        "or 'Check ArgoCD app: kubectl get application data-layer -n cicd --context k3d-k3d-cluster', "
        "or 'Inspect the CloudFormation stack events in the AWS console'.\n\n"
        "Reply in exactly this format:\n"
        "TRANSIENT|PERMANENT\n"
        "<root cause — one sentence>\n"
        "<next step — one sentence, include the exact command if possible>"
```

**Exact new string:**

```python
        "Diagnose the failure:\n"
        "1. Failure mode — choose exactly one:\n"
        "   CLUSTER_FAILURE = remote cluster nodes are UNREACHABLE or NotReady — "
        "CloudFormation or k3s provisioning failed; full acg-up retry needed.\n"
        "   APP_FAILURE = remote cluster nodes are Ready but ArgoCD sync timed out "
        "or pods are CrashLooping — cluster exists, only app deployment failed.\n"
        "2. TRANSIENT or PERMANENT?\n"
        "   TRANSIENT = a retry of acg-up is likely to fix it (e.g. ArgoCD sync lag, "
        "pod not ready yet, API server briefly unreachable).\n"
        "   PERMANENT = human action is required before a retry will work (e.g. "
        "misconfigured manifest, missing secret, quota exceeded, SSH key rejected, "
        "CloudFormation stack in ROLLBACK state).\n"
        "3. Root cause: exactly what component failed and why (name the StatefulSet, "
        "pod, ArgoCD app, or AWS resource if visible in the logs).\n"
        "4. Fix: the single most actionable next step — e.g. 'Run acg-up again', "
        "or 'Check ArgoCD app: kubectl get application data-layer -n cicd --context k3d-k3d-cluster', "
        "or 'Inspect the CloudFormation stack events in the AWS console'.\n\n"
        "Reply in exactly this format:\n"
        "CLUSTER_FAILURE|APP_FAILURE\n"
        "TRANSIENT|PERMANENT\n"
        "<root cause — one sentence>\n"
        "<next step — one sentence, include the exact command if possible>"
```
