# v0.9.5 — Service Mesh: Istio Full Activation

**Branch:** `k3d-manager-v0.9.5`
**Repo for manifests:** `shopping-cart-infra`
**Milestone gate:** All 5 apps reachable through mesh with mTLS verified + frontend accessible via Istio Gateway

---

## Context

Istio is deployed on the app cluster and namespace injection labels are set, but the mesh
is dormant — no policies, no traffic management, no service discovery extensions.
Services communicate via raw Kubernetes DNS + ClusterIP with Envoy sidecars passing
traffic through transparently.

This release activates the mesh with a minimal, correct configuration:
mTLS between all services, L7 authorization replacing the payment NetworkPolicy,
frontend ingress via Istio Gateway, load balancing policy per service, and external
payment gateway registration as a `ServiceEntry`.

This is a prerequisite for:
- **v0.8.0 observability** — Prometheus scrapes Istio sidecar metrics (HTTP rates, error rates, latency per service pair)
- **v1.1.0 EKS** — same mesh posture must carry to cloud providers; retrofitting after multi-cloud is harder

---

## Scope

The infra namespaces.yaml wins via ArgoCD, but the per-repo definition is inconsistent.

This is a PR on `shopping-cart-product-catalog`.

### 2. mTLS — PeerAuthentication (STRICT)

Enable STRICT mTLS mesh-wide. All inter-service traffic requires mutual TLS — plaintext
connections are rejected.

```yaml
# shopping-cart-infra/istio/peer-authentication.yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system   # mesh-wide
spec:
  mtls:
    mode: STRICT
```

One resource, mesh-wide. No per-namespace overrides needed — all three namespaces
(`shopping-cart-apps`, `shopping-cart-payment`, `shopping-cart-data`) are in the mesh.

### 3. AuthorizationPolicy — replace payment NetworkPolicy

The payment service NetworkPolicy operates at L3/L4 (IP + port). Replace with Istio
`AuthorizationPolicy` which operates at L7 — service identity (SPIFFE), not IP address.

**Keep the NetworkPolicy in place as a defence-in-depth layer** — both can coexist.
The Istio policy adds identity-based enforcement on top of the IP-based enforcement.

```yaml
# shopping-cart-infra/istio/authz-payment.yaml

# Default deny all traffic to payment namespace
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: shopping-cart-payment
spec: {}   # empty spec = deny all
---
# Allow order-service → payment-service only
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-order-to-payment
  namespace: shopping-cart-payment
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: payment-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
          - "cluster.local/ns/shopping-cart-apps/sa/order-service"
    to:
    - operation:
        ports: ["8084"]
```

**Note:** Requires `order-service` ServiceAccount to exist in `shopping-cart-apps` namespace.
Verify or create it as part of this milestone.

### 4. Gateway + VirtualService — frontend ingress

Expose the frontend through the Istio ingress gateway. This replaces the NodePort
service currently used for external access.

```yaml
# shopping-cart-infra/istio/gateway.yaml
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: shopping-cart-gateway
  namespace: shopping-cart-apps
spec:
  selector:
    istio: ingressgateway
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "shopping-cart.local"
---
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: frontend
  namespace: shopping-cart-apps
spec:
  hosts:
  - "shopping-cart.local"
  gateways:
  - shopping-cart-gateway
  http:
  - match:
    - uri:
        prefix: /api/orders
    route:
    - destination:
        host: order-service
        port:
          number: 80
  - match:
    - uri:
        prefix: /api/products
    route:
    - destination:
        host: product-catalog
        port:
          number: 80
  - match:
    - uri:
        prefix: /api/cart
    route:
    - destination:
        host: basket-service
        port:
          number: 8083
  - route:
    - destination:
        host: frontend
        port:
          number: 80
```

### 5. DestinationRule — load balancing policy per service

Override kube-proxy random selection with Envoy load balancing. `LEAST_CONN` for
stateful services (order, payment — heavier per-request cost). `ROUND_ROBIN` for
stateless (basket, product-catalog, frontend).

```yaml
# shopping-cart-infra/istio/destination-rules.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: order-service
  namespace: shopping-cart-apps
spec:
  host: order-service
  trafficPolicy:
    loadBalancer:
      simple: LEAST_CONN
    connectionPool:
      http:
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service
  namespace: shopping-cart-payment
spec:
  host: payment-service
  trafficPolicy:
    loadBalancer:
      simple: LEAST_CONN
    connectionPool:
      http:
        http1MaxPendingRequests: 50
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 30s
      baseEjectionTime: 60s
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: basket-service
  namespace: shopping-cart-apps
spec:
  host: basket-service
  trafficPolicy:
    loadBalancer:
      simple: ROUND_ROBIN
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: product-catalog
  namespace: shopping-cart-apps
spec:
  host: product-catalog
  trafficPolicy:
    loadBalancer:
      simple: ROUND_ROBIN
```

### 6. ServiceEntry — external payment gateways

Register external payment gateway endpoints into the mesh so Istio can apply egress
mTLS and metrics to outbound calls from the payment service.

```yaml
# shopping-cart-infra/istio/service-entries.yaml
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: stripe-api
  namespace: shopping-cart-payment
spec:
  hosts:
  - "api.stripe.com"
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  location: MESH_EXTERNAL
  resolution: DNS
---
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: paypal-api
  namespace: shopping-cart-payment
spec:
  hosts:
  - "api.paypal.com"
  - "api.sandbox.paypal.com"
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  location: MESH_EXTERNAL
  resolution: DNS
```

With `STRICT` mTLS and an egress `AuthorizationPolicy`, external calls not registered
as `ServiceEntry` resources will be blocked. This enforces explicit egress control.

## Verification (Gemini — on M2 Air)

```bash
# 1. Confirm sidecars injected — all pods should show 2/2 READY
kubectl get pods -n shopping-cart-apps
kubectl get pods -n shopping-cart-payment
kubectl get pods -n shopping-cart-data

# 2. Verify mTLS is enforced
istioctl x check-inject -n shopping-cart-apps
istioctl authn tls-check <pod> -n shopping-cart-apps

# 3. Confirm frontend reachable via Gateway
kubectl get svc istio-ingressgateway -n istio-system
curl -H "Host: shopping-cart.local" http://<ingress-ip>/

# 4. Confirm order→payment allowed, basket→payment denied
# From order-service pod:
kubectl exec -n shopping-cart-apps deploy/order-service -- \
  curl -s http://payment-service.shopping-cart-payment.svc.cluster.local:8084/health
# From basket-service pod (should be RBAC denied):
kubectl exec -n shopping-cart-apps deploy/basket-service -- \
  curl -s http://payment-service.shopping-cart-payment.svc.cluster.local:8084/health

# 5. Confirm ServiceEntry controls egress
kubectl exec -n shopping-cart-payment deploy/payment-service -- \
  curl -s https://api.stripe.com   # should succeed
kubectl exec -n shopping-cart-payment deploy/payment-service -- \
  curl -s https://api.github.com   # should be blocked (not in ServiceEntry)
```
