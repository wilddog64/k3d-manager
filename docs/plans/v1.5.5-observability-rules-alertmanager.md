## Part 1 — PrometheusRules (shopping-cart-infra)

### File to create: `monitoring/rules/shopping-cart-apps.yaml`

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: shopping-cart-apps
  namespace: monitoring
  labels:
    release: kube-prometheus-stack
spec:
  groups:
    - name: shopping-cart.apps
      interval: 60s
      rules:

        # All pods for a service are down for > 5 min
        - alert: ServiceDown
          expr: |
            sum by (namespace, pod) (
              kube_pod_status_ready{condition="true", namespace=~"shopping-cart.*"}
            ) == 0
          for: 5m
          labels:
            severity: critical
            cluster: hub
          annotations:
            summary: "Service down in {{ $labels.namespace }}"
            description: "No ready pods in namespace {{ $labels.namespace }} for > 5 min."

        # Container restarting > 3 times in 10 min
        - alert: PodCrashLooping
          expr: |
            increase(
              kube_pod_container_status_restarts_total{
                namespace=~"shopping-cart.*"
              }[10m]
            ) > 3
          for: 0m
          labels:
            severity: warning
            cluster: hub
          annotations:
            summary: "Pod crash-looping: {{ $labels.pod }}"
            description: "Container {{ $labels.container }} in {{ $labels.namespace }}/{{ $labels.pod }} restarted > 3 times in 10 min."

        # HTTP 5xx error rate > 10% sustained for 5 min (Istio metrics)
        - alert: HighErrorRate
          expr: |
            (
              sum by (destination_service_name, destination_service_namespace) (
                rate(
                  istio_requests_total{
                    destination_service_namespace=~"shopping-cart.*",
                    response_code=~"5.."
                  }[5m]
                )
              )
            ) /
            (
              sum by (destination_service_name, destination_service_namespace) (
                rate(
                  istio_requests_total{
                    destination_service_namespace=~"shopping-cart.*"
                  }[5m]
                )
              )
            ) > 0.1
          for: 5m
          labels:
            severity: warning
            cluster: hub
          annotations:
            summary: "High error rate: {{ $labels.destination_service_name }}"
            description: "Service {{ $labels.destination_service_name }} in {{ $labels.destination_service_namespace }} has > 10% 5xx errors for 5 min."
```

### File to create: `monitoring/rules/argocd-application.yaml`

Deploy the rules to the Hub k3d cluster (the cluster ArgoCD runs on — not ubuntu-k3s).

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: shopping-cart-rules
  namespace: cicd
  labels:
    app.kubernetes.io/part-of: shopping-cart
    managed-by: k3d-manager
spec:
  project: platform
  source:
    repoURL: https://github.com/wilddog64/shopping-cart-infra.git
    targetRevision: HEAD
    path: monitoring/rules
    directory:
      recurse: false
      include: "shopping-cart-apps.yaml"
  destination:
    server: https://kubernetes.default.svc
    namespace: monitoring
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=false
```

Apply the Application to the Hub cluster once committed:

```bash
kubectl apply -f monitoring/rules/argocd-application.yaml \
  --context k3d-k3d-cluster -n cicd
```

## Part 2 — Alertmanager (k3d-manager)

### Why it is conservative

ACG is an ephemeral sandbox. Rules:
- `group_wait: 5m` — batch alerts before first SMS
- `group_interval: 2h` — no repeat SMS for the same group within 2h
- `repeat_interval: 24h` — same alert won't fire again for 24h
- Inhibit `TargetDown` when `ServiceDown` is firing (same namespace)
- Inhibit `PodCrashLooping` + `HighErrorRate` while `ServiceDown` fires (noise reduction)
- Only `severity: critical` pages via SMS; `severity: warning` is suppressed (silence for ACG)

### File to modify: `scripts/etc/helm/observability/kube-prometheus-stack-values.yaml`

Replace:
```yaml
alertmanager:
  enabled: false
```

With:
```yaml
alertmanager:
  enabled: true
  alertmanagerSpec:
    resources:
      requests:
        memory: 32Mi
        cpu: 10m
      limits:
        memory: 64Mi
        cpu: 50m
  config:
    global:
      smtp_smarthost: 'smtp.gmail.com:587'
      smtp_from: '${ALERTMANAGER_GMAIL_FROM}'
      smtp_auth_username: '${ALERTMANAGER_GMAIL_FROM}'
      smtp_auth_password: '${ALERTMANAGER_GMAIL_APP_PW}'
      smtp_require_tls: true

    route:
      receiver: 'null'
      group_wait: 5m
      group_interval: 2h
      repeat_interval: 24h
      routes:
        - matchers:
            - severity = critical
          receiver: sms-critical

    receivers:
      - name: 'null'
      - name: sms-critical
        email_configs:
          - to: '${ALERTMANAGER_SMS_GATEWAY}'
            send_resolved: false
            headers:
              Subject: '[ALERT] {{ .GroupLabels.alertname }}'
            text: '{{ range .Alerts }}{{ .Annotations.summary }}{{ end }}'

    inhibit_rules:
      - source_matchers:
          - alertname = ServiceDown
        target_matchers:
          - alertname =~ "PodCrashLooping|HighErrorRate|TargetDown"
        equal: [namespace]
```

**Note:** `${ALERTMANAGER_GMAIL_FROM}`, `${ALERTMANAGER_GMAIL_APP_PW}`, and
`${ALERTMANAGER_SMS_GATEWAY}` are `envsubst` placeholders. They must be substituted
at deploy time — they are NOT set in this file. The deploy function in `observability.sh`
will read them from Vault and export them before running `envsubst`.

### File to modify: `scripts/plugins/observability.sh` — `deploy_observability`

After the ApplicationSet apply, add a Vault-backed secret step:

```bash
# Read alertmanager credentials from Vault
_info "[observability] Reading Alertmanager credentials from Vault..."
local _vault_addr="http://127.0.0.1:18200"
local _vault_token
_vault_token=$(kubectl get secret vault-root -n secrets \
  --context k3d-k3d-cluster -o jsonpath='{.data.root_token}' | base64 -d)

local _am_creds
_am_creds=$(curl -sf \
  -H "X-Vault-Token: ${_vault_token}" \
  "${_vault_addr}/v1/secret/data/k3d-manager/alertmanager" 2>/dev/null \
  | python3 -c "import json,sys; d=json.load(sys.stdin)['data']['data']; \
    print(d['gmail_from']+'|'+d['gmail_app_pw']+'|'+d['sms_gateway'])")

if [[ -z "${_am_creds}" ]]; then
  _warn "[observability] Alertmanager Vault secret not found — skipping SMS config"
  _warn "[observability] Run: make alertmanager-secret to configure"
else
  export ALERTMANAGER_GMAIL_FROM="${_am_creds%%|*}"
  local _rest="${_am_creds#*|}"
  export ALERTMANAGER_GMAIL_APP_PW="${_rest%%|*}"
  export ALERTMANAGER_SMS_GATEWAY="${_rest##*|}"
fi
```

The existing `envsubst` call for the ApplicationSet must be extended to also substitute
the three new variables when running the values file through ArgoCD. However, since
kube-prometheus-stack values are rendered by Helm (not envsubst), the substitution must
happen at a different point.

**Correct approach:** The Alertmanager config in the Helm values uses placeholders that
cannot be substituted by ArgoCD Helm rendering. Instead:

1. Do NOT put the credentials inline in the values file.
2. Create a Kubernetes Secret `alertmanager-smtp-secret` in the `monitoring` namespace
   with the rendered config.
3. Set `alertmanager.configSecret: alertmanager-smtp-secret` in the values file
   (not `alertmanager.config`).

**Revised values change** — replace the `alertmanager` block above with:

```yaml
alertmanager:
  enabled: true
  alertmanagerSpec:
    resources:
      requests:
        memory: 32Mi
        cpu: 10m
      limits:
        memory: 64Mi
        cpu: 50m
  configSecret: alertmanager-smtp-secret
```

**Revised deploy step** — in `deploy_observability`, after reading Vault, create the secret:

```bash
# Build and apply the Alertmanager config Secret
local _am_config
_am_config=$(cat <<AMEOF
global:
  smtp_smarthost: 'smtp.gmail.com:587'
  smtp_from: '${ALERTMANAGER_GMAIL_FROM}'
  smtp_auth_username: '${ALERTMANAGER_GMAIL_FROM}'
  smtp_auth_password: '${ALERTMANAGER_GMAIL_APP_PW}'
  smtp_require_tls: true

route:
  receiver: 'null'
  group_wait: 5m
  group_interval: 2h
  repeat_interval: 24h
  routes:
    - matchers:
        - severity = critical
      receiver: sms-critical

receivers:
  - name: 'null'
  - name: sms-critical
    email_configs:
      - to: '${ALERTMANAGER_SMS_GATEWAY}'
        send_resolved: false
        headers:
          Subject: '[ALERT] {{ .GroupLabels.alertname }}'
        text: '{{ range .Alerts }}{{ .Annotations.summary }}{{ end }}'

inhibit_rules:
  - source_matchers:
      - alertname = ServiceDown
    target_matchers:
      - alertname =~ "PodCrashLooping|HighErrorRate|TargetDown"
    equal: [namespace]
AMEOF
)
kubectl create secret generic alertmanager-smtp-secret \
  --context k3d-k3d-cluster \
  -n monitoring \
  --from-literal=alertmanager.yaml="${_am_config}" \
  --dry-run=client -o yaml | _kubectl apply -f -
_info "[observability] Alertmanager config secret created"
```

### File to modify: `Makefile` — add `alertmanager-secret` target

```makefile
## Store Alertmanager credentials in Vault (run once; requires Hub Vault + port-forward)
alertmanager-secret:
	@_tok=$$(kubectl get secret vault-root -n secrets --context k3d-k3d-cluster \
	  -o jsonpath='{.data.root_token}' 2>/dev/null | base64 -d); \
	read -r -p "Gmail from address: " _gmail; \
	read -r -s -p "Gmail app password: " _pw; echo; \
	read -r -p "T-Mobile SMS gateway (10digits@tmomail.net): " _sms; \
	curl -sf -X POST \
	  -H "X-Vault-Token: $$_tok" -H "Content-Type: application/json" \
	  "http://127.0.0.1:18200/v1/secret/data/k3d-manager/alertmanager" \
	  -d "$$(python3 -c "import json,sys; \
	    print(json.dumps({'data':{'gmail_from':sys.argv[1],'gmail_app_pw':sys.argv[2],'sms_gateway':sys.argv[3]}}))" \
	    "$$_gmail" "$$_pw" "$$_sms")" >/dev/null && \
	echo "[alertmanager-secret] Credentials stored in Vault"
```

Add `alertmanager-secret` to `.PHONY`.
