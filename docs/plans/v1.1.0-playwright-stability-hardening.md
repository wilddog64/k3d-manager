# Implementation Plan: Playwright Stability & Chrome CDP Management

## 1. Refactor `scripts/playwright/acg_credentials.js`

### A. Safe Cleanup
- Replace all `await _cdpBrowser.close()` calls with `_cdpBrowser.disconnect()`.
- Ensure the `finally` block handles cleanup for both CDP and persistent contexts correctly.
- Replace `process.exit(1)` with `throw error` in the main `catch` block to ensure the `finally` block runs.

### B. Explicit Page Management
- In the CDP connection block, if no existing Pluralsight page is found, explicitly call `await _cdpContext.newPage()`.
- Ensure all visibility and locator checks are performed on a validated page object.

### C. Cold-Start Polling
- After the initial `page.goto(targetUrl)`, add a loop that detects redirects to `/id` or `/hands-on`.
- If redirected, log: `ACTION REQUIRED: Pluralsight login required. Waiting up to 300s...`
- Use a `while` loop to poll `page.url()` every 2s until it matches the sandbox dashboard pattern.

### D. Global Execution Timeout
- Wrap the `extractCredentials()` call in a `Promise.race` with a 360-second timeout to prevent indefinite hangs.

## 2. Refactor `scripts/plugins/antigravity.sh`

### A. Robust `_browser_launch`
- Check if port 9222 is already responding via `curl`.
- If silent, check if a "Google Chrome" process exists.
- **Scenario 1 (Running without CDP):** If process exists but port is dead, print exact `open -a` instruction and fail fast.
- **Scenario 2 (Not running):** If no process exists, launch Chrome with `--remote-debugging-port=9222`.

## 3. Alignment of Providers
- Update `scripts/plugins/gcp.sh` and `acg.sh` to use the unified `_browser_launch` helper instead of having divergent launch logic.

## 4. Documentation
- Create a new guide `docs/guides/playwright-stability.md` explaining the "Manual Bridge" requirements and how to troubleshoot port 9222 issues.
