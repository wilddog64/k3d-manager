**Branch:** `k3d-manager-v1.0.2`
**Agent:** Codex
**Date:** 2026-03-31

---

## Context

The app cluster (EC2, ubuntu-k3s) runs ESO which fetches secrets from Vault.
Vault runs on the infra cluster (k3d on Mac, `localhost:8200`).

The forward SSH tunnel (`-L 6443:localhost:6443`) lets the Mac reach the EC2 k3s API.
There is no path in the reverse direction — EC2 pods cannot reach `localhost:8200` on the Mac.

Fix: add a reverse SSH tunnel (`-R 8200:localhost:8200`) to the autossh command and launchd
plist so that port 8200 on the EC2 server maps back to Vault on the Mac.
With this in place, the ESO SecretStore Vault address `http://localhost:8200` resolves
correctly from within the EC2 cluster — no socat bridge, no hardcoded IPs.

### 3. Update `tunnel_start` help text — mention the reverse tunnel

**Old:**
```bash
Start the SSH tunnel (${TUNNEL_BIND_ADDR}:${TUNNEL_LOCAL_PORT} -> ${TUNNEL_SSH_HOST}:${TUNNEL_REMOTE_PORT}).
Installs a launchd plist for boot persistence (macOS only).

Config (override via env or scripts/etc/tunnel/vars.sh):
  TUNNEL_SSH_HOST    SSH host alias    (default: ubuntu)
  TUNNEL_LOCAL_PORT  Local port        (default: 6443)
  TUNNEL_REMOTE_PORT Remote port       (default: 6443)
  TUNNEL_BIND_ADDR   Bind address      (default: 0.0.0.0)
```

**New:**
```bash
Start the SSH tunnel:
  Forward: ${TUNNEL_BIND_ADDR}:${TUNNEL_LOCAL_PORT} -> ${TUNNEL_SSH_HOST}:${TUNNEL_REMOTE_PORT} (k3s API)
  Reverse: ${TUNNEL_SSH_HOST}:${TUNNEL_VAULT_PORT} -> localhost:${TUNNEL_VAULT_PORT} (Vault)
Installs a launchd plist for boot persistence (macOS only).

Config (override via env or scripts/etc/tunnel/vars.sh):
  TUNNEL_SSH_HOST    SSH host alias    (default: ubuntu)
  TUNNEL_LOCAL_PORT  Local port        (default: 6443)
  TUNNEL_REMOTE_PORT Remote port       (default: 6443)
  TUNNEL_BIND_ADDR   Bind address      (default: 0.0.0.0)
  TUNNEL_VAULT_PORT  Vault reverse port (default: 8200)
```
