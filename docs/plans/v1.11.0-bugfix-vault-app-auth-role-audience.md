# Bugfix: v1.11.0 — set `audience` on the app-cluster `eso-app-cluster` k8s-auth role

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/plugins/vault.sh`

---

## Problem

During the live `hostinger` in-cluster hub-Vault cutover, configuring the app-cluster
auth role emitted:

```
WARNING! The following warnings were returned from Vault:
  * Role eso-app-cluster does not have an audience. In Vault v1.21+,
  specifying an audience on roles will be required.
```

`configure_vault_app_auth` writes the `eso-app-cluster` role on the `kubernetes-app`
auth mount **without** an `audience`. Auth works today on Vault 1.20.1 (an unset
audience accepts any JWT aud), but Vault **v1.21+ will require** the `audience`
parameter — at which point ESO's k8s-auth login against the in-cluster CSS would
break on upgrade.

**Root cause:** the `vault write auth/${mount}/role/${role}` call in
`configure_vault_app_auth` (vault.sh:1704–1708) omits the `audience` parameter that
verifies the JWT `aud` claim.

---

## Reproduction

```bash
# in-cluster (hostinger) profile, after the cutover flip:
ROOT=$(kubectl --context ubuntu-hostinger -n secrets get secret vault-root \
  -o go-template='{{ index .data "root_token" }}' | base64 -d)
kubectl --context ubuntu-hostinger -n secrets exec vault-0 -- \
  sh -c "VAULT_TOKEN='$ROOT' vault read auth/kubernetes-app/role/eso-app-cluster" \
  | grep -i audience          # expected: audience = https://kubernetes.default.svc.cluster.local
                              # actual:   audience = n/a   (warning on every write)
```

The audience to bind is the standard Kubernetes default SA-token audience, which the
`external-secrets` SA token already carries (verified live on `ubuntu-hostinger`:
`aud: ["https://kubernetes.default.svc.cluster.local", "k3s"]`). Binding the standard
default audience is safe across all providers because every Kubernetes API server
includes it in projected SA tokens. The same default is already used elsewhere in this
file (`K8S_TOKEN_AUDIENCE` at vault.sh:2012).

---

## Fix

### Change 1 — `scripts/plugins/vault.sh`: bind `audience` on the role write

**Exact old block (lines 1699–1708):**

```bash
  # e. Create ESO role bound to app cluster ESO service account
  local safe_mount_role safe_eso_sa safe_eso_ns
  printf -v safe_mount_role '%s' "auth/${mount}/role/${role}"
  printf -v safe_eso_sa '%q' "$eso_sa"
  printf -v safe_eso_ns '%q' "$eso_ns"
  _vault_exec "$ns" "vault write ${safe_mount_role} \
    bound_service_account_names=${safe_eso_sa} \
    bound_service_account_namespaces=${safe_eso_ns} \
    policies=app-cluster-reader \
    ttl=1h" "$release"
```

**Exact new block:**

```bash
  # e. Create ESO role bound to app cluster ESO service account
  local audience="${APP_K8S_TOKEN_AUDIENCE:-https://kubernetes.default.svc.cluster.local}"
  local safe_mount_role safe_eso_sa safe_eso_ns safe_audience
  printf -v safe_mount_role '%s' "auth/${mount}/role/${role}"
  printf -v safe_eso_sa '%q' "$eso_sa"
  printf -v safe_eso_ns '%q' "$eso_ns"
  printf -v safe_audience '%q' "$audience"
  _vault_exec "$ns" "vault write ${safe_mount_role} \
    bound_service_account_names=${safe_eso_sa} \
    bound_service_account_namespaces=${safe_eso_ns} \
    audience=${safe_audience} \
    policies=app-cluster-reader \
    ttl=1h" "$release"
```

The new `audience` local defaults to the standard Kubernetes SA-token audience and is
overridable via `APP_K8S_TOKEN_AUDIENCE`. `%q`-quoting matches the existing
`bound_service_account_*` handling. The role write stays idempotent — re-running
updates the role in place.
