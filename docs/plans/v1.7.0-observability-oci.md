# v1.7.0 — Observability stack on OCI (Prometheus + Grafana)

## Context

Prometheus and Grafana run on OCI, not the laptop, for two reasons:

1. **Monitoring must outlive the monitored** — if the laptop restarts, OCI dashboards
   stay up. Running monitoring on the laptop loses visibility into OCI during reboots.
2. **Always accessible** — Grafana exposed via Cloudflare tunnel (`grafana.3ai-talk.org`);
   on the laptop it is only reachable when the tunnel is up and the laptop is home.

**Decision recorded:** 2026-05-29

### Multi-cluster scraping strategy

| Cluster | Method |
|---------|--------|
| OCI | In-cluster Prometheus; scrapes its own cluster directly |
| Laptop infra | Prometheus agent → `remote_write` to OCI over Cloudflare tunnel |
| ACG (dev) | Prometheus agent → `remote_write` to OCI when sandbox is up; gaps expected |

ACG is ephemeral — metric gaps when ACG is down are acceptable. The OCI Prometheus
is the single aggregation point.

**Prerequisites:** OCI cluster live (`CLUSTER_PROVIDER=k3s-oci`; v1.5.0).

---

## Target Architecture

```
OCI cluster
  ├── namespace: monitoring
  │   ├── kube-prometheus-stack (Prometheus + Grafana + Alertmanager)
  │   │   └── Grafana: Cloudflare tunnel → grafana.3ai-talk.org
  │   └── Prometheus remote_write receiver (HTTPS, Bearer token auth)
  │
Laptop infra cluster
  └── namespace: monitoring
      └── Prometheus agent mode (no local storage)
          └── remote_write → https://prometheus.3ai-talk.org/api/v1/write
          
ACG cluster (when up)
  └── namespace: monitoring
      └── Prometheus agent mode
          └── remote_write → https://prometheus.3ai-talk.org/api/v1/write
```

Grafana on OCI gets a single Prometheus datasource (local). All clusters feed into it.

## File 1: `scripts/etc/argocd/applicationsets/monitoring.yaml`

Deploy `kube-prometheus-stack` only to `environment: prod` (OCI):

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: monitoring
  namespace: ${ARGOCD_NAMESPACE}
  labels:
    managed-by: k3d-manager
    app-type: platform
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            environment: prod
  goTemplate: true
  template:
    metadata:
      name: monitoring
      labels:
        cluster: '{{.name}}'
        environment: prod
    spec:
      project: platform
      destination:
        server: '{{.server}}'
        namespace: monitoring
      source:
        repoURL: https://prometheus-community.github.io/helm-charts
        chart: kube-prometheus-stack
        targetRevision: '${KUBE_PROMETHEUS_STACK_VERSION}'
        helm:
          releaseName: kube-prometheus-stack
          valueFiles:
            - $values/scripts/etc/monitoring/kube-prometheus-values.yaml
      syncPolicy:
        automated:
          prune: false
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - ServerSideApply=true
```

**Note:** `ServerSideApply=true` required — kube-prometheus-stack CRDs are large and hit
`kubectl apply` annotation size limits.

## File 2: `scripts/etc/monitoring/kube-prometheus-values.yaml`

Key settings for OCI ARM64 / resource-constrained:

```yaml
# ARM64: all images must support arm64
prometheus:
  prometheusSpec:
    replicas: 1
    retention: 15d
    storageSpec:
      volumeClaimTemplate:
        spec:
          storageClassName: local-path
          accessModes: [ReadWriteOnce]
          resources:
            requests:
              storage: 10Gi
    # Enable remote_write receiver so laptop/ACG agents can push
    enableRemoteWriteReceiver: true
    externalUrl: https://prometheus.3ai-talk.org
    resources:
      requests:
        cpu: 100m
        memory: 512Mi
      limits:
        memory: 1Gi

alertmanager:
  alertmanagerSpec:
    replicas: 1

grafana:
  replicas: 1
  persistence:
    enabled: true
    storageClassName: local-path
    size: 2Gi
  grafana.ini:
    server:
      root_url: https://grafana.3ai-talk.org
  adminPassword: "${GRAFANA_ADMIN_PASSWORD}"
  resources:
    requests:
      cpu: 50m
      memory: 128Mi
    limits:
      memory: 256Mi

# Disable components not needed on OCI
kubeEtcd:
  enabled: false
kubeControllerManager:
  enabled: false
kubeScheduler:
  enabled: false
```

## File 3: `scripts/etc/monitoring/prometheus-agent-values.yaml`

Minimal agent for laptop/ACG clusters (no local storage, no Grafana):

```yaml
prometheus:
  agentMode: true
  prometheusSpec:
    replicas: 1
    remoteWrite:
      - url: https://prometheus.3ai-talk.org/api/v1/write
        bearerTokenFile: /etc/prometheus/remote-write-token
        tlsConfig:
          insecureSkipVerify: false
    resources:
      requests:
        cpu: 50m
        memory: 128Mi
      limits:
        memory: 256Mi

alertmanager:
  enabled: false

grafana:
  enabled: false

kubeEtcd:
  enabled: false
kubeControllerManager:
  enabled: false
kubeScheduler:
  enabled: false
```

The `remote-write-token` is a Kubernetes Secret mounted at deploy time. OCI generates
the token; `deploy_monitoring_agent` writes it to the agent cluster before installing.

## File 4: `scripts/plugins/monitoring.sh`

```bash
#!/usr/bin/env bash
# monitoring.sh — deploy Prometheus + Grafana (OCI) or Prometheus agent (laptop/ACG)

function deploy_monitoring() {
  local _dir="${SCRIPT_DIR}/etc/monitoring"
  _info "[monitoring] Deploying kube-prometheus-stack (OCI)..."
  KUBE_PROMETHEUS_STACK_VERSION="${KUBE_PROMETHEUS_STACK_VERSION:-68.3.0}" \
    _argocd_deploy_applicationset "monitoring" "${_dir}/monitoring.yaml"
}

function deploy_monitoring_agent() {
  local _env="${1:?env required: laptop|acg}"
  local _dir="${SCRIPT_DIR}/etc/monitoring"
  _info "[monitoring] Deploying Prometheus agent for ${_env}..."
  _kubectl apply -f "${_dir}/prometheus-agent-values.yaml" --context "${_env}"
}
```

## Remote Write Auth

OCI generates a token and stores it as:
- `monitoring/remote-write-token` Secret on OCI (Prometheus reads it as a bearer token)
- Same token pushed to `monitoring/remote-write-token` Secret on laptop/ACG clusters
  (Prometheus agent sends it in `Authorization: Bearer` on remote_write requests)

OCI Prometheus validates the token via `basicAuth` or bearer token in `remoteWriteAuth`
config (exact mechanism to be confirmed at implementation time — kube-prometheus-stack
v68+ supports bearer token validation natively).
