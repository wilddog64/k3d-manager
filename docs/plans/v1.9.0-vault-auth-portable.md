## Problem

`configure_vault_app_auth` (`scripts/plugins/vault.sh:1315`) configures the **hub** Vault's
`kubernetes-app` auth mount so an app cluster's `external-secrets` ServiceAccount can read
secrets (k8s-auth, no secret-zero). It is **fully implemented but has no production caller** —
no provider registration path invokes it. The memory note claiming it was "wired inline into
`_hostinger_register_cluster`" describes the *plan*, not the shipped code (verified: no call
site exists).

Each provider already derives the two inputs the function needs, but **differently**:

| Provider | Register fn | API URL source | CA source |
|----------|-------------|----------------|-----------|
| Hostinger | `_hostinger_register_cluster` (`k3s-hostinger.sh:132`) | `kubectl config view` on `_HOSTINGER_KUBE_CONTEXT` → `.cluster.server` | same context `.cluster.certificate-authority-data` (base64) |
| OCI | `_oci_register_cluster` (`k3s-oci.sh:607`) | `https://${_ip}:6443` | `KUBECONFIG=${_OCI_KUBECONFIG}` context `k3s-oci` `.certificate-authority-data` |
| AWS/Azure | `register_app_cluster` via `bin/cluster-up:683` | env-driven (`ARGOCD_APP_CLUSTER_SERVER`) | env-driven |

**Goal:** factor the API-URL + CA derivation into **one provider-agnostic helper keyed on a
kubeconfig context**, and call that single helper from every provider's registration path so
app-cluster Vault auth works identically on Hostinger / ACG-AWS / Azure / OCI / EKS / AKS /
home-lab. The Vault contract (`kubernetes-app` mount, `eso-app-cluster` role, `external-secrets`
SA in `secrets` ns) is already fixed and shared with `eso.sh:204-205` — keep it.

**Root cause:** Change 4 of the Hostinger cutover was specced as a Hostinger-inline call; it was
never implemented, and `configure_vault_app_auth`'s env-only contract (`APP_CLUSTER_API_URL` +
`APP_CLUSTER_CA_CERT_PATH` as a *file*) is awkward to call from providers that hold a context, not a file.

## Design

### Model: single shared hub-Vault mount, reconfigured per active app cluster

- `configure_vault_app_auth` writes `auth/kubernetes-app/config` with ONE `kubernetes_host` + ONE
  CA. The shopping-cart app-cluster ClusterSecretStore (via `eso.sh`) references the fixed
  `kubernetes-app` mount + `eso-app-cluster` role. The current model runs **one active app cluster
  at a time** (Hostinger is active; ACG/Azure/OCI are alternatives, not simultaneous). So the helper
  **reconfigures the shared mount to point at whichever cluster is being registered** — no per-cluster
  mount needed.
- **Out of scope (note as future seam):** simultaneous multiple app clusters against one hub Vault
  would need per-cluster mounts (`kubernetes-<context>`) + a per-cluster mount rendered into the CSS.
  Not built here — see `project_app_cluster_vault_auth_portability` "open seam".
- **Out of scope:** per-environment Vault *host* (`vault.3ai-talk.org` hardcoded in the static CSS).
  Still the open seam; not touched here.

### New public helper — `configure_vault_app_auth_for_context`

Keyed on a kubeconfig **context** (the universal contract). Derives API URL + CA from the context,
writes the CA to a temp PEM file, exports the two env vars `configure_vault_app_auth` expects, calls
it in a **subshell** (so exports don't leak), and cleans up the temp file. Best-effort by default
(`warn` not `fail`) so a Vault-auth hiccup doesn't abort cluster registration; `APP_VAULT_AUTH_REQUIRED=true`
makes it fatal.

## Fix

### Change 1 — `scripts/plugins/vault.sh`: add `configure_vault_app_auth_for_context`

Add immediately **after** the `configure_vault_app_auth` function (after line 1380, before
`_vault_set_eso_reader`):

**Exact new block:**

```bash
function configure_vault_app_auth_for_context() {
  local app_context="${1:-}"
  local app_kubeconfig="${2:-${KUBECONFIG:-}}"
  if [[ -z "${app_context}" ]]; then
    _err "[vault] configure_vault_app_auth_for_context requires an app cluster kube-context"
  fi

  local -a kctl=(kubectl)
  [[ -n "${app_kubeconfig}" ]] && kctl=(kubectl --kubeconfig "${app_kubeconfig}")

  local server ca_data ca_file
  server="$("${kctl[@]}" config view --raw \
    -o jsonpath="{.clusters[?(@.name==\"${app_context}\")].cluster.server}" 2>/dev/null)"
  ca_data="$("${kctl[@]}" config view --raw \
    -o jsonpath="{.clusters[?(@.name==\"${app_context}\")].cluster.certificate-authority-data}" 2>/dev/null)"
  ca_file="$("${kctl[@]}" config view --raw \
    -o jsonpath="{.clusters[?(@.name==\"${app_context}\")].cluster.certificate-authority}" 2>/dev/null)"

  if [[ -z "${server}" ]]; then
    _warn "[vault] no API server for context '${app_context}' — skipping app-cluster Vault auth"
    [[ "${APP_VAULT_AUTH_REQUIRED:-false}" == "true" ]] && return 1
    return 0
  fi

  local tmp_ca="" ca_path=""
  if [[ -n "${ca_data}" ]]; then
    tmp_ca="$(mktemp "${TMPDIR:-/tmp}/app-cluster-ca.XXXXXX")" || return 1
    printf '%s' "${ca_data}" | base64 -d > "${tmp_ca}" 2>/dev/null || {
      rm -f "${tmp_ca}"
      _warn "[vault] could not decode CA data for '${app_context}' — skipping app-cluster Vault auth"
      [[ "${APP_VAULT_AUTH_REQUIRED:-false}" == "true" ]] && return 1
      return 0
    }
    ca_path="${tmp_ca}"
  elif [[ -n "${ca_file}" && -f "${ca_file}" ]]; then
    ca_path="${ca_file}"
  else
    _warn "[vault] no CA for context '${app_context}' — skipping app-cluster Vault auth"
    [[ "${APP_VAULT_AUTH_REQUIRED:-false}" == "true" ]] && return 1
    return 0
  fi

  local rc=0
  (
    APP_CLUSTER_API_URL="${server}" \
    APP_CLUSTER_CA_CERT_PATH="${ca_path}" \
    configure_vault_app_auth
  ) || rc=$?

  [[ -n "${tmp_ca}" ]] && rm -f "${tmp_ca}"

  if (( rc != 0 )); then
    _warn "[vault] app-cluster Vault auth for '${app_context}' did not complete (rc=${rc})"
    [[ "${APP_VAULT_AUTH_REQUIRED:-false}" == "true" ]] && return "${rc}"
  fi
  return 0
}
```

> Verify `_warn` exists in the sourced libs; if only `_info`/`_err` are available, use `_info` for the
> non-fatal branches. (Check `scripts/lib/` before implementing.)

### Change 2 — `scripts/lib/providers/k3s-hostinger.sh`: call the helper after registration

In `_hostinger_register_cluster`, after the successful registration `_info` line (currently
line 168), add the Vault-auth call. The Hostinger context is `_HOSTINGER_KUBE_CONTEXT` and lives in
the default kubeconfig.

**Exact old block (lines 167–169):**

```bash
  ) || return 1
  _info "[k3s-hostinger] Registered — verify: kubectl get secret ${secret_name} -n ${argocd_ns}"
}
```

**Exact new block:**

```bash
  ) || return 1
  _info "[k3s-hostinger] Registered — verify: kubectl get secret ${secret_name} -n ${argocd_ns}"
  if declare -f configure_vault_app_auth_for_context >/dev/null 2>&1; then
    configure_vault_app_auth_for_context "${_HOSTINGER_KUBE_CONTEXT}" || true
  fi
}
```

> `configure_vault_app_auth_for_context` runs `_kubectl`/`_vault_exec` against the **hub** Vault, so
> it must NOT run inside the `hub_kubectl` registration subshell (which redefines `_kubectl`). Calling
> it here — after the subshell closes, with the default hub-pointing `_kubectl` — is correct for the
> single-hub-Vault model.
