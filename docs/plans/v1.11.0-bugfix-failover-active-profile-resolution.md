# Bugfix: v1.11.0 — failover watchdog reads active profile before sourcing `vars.sh`

**Branch:** `k3d-manager-v1.11.0`
**Files:** `scripts/plugins/vault.sh`

---

## Problem

`vault_failover_hub_into_context` determines the *active* hub-Vault profile with
`local _active="${HUB_VAULT_PROFILE:-laptop}"` (vault.sh:1174) **before** `vars.sh` is ever
sourced. `vars.sh` — which reads the persisted `HUB_VAULT_PROFILE_STATE_FILE` (the file a prior
flip writes) — is only sourced *later*, after the flip decision (vault.sh:1215). The watchdog's
own entrypoint (`bin/k3dm-vault-failover` → `scripts/k3d-manager vault_failover_hub_into_context`)
does **not** pre-source `vars.sh`, so on every watchdog run `HUB_VAULT_PROFILE` is unset and
defaults to `laptop`, regardless of the persisted state.

**Visible symptom (live):** with the state file holding `hostinger` (in-cluster hub Vault is the
active profile after the v1.11.0 cutover), `vault_failover_hub_into_context ubuntu-hostinger
--probe-only` logged `hub Vault healthy (profile=laptop)` and probed the laptop k3d Vault — the
**wrong** Vault. In this state an in-cluster Vault outage would be invisible to the watchdog
(it would keep probing the laptop), so assisted failover would never fire. The whole point of P4
— detect the active hub Vault unreachable and flip — is defeated whenever the active profile was
set via the state file rather than an explicit env var.

**Root cause:** the active-profile read at vault.sh:1174 runs before `vars.sh` (and thus the
state-file resolution in P4 Change 1) has executed in the watchdog dispatch path.

---

## Reproduction

```bash
# Persist hostinger as the active profile (as the v1.11.0 cutover did):
printf 'hostinger\n' > "${HOME}/.local/share/k3d-manager/hub-vault-profile"

# Run the watchdog probe with no HUB_VAULT_PROFILE in the env (the LaunchAgent's situation):
CLUSTER_PROVIDER=k3s-hostinger ./scripts/k3d-manager \
  vault_failover_hub_into_context ubuntu-hostinger --probe-only

# Expected: INFO ... healthy (profile=hostinger)  — probes the in-cluster app-ctx Vault
# Actual:   INFO ... healthy (profile=laptop)     — probes the laptop k3d Vault (wrong)
```

---

## Fix

### Change 1 — `scripts/plugins/vault.sh`: source `vars.sh` before reading the active profile

Source `vars.sh` once at the top of the function (after arg + kube-context validation, before the
active-profile read), mirroring the existing source idiom already used in this function at
vault.sh:1214–1215. `vars.sh` resolves `HUB_VAULT_PROFILE` from the persisted state file only when
it is unset, so an explicit `HUB_VAULT_PROFILE` in the env still wins (behavior-preserving for the
provider/`make refresh` path, which pre-sources `vars.sh`).

**Exact old block (lines 1168–1174):**

```bash
  if ! _kubectl config get-contexts -o name 2>/dev/null | grep -qx "${_app_ctx}"; then
    _err "[vault] kube-context '${_app_ctx}' not found in kubeconfig"
    return 1
  fi

  local _vault_ns="${VAULT_NS:-secrets}"
  local _active="${HUB_VAULT_PROFILE:-laptop}"
```

**Exact new block:**

```bash
  if ! _kubectl config get-contexts -o name 2>/dev/null | grep -qx "${_app_ctx}"; then
    _err "[vault] kube-context '${_app_ctx}' not found in kubeconfig"
    return 1
  fi

  # shellcheck disable=SC1091
  [[ -r "${SCRIPT_DIR}/etc/vault/vars.sh" ]] && source "${SCRIPT_DIR}/etc/vault/vars.sh"

  local _vault_ns="${VAULT_NS:-secrets}"
  local _active="${HUB_VAULT_PROFILE:-laptop}"
```

The added source is byte-for-byte the same idiom as vault.sh:1214–1215 (same guard, same
`# shellcheck disable=SC1091`). It runs before the active-profile read so the persisted state file
is honored; explicit env still wins because `vars.sh` only fills `HUB_VAULT_PROFILE` when unset.
