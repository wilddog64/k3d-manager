# Bugfix: v1.11.0 — failover watchdog fails under launchd (Homebrew tools not on PATH)

**Branch:** `k3d-manager-v1.11.0`
**Files:** `bin/k3dm-vault-failover`

---

## Problem

The `com.k3d-manager.vault-failover` LaunchAgent runs `bin/k3dm-vault-failover` under
launchd's minimal PATH (`/usr/bin:/bin:/usr/sbin:/sbin`). `kubectl` (and `helm`/`jq`) live in
`/opt/homebrew/bin` (Apple Silicon) or `/usr/local/bin` (Intel), which are **not** on that PATH.
Inside `vault_failover_hub_into_context`, `_kubectl config get-contexts` finds no `kubectl`,
returns empty, and the kube-context check fails.

**Visible symptom (live):** after `vault_install_failover_watchdog ubuntu-hostinger`, forcing a
run with `launchctl kickstart gui/<uid>/com.k3d-manager.vault-failover` logged:

```
ERROR: [vault] kube-context 'ubuntu-hostinger' not found in kubeconfig
```

even though the context is present in `~/.kube/config` and an interactive
`vault_failover_hub_into_context ubuntu-hostinger --probe-only` reports
`hub Vault healthy (profile=hostinger)`. The watchdog errors out before it ever probes, so the
LaunchAgent is **non-functional** — assisted failover would never fire on a real outage.

**Root cause:** `bin/k3dm-vault-failover` does not augment PATH, so under launchd's minimal PATH
the Homebrew-installed `kubectl` is not found.

---

## Reproduction

```bash
# Reproduce the launchd minimal-PATH environment:
/usr/bin/env -i HOME="$HOME" PATH=/usr/bin:/bin:/usr/sbin:/sbin CLUSTER_PROVIDER=k3s-hostinger \
  ./scripts/k3d-manager vault_failover_hub_into_context ubuntu-hostinger --probe-only
# Actual:   ERROR: [vault] kube-context 'ubuntu-hostinger' not found in kubeconfig

# Same env with Homebrew prefixes on PATH:
/usr/bin/env -i HOME="$HOME" PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin \
  CLUSTER_PROVIDER=k3s-hostinger \
  ./scripts/k3d-manager vault_failover_hub_into_context ubuntu-hostinger --probe-only
# Expected: INFO: [vault] hub Vault healthy (profile=hostinger); no failover needed
```

---

## Fix

### Change 1 — `bin/k3dm-vault-failover`: prepend Homebrew prefixes to PATH

Prepend the common Homebrew bin directories to PATH right after `set -euo pipefail`, before any
tool is invoked. This covers both Apple Silicon (`/opt/homebrew/bin`) and Intel
(`/usr/local/bin`), is harmless for interactive runs (those dirs are already on PATH), and fixes
every Homebrew tool the failover path needs (`kubectl`, `helm`, `jq`).

**Exact old block:**

```bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

**Exact new block:**

```bash
set -euo pipefail

# launchd runs with a minimal PATH (/usr/bin:/bin:/usr/sbin:/sbin); Homebrew tools
# (kubectl, helm, jq) live outside it. Prepend the common Homebrew prefixes so the
# watchdog finds them when launched by the LaunchAgent.
export PATH="/opt/homebrew/bin:/usr/local/bin:${PATH}"

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```

The 3-line comment is intentional — it documents a non-obvious launchd quirk so the PATH
manipulation is not mistaken for an error.
