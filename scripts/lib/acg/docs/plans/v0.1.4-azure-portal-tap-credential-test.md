# Feature: v0.1.4 — Azure portal TAP login for `make credential-test PROVIDER=az`

**Branch:** `feat/v0.1.4`
**Files:** `bin/acg-credential-test`, `scripts/lib/acg/playwright/providers/azure.js`, `scripts/lib/acg/playwright/lib/sandbox.js`, `CHANGELOG.md`

---

## Problem

The Azure sandbox is TAP/MFA-enforced. The current Azure path can extract portal-visible values, but `bin/acg-credential-test` cannot reliably establish a real `az-cli` session if it trusts cached state or treats an `AADSTS50126` username/password error as success. This must remain fully automated from Slack.

The portal is the source of truth for the current Azure sandbox values:
- portal username (`cloud_user_...@realhandsonlabs.com`)
- Temporary Access Pass (TAP)
- tenant context / tenant domain hint
- subscription ID
- resource group name
- sandbox URL

The CLI auth step must be done from those portal values in a clean Azure CLI session, not from stale local auth.

---

## Borrowed from `azure-dev`

The `ansible/provision-tomcat` `azure-dev` branch shows the same platform shift:
- Azure is TAP/MFA sensitive
- `az account show` is too lenient because it can succeed on a cached session
- `az login --username/--password` is not a stable automation success signal on its own
- a real token probe is required after login

This implementation should follow that direction, not a cached-session shortcut.

---

## Goal

Make `make credential-test PROVIDER=az` fully automated and trustworthy by:
1. extracting the Azure portal fields from the sandbox UI,
2. logging into `az-cli` in a fresh isolated configuration directory,
3. validating the session with a real management-token probe,
4. capturing screenshot/URL/text artifacts when the Azure portal flow fails.

---

## Implementation

### Change 1 — `scripts/lib/acg/playwright/providers/azure.js`: implement portal extraction

Replace the current stub with a real extractor that reads the Azure sandbox page and emits the portal-side Azure values.

Required output fields:
- `AZURE_USERNAME` — the portal user shown on the Microsoft sign-in screen
- `AZURE_PASSWORD` — the Temporary Access Pass (TAP) shown on the next screen
- `AZURE_TENANT_ID` — the tenant ID if visible, otherwise derive the tenant/domain hint from the portal URL / OIDC metadata and persist it
- `AZURE_SUBSCRIPTION_ID` — the subscription ID from the Azure portal essentials panel
- `AZURE_RESOURCE_GROUP` — the resource group name shown in the portal header/essentials

Implementation notes:
- Scope lookups to the Azure provider card / portal view only.
- If the Azure panel closes while waiting, reopen it once and continue.
- If extraction fails, save a screenshot and print the screenshot path, current URL, and visible page text summary before throwing.
- Do not treat portal extraction as a CLI validation success.

---

### Change 2 — `bin/acg-credential-test`: validate Azure CLI in a clean session

Update the Azure validation block so it does not rely on cached Azure CLI auth.

Required behavior:
- Create an isolated temporary `AZURE_CONFIG_DIR` for the login attempt.
- Run `az logout` before the login attempt in that clean config directory.
- Prefer portal-derived TAP login for the Azure sandbox:
  - `az login --username "$AZURE_USERNAME" --password "$AZURE_PASSWORD" --tenant "$AZURE_TENANT_ID" --allow-no-subscriptions --output none`
- After login, validate auth with a real token probe such as:
  - `az account get-access-token --resource https://management.azure.com/ --output none`
- Do not use `az account show` as the proof of authentication.
- If Azure CLI returns `AADSTS50126` or the MFA warning text, do not report success just because `az account show` has an account.
- If the first login/probe fails, restart the sandbox once, re-extract, and retry the clean login path.
- On final failure, print the Azure CLI error, the current portal URL, and the saved screenshot path.

If `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` are also present later, they may remain as a fallback path, but the current TAP/portal flow is the primary path for this implementation.

---

### Change 3 — `scripts/lib/acg/playwright/lib/sandbox.js`: keep Azure retry flow failure-artifact friendly

If the Azure portal flow bounces or the panel closes while extraction is in progress:
- capture a screenshot,
- log the current URL,
- log a short visible text summary,
- and do not keep looping on the same closed-panel message indefinitely.

This change should be minimal and only support the Azure extraction flow; it should not change unrelated AWS/GCP behavior.

## Acceptance Criteria

- `make credential-test PROVIDER=az` can authenticate `az-cli` automatically from the portal-extracted Azure sandbox values.
- The validation is rejected unless a real Azure token probe succeeds in a clean config directory.
- `AADSTS50126` / MFA warnings do not produce a false success message.
- When the portal flow fails, the job leaves behind a screenshot and enough context to debug the next run.
- The implementation remains fully automatable from Slack; no manual portal work is required.
