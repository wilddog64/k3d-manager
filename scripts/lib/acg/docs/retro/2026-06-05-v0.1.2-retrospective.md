# Retrospective — v0.1.2

**Date:** 2026-06-05
**Milestone:** Start Sandbox button outside viewport fix
**PR:** #37 — merged to main (`4d5aa477`)
**Participants:** Claude, Codex, Copilot

## What Went Well
- Root cause identified immediately from Gemini failure analysis (viewport error in log)
- Fix was minimal: three `scrollIntoViewIfNeeded()` calls, one file
- Codex implemented correctly on first attempt
- CI green on first push; Copilot found only doc consistency issues, not code issues

## What Went Wrong
- Bug spec template still generating inconsistent "Files" and "What NOT to Do" lines — Copilot caught 3/6 findings on the same spec doc again (same pattern as v0.1.1)
- memory-bank was not updated to reflect v0.1.2 work before PR creation — Copilot caught 2 more findings
- Two fix PRs in the same session (v0.1.1 + v0.1.2) means two subtree pulls needed; the "batch fixes" approach discussed but not yet adopted

## Process Rules Added

| Rule | Where |
|------|-------|
| Bug spec "Files" line must list ALL files Codex is expected to touch — code file + CHANGELOG.md + memory-bank | Bug spec template |
| Bug spec "Rules" must explicitly name CHANGELOG.md in the scope statement | Bug spec template |
| Bug spec "What NOT to Do" must say "code file other than X" not "any file other than X" | Bug spec template |

## Decisions Made
- Batch related Playwright fixes on the same branch until `acg-up` runs end-to-end cleanly — one PR, one CI run, one subtree pull — instead of PR-per-fix

## Theme
A straightforward viewport scrolling fix revealed a recurring pattern: the bug spec template keeps producing internally inconsistent "Files" and "What NOT to Do" sections that Copilot flags every time. The fix took one commit; cleaning up the spec doc took another. The real output of this milestone is the decision to batch related fixes rather than PR each one individually.
