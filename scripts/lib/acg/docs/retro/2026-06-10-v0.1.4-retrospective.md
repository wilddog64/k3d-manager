# Retrospective — v0.1.4

**Date:** 2026-06-10
**Milestone:** Azure credential validation — SP/CLI-first auth, screenshot archival
**PR:** #39 — merged to main (`17e5e4d78903b1158629fdfe2382fed8571f2683`)
**Participants:** Claude, Codex, Copilot

## What Went Well

1. **Provider-scoped architecture held steady** — the foundational v0.1.3 refactor to scope button lookups and credentials to provider cards (AWS/GCP/Azure) prevented cascading failures when adding Azure. All 16+ related bugs (shared containers, panel cycling, credential short-circuits) traced back to this pattern; fixing it once at the architectural level eliminated entire categories of regressions.

2. **Azure SP validation clarity** — defining CLI auth branches as exclusive (SP → identity → portal, never silent fallback) forced clarity on auth method ordering. Bug doc `2026-06-10-azure-cli-auth-method-selection.md` captured why SP must be tried before portal (SP failures are hard failures, not "try the next provider"), preventing future regressions where CLI auth failures silently became portal retries.

3. **Screenshot archival coordination** — moving debug screenshots from `/tmp` to `~/.local/share/k3d-manager/screenshots/` required coordinating between lib-acg (Playwright) and k3d-manager (webhook ingestion). The shared path convention surfaced immediately in test expectations; no silent mismatches.

4. **Toast dismissal architecture matured** — the `addLocatorHandler` pattern (set once, active for all pointer operations) proved stable across Azure's unique panel lifecycle. No regressions when Azure panel auto-closes after credential population.

5. **MFA graceful degradation** — treating Azure CLI MFA failures (`AADSTS50126`) as non-fatal warnings allowed the tool to skip direct auth validation while still reporting that the credential exists and passes basic syntax checks. No silent false-negatives; portal/TAP path remains available as fallback.

## What Went Wrong

1. **Test coverage gap for portal validation** — Jest tests did not exercise the Azure portal username/password path with `az login --username ... --password ...` until late (2026-06-09). Early commits assumed `az account show` would always work after portal login; real Azure behavior is more complex (cached auth, domain discovery, subscription absence). Added test after the fact.

2. **OIDC tenant discovery ordering** — the flow was: try portal UI tenant → fallback to username domain parsing. If domain parsing was wrong, silent success with empty tenant followed by `az login` failure. Bug doc `2026-06-10-azure-cli-auth-method-selection.md` now enforces: always query OIDC endpoint for tenant when username domain is available, avoiding silent mismatches. Codex implementation initially skipped the OIDC query.

3. **Copilot review lag on PR #39** — Copilot did not post review comments until after the PR was already merged to main. The bot was tagged but the review pipeline appears to have lagged. No blocking issues caught, but the workflow gap means we cannot rely on Copilot review for pre-merge gating on this project.

4. **Panel re-open flakiness on Azure** — the early-exit check for `credentialsAlreadyVisible` ran too early in v0.1.2, before panel deletion attempt. The fix (move early-exit after deletion check) was correct but scattered across multiple commits (fa10572, e1ff241); by v0.1.4 it was coherent, but the intermediate state was fragile.

## Process Rules Added

1. **Azure CLI auth method ordering is canonical** — when adding new cloud providers with multiple auth paths, define the order explicitly (SP → identity → user/password → portal/TAP) and treat failures in higher-precedence paths as hard failures, never silent fallthrough. Document in bug spec why the order matters (trust, audit trail, token lifetime).

2. **Playwright toast handlers must be architecture-level** — `addLocatorHandler` is placed once per navigation cycle and remains active for all pointer operations. Do not try to manage toast state inside polling loops; the handler pattern decouples toast presence from script logic.

3. **Screenshot paths must be shared constants** — when multiple repos coordinate on artifact ingestion, define the path as a constant in both places (lib-acg and k3d-manager) and verify in test expectations. No ad-hoc path selection.

4. **Portal validation requires clean auth context** — never reuse cached Azure credentials for portal validation. The v0.1.4 flow creates a clean `AZURE_CONFIG_DIR` for `az login` calls, preventing stale tokens from masking real failures.

## Decisions Made

1. **MFA failures are non-fatal** — Azure portal username/password auth is optional (Pluralsight may expose SP creds instead). If MFA blocks CLI validation, emit a WARN and continue; the credential exists and passed extraction. Portal-only users will use the SP path or manual validation.

2. **Keep portal/TAP as fallback** — do not remove portal username/password validation even after SP/identity paths succeed. Keep it as a fallback in case UI changes expose portal creds instead of SP/identity. This adds robustness against future Pluralsight layout changes.

3. **Screenshot archival is per-provider** — Azure and AWS both save debug screenshots on failure, but to the same shared directory. The path is set at Playwright level (lib-acg) but ingested by k3d-manager webhook; no per-provider paths.

## Theme

**Multi-Path Authentication with Graceful Fallback** — v0.1.4 proved that Pluralsight's UI can expose three distinct Azure auth methods (service principal, managed identity, portal username/password) often on the same page. Hardening the auth priority (SP → identity → portal) and making CLI validation graceful (warnings for MFA, no false-negatives) allows the tool to adapt to layout changes without silent failures. The key: define fallback order **before** implementing any path, not after.
