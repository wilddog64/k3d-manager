# Retrospective — v0.1.7

**Date:** 2026-06-12
**Milestone:** Azure SP validation restart + sandbox conflict detection
**PR:** #44 — merged to main (`f5e1f6a`)
**Participants:** Claude, Codex, Copilot

## What Went Well
- Conflict-detection root cause was pinned precisely from debug screenshots: the AWS credential panel is a full-screen modal that hides Azure's controls, so `_deleteConflictingSandbox` found the conflict via banner text but couldn't click any Azure button. The fix (`_closeOpenPanel` before button search) was small and targeted.
- The `_waitForCredentials` fall-through bug was caught in the same pass — `inputCount > 0` with a null `panelStartBtn` fell through to the reopen loop, producing the "panel stayed closed after 3 attempts" error.
- Codex implemented both changes exactly to spec (`cdd9ba4`); verification confirmed the diff matched, `node --check` passed, and only the one file was touched.
- Copilot caught a real latent bug: `isDisabled()` with the default 30s timeout could throttle the credential wait loop. Fixed in `87a7ff3` to match the surrounding `{ timeout: 300 }` cadence.
- AWS credential extraction smoke-tested clean post-fix (all 4 fields, no conflict banner).

## What Went Wrong
- **Auto-merge without user review.** Claude squash-merged PR #44 immediately after CI passed and the Copilot thread resolved, before the user had a chance to review the diff. The user objected. New rule added (see below).
- **Subtree synced the wrong way first.** The initial `git subtree pull` (non-squash) hit "unrelated histories"; Claude resolved it with `--allow-unrelated-histories`, which grafted 253 lib-acg commits into the k3d-manager branch. Caught before push, reset, and redone with `--squash` (the established pattern: one squash artifact + merge commit).
- **Version bookkeeping drifted.** v0.1.6 (PR #43) was merged but never tagged; its changelog entries sat in `[Unreleased]` mixed with v0.1.7 work. Resolved this milestone by tagging both v0.1.6 and v0.1.7 and splitting the changelog.

## Process Rules Added
| Rule | Where |
|------|-------|
| Never auto-merge — notify the user to review the PR and wait for explicit go, even when they said "create a PR and merge" | `feedback_notify_pr_review_before_merge` (memory) |
| Subtree sync uses `git subtree pull --squash` — never `--allow-unrelated-histories` (grafts full upstream history) | this retro + lib subtree discipline |

## Decisions Made
- Tag both v0.1.6 (`696ca83`) and v0.1.7 (`f5e1f6a`) for continuous release history; split `[Unreleased]` into dated `[0.1.6]` + `[0.1.7]` sections on the next branch (`feat/v0.1.8`).
- Azure SP slow-population (5–6 reprovision cycle worst case) deferred — spec filed in k3d-manager `docs/bugs/2026-06-12-azure-sp-oauth-population-timing.md`; not reproducing currently, instrumentation (read-only probe + `AZURE-TIMING` logs) ready for when it recurs. The real bug there is the reprovision budget (deadline 420s + cap <3 → only ~2–3 cycles), not the 60s trigger.

## Theme
A focused reliability pass on the Azure credential path: the conflict-deletion flow was silently no-op'ing because a full-screen modal hid the controls it needed to click, and the credential wait loop had a fall-through that turned a recoverable state into a hard failure. Both fixed and verified. The process lessons were sharper than the code ones — auto-merging skipped the user's review gate, and a subtree sync took the wrong path before being caught and redone. The deferred SP-population bug is now correctly understood as a budget-sizing problem, waiting on real data.
